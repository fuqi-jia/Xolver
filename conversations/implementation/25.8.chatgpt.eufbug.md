你的定位是对的：**这是一个必须修的 soundness/completeness 级问题，不是简单测试 bug。**

`SharedEqualityManager::explainEquality()` 为空，会把本来应该是：

```text
Eq(y,v2) ∧ Eq(v2,v1) ∧ Eq(v1,y) ∧ ¬Eq(x,y)  -> false
```

错误压缩成：

```text
Eq(x,y)
```

或者等价地只给 SAT 一个“强迫 Eq(x,y) 为真”的单子句。这样 SAT 不会回溯 guessed arrangement/shared equality，而是直接在 level 0 判 unsat。这个确实会导致 `test5_diseq_sat` 误报 unsat。

---

# 结论

建议这样做：

```text
1. SharedEqualityManager 必须实现 explanation。
2. 不要继续保留 explainEquality stub。
3. 不建议直接把 SatLit reason 强行塞进所有 RollbackUnionFind 使用场景。
4. 推荐新增 reason-bearing UF，或者让 RollbackUnionFind 模板化/可选 reason。
5. EUF 内部不要随便传 dummy reason；如果 EUF 需要产生 lemma，也必须有自己的 explanation 机制。
```

你的改动方向整体是对的，但要避免把 **SharedEqualityManager 的 SAT-level explanation** 和 **EUF e-graph 的 congruence explanation** 混成一个东西。

# 1. EUF solver 内部的 RollbackUnionFind 也需要 explanation 吗？

答案分两层。

## Stage A 当前这个 bug：只需要 SharedEqualityManager 有 explanation

`test5_diseq_sat` 这个问题发生在组合层：

```text
SAT guessed some shared equality literals true
SharedEqualityManager 合并 shared terms
另一个 disequality literal 为 true
于是 combination 层发现 x 和 y 被合并，但同时 x ≠ y
```

这个冲突引理应该由 `SharedEqualityManager` 解释。

所以当前必须修的是：

```cpp
SharedEqualityManager::explainEquality(a,b)
```

而不是先大改 EUF。

---

## 但 EUF 如果向 SAT 返回 lemma/conflict，也必须有 explanation

如果 EUF solver 只是内部检查，不向 SAT 传播任何 equality lemma，那么它可以暂时不暴露完整 explanation。

但只要 EUF 做这些事情：

```text
1. EUF conflict:
   x = y, f(x) != f(y)

2. EUF propagation:
   x = y  ->  f(x) = f(y)

3. NO shared equality propagation:
   EUF deduces sharedTermA = sharedTermB
```

那么 EUF 就必须能给出 reason。

区别在于：EUF 的 reason 通常不是简单的一条 `SatLit`，而可能是：

```text
- asserted equality literal
- interface equality literal
- bridge definition
- congruence reason
- recursive equality explanation
```

例如 EUF congruence explanation 可能是：

```text
x = y  ->  f(x) = f(y)
```

这里 `f(x) = f(y)` 的 reason 不是 UF 的一条边，而是 congruence closure 的推导，需要 e-graph explanation。

所以：

```text
SharedEqualityManager:
    reason = SAT-assigned shared equality literal
    explanation = UF path reasons

EUF solver:
    reason = asserted equality / interface equality / congruence proof
    explanation = e-graph explanation
```

不要让 EUF 内部 `uf_.unite()` 传 dummy reason。dummy reason 后面一旦进入 lemma，就会产生不可解释甚至错误的 clause。

---

# 2. explain 接口应该放在哪里？

我建议不要把 `SatLit reason` 硬塞进现有通用 `RollbackUnionFind`，因为这个类现在被 EUF 也用。直接改成：

```cpp
struct Change {
    int childRoot;
    int parentRoot;
    int oldParentSize;
    SatLit reason;
};
```

会把 SAT 层概念污染到所有 UF 使用场景。

更好的方案有两个。

---

## 推荐方案 A：新增 `ExplainableRollbackUnionFind<Reason>`

例如：

```cpp
template <typename Reason>
class ExplainableRollbackUnionFind {
public:
    using Snapshot = size_t;

    explicit ExplainableRollbackUnionFind(size_t n);

    int find(int x) const;

    Snapshot snapshot() const;
    void rollback(Snapshot snap);

    bool unite(int a, int b, const Reason& reason);

    std::vector<Reason> explain(int a, int b) const;

private:
    struct Change {
        int childRoot;
        int parentRoot;
        int oldParentSize;
        std::optional<Reason> oldEdgeReason;
    };

    std::vector<int> parent_;
    std::vector<int> size_;
    std::vector<std::optional<Reason>> edgeReason_;
    std::vector<Change> changes_;
};
```

然后在 `SharedEqualityManager` 里用：

```cpp
ExplainableRollbackUnionFind<SatLit> uf_;
```

EUF 如果以后也需要，可以用：

```cpp
ExplainableRollbackUnionFind<EufReason> uf_;
```

这样 UF 只知道“边有 reason”，但不知道 reason 是 SAT literal 还是 EUF proof object。

---

## 备选方案 B：SharedEqualityManager 自己维护 reason forest

如果你不想动通用 UF，也可以在 `SharedEqualityManager` 中维护：

```cpp
std::vector<SharedTermId> parent_;
std::vector<int> size_;
std::vector<std::optional<SatLit>> edgeReason_;
std::vector<Change> changes_;
```

也就是把 reason-bearing UF 只做在 `SharedEqualityManager` 内部。

这个实现更局部，风险更小。缺点是会和现有 `RollbackUnionFind` 代码重复。

---

## 不推荐方案：直接让所有 RollbackUnionFind 都依赖 SatLit

不推荐这样：

```cpp
RollbackUnionFind::unite(a, b, SatLit reason)
```

因为 EUF、LRA、其他 theory 内部 UF 不一定有 SAT literal 作为直接 reason。强行传 dummy reason 是后续 soundness 隐患。

# 3. EUF 内部是否应该路径压缩？

从你描述看，`find()` 没有路径压缩，这是正确的设计意图。

对于 rollback union-find，通常不做路径压缩，原因有两个：

```text
1. 路径压缩会修改大量 parent，需要记录大量 rollback change。
2. explanation 需要保留 parent edge 上的 reason，路径压缩会破坏原始 explanation forest。
```

所以用于 explanation 的 UF 应该保持：

```text
union by size/rank: 可以
path compression: 不要
```

`explain(a,b)` 依赖从 `a` 到 root、从 `b` 到 root 的路径边 reason。如果路径压缩了，边 reason 就要被重写成组合 explanation，复杂很多，第一版不应该做。

因此：

```cpp
int find(int x) const {
    while (parent_[x] != x) x = parent_[x];
    return x;
}
```

不要写成压缩版本：

```cpp
parent_[x] = find(parent_[x]); // 不要
```

# 4. 正确的 conflict lemma 应该怎么生成

这是最关键的语义点。

假设当前 SAT assignment 中：

```text
r1 = Eq(y, v2)     true
r2 = Eq(v2, v1)    true
r3 = Eq(v1, y)     true
d  = ¬Eq(x, y)     true
```

SharedEqualityManager 发现：

```text
r1, r2, r3 imply Eq(x,y)
but d says ¬Eq(x,y)
```

那么 conflict clause 应该是：

```text
¬r1 ∨ ¬r2 ∨ ¬r3 ∨ ¬d
```

由于：

```text
d = ¬Eq(x,y)
¬d = Eq(x,y)
```

所以如果你内部只存 canonical positive equality atom，也可以表示成：

```text
¬Eq(y,v2) ∨ ¬Eq(v2,v1) ∨ ¬Eq(v1,y) ∨ Eq(x,y)
```

但实现上我建议统一使用 **实际赋值为 true 的 reason literal**：

```cpp
// equality reason:
r_i = Eq(a,b)      // assigned true

// disequality reason:
d = ¬Eq(x,y)       // assigned true
```

然后 conflict lemma 统一生成：

```cpp
for (SatLit r : equalityReasons)
    clause.push_back(negate(r));

clause.push_back(negate(disequalityReason));
```

这样不会搞错符号。

# 5. SharedEqualityManager 应该这样改

建议接口改成：

```cpp
struct DisequalityRecord {
    SharedTermId a;
    SharedTermId b;
    SatLit reasonLit;      // actual assigned literal, usually ¬Eq(a,b)
    int decisionLevel;
};

class SharedEqualityManager {
public:
    void assertEquality(
        SharedTermId a,
        SharedTermId b,
        SatLit reasonLit,
        int decisionLevel);

    void assertDisequality(
        SharedTermId a,
        SharedTermId b,
        SatLit reasonLit,
        int decisionLevel);

    std::optional<TheoryLemma> checkDisequalityConflict() const;

    std::vector<SatLit> explainEquality(
        SharedTermId a,
        SharedTermId b) const;
};
```

其中：

```cpp
assertEquality(a,b, reasonLit, level)
```

要求：

```text
reasonLit 必须是当前 SAT assignment 中为 true 的 equality literal。
例如 Eq(a,b)。
```

而：

```cpp
assertDisequality(a,b, reasonLit, level)
```

要求：

```text
reasonLit 必须是当前 SAT assignment 中为 true 的 disequality literal。
例如 ¬Eq(a,b)。
```

# 6. reason-bearing UF 的核心实现

大致如下。

```cpp
bool unite(int a, int b, SatLit reason) {
    int ra = find(a);
    int rb = find(b);

    if (ra == rb) {
        return false;
    }

    if (size_[ra] > size_[rb]) {
        std::swap(ra, rb);
    }

    changes_.push_back(Change{
        .childRoot = ra,
        .parentRoot = rb,
        .oldParentSize = size_[rb],
        .oldEdgeReason = edgeReason_[ra]
    });

    parent_[ra] = rb;
    size_[rb] += size_[ra];
    edgeReason_[ra] = reason;

    return true;
}
```

因为 `ra` 是 root，所以正常情况下：

```cpp
edgeReason_[ra] == std::nullopt
```

但保存 `oldEdgeReason` 更稳。

rollback：

```cpp
void rollback(Snapshot snap) {
    while (changes_.size() > snap) {
        auto ch = changes_.back();
        changes_.pop_back();

        parent_[ch.childRoot] = ch.childRoot;
        size_[ch.parentRoot] = ch.oldParentSize;
        edgeReason_[ch.childRoot] = ch.oldEdgeReason;
    }
}
```

# 7. explain(a,b) 的实现

因为没有路径压缩，路径解释很直接。

```cpp
std::vector<SatLit> explain(int a, int b) const {
    assert(find(a) == find(b));

    std::vector<SatLit> reasons;

    std::unordered_map<int, size_t> pos;
    std::vector<int> pathA;

    int x = a;
    while (true) {
        pos[x] = pathA.size();
        pathA.push_back(x);
        if (parent_[x] == x) break;
        x = parent_[x];
    }

    std::vector<int> pathB;
    x = b;
    int lca = -1;
    while (true) {
        auto it = pos.find(x);
        if (it != pos.end()) {
            lca = x;
            break;
        }
        pathB.push_back(x);
        if (parent_[x] == x) break;
        x = parent_[x];
    }

    // collect reasons from a up to lca
    x = a;
    while (x != lca) {
        assert(edgeReason_[x].has_value());
        reasons.push_back(*edgeReason_[x]);
        x = parent_[x];
    }

    // collect reasons from b up to lca
    x = b;
    while (x != lca) {
        assert(edgeReason_[x].has_value());
        reasons.push_back(*edgeReason_[x]);
        x = parent_[x];
    }

    return reasons;
}
```

可以后续去重：

```cpp
sort/re unique by SatLit
```

但第一版不强制。

# 8. checkDisequalityConflict 应该这样写

```cpp
std::optional<TheoryLemma>
SharedEqualityManager::checkDisequalityConflict() const {
    for (const auto& diseq : disequalities_) {
        if (uf_.find(diseq.a) != uf_.find(diseq.b)) {
            continue;
        }

        std::vector<SatLit> eqReasons = uf_.explain(diseq.a, diseq.b);

        TheoryLemma lemma;
        lemma.kind = TheoryLemmaKind::Conflict;

        for (SatLit r : eqReasons) {
            lemma.lits.push_back(r.negated());
        }

        lemma.lits.push_back(diseq.reasonLit.negated());

        return lemma;
    }

    return std::nullopt;
}
```

这里的核心 invariant 是：

```text
所有 eqReasons 在当前 SAT assignment 中为 true。
diseq.reasonLit 在当前 SAT assignment 中也为 true。
生成的 conflict clause 是这些 true literals 的否定析取。
```

这就是标准 CDCL(T) conflict clause。

# 9. 重复 equality 的处理

如果调用：

```cpp
assertEquality(a,b, reasonLit, level)
```

但 `a,b` 已经在同一类里，`unite()` 会返回 false。

这时不要强行把 reason 加到 UF 路径里。因为没有结构变化，就没有 edge 可以挂 reason。

第一版可以忽略这个重复 reason：

```cpp
if (!uf_.unite(a, b, reasonLit)) {
    return;
}
```

这不会影响 soundness，因为已有路径已经可以解释 `a=b`。

如果后续想优化解释长度，可以维护 alternative reasons，但不需要在 Stage A 做。

# 10. decision-level rollback 怎么处理

`SharedEqualityManager` 的 snapshot 应该包含两个部分：

```cpp
struct Snapshot {
    ExplainableUF::Snapshot ufSnap;
    size_t disequalitySize;
};
```

创建 snapshot：

```cpp
Snapshot snapshot() const {
    return Snapshot{
        .ufSnap = uf_.snapshot(),
        .disequalitySize = disequalities_.size()
    };
}
```

rollback：

```cpp
void rollback(Snapshot snap) {
    uf_.rollback(snap.ufSnap);
    disequalities_.resize(snap.disequalitySize);
}
```

不要只 rollback UF，而忘了 disequality records。

否则回溯后旧的 `¬Eq(a,b)` 还留着，会继续制造假冲突。

# 11. 对你三个问题的直接回答

## 问题 1：EUF solver 内部的 RollbackUnionFind 也需要 explanation 吗？

**当前 bug 不需要先改 EUF。**

必须先改的是 `SharedEqualityManager` 的 reason-bearing explanation。

但长期看：

```text
EUF 如果产生 SAT lemma/conflict，就必须有 EUF explanation。
EUF 不应该传 dummy reason。
EUF 的 reason 类型不一定是 SatLit，可能是 EufReason / proof object。
```

所以不要为了 SharedEqualityManager 的 bug，把 EUF 内部 UF 也强行改成 `SatLit reason`。

---

## 问题 2：explain 接口应该放在哪里？

推荐：

```text
新增 ExplainableRollbackUnionFind<Reason>
```

或者：

```text
在 SharedEqualityManager 内部维护独立 reason-bearing UF
```

不推荐：

```text
把 SatLit reason 直接塞进所有 RollbackUnionFind
```

如果你想复用代码，做模板化最好：

```cpp
ExplainableRollbackUnionFind<SatLit>         // SharedEqualityManager
ExplainableRollbackUnionFind<EufReason>      // future EUF explanation
```

---

## 问题 3：EUF 内部是否已经做路径压缩？

从你描述看，`find()` 没有压缩。这应该是刻意设计，而且是对的。

对于 rollback + explanation：

```text
不要路径压缩。
可以 union by size/rank。
```

路径压缩会破坏 explanation path，除非你额外记录压缩前后的 reason，这没有必要。

# 最终建议

你可以这样执行：

```text
1. 不改 EUF 内部 UF 的 reason 类型。
2. 新增 ExplainableRollbackUnionFind<SatLit>，或者在 SharedEqualityManager 内部实现 reason-bearing UF。
3. SharedEqualityManager::assertEquality 传入实际为 true 的 equality literal。
4. SharedEqualityManager::assertDisequality 传入实际为 true 的 disequality literal。
5. explainEquality(a,b) 返回 UF 路径上的 equality reason literals。
6. checkDisequalityConflict 生成：
   ¬r1 ∨ ... ∨ ¬rk ∨ ¬d
7. rollback 同时恢复 UF 和 disequality records。
8. 保持 find() 无路径压缩。
```

一句话：**这个 bug 的正确修法是在组合层实现可回滚、带 reason 的 equality explanation；不要用空 explanation，也不要把 EUF 的内部 congruence explanation 和 SharedEqualityManager 的 SAT-level explanation 混在一起。**
