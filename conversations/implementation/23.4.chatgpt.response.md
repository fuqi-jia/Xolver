**结论：不接受当前版本。**

这版已经非常接近可实现，而且比上一版干净很多：范围已经收缩为 **Incremental EUF baseline only**，model construction、theory propagation、mixed fallback 都明确 out of scope；decision-level snapshot、scope snapshot、Bool formula-position、`true != false`、`pendingConflict_` 生命周期、`ExplainResult`、`MergeStatus` 等关键红线也基本补齐了。

但是还有一个核心实现 blocker：**RollbackUnionFind 的 winner/loser 与 IncrementalEGraph 的 members 合并假设不一致**。这个不修，rollback 后 `members_` 会错，继而影响 congruence closure。

# 1. 最大 blocker：`uf_.unite(ra, rb)` 的 winner 不一定是 `ra`

当前 `RollbackUnionFind::unite(a,b)` 规则是：

```text
union-by-size；
attach smaller root to larger root；
returns true iff merged。
```

但 `IncrementalEGraph::merge()` 里写的是：

```cpp
// unite (ra is winner, rb is loser)
uf_.unite(ra, rb);

// merge members: append rb members to ra
memberTrail_.push_back({ra, rb, members_[ra].size()});
members_[ra].insert(members_[ra].end(), members_[rb].begin(), members_[rb].end());
```

这里假设 **`ra` 一定是 winner，`rb` 一定是 loser**。但 union-by-size 不保证这一点。
如果 `rb` 的 class 更大，`uf_.unite(ra, rb)` 会让 `rb` 成为 winner。此时你却把 `members_[rb]` append 到 `members_[ra]`，导致：

```text
UF root = rb
members_[rb] 没更新
members_[ra] 被更新但 ra 已经不是 root
```

而 hard rule 又说：

```text
members_[nonRoot] may be stale；only read members_[root]
```

于是后续 `collectParents(rep)` 会读到错误的 root members，congruence closure 可能漏合并。

## 必须修改

让 `RollbackUnionFind::unite()` 返回实际 winner/loser。

建议改成：

```cpp
struct UniteResult {
    bool merged;
    EufClassId winner;
    EufClassId loser;
};

UniteResult RollbackUnionFind::unite(EufClassId a, EufClassId b);
```

实现：

```cpp
UniteResult RollbackUnionFind::unite(EufClassId a, EufClassId b) {
    EufClassId ra = find(a);
    EufClassId rb = find(b);

    if (ra == rb) {
        return {false, ra, rb};
    }

    if (size_[ra] < size_[rb]) {
        std::swap(ra, rb);
    }

    trail_.push_back({
        .childRoot = rb,
        .parentRoot = ra,
        .oldParentSize = size_[ra]
    });

    parent_[rb] = ra;
    size_[ra] += size_[rb];

    return {true, ra, rb};
}
```

然后 `IncrementalEGraph::merge()` 改成：

```cpp
auto ur = uf_.unite(ra, rb);
if (!ur.merged) continue;

EufClassId dst = ur.winner;
EufClassId src = ur.loser;

memberTrail_.push_back({dst, src, members_[dst].size()});
members_[dst].insert(
    members_[dst].end(),
    members_[src].begin(),
    members_[src].end()
);
```

`MergeRecord` 仍然可以记录原始 `lhsRootBefore = ra` / `rhsRootBefore = rb`，但 members 合并必须按 `winner/loser` 来做。

---

# 2. `ensureSnapshotForLevel()` 还需要处理同 level 重入与 level 回退后的新分支

当前规则是：

```cpp
if (levelSnapshots_.empty() || levelSnapshots_.back().level < level) {
    levelSnapshots_.push_back(...);
}
```

这通常可行，但需要明确一个前提：

```text
CDCL decision levels 单调进入；
backtrack 后会弹掉更高 levelSnapshots_；
同一 level 的后续 assertLit 不再重复创建 snapshot。
```

你的 `backtrackToLevel()` 已经会弹掉 `level > target`，所以基本成立。
但建议加一个防御：

```cpp
assert(levelSnapshots_.empty() || levelSnapshots_.back().level <= level);
```

如果出现 `level < levelSnapshots_.back().level`，说明调用顺序异常，应先 backtrack，而不是继续 assert。

这不是 blocker，但建议写进实现。

---

# 3. Bool formula-position 分支没有检查 `merge()` 返回值

当前 Bool 分支：

```cpp
if (payload.kind == EufAtomKind::BoolTermAsFormula) {
    EufTermId target = value ? trueTerm : falseTerm;
    trail_.push_back({level, reason, atom, value});
    egraph_.merge(lhs, target, MergeReason::asserted(reason));
    return;
}
```

这里也可能返回 `MergeStatus::SortMismatch`。虽然按正确 Atomizer/SortManager 不该发生，但 solver 层不应静默忽略。

建议改成：

```cpp
auto status = egraph_.merge(lhs, target, MergeReason::asserted(reason));
if (status == MergeStatus::SortMismatch) {
    pendingUnknown_ = true;
}
```

---

# 4. `pendingConflict_` 的设置条件没有真正实现

文档写：

```text
pendingConflict_ is set in assertLit() when a disequality is violated by an incremental merge.
```

但 `assertLit()` 的伪代码里没有实际设置 `pendingConflict_`，只是在 `check()` 扫 disequalities。
这不是 soundness blocker，因为 `check()` 会发现冲突；但文档和代码不一致。

建议二选一：

## 方案 A：删掉这句话

保留当前设计：

```text
assertLit() 不设置 pendingConflict_；
check() 统一扫描 true/false 和 disequalities。
```

这样最简单。

## 方案 B：真的实现 incremental conflict detection

每次 equality merge 后检查 active disequalities 是否被违反，并设置 `pendingConflict_`。

V1 baseline 我建议用方案 A：**删掉 pendingConflict_ 在 assertLit 设置的描述**，只保留它作为未来优化或外部冲突缓存。否则实现模型会困惑。

---

# 5. 测试名有一个语义不清

测试表里：

```text
decision-level snapshot | level 1 merge survives backtrack to level 0 then push to level 1
```

“level 1 merge survives backtrack to level 0” 这句话语义不对。
如果 backtrack 到 level 0，level 1 merge 应该被撤销，不应 survive。

你想测的应该是：

```text
level 1 merge survives backtrack to level 1
level 2 merge is removed
```

建议改成：

```text
level 1: merge(a,b)
level 2: merge(b,c)
backtrackToLevel(1)
expect same(a,b) == true
expect same(b,c) == false
```

这个测试正好覆盖上一轮的关键问题。

---

# 最小修改后即可接受

只要改下面几处，我认为下一版可以接受：

```text
1. RollbackUnionFind::unite 返回 winner/loser，不只返回 bool。
2. IncrementalEGraph::merge 按 winner/loser 合并 members_。
3. BoolTermAsFormula 分支检查 MergeStatus，SortMismatch 设置 pendingUnknown_。
4. pendingConflict_ 文档与伪代码统一：要么实现，要么删掉“assertLit 设置 pendingConflict_”的说法。
5. decision-level snapshot 测试改成：level 1 merge survives backtrackToLevel(1)，level 2 merge is removed。
```

# 最终意见

**不接受当前版本。**

不是架构问题，而是一个关键实现细节会破坏 congruence closure：`RollbackUnionFind` 的 union-by-size winner 与 `members_` 合并目标不一致。修掉这个之后，这版基本可以进入实现。
