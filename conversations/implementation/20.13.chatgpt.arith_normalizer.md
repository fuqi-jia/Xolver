对，**必须全局统一**。现在这个 bug 的本质不是某一处 `-linearRhs` 写错，而是系统里缺少一个唯一的“线性约束标准化出口”。
我的建议是：新增一个专门模块，所有地方都只能通过它把 `p rel 0` 转成 `lhs rel rhs`，禁止 generator / adapter / solver 直接手写移项。

---

# 总体方案：建立 `LinearConstraintNormalizer`

核心原则：

```text
内部统一表示：    constant + Σ a_i x_i  rel  0
注册到 LIA/LRA：  Σ a_i x_i             rel -constant
```

也就是说，所有 McCormick cut、square cut、abstraction lemma、mirror bound，都先生成：

```cpp
ZeroLinearConstraint
```

最后只有一个地方负责转换成：

```cpp
LinearAtomSpec
```

这样不会再出现到处写：

```cpp
rhs = -linearRhs;
rhs = linearRhs;
rhs += 1;
rhs -= 1;
```

导致符号混乱的问题。

# 1. 新增统一数据结构

建议放到：

```text
src/theory/arith/linear/LinearConstraintNormalizer.h
src/theory/arith/linear/LinearConstraintNormalizer.cpp
```

或者如果你想和 linearizer 放近一点：

```text
src/theory/arith/linearizer/LinearConstraintNormalizer.h
src/theory/arith/linearizer/LinearConstraintNormalizer.cpp
```

核心结构：

```cpp
using Rat = mpq_class;

struct LinearTerm {
    std::string var;
    Rat coeff;
};

// Represents: constant + Σ coeff_i * var_i
struct LinearExpr {
    Rat constant = 0;
    std::vector<LinearTerm> terms;
};

// Represents: expr rel 0
struct ZeroLinearConstraint {
    LinearExpr expr;
    Relation rel;
    SortKind sort;          // Int for NIA/LIA, Real for NRA/LRA
    std::string debugTag;
};

// Represents: lhs rel rhs
// This is exactly what getOrCreateLinearBoundAtom expects.
struct LinearAtomSpec {
    LinearFormKey lhs;
    Relation rel;
    Rat rhs;
    SortKind sort;
    std::string debugTag;
};
```

注意：**业务层永远生成 `ZeroLinearConstraint`，不要直接生成 `LinearAtomSpec`。**

# 2. 唯一移项函数

最关键的是这个函数：

```cpp
class LinearConstraintNormalizer {
public:
    static ZeroLinearConstraint fromPolynomialZero(
        const PolynomialKernel& kernel,
        PolyId poly,
        Relation rel,
        SortKind sort,
        std::string debugTag);

    static LinearAtomSpec toLinearAtomSpec(
        const ZeroLinearConstraint& c);

    static SatLit registerLinearConstraint(
        TheoryAtomRegistry& registry,
        const ZeroLinearConstraint& c,
        TheoryId targetTheory);
};
```

实现逻辑：

```cpp
LinearAtomSpec LinearConstraintNormalizer::toLinearAtomSpec(
    const ZeroLinearConstraint& c) {

    LinearAtomSpec out;
    out.rel = c.rel;
    out.sort = c.sort;
    out.debugTag = c.debugTag;

    // expr = constant + Σ a_i x_i
    // expr rel 0
    // Σ a_i x_i rel -constant

    LinearExpr e = canonicalize(c.expr);

    out.lhs = makeLinearFormKey(e.terms);
    out.rhs = -e.constant;

    if (out.sort == SortKind::Int) {
        normalizeStrictIntegerRelation(out.rel, out.rhs);
    }

    return canonicalizeAtomSpec(out);
}
```

对你当前的例子：

```text
linearizedPoly = __nl_aux_0 - 4
```

内部表示是：

```text
constant = -4
terms = { +1 * __nl_aux_0 }
rel = =
```

统一出口得到：

```text
rhs = -constant = 4
```

所以注册结果必然是：

```text
__nl_aux_0 = 4
```

不会再误变成：

```text
__nl_aux_0 = -4
```

# 3. 禁止业务代码直接调用 `getOrCreateLinearBoundAtom`

现在 plan 里 linearizer cut 会通过 `getOrCreateLinearBoundAtom` 注册到 LIA/LRA，这是方向对的；cuts 本来就应该作为 linear atoms 路由给 LIA/LRA。你的 plan 也明确写了 NIA cuts → LIA，NRA cuts → LRA 的动态路由原则。

但要加一个工程约束：

```cpp
// 禁止这样直接调用：
registry->getOrCreateLinearBoundAtom(lhs, rel, rhs, targetTheory);
```

改成：

```cpp
SatLit lit = LinearConstraintNormalizer::registerLinearConstraint(
    *registry,
    zeroConstraint,
    targetTheory

```

也就是说，`getOrCreateLinearBoundAtom` 应该只在 normalizer 内部出现。
如果不想改 registry 可见性，至少用 grep / 单测 / 代码审查强约束：


只允许出现在：

```text
LinearConstraintNormalizer.cpp
NiaLinearizationAdapter.cpp  // 临时兼容，后续也迁走
NraLinearizationAdapter.cpp  // 临时兼容，后续也迁走
```

最终最好只剩 normalizer 一个入口。

# 4. 所有 generator 只生成 `expr rel 0`

例如 square nonnegativity：

原来可能到处写：

```text
s >= 0
```

统一改成：

```cpp
ZeroLinearConstraint makeSquareNonneg(const AuxTerm& s) {
    LinearExpr e;
    e.terms.push_back({s.name, Rat(-1)});

    // -s <= 0  等价于 s >= 0
    return {
        .expr = canonicalize(e),
        .rel = Relation::Leq,
        .sort = s.sort,
        .debugTag = "square_nonneg"
    };
}
```

也可以写成：

```cpp
s >= 0
```

但为了全局统一，我建议 **所有 cut 都输出 `<= 0`**，除了 abstraction lemma 可以保留原始关系。

例如：

```text
s >= 0
```

统一写成：

```text
-s <= 0
```

这样 cut generator 内部只有一种方向，减少后续冲突。

# 5. McCormick / Square cut 全部改成零右端形式

你的 plan 中 McCormick 和 square cuts 已经基本采用了 `... <= 0` 的形式，这很好。比如 square nonnegativity 写成 `s >= 0 -> -s <= 0`，secant 写成 `s - (l+u)x + lu <= 0`，tangent 写成 `-s + 2ax - a^2 <= 0`。

建议把这个变成强制接口：

```cpp
struct LinearCut {
    ZeroLinearConstraint zero;
    std::vector<SatLit> reasons;
    std::string debugTag;
};
```

不要再写：

```cpp
struct LinearCut {
    LinearFormKey lhs;
    Relation rel;
    mpq_class rhs;
};
```

而是改成：

```cpp
struct LinearCut {
    ZeroLinearConstraint constraint;
    std::vector<SatLit> reasons;
    std::string debugTag;
};
```

这样 cut generator 没有机会手动填 `rhs`。

# 6. Abstraction lemma 也走同一个出口

对于 nonlinear constraint：

```text
p rel 0
```

abstraction 后得到：

```text
p_L rel 0
```

然后统一：

```cpp
ZeroLinearConstraint z =
    LinearConstraintNormalizer::fromPolynomialZero(
```

你的 plan 里 abstraction lemma 是 `¬C ∨ p_L rel 0`，cut lemma 是 `¬C ∨ ¬bound_reasons... ∨ cut`，这个结构保留即可。

# 7. 整数严格不等式也统一处理

这类也很容易出错：

```text
x < 4    over Int  => x <= 3
x > 4    over Int  => x >= 5
```

不要让每个地方自己写。

统一放到：

```cpp
static void normalizeStrictIntegerRelation(Relation& rel, Rat& rhs) {
    if (rel == Relation::Lt) {
        rel = Relation::Leq;
        rhs -= 1;
    } else if (rel == Relation::Gt) {
        rel = Relation::Geq;
        rhs += 1;
    }
}
```

只在：

```cpp
toLinearAtomSpec()
```

里面调用。

这样所有 NIA → LIA 的 linear atom 都自动规范化。你的 plan 里也已经提到 NIA 下 strict linear atoms 要在注册前做整数归一化，这个设计可以把它变成强制规则。

# 8. Effective relation 也要统一

`mirrorActiveLinearBounds` 里面还有一个容易错的地方：

```text
原始 atom 被赋值 false 时，要 mirror 它的否定，而不是原始关系。
```

你 plan 里已经强调 mirror 要用 effective constraint，不是 raw syntactic atom。

建议也放进 normalizer：

```cpp
std::optional<ZeroLinearConstraint>
LinearConstraintNormalizer::makeEffectiveConstraint(
    const LinearAtomPayload& payload,
    bool assignedValue,
    SortKind sort);
```

关系转换规则：

```text
assigned true:
  rel 原样

assigned false:
  <=  -> >
  <   -> >=
  >=  -> <
  >   -> <=
  =   -> !=
  !=  -> =
```

但是 V1 要注意：

```text
not (=) 是 disequality，不是一个 linear bound。
```

所以 V1 可以这样处理：

```cpp
if (effectiveRel == Relation::Neq) {
    return std::nullopt; // V1 不 mirror disequality
}
```

不要强行把：

```text
x != 4
```

拆成：

```text
x <= 3 or x >= 5
```

因为这是 disjunction，不是单个 linear atom。除非你专门生成 split lemma，否则 V1 先跳过更安全。

# 10. 高效实现细节

为了效率，normalizer 不能每次都做很重的字符串操作。建议这样做。

## 10.1 变量用 ID，不要长期用 string

短期可以用：

```cpp
std::string var;
```

但最终最好用：

```cpp
VarId var;
```

结构改成：

```cpp
struct LinearTerm {
    VarId var;
    Rat coeff;
};
```

`debugTag` 里再保留变量名。

## 10.2 canonicalize 用排序 + 合并

```cpp
LinearExpr canonicalize(LinearExpr e) {
    std::sort(e.terms.begin(), e.terms.end(),
              [](const auto& a, const auto& b) {
                  return a.var < b.var;
              });

    std::vector<LinearTerm> merged;

    for (auto& t : e.terms) {
        if (t.coeff == 0) continue;

        if (!merged.empty() && merged.back().var == t.var) {
            merged.back().coeff += t.coeff;
            if (merged.back().coeff == 0) {
                merged.pop_back();
            }
        } else {
            merged.push_back(t);
        }
    }

    e.terms = std::move(merged);
    return e;
}
```

## 10.3 可以缓存 `ZeroLinearConstraint -> LinearAtomSpec`

很多 cut 会重复生成，建议加一个 lightweight cache：

```cpp
struct ZeroLinearConstraintKey {
    Relation rel;
    SortKind sort;
    Rat constant;
    std::vector<std::pair<VarId, Rat>> terms;
};

std::unordered_map<ZeroLinearConstraintKey, LinearAtomSpec> atomSpecCache_;
```

不过注意：这个 cache 只缓存 **规范化后的 atom spec**，不要替代你原来的 `LinearizationCache`。
原来的 `LinearizationCache` 仍然要包含 `nonlinearReason`、bound values、cut index，避免 guard 丢失。你的 plan 里已经明确 cache key 需要包含 nonlinear reason 和 bound values，这个设计要保留。

最小测试集：

```text
1. __nl_aux_0 - 4 = 0
   => __nl_aux_0 = 4

2. __nl_aux_0 + 4 = 0

   => x <= 3

   => x >= 5
```

# 13. 和 LIA tableau 的接口也要统一语义

你的 LIA/simplex plan 里已经明确 row 语义应该固定成：

```text
x_b = rhs + Σ a_j x_j
```

不要混用其他形式。

这和这里的 normalizer 是同一个思想：
**外部线性约束统一为 `lhs rel rhs`，进入 simplex tableau 时再统一改写为 row 语义。**

所以建议分两层：

```text
LinearConstraintNormalizer:
  p rel 0  -> lhs rel rhs

GeneralSimplex:
  lhs rel rhs -> slack/basic row
```

不要让 simplex 再去理解 polynomial constant，也不要让 linearizer 理解 tableau row。

# 最后建议的接口形态

最干净的最终代码应该是这样：

```cpp
ZeroLinearConstraint z =
    LinearConstraintNormalizer::fromPolynomialZero(
        kernel_,
        linearizedPoly,
        originalRel,
        auxSort,
        "abstraction"
    );

SatLit lit =
    LinearConstraintNormalizer::registerLinearConstraint(
        *registry_,
        z,
        targetLinearTheory
    );
```

cut generator 则是：

```cpp
LinearCut cut;
cut.constraint = ZeroLinearConstraint{
    .expr = LinearExpr{
        .constant = -4,
        .terms = {{auxVar, 1}}
    },
    .rel = Relation::Eq,
    .sort = SortKind::Int,
    .debugTag = "abstraction:x2_eq_4"
};
```

最终注册结果由 normalizer 保证：

```text
__nl_aux_0 = 4
```

而不是由调用者猜：

```text
rhs 到底该传 4 还是 -4？
```

一句话：**V1 现在最应该补的不是更多 cut，而是把“线性约束标准化”做成唯一、可测试、不可绕过的基础设施。**
