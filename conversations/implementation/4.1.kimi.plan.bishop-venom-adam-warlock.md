# Plan: QF_LIA Solver (Phase 1 â€” Minimal LIA)

## Goal

Add quantifier-free linear integer arithmetic (QF_LIA) support to NLColver. Phase 1 implements the minimal closed loop:

```
GeneralSimplex (LRA relaxation)
  â†’ integral model check
  â†’ disequality split (reuses LRA design)
  â†’ branch-and-bound split lemma
  â†’ model validation
```

## Current State

- `GeneralSimplex` is now a sparse indexed tableau (tested, 18 tests pass).
- `SimplexSolver` wraps `GeneralSimplex` for LRA. It handles `CoreExpr` â†’ linear form extraction, atom registration, bound assertion, conflict translation.
- `NiaSolver` is an empty placeholder returning `unknown()`.
- `Solver.cpp` registers `SimplexSolver` (LRA) and `CdcacSolver` (NRA). No LIA solver exists.
- `CoreExpr` carries a `SortId`, but the sort system has no explicit int/real/bool classification yet.

## Design Decision

**Extend `SimplexSolver` with a LIA mode** rather than creating a separate `LiaSolver` class.

Rationale:
- LIA = LRA + integer layer. The constraint extraction, atom registration, bound mapping, and conflict translation are identical.
- A separate `LiaSolver` would duplicate ~300 lines of `SimplexSolver` logic (or require a non-trivial base-class extraction refactor).
- The integer layer is a thin wrapper around the existing `check()` flow.

Trade-off: `SimplexSolver::id()` currently returns `TheoryId::LRA`. For LIA mode it should conceptually be `TheoryId::LIA`, but the `TheoryManager` does not filter solvers by theory ID at check time. The SAT atomizer feeds all theory atoms to all solvers; each solver ignores atoms it cannot handle. So a single `SimplexSolver` instance can safely process both LRA and LIA atoms if it knows which variables are integer.

Future refactoring: when QF_LIRA (mixed) is needed, extract a `LinearSolverBase` and split into `LraSolver` + `LiaSolver`.

## Implementation Steps

### Step 1: Add integer variable tracking to `SimplexSolver`

Add to `SimplexSolver`:

```cpp
bool liaMode_ = false;
std::unordered_set<int> integerVars_;  // GeneralSimplex var indices that are integer
```

When `registerAtom` extracts a linear constraint, inspect the `CoreExpr` sort of each variable node. If `liaMode_` is true and the sort is integer, add the corresponding GS var to `integerVars_`.

**Sort classification**: The `CoreIr` stores `SortId` per expression, but there is no global sort name table. For Phase 1, add a minimal sort classification to `CoreIr`:

```cpp
// In CoreIr or a separate SortRegistry
enum class SortClass { Unknown, Bool, Int, Real, BV, FP };
SortClass classifySort(SortId s) const;
```

Populate it during `FrontendAdapter::importProblem()` by inspecting SOMTParser sort names (`"Int"`, `"Real"`, `"Bool"`). Store a `std::vector<SortClass>` indexed by `SortId`.

### Step 2: Implement integral model check

In `SimplexSolver::check()`:

```cpp

if (r == GeneralSimplex::Result::Unsat) {
    return TheoryCheckResult::mkConflict(translateConflict());
}

// LIA mode: check if all integer variables have integral values
if (liaMode_) {
    bool allIntegral = true;
    int fractionalVar = -1;
    DeltaRational fractionalValue;
    for (int v : integerVars_) {
        auto val = gs_.value(v);
        if (val.b != 0 || val.a.get_den() != 1) {
            allIntegral = false;
            fractionalVar = v;
            fractionalValue = val;
            break; // or collect all fractional vars for better branching
        }
    }

    if (!allIntegral) {
        return TheoryCheckResult::mkLemma(
            buildBranchSplitLemma(fractionalVar, fractionalValue));
    }
}

// Check disequalities (existing logic)
...

return TheoryCheckResult::consistent();
```

### Step 3: Implement branch split lemma builder

```cpp
TheoryLemma SimplexSolver::buildBranchSplitLemma(int var, const DeltaRational& value) {
    mpz_class lo = floor(value.a);
    mpz_class hi = lo + 1;  // ceil

    // Lemma: (x <= lo) âˆ¨ (x >= hi)
    // These are new SAT literals; for Phase 1 we encode them as
    // unit clauses with fresh SAT vars, or we reuse existing atoms.
    // V1: return a lemma containing two fresh placeholder literals.
    //     The SAT solver will need to learn and assign them.
    //     In practice, the SAT solver will try both branches.

    // For testing, we can create explicit bound assertions:
    SatLit left = SatLit{freshSatVar_, true};   // x <= lo
    SatLit right = SatLit{freshSatVar_, true};  // x >= hi
    return TheoryLemma{{left, right}};
}
```

**Important**: The branch split lemma `x <= floor(q) âˆ¨ x >= ceil(q)` is a theory tautology for integer variables. It does not need premises/guarding. The SAT solver will add this clause and eventually assign one of the literals true, which on the next `check()` iteration will assert the corresponding bound.

However, the new literals `x <= lo` and `x >= hi` must be **registered as atoms** so the SAT solver can use them and the theory solver can interpret them. For Phase 1:

- Create them dynamically in `SimplexSolver`.
- Store them in a `branchAtoms_` map: `var + bound -> ExprId`.
- When `assertLit` is called for these new atoms, translate them to `gs_.assertLower/Upper` as usual.

### Step 4: Dynamic atom creation for branch bounds

When a branch split is returned, the SAT solver receives a clause with fresh literals. On the next solve, if SAT assigns one of these literals true, `TheoryManager::check()` will call `assertLit` with the new atom.

`SimplexSolver` must handle these dynamically created atoms:

```cpp
void SimplexSolver::assertLit(const TheoryAtom& atom, bool value, const CoreIr& ir) {
    // Check if it's a dynamically created branch bound atom
    auto it = branchAtoms_.find(atom.exprId);
    if (it != branchAtoms_.end()) {
        assertBound(it->second.auxVar, it->second.rel, value, ...);
        return;
    }
    // ... existing logic
}
```

For Phase 1, the simplest approach is:
- `buildBranchSplitLemma` creates two new `CoreExpr` nodes (`Leq` and `Geq`) via a mutable `CoreIr*` reference, or stores them separately.
- This requires `SimplexSolver` to have a reference to the current `CoreIr`.

**Simpler alternative for Phase 1**: Instead of creating real SAT atoms, have the branch split lemma use the existing disequality split mechanism. Return a lemma that contains the branch bounds as unit clauses that the SAT solver treats as decisions.

Actually, the cleanest Phase 1 design:

1. `buildBranchSplitLemma` returns `TheoryLemma` with 2 fresh `SatLit`s.
2. The SAT solver adds this clause.
3. On the next SAT solve, one of the literals is assigned.
4. The SAT solver calls `theoryManager_.assertLit` for the new literal.
5. But the literal's `ExprId` doesn't exist in the original problem!

This is the core challenge: **dynamically creating theory atoms**.

For Phase 1, work around this by storing a side table in `SimplexSolver`:

```cpp
struct BranchAtom {
    int auxVar;      // GeneralSimplex aux var
    Relation rel;    // Leq or Geq
    mpq_class rhs;   // bound value
};
std::unordered_map<ExprId, BranchAtom> branchAtoms_;
ExprId nextBranchExprId_ = ...; // starting from some high number
```

When creating a branch split, allocate a new `ExprId` (not in CoreIr, just a unique ID), map it to the branch bound, and return `SatLit`s referencing these IDs. When `assertLit` is called with these IDs, look up the bound and assert it.

This avoids modifying CoreIr but is a bit of a hack. For a proper implementation, we need a mutable expression registry.

## Risks & Mitigations

| Risk | Mitigation |
|------|------------|
| Dynamic branch atom creation is hacky | Phase 1 uses side-table hack; Phase 2 adds proper mutable expression registry |
| Sort classification incomplete | Only classify Int/Real/Bool for now; other sorts map to Unknown |
| Branch split may cause infinite loops without fair branching | Bland's rule on simplex + SAT solver's learned clauses prevent cycling in practice; add iteration limit safeguard |
| `SimplexSolver` growing too large | If LIA logic exceeds ~150 lines, extract `LiaBranching` helper class |
| No gcd reasoning in Phase 1 | Some LIA problems will branch excessively; Phase 2 adds cheap gcd reasoning |
