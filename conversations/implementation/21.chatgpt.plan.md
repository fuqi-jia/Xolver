下面给你一版 **IncrementalLinearizer 后续增强版 plan**。前提是当前 V1 已经批准：

```text
V1:
  x*y McCormick
  x² square nonneg/secant/tangent
  aux var
  abstraction lemma
  guarded cut lemma
  NIA -> LIA
  NRA -> LRA
  mirror active linear bounds
  pending lemma + CutCacheKey
```

接下来目标不是简单“多支持几个算子”，而是把它升级成一个真正强大的 **非线性松弛与割平面引擎**。

---

# 0. 总体定位

增强后的 IncrementalLinearizer 应该变成：

```text
非线性约束
  ↓
结构识别
  ↓
auxiliary abstraction
  ↓
bound-aware relaxation
  ↓
model-based separation
  ↓
cut selection / cache / scheduling
  ↓
LIA/LRA relaxation strengthening
```

它仍然不直接做 SAT / UNSAT，而是负责生成 sound linear lemmas：

```text
原非线性约束 active
+ 当前 bounds active
⇒ 某个 linear relaxation cut active
```

最终 UNSAT 由 LIA/LRA 证明；最终 SAT 仍然必须回到 NIA/NRA exact validator。

---

# 1. 架构升级目标

后续 linearizer 应支持四类能力：

```text
A. 更广泛的非线性项抽象
   x*y, x², x^k, x*y*z, x²*y, general monomial

B. 更强的线性松弛
   McCormick, recursive McCormick, square/power tangent/secant,
   RLT cuts, bound-implied cuts, integer-strengthened cuts

C. 更好的增量效率
   term cache, cut cache, bound versioning, violation-based separation,
   watchlist, cut scoring, cut budget

D. 更深的 solver 集成
   ICP 提供 tighter box；
   LIA/LRA 负责 relaxation；
   NIA/NRA exact validator 过滤 false SAT；
   LocalSearch 提供 separating point；
   MCSAT 未来使用 cuts 作为 theory lemmas。
```

# 3. V2：支持一般 monomial

当前 V1 只支持：

```text
x*y
x²
```

V2 要支持：

```text
x*y*z
x²*y
x*y²
x³
x^k
c*x²*y³
```

## 3.1 Monomial key

使用 canonical monomial powers：

```cpp
struct MonomialKey {
    std::vector<std::pair<std::string, int>> powers; // sorted by variable name
};

examples:
  x*y      -> {x:1, y:1}
  y*x      -> {x:1, y:1}
  x²*y³    -> {x:2, y:3}
```

不要用 `PolyId children` 做 key，因为重建 poly 后 id 可能不稳定。

---

## 3.2 Recursive product abstraction

对 monomial：

```text
x²*y³
```

递归分解：

```text
s_x2 = x*x
s_y2 = y*y
s_y3 = s_y2*y
t = s_x2*s_y3
```

或者使用更平衡的乘法树：

```text
x²*y³ = (x*x) * (y*y*y)
```

推荐 V2 采用 **balanced product tree**，减少 aux 深度：

```text
product(v1, v2, v3, v4)
  -> t1 = v1*v2
  -> t2 = v3*v4
  -> t  = t1*t2
```

---

## 3.3 Aux variable sort

规则：

```text
NIA:
  所有 aux term 是 Int

NRA:
  所有 aux term 是 Real
```

注意：NIA 中如果原变量都是 Int，monomial 结果仍是 Int。

---

## 3.4 Abstraction lemma

原约束：

```text
C: 3*x²*y + z <= 5
```

抽象后：

```text
t = x²*y
L: 3*t + z <= 5
```

加入：

```text
¬C ∨ L
```

这个 lemma 是 conservative extension lemma。它不是普通 LIA theory-valid lemma，而是 fresh aux 下的 relaxation extension。最终 SAT 必须 validate 原非线性约束。

# 4. V3：递归 McCormick

对每个 product aux：

```text
t = a*b
```

只要 `a,b` 有 finite complete bounds，就生成 McCormick envelope：

```text
t ≥ la*b + lb*a - la*lb
t ≥ ua*b + ub*a - ua*ub
t ≤ ua*b + lb*a - ua*lb
t ≤ la*b + ub*a - la*ub
```

## 4.1 Aux term bounds

递归 product 需要给 aux 也维护 bounds。

例如：

```text
x ∈ [0,10]
y ∈ [0,10]
t = x*y
```

可以推出：

```text
t ∈ [0,100]
```

这个 bound 可作为后续：

```text
u = t*z
```

的 McCormick 输入。

所以 Linearizer 需要维护：

```cpp
struct AuxBoundInfo {
    BoundInfo bound;
    std::vector<SatLit> reasons;
    bool derivedByIntervalEval;
};
```

这些 aux bounds 来自 interval multiplication，是 over-approx。

---

## 4.2 Aux bound reasons

对于：

```text
t = x*y
x ∈ [lx,ux]
y ∈ [ly,uy]
```

推导 `t∈[lt,ut]` 的 reason 应包含：

```text
x lower/upper reasons
y lower/upper reasons
definition context / nonlinear reason
```

如果 reasons 不完整，不能用这些 bound 生成 guarded cuts。

V1 可以先只把 aux bounds 作为 internal heuristic，不生成 outward bound lemma；V3 需要 reasons 完整化。

---

# 5. V4：convex / concave cuts

除了 McCormick，还要给常见 convex terms 生成 tangent / secant cuts。

## 5.1 Convex univariate term

对：

```text
s = f(x)
```

如果 `f` 在当前 interval 上 convex，则 tangent 是全局下界：

```text
s ≥ f(a) + f'(a)(x-a)
```

转成：

```text
-s + f'(a)x + f(a) - f'(a)a <= 0
```

适用于：

```text
x²
x^4
exp(x)   // 未来超越函数时
```

当前多项式阶段先支持：

```text
x²
x^4
even power on sign-stable interval
```

---

## 5.2 Concave univariate term

如果 `f` concave，则 tangent 是上界。

例如在正区间上的某些根函数以后可以用，但 V2/V3 不急。

---

## 5.3 Secant cuts

对 convex `f` 在 `[l,u]` 上，secant 是上界：

```text
s <= slope*x + intercept
```

对 concave 则相反。

当前优先：

```text
x² secant upper
x² tangent lower
x^k tangent/secant for small k
```

---

# 6. V5：model-based separation

现在 V1 默认 emit all McCormick cuts。后续为了效率，应该加入 separation 模式。

输入 current relaxation model：

```text
x0, y0, t0
```

对 product `t=x*y`：

```text
evaluate 4 McCormick cuts
只发 violated cuts
```

对 square `s=x²`：

```text
选择 tangent point a = x0
如果当前 s0 < x0²，则 tangent likely violated
```

## 6.1 Separation mode

配置：

```cpp
enum class CutGenerationMode {
    EmitAll,
    ViolatedOnly,
    Hybrid
};
```

建议：

```text
early stage:
  EmitAll for small problems

large benchmark:
  ViolatedOnly or Hybrid
```

---

## 6.2 Cut violation score

每个 cut 计算：

```text
violation = lhs(model) - rhs
```

只发：

```text
violation > threshold
```

并排序：

```text
1. violation magnitude
2. term activity
3. bound tightness
4. previous success score
```

---

# 7. V6：cut selection / cut pool

如果每个 product 都发 4 条，每个 square 又发 2-3 条，很快爆炸。

需要 cut pool：

```cpp
struct CutCandidate {
    TheoryLemma lemma;
    CutCacheKey cacheKey;
    double violationScore;
    double expectedStrength;
    int age;
    int timesGenerated;
};
```

选择策略：

```text
1. hard limit maxCutsPerRound
2. prefer violated cuts
3. prefer cuts using tight bounds
4. avoid near-duplicate cuts
5. age-based fairness
```

配置：

```cpp
struct LinearizationConfig {
    int maxCutsPerRound = 32;
    int maxCutsPerTerm = 4;
    int maxAuxTermsPerRound = 64;
    bool emitAllMcCormickInitially = true;
    bool useViolationScoring = true;
};
```

# 8. V7：RLT cuts

RLT（Reformulation-Linearization Technique）对有 bounds 的线性约束和变量 bounds 相乘，可以得到更强 cuts。

例如已有线性约束：

```text
a*x + b*y <= c
```

变量 bound：

```text
z >= lz
```

因为：

```text
z - lz >= 0
c - a*x - b*y >= 0
```

两者乘积非负：

```text
(z-lz)(c-a*x-b*y) >= 0
```

展开后产生二次项，再用 aux 替换，得到线性 cut。

这能显著增强 polynomial relaxation。

## 8.1 V7 支持范围

第一版 RLT 只做：

```text
linear inequality × variable bound
```

并且只在所有产生的 bilinear terms 已经有 aux 时使用。

不新增复杂 monomial。

---

# 9. V8：integer strengthening

NIA 下可以利用 integrality 加强 linearization。

## 9.1 Product bound strengthening

若：

```text
x,y ∈ Int
x ∈ [lx,ux]
y ∈ [ly,uy]
t = x*y
```

McCormick 是 continuous relaxation，对 integer hull 不够强。

可以加入：

```text
t mod gcd-related constraints
parity cuts
domain-specific finite hull cuts
```

V8 初始做简单整数 strengthening：

```text
if x,y small finite domains:
  compute exact convex hull of points (x,y,t)
```

但这个昂贵。限制：

```text
domain size <= threshold
```

例如：

```text
|Dx| * |Dy| <= 100
```

生成 valid linear inequalities 或直接 finite table cut。

## 9.2 Parity / modular cuts

如果：

```text
t = x²
```

则：

```text
x² mod 4 ∈ {0,1}
```

这个不是线性 cut，但可以生成 modular lemma。
建议放到 NIA modular reasoner，不放 linearizer。

---

# 10. V9：piecewise linearization

当前 McCormick 的强度依赖 box 大小。box 越小越强。

所以后续要支持 piecewise refinement：

```text
x ∈ [l,u]
split at m
use different McCormick cuts on [l,m] and [m,u]
```

生成 split lemma：

```text
x <= m ∨ x >= m+1     // NIA
x <= m ∨ x > m        // NRA
```

然后在每个分支上生成更紧 cuts。

这和 ICP / B&B 配合非常强。

## 10.1 触发条件

```text
1. relaxation model violates original product badly；
2. McCormick gap large；
3. interval width large；
4. repeated spurious SAT candidate；
5. local search / exact validator 失败。
```

## 10.2 不要在 linearizer 内部递归搜索

Linearizer 只返回：

```text
SplitSuggestion
```

由 NIA/NRA adapter 或 B&B client 决定是否生成 split lemma。

---

# 11. V10：与 ICP 的深度协同

ICP 和 Linearizer 的协同机制：

```text
ICP:
  tighten variable bounds

Linearizer:
  use tighter bounds to generate stronger cuts

LIA/LRA:
  derive linear conflict or improve model

Conflict / failed validation:
  trigger more ICP / split / new cuts
```

需要建立 bound versioning：

```cpp
struct BoundSignature {
    Var var;
    Bound lower;
    Bound upper;
    uint64_t lowerVersion;
    uint64_t upperVersion;
};
```

当 ICP 更新 bounds：

```text
x: [0,100] -> [0,10]
```

Linearizer 应重新生成依赖 `x` 的 McCormick cuts。

---

# 12. V11：equality handling

非线性 equality：

```text
x*y = z
```

不能只做一边。

抽象：

```text
t = x*y
t = z
```

linearized equality 可以拆成：

```text
t <= z
t >= z
```

McCormick 对 `t` 提供 relaxation。

对于 original constraint：

```text
C: x*y = z
```

abstraction lemma：

```text
¬C ∨ (t = z)
```

cut lemmas：

```text
¬C ∨ bounds guards ∨ mccut_i
```

注意：

```text
relaxation SAT 不代表 equality 真满足
```

SAT 前 exact validate。

---

# 13. V12：disequality handling

非线性 disequality：

```text
x*y != z
```

Linearizer 不适合直接处理。V1-V3 建议：

```text
NoChange
```

或者仅在 equality relaxation 已经证明 impossible 时间接冲突。

真正处理 disequality 应交给：

```text
NIA/NRA solver:
  p != 0 -> p < 0 ∨ p > 0 split
```

linearizer 可以处理 split 后的 `<` 或 `>` 分支。

---

# 14. V13：cut proof / trace

每条 cut 应能解释来源：

```cpp
enum class LinearizationProofKind {
    Abstraction,
    McCormick,
    SquareNonneg,
    SquareTangent,
    SquareSecant,
    RecursiveProduct,
    RLT,
    IntegerHull,
};
```

记录：

```cpp
struct LinearizationProof {
    LinearizationProofKind kind;
    NonlinearTermKey term;
    std::vector<SatLit> guards;
    LinearCut cut;
    std::string debug;
};
```

ProofManager 未来把它导出为 extension lemma / theory lemma。

# 15. 效率设计

## 15.1 Watchers

为每个 variable 建立 watcher：

```text
var x -> terms involving x
var x -> constraints involving x
```

当 `x` bound 改变：

```text
只重新处理相关 terms
```

---

## 15.2 Cut cache

cache key 必须包含：

```text
term
nonlinear reason
bound values
cut kind
cut index
```

对于 model-based cuts，还要包含：

```text
tangent point a
```

例如 square tangent：

```cpp
struct SquareTangentKey {
    NonlinearTermKey term;
    SatLit nonlinearReason;
    mpq_class a;
};
```

---

## 15.3 Budget

```cpp
struct LinearizationBudget {
    int maxNewAuxTerms;
    int maxNewCuts;
    int maxProcessedConstraints;
    int maxProcessedTerms;
};
```

budget 用完：

```text
NoChange / BudgetExceeded
```

不能 UNSAT。

---

## 15.4 Redundancy filtering

不要发明显弱 cut：

```text
1. cut 已被已有 cut dominated；
2. cut 在 current model 下 slack 很大；
3. cut 与已有 cut 系数比例相同但 rhs 更弱；
4. bound values 没变化。
```

第一版只做 exact duplicate；后续做 dominance。

# 16. Solver pipeline 位置

推荐后续 NIA pipeline：

```text
1. Assert active atoms
2. LinearDomainReasoner
3. Square / SumSquares old reasoners
4. ICP
5. Mirror active linear bounds to LIA
6. IncrementalLinearizer
7. LIA relaxation check
8. Modular / GCD / factor
9. Bounded enumeration
10. Local search SAT finder
11. Branch / split
```

NRA pipeline：

```text
1. Normalize polynomial constraints
2. Real bound extraction
3. ICP
4. Mirror active linear bounds to LRA
5. IncrementalLinearizer
6. LRA relaxation check
7. CDCAC / NLSAT / exact NRA engine
8. Split if needed
```

# 17. 测试扩展计划

## 17.1 Monomial abstraction

```text
x*y*z <= 10
```

应创建：

```text
t1 = x*y
t2 = t1*z
t2 <= 10
```

## 17.2 Recursive McCormick

```text
0<=x,y,z<=2
x*y*z > 8
```

应证明 UNSAT，因为最大为 8。

## 17.3 Square strengthening

```text
x² + y² <= 1
x = 2
```

Mirror `x=2`，square nonnegativity，tangent at 2，证明 UNSAT。

## 17.4 Bounds update triggers new cuts

```text
x,y initially [0,100]
ICP tightens to [0,10]
```

Linearizer 应重新生成 tighter McCormick cuts。

## 17.5 Missing bound

```text
x bounded, y unbounded, x*y <= 5
```

允许 abstraction lemma，但不生成 McCormick envelope.

## 17.6 Different nonlinear reasons

```text
C1: x*y + z <= 5
C2: x*y - w >= 3
```

相同 term、相同 bounds，但不同 nonlinearReason，应生成不同 guarded lemmas。

# 18. 推荐扩展顺序

不要一口气做完。建议按这个顺序：

```text
V2. Monomial abstraction + recursive product aux
V3. Recursive McCormick
V4. Model-based separation
V5. Cut pool / scoring / budget
V6. Power cuts for x^k
V7. RLT cuts
V8. Integer strengthening for small finite domains
V9. Piecewise linearization / split suggestion
V10. Proof trace
```

其中最值得优先做的是：

```text
V2 + V3 + V5
```

因为它们直接扩大可处理范围，并控制 cut 爆炸。

---

# 19. 红线

```text
1. Linearizer 不返回 SAT。
2. Relaxation SAT 不代表 original SAT。
3. 所有 bounds-dependent cuts 必须 guarded by complete bound reasons。
4. aux vars 不进入 public model。
5. NIA aux 是 Int，NRA aux 是 Real。
6. generated cuts route to LIA/LRA。
7. original nonlinear atoms stay with NIA/NRA。
8. budget exhausted 不返回 UNSAT。
9. model-based cut 只能作为 separation，不可作为 validation。
10. final SAT 必须 validate original nonlinear constraints。
```

# 20. 最终总结

增强版 IncrementalLinearizer 的目标是：

```text
从“支持 x*y 和 x² 的简单 cut generator”
升级为
“面向 NIA/NRA 的增量式线性松弛引擎”。
```

核心路线：

```text
更广泛抽象：
  monomial / power / recursive product

更强 cut：
  recursive McCormick / tangent / secant / RLT / integer strengthening

更高效率：
  cache / watcher / cut pool / violation scoring / budget

更好集成：
  ICP bounds -> linearizer cuts -> LIA/LRA relaxation -> exact validation
```

最优先扩展：

```text
1. Monomial abstraction
2. Recursive McCormick
3. Cut pool + violation-based selection
4. Bound-version-triggered regeneration
```

这几项做完后，linearizer 才真正从 V1 的“小工具”变成一个有研究价值、工程价值都很强的 nonlinear relaxation subsystem。
