# 1. 4.12 的 `LiaSolver::check()` 伪代码还是旧版本

前面 4.8 已经说 `GomoryCut` 有三种 kind：

```cpp
enum class GomoryCutKind {
    LinearCut,
    GuardedConflict,
    Skip
};
```

并且写了应该如何分支处理。这个设计是对的。

但 4.12 的最终 `LiaSolver::check()` 伪代码里，仍然是旧逻辑：

```cpp
auto cut = gomory_.generate(snapshot, integerValuedVars_);

if (cut) {
    SatLit cutLit = registry_.getOrCreateLinearBoundAtom(
        cut->lhs, cut->rel, cut->rhs
    );

    TheoryLemma lemma =
        buildGuardedCutLemma(cut->premises, cutLit);

    assert(lemmaValidator_.validateGomoryCut(lemma));

    return TheoryCheckResult::mkLemma(lemma);
}
```

这会无条件把所有 cut 当作 `LinearCut` 注册 dynamic atom，和前面的 `GuardedConflict / Skip` 设计矛盾。必须把 4.12 改成真正的 kind 分支：

```cpp
if (options_.enableGomoryCuts) {
    auto snapshot = gs_.snapshot();
    auto cut = gomory_.generate(snapshot, integerValuedVars_);

    if (cut) {
        if (cut->kind == GomoryCutKind::GuardedConflict) {
            TheoryLemma lemma =
                buildGuardedConflictLemma(cut->premises);

            assert(lemmaValidator_.validateGomoryCut(lemma));
            return TheoryCheckResult::mkLemma(lemma);
        }

        if (cut->kind == GomoryCutKind::LinearCut) {
            SatLit cutLit = registry_.getOrCreateLinearBoundAtom(
                cut->lhs, cut->rel, cut->rhs
            );

            TheoryLemma lemma =
                buildGuardedCutLemma(cut->premises, cutLit);

            assert(lemmaValidator_.validateGomoryCut(lemma));
            return TheoryCheckResult::mkLemma(lemma);
        }

        if (cut->kind == GomoryCutKind::Skip) {
            // continue to branch split
        }
    }
}
```

否则前面新增 `GomoryCutKind` 的 correctness 设计没有真正落到最终 check 流程里。

# 2. UnitCubeTester 仍然缺少 normalized active constraints 入口

Phase 3 现在仍然只写：

```text
A x <= b
a_i · z <= b_i - 1/2 * ||a_i||_1
```

这只是数学核心，不足以指导正确实现。还必须明确 `UnitCubeTester` 接收的不是原始 atom，而是 **normalized effective active constraints**。

需要补一段：

```text
UnitCubeTester must operate on normalized effective active constraints:

1. If the current SAT assignment sets a theory atom to false,
   first negate the relation.

2. Convert all constraints to <= form:
   a·x <= c      keep
   a·x >= c      convert to -a·x <= -c
   a·x = c       add both a·x <= c and -a·x <= -c

3. For QF_LIA strict constraints:
   a·x < c       convert to a·x <= ceil(c) - 1
   a·x > c       convert to a·x >= floor(c) + 1,
                 then to -a·x <= -(floor(c)+1)

4. Disequality does not enter the cube system.
   It is handled earlier as a disequality obligation.

5. Clear denominators before integer-aware normalization.
```

这个不补，`>=`、`=`、`<`、`>`、false literal 很容易被 cube test 错建系统。

# 3. Phase 2 最好补一句统一入口

现在 GCD inequality 已经写了“经过归一化（取反、strict 转非严格、clear denominators）后的 effective active constraint”，但 GCD equality conflict 仍然直接写 `a1*x1 + ... + an*xn = c`。建议在 Phase 2 开头统一补一句，避免每个 rule 各写各的：

```text
All IntegerReasoner rules operate on normalized effective active constraints:
  - false literal has been negated;
  - strict LIA relation has been converted to non-strict integer relation;
  - rational coefficients and rhs have been cleared to integer coefficients;
  - gcd is always computed over integer coefficients.
```

这不是大问题，但能避免实现时 equality conflict 漏掉 clear denominators。
