# 1. `Square nonnegativity` 写在 config 里，但 Step 5 没有真正设计

你现在有：

```cpp
bool emitSquareNonneg = true;   // s = x^2 -> s >= 0
```

但 `SquareCutGenerator` 的责任仍写成“given finite bounds, generate secant upper and tangent lower”。这会让实现者以为 square cuts 都需要 finite bounds。

必须明确：

```text
s = x² -> s >= 0
```

这个 cut **不需要 finite bounds**，应该总是可以发。

线性形式：

```text
s >= 0
等价于
-s <= 0
```

建议改成：

```text
SquareCutGenerator emits:
1. Nonnegativity cut: s >= 0, no finite bounds required.
2. Secant upper: requires finite [l,u].
3. Tangent lower: requires finite [l,u] or model point a.
```

否则你的测试：

```text
x²+y²≤1, x=2 -> UNSAT
```

不稳。因为要推出 `s_y >= 0`，必须有 square nonnegativity cut。计划里这个测试本身依赖这一点。

# 2. `p_L rel 0` lemma 的 guard 不应带 bound reasons

对原约束：

```text
C: x*y + z <= 5
```

抽象后：

```text
L: aux_xy + z <= 5
```

lemma：

```text
¬C ∨ L
```

这个是 definitional / extension relaxation 的一部分，不依赖 `x,y` 的 bounds。
只有 McCormick / secant / tangent 这些 cut 依赖 bounds。

所以要区分两类 lemma：

```text
A. abstraction lemma:
   ¬C ∨ p_L rel 0
   只 guard nonlinear reason C

B. envelope/cut lemma:
   ¬C ∨ ¬bound_reasons... ∨ cut
   guard nonlinear reason + bound reasons
```

现在计划的 `buildGuardedLemma(nonlinearReason, boundReasons, cutLit)` 是统一接口，容易让实现者把 `p_L rel 0` 也加 bound guard，或者把 envelope cut 漏掉 bound guard。

建议拆成两个函数：

```cpp
TheoryLemma buildAbstractionLemma(SatLit nonlinearReason, SatLit linearizedLit);

TheoryLemma buildCutLemma(
    SatLit nonlinearReason,
    const std::vector<SatLit>& boundReasons,
    SatLit cutLit);
```

# 3. `BoundInfo` 仍缺少 “global / reasonsComplete” 字段

计划里虽然写了：

```text
Empty reasons do NOT automatically mean unconditional.
```

这是对的。

但 `BoundInfo` 结构还是：

```cpp
struct BoundInfo {
    mpq_class lower;
    mpq_class upper;
    std::vector<SatLit> lowerReasons;
    std::vector<SatLit> upperReasons;
    bool hasLower = false;
    bool hasUpper = false;
};
```

这不足以区分：

```text
1. 真正全局 bound；
2. reason 丢失；
3. adapter 没填 reason；
4. heuristic/model 产生的临时 bound。
```

建议改成：

```cpp
struct BoundInfo {
    mpq_class lower;
    mpq_class upper;
    std::vector<SatLit> lowerReasons;
    std::vector<SatLit> upperReasons;

    bool hasLower = false;
    bool hasUpper = false;

    bool lowerIsGlobal = false;
    bool upperIsGlobal = false;
    bool lowerReasonComplete = false;
    bool upperReasonComplete = false;

    bool hasFiniteCompleteBounds() const {
        return hasLower && hasUpper &&
               ((lowerIsGlobal || lowerReasonComplete) &&
                (upperIsGlobal || upperReasonComplete));
    }
};
```

V1 规则：

```text
如果 finite bound 没有 complete reason 且不是 global：
  skip cut
或使用 all-active coarse guard。
```

不要只靠 `reasons.empty()` 判断。

# 4. `maxCutsPerTerm=4` 和 square cuts 数量冲突

Product 的 McCormick 正好 4 条。
Square 如果发：

```text
1. nonnegativity
2. secant
3. tangent
```

是 3 条，没有问题。

但如果一个 polynomial 有多个 square，例如：

```text
x² + y² <= 1
```

每个 square 都会有若干 cut，加上 abstraction lemma，总 lemma 数可能超过 `maxLemmas=10`。这不是错误，但要定义行为：

```text
如果超出 maxLemmas：
  返回前 maxLemmas 条；
  剩余进入 pending queue？
  还是下次重新生成？
```

现在 adapter 有 pending queue，但 linearizer 内部只返回 `lemmas`，没有 “还有未发完” 的状态。

建议简单处理：

```text
V1: Linearizer run() 最多返回 maxLemmas；
未返回的下次 run() 可重新生成，但 cache 只能在 lemma 真正进入 pending queue 后 mark。
```

否则可能发生：

```text
cut 生成了但因 maxLemmas 没返回；
cache 已经 mark；
以后永远不发。
```

这是一个很隐蔽的 bug。

所以 cache 规则应改成：

```text
只有当 lemma 被加入 pendingLinLemmas_ 或成功返回给 propagator 后，才 markEmitted。
```

不要在生成阶段就 mark。

# 5. `NonlinearTermKey` 用 `PolyId children` 排序可能不稳定

Product key 写：

```cpp
std::vector<PolyId> children;  // sorted for Product
```

这需要保证：

```text
PolyId(x) 和 PolyId(y) 的排序稳定；
同一个变量在不同重建 polynomial 后 PolyId 不变；
x*y 和 y*x 经过 terms() 后能得到相同 key。
```

如果 `PolyId` 是内部对象 id，重建时可能不同。更稳的是 key 用变量名/monomial powers：

```cpp
struct NonlinearTermKey {
    NonlinearKind kind;
    std::vector<std::pair<std::string, int>> powers; // sorted by var name
};
```

例如：

```text
x*y -> {("x",1),("y",1)}
x²  -> {("x",2)}
```

这样比 `children PolyId` 更适合从 `terms()` 输出 canonicalize。

# 6. `x*y>200` 的 NIA strict 归一化要再检查符号

计划里写：

```cpp
if (rel == Relation::Gt) { rel = Relation::Geq; rhs += 1; }  // p > 0 -> p >= 1
if (rel == Relation::Lt) { rel = Relation::Leq; rhs -= 1; }  // p < 0 -> p <= -1
```

这个只在 atom 已经是：

```text
p rel 0
```

且 `rhs=0` 时显然正确。若 `getOrCreateLinearBoundAtom(linearLhs, rel, rhs)` 的语义是：

```text
linearLhs rel rhs
```

则一般形式应该是：

```text
linearLhs > rhs  -> linearLhs >= rhs + 1
linearLhs < rhs  -> linearLhs <= rhs - 1
```

所以逻辑是对的，但注释要从 `p > 0` 改成一般形式，避免误用。

另外，NIA cut 中 aux 变量必须是 Int sort，否则 LIA strict 规约不成立。要补：

```text
NIA owner 下所有 aux var 注册为 Int；
NRA owner 下 aux var 注册为 Real。
```

# 7. Co-register LIA/LRA 会带来 “原始 atoms 被谁接收” 的问题

计划说：

```text
LIA/LRA solvers silently ignore polynomial atoms; NIA/NRA solvers silently ignore linear atoms.
```

这个前提依赖 `TheoryAtomRecord.theory` dispatch。

需要加一个 regression：

```text
QF_NIA 中原始 linear atom x<=10 应该由 NIA 还是 LIA 接？
```

如果原始 linear atoms 仍标成 NIA，那么 LIA 只接 linearizer cuts；NIA 自己处理原始 linear domain，这可以。
如果原始 linear atoms 被 Atomizer 标成 LIA，那么 NIA 的 DomainStore 可能拿不到 bounds，linearizer 也拿不到 bounds。

所以必须明确：

```text
QF_NIA:
  原始 arithmetic atoms 仍 route to NIA；
  linearizer-generated cuts route to LIA。

QF_NRA:
  原始 arithmetic atoms route to NRA；
  linearizer-generated cuts route to LRA。
```

否则 DomainStore / bounds 会断。
