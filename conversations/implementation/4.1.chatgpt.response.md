有问题，而且是**正确性级别的问题**。这版不能 Approve。建议选 **4 Revise**，并且要求它先把 **branch lemma 的动态 atom 注册机制**做对，再实现 LIA branch。

最核心的问题是：计划里已经意识到“fresh literal 没有原始 ExprId / theory atom”是核心挑战，但仍然打算用 `nextBranchExprId_` 和 side table hack 绕过去。这个会破坏 SAT literal、TheoryAtom、CoreExpr/ExprId 之间的一致性，存在 soundness 风险。文档里还写了 `SatLit left = SatLit{freshSatVar_, true}; SatLit right = SatLit{freshSatVar_, true};` 这种“placeholder literal”方案，并承认新 literal 必须注册为 atom 才能被 theory solver 解释，这正是不能 Approve 的地方。

# 1. 最大正确性问题：不能用“假 ExprId + side table”注册 branch atom

计划里说：

```text id="e4iesd"
allocate a new ExprId (not in CoreIr, just a unique ID)
map it to branch bound
return SatLits referencing these IDs
```

这不应该做。原因很简单：

```text id="jtfgo1"
SAT solver 只认识 SatVar / SatLit；
TheoryManager 认识 TheoryAtom；
Atomizer/CoreIr 认识 ExprId；
SimplexSolver 认识 auxVar + bound。
```

你不能凭空造一个不在 CoreIr、不在 Atomizer、不在 TheoryManager 的 ExprId，然后希望 SAT solver 下一轮能“自然调用 assertLit”。如果没有统一注册路径，会出现：

```text id="7be3zv"
1. SAT assign 了 fresh literal，但 TheoryManager 不知道它代表什么；
2. TheoryManager 知道 ExprId，但 Atomizer 不知道 SatVar -> ExprId 映射；
3. SimplexSolver 有 branchAtoms_ side table，但外层根本不会把这个 literal 传给它；
4. 更糟糕：freshSatVar_ 如果没有通过 SAT backend newVar() 创建，会直接破坏 SAT solver 内部状态。
```

这不是工程 hack 的问题，是**语义对象没有闭环**。

正确做法必须是引入一个统一的 atom registry：

```cpp id="r8p0r9"
struct TheoryAtomRecord {
    SatLit lit;
    ExprId expr;          // 可选，如果是 CoreIr 原生 atom
    TheoryId theory;
    Relation rel;
    LinearFormKey form;
    mpq_class rhs;
    bool isDynamic;
};
```

然后所有动态 branch atom 都必须通过同一个接口创建：

```cpp id="8xq37t"
SatLit TheoryAtomRegistry::getOrCreateLinearBoundAtom(
    LinearFormKey form,
    Relation rel,
    mpq_class rhs
);
```

它必须同时完成：

```text id="jxp41a"
1. 创建或复用 LinearFormKey 对应的 GeneralSimplex aux；
2. 创建或复用 TheoryAtomRecord；
3. 调 sat->newVar() 创建 SAT var；
4. 建立 SatVar -> TheoryAtomRecord 映射；
5. 后续 TheoryManager::assertLit 能把该 literal 传回 SimplexSolver。
```

没有这个，就不能做 branch lemma。

# 2. `freshSatVar_` 示例本身是错误的

计划里的代码：

```cpp id="3sn381"
SatLit left = SatLit{freshSatVar_, true};   // x <= lo
SatLit right = SatLit{freshSatVar_, true};  // x >= hi
return TheoryLemma{{left, right}};
```

这里 left 和 right 用了同一个 `freshSatVar_`。即使只是伪代码，也必须指出：**这会把 `(x <= lo) ∨ (x >= hi)` 变成同一个 literal 的重复 clause**，完全错误。

正确必须是：

```cpp id="bmrcg1"
SatLit left  = registry.getOrCreateLinearBoundAtom(form, LEQ, lo);
SatLit right = registry.getOrCreateLinearBoundAtom(form, GEQ, hi);
return TheoryLemma{{left, right}};
```

# 3. Sort classification 如果失败，不能继续当 LRA 处理

计划里说 CoreExpr 有 `SortId`，但现在没有明确 int/real/bool 分类，所以 Phase 1 添加 `SortClass`，并在 import 时通过 SOMTParser sort name 填充。

这个方向可以，但必须加一条硬规则：

```text id="vtf9cj"
在 liaMode_ 下，任何 arithmetic variable 的 sort 如果不是明确 Int，就不能当 Real 继续求。
```

否则会出现严重 unsound：

```smt2 id="lhggwi"
(set-logic QF_LIA)
(declare-const x Int)
(assert (= (* 2 x) 1))
(check-sat)
```

如果 `x` 没被识别成 Int，solver 会按 LRA 得到 `x = 1/2`，然后返回 sat。这是错误的。

所以在 QF_LIA 下：

```text id="4dsj8u"
sort = Int      -> add to integerVars_
sort = Unknown  -> return Unknown / internal error
sort = Real     -> logic mismatch
```

不能 silently fallback to LRA。

# 4. 不应该在 QF_LIA 下继续注册 CdcacSolver

计划里的 Solver 注册逻辑：

```cpp id="x19jmi"
if (logic == "QF_LIA" || logic == "LIA") {
    auto simplex = std::make_unique<SimplexSolver>();
    simplex->setLiaMode(true);
    theoryManager_.registerSolver(std::move(simplex));
} else {
    theoryManager_.registerSolver(std::make_unique<SimplexSolver>());
}
theoryManager_.registerSolver(std::make_unique<CdcacSolver>(...));
```

这有风险。QF_LIA 下不应该无条件注册 NRA/CAC solver。尤其你前面说 TheoryManager 不按 theory ID filter，而是“SAT atomizer feeds all theory atoms to all solvers; each solver ignores atoms it cannot handle”。

这会导致：

```text id="4vmr10"
1. CdcacSolver 可能把 Int 变量当 Real 变量处理；
2. LIA 和 NRA 两个 solver 对同一 atom 都给结论；
3. 一个 solver 返回 sat，另一个返回 unknown/conflict，组合语义混乱；
4. 对 LIA branch lemma 产生的新 atom，CdcacSolver 未必能解释。
```

Phase 1 应该非常明确：

```text id="h7v20x"
QF_LIA:
  register only SimplexSolver(liaMode=true)

QF_LRA:
  register only SimplexSolver(liaMode=false)

QF_NRA:
  register CdcacSolver / NRA solver

Mixed / unknown:
  return Unknown or use future TheoryDispatcher
```

不要让多个 arithmetic solver 同时吃同一批 atom。

# 5. integral check 和 disequality check 的顺序建议调整

计划中顺序是：

```text id="x7k07f"
LRA check
→ if liaMode, check integral
→ if fractional, branch split
→ Check disequalities
```

这在 soundness 上一般没错，因为 branch split 是 theory tautology。但不够好，可能造成多余 branch。建议改成：

```text id="8znqen"
1. gs_.check()
2. if LRA unsat -> conflict
3. check disequality obligations
   - LRA violated disequality -> split lemma
   - LIA violated disequality -> stronger integer split
4. if liaMode, check integrality
5. if fractional -> integer branch split
6. model validation
```

原因是 disequality split 直接由当前 active literal 触发，通常比整数 branch 更局部。比如：

```smt2 id="r1pbqg"
(assert (not (= x 0)))
(assert (= x 0))
```

应该优先返回 disequality split / conflict，而不是先对某个无关 fractional 变量 branch。

---

# 6. LIA disequality 不能“复用 LRA design”作为最终 Phase 1

计划目标里写：

```text id="r4e88a"
disequality split (reuses LRA design)
```

Phase 1 当然可以先复用 LRA 的 `p < 0 ∨ p > 0`，但在 LIA 中更专业、更强的是：

```text id="rcy7nd"
p != 0  =>  p <= -g ∨ p >= g
```

其中：

```text id="ck3pk7"
g = gcd(coefficients of p)
```

如果 `g ∤ -constant`，则 `p = 0` 不可能，`p != 0` 恒真。这一点最好现在就加，因为它不复杂，而且避免大量无意义 δ split。

---

# 7. “Branch split may cause infinite loops”的 mitigation 不正确

计划说：

```text id="p5ykvg"
Bland's rule on simplex + SAT solver's learned clauses prevent cycling in practice
```

这句话不对。Bland's rule 只保证 simplex pivot 不循环，不能保证 integer branching 不无限推进。

Phase 1 的正确说法应是：

```text id="3kg49g"
branch lemma x <= floor(q) ∨ x >= ceil(q) 是 sound 的；
但没有 cuts/gcd/complete fallback 时，LIA solver 可能不完整或性能很差；
若有资源限制，允许返回 Unknown；
不要声称 learned clauses + Bland 保证 LIA 完整性。
```

尤其无界整数变量和某些 diophantine 结构会让纯 branch split 很差。Phase 1 可以是 sound but incomplete/practical，不要夸成完整 LIA。

# 结论

这版**不能 Approve**。主要问题不是 LIA 思路，而是动态 branch atom 方案没有语义闭环：计划里自己也写到了“fresh literals 没有原始 ExprId 是核心挑战”，但提出的 side-table hack 仍然不够严格。

我的选择：**[4] Revise**。
必须先把 **SAT literal ↔ TheoryAtom ↔ Linear bound ↔ Simplex assert** 这条链打通，再做 LIA branch。
