# 1. Bug 3 的描述不一致：`x^3 - 1331 = 0` 不是“零常数项”

计划里写：

```text
UnivariateIntegerReasoner — 零常数项高次多项式伪完整
示例: x^3 - 1331 = 0
```

但 `x^3 - 1331` 的常数项是 `-1331`，不是 0。这个 bug 本质上更像是：

```text
RRT divisor enumeration / fallback root search 不完整，却错误标记 Complete。
```

如果当前实现真的是 “a0 == 0 时只测试 [-10,10]”，那应该用这种例子：

```text
x * (x - 11) = 0
```

或者：

```text
x^2 - 11x = 0
```

它们常数项为 0，整数根包括 11。正确 roots 是 `{0, 11}`。如果只测 `[-10,10]`，会漏 11。

建议改：

```text
Bug 3 改名：
UnivariateIntegerReasoner — incomplete root enumeration incorrectly marked Complete

测试分两类：
1. 常数项非零 RRT：
   x^3 - 1331 = 0 -> root {11}
   应通过 Rational Root Theorem 枚举 divisors of 1331 找到 11。

2. 常数项为 0：
   x*(x-11)=0 或 x^2-11x=0 -> roots {0,11}
   如果只测试 [-10,10] 会漏 11，不能标记 Complete。
```

# 2. 类别 D 的 GCD conflict 例子 `4x = 2 → UNSAT` 是错的

计划写：

```text
GCD conflict 简单: 4x = 2 → UNSAT（GCD=4 不整除 2）
```

这当然 UNSAT，但 reason 是 `4 ∤ 2`。不过如果内部 normalized polynomial 是：

```text
4x - 2 = 0
```

那 `q=4x, c=-2`，检查的是：

```text
g = 4
(-c) = 2
4 ∤ 2
```

没问题。

但这个测试跟 LinearNiaDomainReasoner 的单变量线性等式 `a*x+c=0` 会重叠。更适合 Algebraic GCD 的例子应该是非线性：

```text
2x^2 + 4yz = 3
```

或者单变量非线性：

```text
4x^2 = 2
```

建议保留 `4x=2` 作为 LinearNiaDomainReasoner / normalized linear conflict 测试；Algebraic GCD 用：

```text
4x^2 - 2 = 0 -> UNSAT
2x^2 + 4yz - 3 = 0 -> UNSAT
```

这样更能覆盖 NIA algebraic reasoner。

# 3. 类别 D 第 5 条“多变量 GCD conflict 跳过”不应该写成已知限制

计划里写：

```text
GCD conflict 多变量: 6x + 9y = 5 → 当前应为 NoChange（多变量跳过）
```

这不合理。`6x + 9y = 5` 的 gcd conflict 是最基础的整数线性/多项式系数 GCD 检查：

```text
gcd(6,9)=3
3 ∤ 5
```

应该直接 UNSAT。你前面的 NIA-Core plan 已经有 “Coefficient GCD equality conflict”，而且例子就是多项式多变量系数 gcd。这个测试不应标注为已知限制。

建议改成：

```text
GCD conflict 多变量:
  6x + 9y = 5 -> UNSAT

GCD no conflict 多变量:
  6x + 9y = 15 -> NoChange / continue
```

如果当前实现确实跳过多变量，那这就是要发现的 bug，不应该把它设成 expected NoChange。

# 4. 严格不等式冲突例子写错：`x < 3` 与 `x > 3` 确实直接冲突

计划写：

```text
x < 3 → x <= 2，x > 3 → x >= 4，2 < 4 无直接 bound 冲突，需要 SAT 搜索
```

这句话不对。`x <= 2` 和 `x >= 4` 对同一个整数变量是直接 domain conflict，DomainStore 或 LinearNiaDomainReasoner 应该直接发现 empty domain。

所以类别 F 第 3 条应改成：

```text
Lt + Gt 冲突:
  x < 3, x > 3
  normalize:
    x <= 2
    x >= 4
  DomainStore 直接 empty，UNSAT。
```

这正好测试 strict normalization + domain conflict，不需要 SAT 搜索。

# 5. Bool + NIA UNSAT 的例子不对

计划写：

```text
Bool + NIA UNSAT: (and p (= (* x x) 2)) + (assert p) → unsat
```

如果公式真是：

```smt2
(assert (and p (= (* x x) 2)))
(assert p)
```

那么 UNSAT 的原因只是 `x^2=2`，`p` 没什么保护作用。这个测试不能很好覆盖 CDCL(T) 分支交互。

建议换成：

```smt2
(set-logic QF_NIA)
(declare-const p Bool)
(declare-const x Int)
(assert (or (not p) (= (* x x) 2)))
(assert p)
(check-sat)
```

期望 `unsat`。这个测试表示：布尔分支强制启用一个 NIA 不可满足约束。

SAT 分支覆盖测试可以保留：

```smt2
(assert (or (= (* x x) 4) (= (* x x) 2)))
```

期望 `sat`，因为存在 `x^2=4` 分支。

# 7. 测试计划还缺两个关键回归

建议增加两个测试。

## A. false literal effective relation 回归

之前 NIA normalizer 的核心 bug 风险是 false literal 取反 strictness。应该加：

```smt2
(set-logic QF_NIA)
(declare-const x Int)
(assert (not (<= x 0)))
(assert (<= x 0))
(check-sat)
```

期望 `unsat`。

原因：

```text
not (x <= 0) -> x > 0 -> x >= 1
同时 x <= 0
冲突。
```

如果错误把 `not (x<=0)` 变成 `x>=0`，就会错误 SAT with x=0。

## B. dynamic polynomial atom route 回归

Factor lemma 需要 dynamic atom route。加：

```smt2
(set-logic QF_NIA)
(declare-const x Int)
(declare-const y Int)
(assert (= (* x y) 0))
(assert (not (= x 0)))
(assert (not (= y 0)))
(check-sat)
```

计划里已有 factor unsat CLI，但单元测试里最好明确检查 dynamic atom 被注册和回传。否则 `p*q=0` lemma 可能生成了但后续不生效。
