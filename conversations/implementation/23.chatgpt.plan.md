下面给你一版 **incremental e-graph / incremental congruence closure** 的完整实现计划。这里的 e-graph 指的是 SMT/EUF 里的 **可回溯增量同余闭包结构**，不是 equality saturation 里的 `egg` 那种 rewrite e-graph。

目标是把之前 V1 的：

```text
每次 check() full rebuild EUF graph
```

升级成：

```text
assertLit 增量加入 equality / disequality
merge 后即时维护 congruence closure
支持 decision-level backtrack
支持 explanation / conflict / propagation
支持 CDCL(T) theory solver 集成
```

---

# 0. 总体目标

最终 EUF solver 应该支持：

```text
1. 增量 assert equality；
2. 增量 assert disequality；
3. 每次 equality merge 后维护 congruence closure；
4. disequality 被 merge 后立即产生 conflict；
5. 支持 rollback 到任意 decision level；
6. 支持 explanation：为什么 a=b；
7. 支持 theory propagation：如果当前 e-graph 推出 s=t，则返回 lemma；
8. 支持 model construction；
9. 后续能接 EqualityManager / MCSAT / ProofManager。
```

核心 invariant：

```text
任意两个 function applications：
  f(a1,...,an)
  f(b1,...,bn)

如果所有 ai 与 bi 当前等价，
那么这两个 application term 必须在同一个 equivalence class 中。
```

也就是维护 congruence closure。

---

# 1. V1 full rebuild vs incremental e-graph

之前 full rebuild 的方式是：

```text
check():
  清空 egraph
  重新加入 active equalities
  反复扫描所有 applications
  直到同余闭包稳定
  检查 disequality
```

优点：

```text
简单、正确性好。
```

缺点：

```text
每次 check 都重复工作；
CDCL 中频繁 backtrack / propagate 时很慢；
难以做即时 propagation。
```

incremental e-graph 改成：

```text
assert equality:
  merge class
  只检查受影响的 parent applications
  递归触发 congruence merge

backtrack:
  回滚 union-find、signature table、pending structures
```

# 3. Term Manager：不可回滚，term 永久 intern

Term interning 建议是 **monotonic** 的：创建过的 term 不删除。Backtrack 只回滚 equality，不回滚 term DAG。

这样实现简单很多。

## 3.1 TermId / SymbolId

```cpp
using EufTermId = uint32_t;
using EufClassId = uint32_t;
using EufSymbolId = uint32_t;
```

---

## 3.2 ENode

```cpp
struct ENode {
    EufTermId id;
    EufSymbolId symbol;
    std::vector<EufTermId> args;
    SortId sort;
    ExprId origin;
};
```

常量是 0 元函数：

```text
a        -> symbol=a, args=[]
f(a,b)   -> symbol=f, args=[a,b]
```

---

## 3.3 Term interning key

```cpp
struct TermKey {
    EufSymbolId symbol;
    std::vector<EufTermId> args;

    bool operator==(const TermKey&) const;
};
```

`EufTermManager`：

```cpp
class EufTermManager {
public:
    EufTermId intern(EufSymbolId sym,
                     std::vector<EufTermId> args,
                     SortId sort,
                     ExprId origin);

    const ENode& node(EufTermId t) const;

    const std::vector<EufTermId>& parentsOf(EufTermId t) const;

    size_t numTerms() const;

private:
    std::vector<ENode> nodes_;
    std::unordered_map<TermKey, EufTermId, TermKeyHash> intern_;
    std::vector<std::vector<EufTermId>> parents_;
};
```

`parentsOf(t)` 是 immutable use-list：

```text
t -> 所有以 t 为参数的 function applications
```

例如：

```text
parentsOf(a) = { f(a), g(a,b), h(c,a) }
```

这个不随 equality merge 改变，所以不用 rollback。

# 4. Rollback Union-Find

## 4.1 不使用路径压缩

为了 rollback 简单，建议第一版 rollback UF 不使用 path compression，只用 union-by-size/rank。

```cpp
class RollbackUnionFind {
public:
    EufClassId addNode();

    EufClassId find(EufClassId x) const;

    bool same(EufClassId a, EufClassId b) const;

    bool unite(EufClassId a, EufClassId b, MergeId reason);

    size_t snapshot() const;
    void rollback(size_t snap);

private:
    std::vector<EufClassId> parent_;
    std::vector<uint32_t> size_;

    struct UfChange {
        EufClassId childRoot;
        EufClassId parentRoot;
        uint32_t oldParentSize;
        bool merged;
    };

    std::vector<UfChange> trail_;
};
```

`find`：

```cpp
EufClassId find(EufClassId x) const {
    while (parent_[x] != x) x = parent_[x];
    return x;
}
```

复杂度足够好。

---

## 4.2 Snapshot / rollback

```cpp
size_t RollbackUnionFind::snapshot() const {
    return trail_.size();
}

void RollbackUnionFind::rollback(size_t snap) {
    while (trail_.size() > snap) {
        auto ch = trail_.back();
        trail_.pop_back();

        if (!ch.merged) continue;

        parent_[ch.childRoot] = ch.childRoot;
        size_[ch.parentRoot] = ch.oldParentSize;
    }
}
```

# 6. Signature table

Congruence closure 需要维护：

```text
signature = symbol + representative classes of arguments
```

如果两个 application 有相同 signature，则它们必须 merge。

## 6.1 Signature

```cpp
struct AppSignature {
    EufSymbolId symbol;
    std::vector<EufClassId> argReps;

    bool operator==(const AppSignature&) const;
};
```

对 term `f(a,b)`：

```cpp
AppSignature signature(EufTermId t) {
    const auto& n = tm_.node(t);
    AppSignature s;
    s.symbol = n.symbol;
    for (auto arg : n.args) {
        s.argReps.push_back(rep(arg));
    }
    return s;
}
```

常量没有参数，一般不需要进 signature table，因为常量同名已经 intern 成同一 term。也可以统一处理。

---

## 6.2 Signature table

```cpp
class SignatureTable {
public:
    std::optional<EufTermId> find(const AppSignature& sig) const;

    void insert(const AppSignature& sig, EufTermId term);

    void erase(const AppSignature& sig);

    size_t snapshot() const;
    void rollback(size_t snap);

private:
    std::unordered_map<AppSignature, EufTermId, AppSignatureHash> table_;

    struct SigChange {
        enum Kind { InsertedNew, Overwritten, Erased };
        Kind kind;
        AppSignature sig;
        std::optional<EufTermId> oldValue;
    };

    std::vector<SigChange> trail_;
};
```

但是 signature key 会随着 class merge 变化。旧 signature table 里的 key 会 stale。所以更稳的设计是：

```text
V1 incremental:
  不全局保证 sigTable 完全无 stale；
  每次处理 affected applications 时重新计算 signature；
  插入新 signature；
  如果发现旧 entry stale，则忽略并更新。
```

# 7. Stale signature 处理

这是 incremental e-graph 最容易写错的地方。

假设之前有：

```text
f(a) key = f(class(a))
f(b) key = f(class(b))
```

后来 merge(a,b)，原来的 key 都 stale。

可以做两种方案。

---

## 方案 A：严格删除旧 signature

每个 application term 记录它当前注册到 signature table 的 signature。

```cpp
std::vector<std::optional<AppSignature>> currentSig_;
```

当 term 被重新处理：

```cpp
void refreshSignature(EufTermId t) {
    if (currentSig_[t]) {
        sigTable_.erase(*currentSig_[t]);
        currentSig_[t] = std::nullopt;
    }

    auto sig = signature(t);
    auto old = sigTable_.find(sig);

    if (!old) {
        sigTable_.insert(sig, t);
        currentSig_[t] = sig;
    } else if (!same(*old, t)) {
        enqueueMerge(*old, t, MergeReason::congruence(*old, t));
    }
}
```

问题：如果两个 terms 旧 signature 相同，erase 时可能删掉别人的 entry，需要 table entry 支持 owner 检查。

更稳：

```cpp
void eraseIfOwner(sig, t);
```

---

## 方案 B：允许 stale entry，find 时验证

`sigTable` 中可能保留 stale `(oldSig -> term)`。

当查到 `old` 时，重新计算：

```cpp
if (signature(old) != sig) {
    // stale, ignore/overwrite
}
```

伪代码：

```cpp
void processApp(EufTermId t) {
    AppSignature sig = signature(t);

    auto oldOpt = sigTable_.find(sig);

    if (!oldOpt) {
        sigTable_.insertOrAssign(sig, t);
        return;
    }

    EufTermId old = *oldOpt;

    if (signature(old) != sig) {
        // stale entry
        sigTable_.insertOrAssign(sig, t);
        return;
    }

    if (!same(old, t)) {
        enqueueMerge(old, t, MergeReason::congruence(old, t));
    }
}
```

这个方案实现简单，但 stale entry 可能积累。可以定期 rebuild signature table。

我建议 V1 incremental 用 **方案 B + periodic rebuild**，因为最容易正确。

# 8. Parent use-list 与 affected applications

当两个 classes merge：

```text
merge(A,B)
```

只有依赖 A 或 B 的 parent applications 可能产生新的 congruence。

term manager 有 immutable parents：

```text
parentsOf(term)
```

但 class 由多个 terms 组成。需要知道 class members。

---

## 8.1 Class members with rollback

为了获取 class 的所有 parent applications，需要维护 members。

简单做法：每次 merge 时把小类 members 移到大类。

```cpp
std::vector<std::vector<EufTermId>> members_;
```

Rollback 时要恢复。

Trail：

```cpp
struct MemberChange {
    EufClassId dstRoot;
    EufClassId srcRoot;
    size_t oldDstSize;
};
```

merge：

```cpp
void mergeMembers(dst, src) {
    trail.push({dst, src, members_[dst].size()});

    members_[dst].insert(members_[dst].end(),
                         members_[src].begin(),
                         members_[src].end());
}
```

rollback：

```cpp
members_[src].assign(members_[dst].begin() + oldDstSize,
                     members_[dst].end());
members_[dst].resize(oldDstSize);
```

这个可行，但注意 union-find root rollback 顺序。

---

## 8.2 Affected parents

```cpp
std::vector<EufTermId> affectedParents(EufClassId root) {
    std::vector<EufTermId> out;
    for (auto t : members_[root]) {
        for (auto p : tm_.parentsOf(t)) {
            out.push_back(p);
        }
    }
    return out;
}
```

当 merge `ra, rb` 后：

```text
重新处理 merged class 中所有 members 的 parents
```

更精确是处理两个旧类的 parents，但 merge 后旧 root 可能消失。实现时在 unite 前先取：

```cpp
auto parentsA = collectParents(ra);
auto parentsB = collectParents(rb);
```

然后 enqueue。

# 9. Merge queue

不要递归 merge，否则 explanation / trail / queue 容易乱。

使用队列：

```cpp
struct PendingMerge {
    EufTermId a;
    EufTermId b;
    MergeReason reason;
};
```

主循环：

```cpp
void processMergeQueue() {
    while (!mergeQueue_.empty()) {
        auto m = mergeQueue_.front();
        mergeQueue_.pop_front();

        if (same(m.a, m.b)) continue;

        performMerge(m.a, m.b, m.reason);
    }
}
```

---

# 10. performMerge 详细流程

```cpp
void IncrementalEGraph::performMerge(
    EufTermId a,
    EufTermId b,
    MergeReason reason
) {
    auto ra = rep(a);
    auto rb = rep(b);
    if (ra == rb) return;

    auto parentsA = collectParents(ra);
    auto parentsB = collectParents(rb);

    MergeId mid = recordMerge(a, b, reason, ra, rb);

    auto newRoot = uf_.unite(ra, rb, mid);

    mergeMembers(newRoot, oldRoot);

    // affected parents may now have changed signatures
    for (auto p : parentsA) enqueueCongruenceCheck(p);
    for (auto p : parentsB) enqueueCongruenceCheck(p);

    processCongruenceQueue();
}
```

注意：

```text
recordMerge 必须发生在 uf_.unite 之前，
因为 explanation 需要知道 merge 前的两个 class。
```

# 11. Congruence queue

```cpp
void processCongruenceQueue() {
    while (!congruenceQueue_.empty()) {
        EufTermId app = congruenceQueue_.front();
        congruenceQueue_.pop_front();

        if (!isApplication(app)) continue;

        AppSignature sig = computeSignature(app);

        auto other = sigTable_.lookupFresh(sig, *this);

        if (!other) {
            sigTable_.insertOrAssign(sig, app);
            continue;
        }

        if (!same(*other, app)) {
            MergeReason r = MergeReason::congruence(*other, app, sig);
            mergeQueue_.push_back({*other, app, r});
            processMergeQueue();
        }
    }
}
```

`lookupFresh` 要处理 stale：

```cpp
std::optional<EufTermId> lookupFresh(sig) {
    auto it = table.find(sig);
    if (it == table.end()) return nullopt;

    EufTermId t = it->second;
    if (computeSignature(t) == sig) return t;

    return nullopt; // stale
}
```

也可以发现 stale 后 overwrite。

# 12. MergeReason

```cpp
enum class MergeReasonKind {
    AssertedEquality,
    Congruence
};

struct MergeReason {
    MergeReasonKind kind;

    // asserted equality
    SatLit literal;

    // congruence
    EufTermId app1;
    EufTermId app2;

    // optional cached arg pairs
    std::vector<std::pair<EufTermId, EufTermId>> argPairs;
};
```

对于 congruence merge：

```text
app1 = f(a1,...,an)
app2 = f(b1,...,bn)
```

存：

```text
argPairs = [(a1,b1),...,(an,bn)]
```

之后解释时递归解释每个 `ai=bi`。

# 13. MergeRecord / explanation graph

每次真正 merge 记录：

```cpp
using MergeId = uint32_t;

struct MergeRecord {
    MergeId id;
```

# 14. Explanation：解释 a=b

rollbackable e-graph 中 explanation 最简单可靠的做法：

```text
保留 active merge records；
解释时在 mergeRecords_ 构造临时无向图；
BFS 找 lhs 到 rhs 的路径；
展开每条 merge reason。
```

虽然不是最高效，但 correctness 好。

## 14.1 Active merge records

Backtrack 时需要回滚 `mergeRecords_` size。

snapshot 记录：

```cpp
struct EGraphSnapshot {
    size_t ufSnap;
    size_t sigSnap;
    size_t mergeRecordSize;
    size_t trailSize;
    size_t mergeQueueSize;
    size_t congruenceQueueSize;
};
```

rollback：

```cpp
mergeRecords_.resize(snapshot.mergeRecordSize);
```

---

## 14.2 explainEquality

```cpp
std::vector<SatLit> explainEquality(EufTermId a, EufTermId b) {
    if (a == b) return {};

    if (!same(a,b)) {
        return {}; // internal error
    }

    auto path = findMergePath(a, b, mergeRecords_);

    std::vector<SatLit> out;
    for (auto mid : path) {
        auto rs = explainMergeReason(mergeRecords_[mid].reason);
        appendUnique(out, rs);
    }
    return out;
}
```

---

## 14.3 findMergePath

Build graph:

```text
node = EufTermId 或 class representative?
edge = mergeRecord(lhs, rhs)
```

建议用 **term-level graph**：

```cpp
adj[record.lhs].push(record.rhs, record.id);
adj[record.rhs].push(record.lhs, record.id);
```

但 congruence merge 可能 merge app terms，asserted merge merge arbitrary terms。这个足够。

如果路径找不到，但 `same(a,b)` 为 true，说明 record graph 不完整，是 bug。

---

## 14.4 explainMergeReason

```cpp
std::vector<SatLit> explainMergeReason(const MergeReason& r) {
    if (r.kind == AssertedEquality) {
        return {r.literal};
    }

    if (r.kind == Congruence) {
        std::vector<SatLit> out;
        for (auto [ai, bi] : r.argPairs) {
            appendUnique(out, explainEquality(ai, bi));
        }
        return out;
    }
}
```

必须防止递归环。加 visited pair：

```cpp
std::unordered_set<Pair<EufTermId,EufTermId>> explaining_;
```

```cpp
struct ActiveDisequality {
```

```cpp
std::optional<TheoryLemma> findPropagation() {
```

# 17. Backtracking

## 17.1 Solver-level trail

`EufSolver::assertLit` 存：

```cpp

    SatLit lit;
```

Backtrack:

```cpp
void EufSolver::backtrackToLevel(int level) {
    while (!activeAssignments_.empty() &&
           activeAssignments_.back().level > level) {
        activeAssignments_.pop_back();

    egraph_.rollback(levelSnapshots_[level]);

```

但如果 assertLit 时就增量 merge，那么需要在每个 decision level 创建 snapshot。


## 17.3 assertLit 增量处理

```cpp
TheoryCheckResult EufSolver::assertLit(SatLit lit,
                                       const TheoryAtom& atom,
                                       bool value,
                                       int level) {
    ensureSnapshotForLevel(level);

    auto eufAtom = extractor_.extract(atom);
    auto effective = applyValue(eufAtom, value);

    activeAssignments_.push_back(...);

    if (effective.rel == Eq) {
        auto r = egraph_.merge(effective.lhs,
                               effective.rhs,
                               MergeReason::asserted(lit));
        if (r.conflict) return ...
    } else {
        disequalities_.push_back(...);

        if (egraph_.same(effective.lhs, effective.rhs)) {
            return makeDiseqConflict(...);
        }
    }

    if (auto c = checkDisequalitiesTouched(...)) return conflict;

    return NoChange;
}
```

V1 可以 `assertLit` 增量 merge，但 disequality conflict 在 `check()` 里统一检查。

# 18. Snapshot 内容

`EGraphSnapshot`：

```cpp
struct EGraphSnapshot {
```

如果 signature table 采用 stale entry，不严格 rollback table，也可以在 rollback 后 rebuild sigTable。

推荐简单方案：

```text
Rollback equality state；

下一次 processCongruence 前 rebuild signature table from current classes。
```

这会简单很多。

# 19. Signature table rollback 推荐方案

严格 rollback signature table 很复杂。建议使用混合方案：

```text
1. Union-find / members / mergeRecords 严格 rollback；
2. Backtrack 后 sigTableDirty_=true；
3. 下次 assert equality 或 check 前，如果 dirty，则 rebuild signature table。
```

Rebuild:

```cpp
void rebuildSignatureTable() {
    sigTable_.clear();

    for (EufTermId t = 0; t < tm_.numTerms(); ++t) {
        if (!isApplication(t)) continue;
        processAppForSignature(t);
    }

    sigTableDirty_ = false;
}
```

这仍然是 incremental，因为正常 forward search 不重建；只有 backtrack 后重建。

更稳，更容易实现。

# 20. Recommended incremental strategy

我建议分三档实现：

## Inc-EGraph V1：增量 merge + backtrack 后 rebuild signature

```text
assert equality:
  incremental merge
  affected parents queue

backtrack:
  rollback UF/members/mergeRecords
  mark sigTable dirty

next assert/check:
  rebuild sigTable
```

这是最稳版本。

---

## Inc-EGraph V2：signature table rollback

```text
sigTable changes trail
currentSig per app
eraseIfOwner
```

性能更好，但复杂。

---

## Inc-EGraph V3：incremental explanation optimization

# 21. Congruence closure algorithm: V1 recommended

```cpp
void IncrementalEGraph::merge(a,b,reason) {
    if (sigTableDirty_) rebuildSignatureTable();

    mergeQueue_.push_back({a,b,reason});
    processMergeQueue();
}

void processMergeQueue() {
    while (!mergeQueue_.empty()) {
        auto [a,b,reason] = pop();

        if (same(a,b)) continue;

        auto ra = rep(a);
        auto rb = rep(b);

        auto affected = collectParents(ra);
        append affected with collectParents(rb);

        MergeId mid = recordMerge(a,b,reason,ra,rb);

        uf_.unite(ra,rb,mid);
        mergeMembers(ra,rb);

        for (auto p : affected) {
            congruenceQueue_.push_back(p);
        }

        processCongruenceQueue();
    }
}

void processCongruenceQueue() {
    while (!congruenceQueue_.empty()) {
        auto app = pop();

        if (!isApplication(app)) continue;

        auto sig = signature(app);
        auto other = sigTable_.findFresh(sig);

        if (!other) {
            sigTable_.insertOrAssign(sig, app);
        } else if (!same(app,*other)) {
            auto reason = MergeReason::congruence(
                app, *other, argPairs(app,*other));
            mergeQueue_.push_back({app,*other,reason});
        }
    }
}
```

```cpp
TheoryConflict EufSolver::makeConflict(ActiveDisequality d) {
    auto reasons = egraph_.explainEquality(d.lhs, d.rhs);

    std::vector<SatLit> clause;
    for (auto r : reasons) clause.push_back(r.negated());
    clause.push_back(d.reason.negated());

    return TheoryConflict{clause};
}
```

# 26. Common pitfalls

## 26.1 Path compression breaks rollback

不要使用 destructive path compression，除非你记录所有 parent changes。

---

## 26.2 Signature table stale bug

如果不处理 stale signatures，会漏 congruence merge 或错误 merge。

必须：

```text
findFresh verifies computeSignature(old)==sig
```

---

## 26.3 Explanation after rollback

mergeRecords 必须 rollback，否则 explanation 会使用已撤销 equality。

---

## 26.4 Class members rollback

如果 members 不 rollback，collectParents 会包含撤销 class 的 members，导致 spurious congruence。

---

## 26.5 Congruence reason missing arg pairs

`MergeReason::congruence` 必须记录当时的 app pair，并能重建 arg pairs：

```text
f(a1,..,an), f(b1,..,bn)
```

否则 explanation 无法递归。

---

## 26.6 Disequality reason sign

如果 active literal 是 false Eq 转 Neq，disequality reason 仍然是那个 assigned literal。Conflict 中放 `reason.negated()`。

# 29. 最终建议

最稳的 incremental e-graph 方案是：

```text
TermManager 永久 intern；
UnionFind 可回滚；
members 可回滚；
mergeRecords 可回滚；
signature table 正常前进时增量维护；
backtrack 后 lazy rebuild signature table；
explanation 用 active mergeRecords BFS + recursive congruence expansion。
```

这比一开始就实现完全 rollbackable signature table 稳得多。

推荐第一版目标：

```text
incremental merge + congruence closure + rollback correctness
```

不要一开始追求最优复杂度。先把：

```text
same / merge / backtrack / explain / conflict
```

全部做对，再优化 signature table rollback 和 explanation speed。
