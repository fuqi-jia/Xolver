如果你问的是**刚上传这份计划**，那它其实更像是 **ICP Framework plan**，不是 IncrementalLinearizer plan。它的总体方向是对的：把 NIA-private interval evaluation 抽成 theory-agnostic ICP common framework，并让 NIA、NRA、IncrementalLinearizer、BoundedNiaSolver、LocalSearchAdvisor 都作为 client。这个定位是正确的。

但现在还不能直接 approve。建议选 **Revise**。主要问题如下。

---

# 1. `Contractor` 接口现在只接受 `ReasonedBoxQ`，和 NIA 不兼容

计划里写：

```cpp
class Contractor {
public:
    virtual ContractorResult contract(ReasonedBoxQ& box) = 0;
};
```

但你的 ICP 目标是同时服务：

```text
NIA -> ReasonedBoxZ
NRA -> ReasonedBoxQ
```

如果 contractor 只接受 `ReasonedBoxQ`，那 NIA adapter 要么做 Z→Q→Z 转换，要么不能复用 contractor。这个会很快变乱。

建议改成两种方案之一：

```text
方案 A：模板化 Contractor<IntervalT, BoxT>
方案 B：保留两套具体接口：
  ContractorZ
  ContractorQ
```

第一版为了少折腾，可以选 B：

```cpp
class ContractorZ {
public:
    virtual ContractorResult contract(ReasonedBoxZ& box) = 0;
};

class ContractorQ {
public:
    virtual ContractorResult contract(ReasonedBoxQ& box) = 0;
};
```

否则 Phase 4 的 `NiaIcpAdapter` 会和前面的公共 contractor 设计冲突。

---

# 2. `IcpConstraint` 只有 `PolyId poly`，但 backward contractor 需要表达式 DAG

计划的 `IcpConstraint` 是：

```cpp
struct IcpConstraint {
    PolyId poly;
    Relation rel;
    SatLit reason;
    TheoryId owner;
};
```

这足够做 forward interval evaluation，但不够做：

```text
AddContractor: z = x + y
SubContractor: z = x - y
MulContractor: z = x * y
SquareContractor: z = x^2
```

因为 backward contraction 需要知道表达式结构。例如：

```text
p = x*y + z - 5
```

单靠 `PolyId` 很难知道应该构造哪个中间变量、哪个 contractor 负责 `x*y`，除非你做 polynomial term decomposition 和 expression DAG reconstruction。

所以现在 Phase 3 写的 primitive backward contractors 和当前输入表示不匹配。

建议改成：

```cpp
struct IcpConstraint {
    std::optional<ExprId> expr;   // 原始表达式 DAG，可选
    PolyId poly;                 // fallback for interval evaluation
    Relation rel;
    SatLit reason;
    TheoryId owner;
};
```

V1 可以只做：

```text
PolyId forward evaluation
+ pattern contractor: square / sum-of-squares
```

真正的 Add/Sub/Mul backward contractor 放到 V2，等表达式 DAG adapter 准备好再做。

---

# 3. Phase 2 的 “Relation Forward Conflict” 不能默认支持多变量

计划说：

```text
p <= 0: if p.lo > 0 -> Empty conflict
```

这个前提是 `p.interval` 是 sound over-approx。可是当前状态写明：

```text
IntervalEvaluator — single-var polynomial forward eval (multi-var → NoChange)
```

所以如果 `p` 是：

```text
x*y + z - 5
x^2 + y^2 - 1
```

不能直接拿不到 interval 后还判断 conflict。

建议写死：

```text
RelationContractor V1:
  if poly is single-var and IntervalEvaluator succeeds:
      use interval conflict rule
  else if pattern is recognized, e.g. sum-of-squares:
      use specialized contractor
  else:
      NoChange
```

不能让实现者误以为 Phase 2 已经支持任意多变量 `p rel 0`。

---

# 4. `SquareBoundReasoner` 不建议直接删除/替换

计划里 Phase 4 写：

```text
replace/remove old SquareBoundReasoner / SumOfSquaresBoundReasoner direct logic; delegate to ICP
```

这个太激进。

`SquareBoundReasoner` 和 `SumOfSquaresBoundReasoner` 已经是 NIA 里比较稳的 sound rules。ICP 新框架第一版还没经过回归，不应该直接替换掉。

建议改成：

```text
Phase 4:
  先并行接入 NiaIcpAdapter；
  保留现有 SquareBoundReasoner / SumOfSquaresBoundReasoner；
  等 ICP 版本通过完全相同 regression 后，再考虑迁移/删除旧实现。
```

也就是先：

```text
linearDomain
→ squareBound(old)
→ sumSquares(old)
→ algebraic
→ ICP(new)
→ bounded
```

而不是直接删旧逻辑。

---

# 5. reason tracking 还不够具体

计划原则里说：

```text
every contraction carries SatLit reasons; empty box conflicts are explainable
```

这是对的。

但后面 `DomainUpdate` 没明确怎么构造 reasons。比如：

```text
x^2 + y^2 <= 1
x ∈ [-2,2]
y ∈ [-2,2]
```

收缩 `x` 到 `[-1,1]` 时，新 bound reason 至少应该包含：

```text
constraint reason: x^2+y^2<=1
+ old bound reasons used by contractor
```

建议明确：

```cpp
struct BoundUpdate {
    std::string var;
    Interval newInterval;
    std::vector<SatLit> reasons;
};
```

并规定：

```text
NiaIcpAdapter 更新 DomainStore 时，必须把 BoundUpdate.reasons 写入 lower/upper reasons。
如果 reasons 不完整，只能 internal update；不能用它生成 theory conflict。
```

---

# 6. open interval / strict relation 没有落到 contraction 规则里

`IntervalQ` 里有：

```cpp
bool loOpen = false, hiOpen = false;
```

但是 Phase 2 的规则只写了：

```text
p <= 0: p.lo > 0
p >= 0: p.hi < 0
p = 0: 0 ∉ p.interval
```

要补 strict：

```text
p < 0:
  definitely violated if p.lo >= 0 and lo is closed enough
p > 0:
  definitely violated if p.hi <= 0
```

但为了 V1 简单，建议先把 strict 在 adapter 层规范化：

```text
NIA:
  p < 0  -> p <= -1
  p > 0  -> p >= 1

NRA:
  保留 strict，RelationContractor 必须正确处理 open bound。
```

如果 V1 不想处理 open interval，那就明确：

```text
NRA strict constraints in ICP V1 return NoChange.
```

不能半支持。

---

# 7. `MulContractor` backward 的“div-by-zero → NoChange”太弱但 sound，需要明确“不 conflict”

计划写：

```text
MulContractor — div-by-zero interval → NoChange
```

这个是 sound 的。但要强调：

```text
如果 denominator interval contains 0：
  不做 backward contraction；
  也不能因为 division unavailable 返回 Empty。
```

否则很容易被误实现成错误 pruning。

---

# 8. `Candidate` 不建议放在 ICP V1

`IcpResult` 里有：

```cpp
std::optional<ModelCandidate> candidate;
```

ICP 可以给 sample point，但这会让职责有点扩张。你现在更需要的是：

```text
Conflict
DomainUpdate
SplitSuggestion
NoChange
UnknownBudget
```

Candidate 可以先放到 LocalSearchAdvisor 或 B&B client 里。ICP core 只提供 box，不直接产生候选模型。

建议 V1 删除 `Candidate`，或者标成：

```text
reserved for future; not used in V1
```

---

# 9. P6 验收标准过大

计划里 P6 写：

```text
大域 B&B 能在合理时间内证明 UNSAT（budget 内）。
```

这个验收太泛，容易变成无底洞。B&B 是否能证明 UNSAT 强依赖具体例子、split 策略、contractor 强度、budget。

建议换成明确测试：

```text
0 <= x <= 100000 ∧ x^2 = 99980001 -> SAT
0 <= x <= 100000 ∧ x^2 = 99980002 -> UNSAT
x^2 + 1 <= 0 over bounded box -> UNSAT by interval conflict
```

不要用“合理时间内证明大域 UNSAT”这种不可操作标准。

---

# 10. 和 IncrementalLinearizer 的关系要补一句

这份计划目标里提到 IncrementalLinearizer 会作为 ICP client。

但 Linearizer 实际需要的是：

```text
ReasonedBox
bound reasons
interval width / finite bounds
```

不一定需要调用完整 `IcpEngine`。建议补一句：

```text
IncrementalLinearizer should depend only on ReasonedBox / IntervalOps / bound reasons.
It should not depend on IcpEngine worklist or contractors.
```

否则两个 agent 并行开发时容易互相耦合。

---

# 建议给 agent 的 Revise 文本

```text
Revise before approve:

1. Contractor interface cannot be fixed to ReasonedBoxQ only.
   ICP must support both NIA and NRA.
   Either template the contractor over Box type or define ContractorZ / ContractorQ.

2. IcpConstraint with only PolyId is insufficient for Add/Sub/Mul backward contractors.
   Add optional ExprId / expression-DAG input, or defer primitive backward contractors to V2.
   V1 should only do forward interval conflict + square/sum-of-squares pattern contractors.

3. RelationContractor V1 must not assume multivar interval evaluation works.
   If IntervalEvaluator cannot soundly over-approx p, return NoChange.

4. Do not immediately remove old SquareBoundReasoner / SumOfSquaresBoundReasoner.
   Keep them until ICP replacements pass identical regression tests.

5. BoundUpdate must carry reasons.
   NiaIcpAdapter must store those reasons into DomainStore.
   If reasons are incomplete, do not generate conflict from that update.

6. Clarify strict relations:
   NIA strict can be normalized to non-strict.
   NRA strict either handled with open intervals or returns NoChange in V1.

7. Candidate output should be disabled or reserved in V1.
   ICP core should return Conflict / DomainUpdate / SplitSuggestion / NoChange / UnknownBudget.

8. Replace vague P6 acceptance with concrete SMT2 cases.

9. IncrementalLinearizer should depend on ReasonedBox/IntervalOps, not the full IcpEngine.
```

---

# 结论

这份计划的**架构方向是对的**，尤其是 theory-agnostic ICP、over-approx-only、reason tracking、client-owned B&B 这些原则都正确。

但现在的问题是：**Phase 3 的 backward contractors 与当前 `PolyId` 输入结构不匹配，Contractor 接口只支持 `ReasonedBoxQ`，以及过早替换 NIA 现有 square/sum-of-squares 逻辑。**

所以我建议：**Revise 后再执行**。第一版最好收敛成：

```text
IntervalQ + ReasonedBoxQ/Z
Relation forward conflict
Square/SumSquares contractors
NIA adapter
NRA optional adapter
不做 Add/Sub/Mul backward
不删除旧 SquareBoundReasoner
```

这样更稳。
