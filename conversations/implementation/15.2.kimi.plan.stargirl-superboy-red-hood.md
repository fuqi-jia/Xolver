## Chosen Approach: Phase 5 Core + Skeleton

### Part A: PolynomialKernel::terms() Interface (Optional)

Extend `PolynomialKernel` with a monomial term iterator that is **purely optional** — any reasoning that depends on it must return `NoChange` on failure, never `UNSAT`.

```cpp
struct MonomialTerm {
    mpz_class coefficient;           // exact integer coefficient ONLY
    std::vector<std::pair<std::string, int>> powers; // [(var, exp), ...]
};

// Return all monomial terms. nullopt if decomposition fails,
// coefficients are not exact integers, or backend unsupported.
virtual std::optional<std::vector<MonomialTerm>> terms(PolyId a) const = 0;
```

**Conservative rule:**
```
If terms() returns nullopt, the caller must return NoChange.
No UNSAT decision may depend on terms() succeeding.
```

**Implementation in LibPolyKernel:**
- libpoly C API has `lp_polynomial_traverse_f` and `lp_polynomial_to_monomial`.
- C++ `polyxx` does not expose term iteration.
- Implement via C API on underlying `lp_polynomial_t*` (`poly::Polynomial::get_internal()`).
- Map `lp_variable_t` back to string via `revVarMap_`.
- If C API fails, coefficient is non-integer, or polynomial is constant → return `nullopt`.

**Files modified:**
- `src/theory/arith/poly/PolynomialKernel.h` — add `terms()` virtual
- `src/theory/arith/poly/LibPolyKernel.h` — add override
- `src/theory/arith/poly/LibPolyKernel.cpp` — implement via libpoly C API

### Part B: Multivariable GCD Conflict

Current `checkGcdConflict` only handles univariate. Extend using `terms()`.

**Algorithm:**
1. Use `terms()` to get all monomials.
2. Collect coefficient of each non-constant monomial.
3. Compute `g = gcd(|coeff_1|, ..., |coeff_n|)`.
4. If `g > 0` and constant term `c % g != 0` → UNSAT.

**Examples:**
- `2x + 4y = 3` → coefficients [2, 4], g=2, 3%2≠0 → UNSAT ✓
- `6x + 9y = 5` → g=3, 5%3≠0 → UNSAT ✓
- `xy + x + y = 0` → coefficients [1, 1, 1], g=1 → no conflict ✓
- `2x + 4y = 6` → g=2, 6%2=0 → no conflict ✓

**Safety:** If `terms()` returns `nullopt`, return `NoChange`.

**Files modified:**
- `src/theory/arith/nia/AlgebraicIntegerReasoner.cpp` — rewrite `checkGcdConflict`

#### C2. IntervalTypes.h

```cpp
struct IntervalZ {
    mpz_class lo; // inclusive
    mpz_class hi; // inclusive
    bool isEmpty() const { return lo > hi; }
};

struct ReasonedInterval {
    IntervalZ interval;
    std::vector<SatLit> reasons; // all SAT literals contributing to this bound
};
```

Design decision: `reasons` is a flat `vector<SatLit>`, not a tagged enum.
The NIA adapter (not `ReasonedBox`) knows which reasons belong to which bounds.

#### C3. ReasonedBox

```cpp
class ReasonedBox {
public:
    // Get interval for a variable
    std::optional<ReasonedInterval> get(const std::string& var) const;

    // Set initial interval (e.g. from DomainStore)
    void set(const std::string& var, ReasonedInterval ri);

    // Narrow: intersect with new interval, merge reasons.
    // Returns true if interval actually changed.
    bool narrow(const std::string& var, IntervalZ newInterval,
                const std::vector<SatLit>& reasons);

    bool isEmpty() const; // any interval empty?
    void clear();

private:
    std::unordered_map<std::string, ReasonedInterval> box_;
};
```

**No `buildConflict()` on `ReasonedBox`.** Conflict construction is the responsibility of the caller (`NiaSolver` adapter) that knows which constraint was violated.

#### C4. IntervalEvaluator (Refactored)

1. **Move** core interval arithmetic (`intervalAdd`, `intervalSub`, `intervalNeg`, `intervalMul`, `intervalPow`) to `IntervalOperations.h`.
2. **Move** `IntervalZ` to `IntervalTypes.h`.
3. **Move** `IntervalEvaluator` class to `interval/IntervalEvaluator.h/.cpp`.
4. **Keep single-var logic** from Phase 3. The evaluator still skips multi-variable polynomials.
5. **Return used reasons:** when a conflict is found, return the violated constraint reason PLUS the variable interval reasons that defined the bounds.

```cpp
struct IntervalEvalResult {
    NiaReasoningKind kind;
    std::optional<TheoryConflict> conflict;
    std::vector<SatLit> usedReasons; // reasons of intervals that participated
};
```

**NIA adapter responsibility:**
```cpp
auto ir = intervalEvaluator_.run(normalized, box);
if (ir.kind == Conflict) {
    // Build conflict: violated constraint reason + all used interval reasons
    auto conflictLits = ir.conflict->lits;
    conflictLits.insert(conflictLits.end(), ir.usedReasons.begin(), ir.usedReasons.end());
    return {Conflict, TheoryConflict{conflictLits}, ...};
}
```

#### C5. NIA Integration

In `NiaSolver::check()`:
```cpp
// 9. Interval evaluation (via common framework)
ReasonedBox box;
for (auto& [var, domain] : domains_.entries()) {
    if (domain.hasLower && domain.hasUpper) {
        std::vector<SatLit> reasons = domain.lowerReasons; // TODO: track reasons in DomainStore
        reasons.insert(... domain.upperReasons ...);
        box.set(var, ReasonedInterval{IntervalZ{domain.lower, domain.upper}, reasons});
    }
}
auto ir = intervalEvaluator_.run(normalized, box);
if (ir.kind == NiaReasoningKind::Conflict) { ... }
```

**Note:** `DomainStore` may need to track `reasons` per bound. If not yet tracked, use coarse fallback (all active reasons) — same as Phase 3.
