## 1. `sat_->value()` 不能在 `cb_check_found_model` 中使用

这个判断很明确。既然 CaDiCaL 已经报：

```text
CaDiCaL::Solver::val() -> solver in invalid state
```

那就说明 `val()` 在这个 callback 状态下不是合法 API。不要再试图在 `TheoryManager::check()` 里面调用它。

`TheoryManager` 本来也不应该知道 CaDiCaL 的状态机。它应该依赖一个抽象接口：

```cpp
enum class LitValue {
    True,
    False,
    Unknown
};

class TheoryAssignmentView {
public:
    virtual LitValue value(SatLit lit) const = 0;
};
```

然后：

```cpp
class TheoryManager {
public:
    void setAssignmentView(const TheoryAssignmentView* view) {
        assignment_ = view;
    }

private:
    const TheoryAssignmentView* assignment_ = nullptr;
};
```

`TheoryManager::check()` 里不要再写：

```cpp
bool satVal = sat_->value(eqLit.var);
```

而是写：

```cpp
LitValue v = assignment_->value(eqLit);
```

CaDiCaL 的具体赋值状态由 `CadicalTheoryPropagator` 管。

---

## 2. 不建议只用 `deducedEqCache_ + lemmaDb.insertIfNew()`

你提出的方案：

> 彻底移除 `sat_->value()`，仅依靠 `deducedEqCache_ + lemmaDb.insertIfNew()` 去重。

这个可以作为临时止血，但不应该作为正式实现。

原因是你仍然需要区分三种情况：

```text
Eq(a,b) 已经为 true：
    不需要生成 lemma。

Eq(a,b) 已经为 false：
    reasons -> Eq(a,b) 是 conflict lemma。

Eq(a,b) 未赋值：
    reasons -> Eq(a,b) 是 propagation lemma。
```

如果完全不知道 literal 当前值，你会把这三种情况混在一起。虽然 clause database 可以去重，但这会造成无意义 lemma、潜在重复 callback，甚至在 found-model callback 中无法清楚地区分“剪掉当前模型”还是“发现理论冲突”。

所以正确做法不是“不查 assignment”，而是 **不通过 CaDiCaL::val() 查 assignment**。

## 3. 在 `CadicalTheoryPropagator` 层维护 assignment/model cache

建议实现一个 `CadicalAssignmentView`：

```cpp
class CadicalAssignmentView final : public TheoryAssignmentView {
public:
    LitValue value(SatLit lit) const override {
        auto var = lit.var;
        if (var >= values_.size()) return LitValue::Unknown;

        int v = values_[var]; // 1 true, -1 false, 0 unknown
        if (v == 0) return LitValue::Unknown;

        bool varTrue = (v > 0);
        bool litTrue = lit.isPositive() ? varTrue : !varTrue;
        return litTrue ? LitValue::True : LitValue::False;
    }

    void clear() {
        values_.clear();
    }

    void setVarValue(SatVar var, bool value) {
        if (var >= values_.size()) values_.resize(var + 1, 0);
        values_[var] = value ? 1 : -1;
    }

private:
    std::vector<int8_t> values_;
};
```

在 `cb_check_found_model(...)` 中，不要调用 `solver->val()`，而是用 callback 传进来的 model 信息刷新这个 view。

伪代码：

```cpp
bool CadicalTheoryPropagator::cb_check_found_model(...) {
    assignmentView_.clear();

    for (int lit : model) {
        SatVar v = std::abs(lit);
        assignmentView_.setVarValue(v, lit > 0);
    }

    theoryManager_.setAssignmentView(&assignmentView_);
    return theoryManager_.check(lemmaDb_);
}
```

具体怎么从 `model` 里解析，取决于你现在的 `cb_check_found_model` 签名。如果传入的是完整 model vector，就从里面读；如果传入的是 assigned literals 列表，就按 literal 正负号填 cache。核心原则是：**callback 给了什么状态，就用 callback 参数，不要在 callback 里再反查 CaDiCaL。**

## 4. `TheoryManager::check()` 里的 deduced equality 逻辑应改成这样

```cpp
for (const auto& prop : solver->getDeducedSharedEqualities()) {
    auto eqLit = registry_->getOrCreateSharedEqualityAtom(prop.a, prop.b);

    LitValue val = assignment_->value(eqLit);

    if (val == LitValue::True) {
        // SAT already assigned Eq(a,b), nothing to do.
        continue;
    }

    TheoryLemma lemma;
    lemma.lits = prop.reasonNegations;
    lemma.lits.push_back(eqLit);

    if (val == LitValue::False) {
        // Current assignment has ¬Eq(a,b), so this is a conflict lemma.
        return TheoryCheckResult::conflict(lemma);
    }

    // Unknown / unassigned: propagation lemma.
    if (deducedEqCache_.insert(prop.key()).second &&
        lemmaDb.insertIfNew(lemma)) {
        return TheoryCheckResult::lemma(lemma);
    }
}
```

注意：`deducedEqCache_` 只能用来防重复，不能替代 assignment 查询。

## 5. `x = v1` 这个 EUF 推导必须修，不能忽略

对于测试 1：

```smt2
(assert (> (f x) 0))
```

合理的 purification 应该类似：

```text
v1 = f(x)      // EUF bridge
v1 > 0         // LRA atom
```

这里 **EUF 绝对不应该推出 `x = v1`**。

`x` 是函数参数，`v1` 是函数应用 `f(x)` 的结果。除非原公式显式断言 `x = f(x)`，否则 EUF 不能把它们合并。

所以 `x = v1` 的出现说明 EUF 层仍有 bug。最可能的位置有三个：

### A. EUF bridge 编码错了

错误编码可能是：

```text
merge(x, v1)
```

正确编码应是：

```text
appNode = f(x)
merge(v1, appNode)
```

也就是说，`v1 = f(x)` 是 **结果常量和函数应用节点相等**，不是 `v1` 和参数 `x` 相等。

### B. `termManager_.intern()` 把 variable node 和 bridge variable node 复用了

要检查：

```text
x
v1
f(x)
```

三者必须是三个不同的 e-graph node：

```text
const(x)      != const(v1)
app(f, x)     != const(x)
app(f, x)     != const(v1) before asserting bridge equality
```

bridge equality 后只能有：

```text
const(v1) == app(f, const(x))
```

不能有：

```text
const(v1) == const(x)
```

### C. `getDeducedSharedEqualities()` 报告了非 shared-constant 等价类

`getDeducedSharedEqualities()` 只能报告：

```text
registered SharedTermId constant 之间的 equality
```

不能报告：

```text
函数应用节点 与 参数节点
函数应用节点 与 任意内部节点
同一个 EUF term 的内部结构关系
```

建议加过滤：

```cpp
if (!sharedTermRegistry_->isRegisteredSharedConstant(a)) continue;
if (!sharedTermRegistry_->isRegisteredSharedConstant(b)) continue;
if (a == b) continue;
if (!egraph.areEqual(nodeOf(a), nodeOf(b))) continue;
```

更重要的是，`nodeOf(a)` 必须是 shared constant 的 e-graph node，不是 `a` 对应表达式内部随便某个节点。

## 6. 现在应该怎么推进

我建议按下面顺序做，不要跳步：

### 第一步：移除 `TheoryManager` 对 `sat_->value()` 的直接依赖

这是架构修复，必须做。

```text
TheoryManager 不再持有 CaDiCaL backend。
TheoryManager 只持有 TheoryAssignmentView。
CadicalTheoryPropagator 在 callback 中构造/更新 AssignmentView。
```

### 第二步：保留 assignment 判断，不要只靠 lemma 去重

改成：

```text
Eq true    -> skip
Eq false   -> conflict lemma
Eq unknown -> propagation lemma
```

### 第三步：立刻加 EUF deduced equality debug log

在 `getDeducedSharedEqualities()` 里打印：

```text
deduced equality:
  shared term a id/name/expr
  shared term b id/name/expr
  egraph node a
  egraph node b
  reason
  whether each side is a shared constant
```

对于测试 1，只要看到：

```text
x = v1
```

就直接 abort：

```cpp
assert(!(name(a) == "x" && name(b).starts_with("__purify")));
```

不是为了长期保留这个断言，而是为了强制定位错误合并来源。

### 第四步：追踪 EUF merge 来源

给 EUF 的每次 merge 加 reason：

```cpp
enum class EufMergeReasonKind {
    AssertedEquality,
    BridgeDefinition,
    Congruence,
    InterfaceEquality
};

struct EufMergeReason {
    EufMergeReasonKind kind;
    ExprId sourceExpr;
    SatLit reasonLit;
};
```

然后打印 `x` 和 `v1` 为什么会在同一个 class。

如果原因是 `BridgeDefinition`，说明 bridge 编码错了。
如果原因是 `Congruence`，说明 congruence rule 把参数和结果混了。
如果原因是 `InterfaceEquality`，说明 combination 层错误传播了 `x = v1`。
如果没有 reason，说明 union-find merge 调用点缺失记录，也要补。

## 7. 对你给的三个选项的明确回答

### 选项 1：先绕过 `val()`，让测试 1 跑通，再处理 EUF 误推导？

**不建议。**

可以临时注释 `sat_->value()` 验证是不是唯一崩溃点，但不要把“只靠 cache 和 lemmaDb”作为正式方案。否则你只是绕过了 CaDiCaL 状态问题，但留下了 assignment 语义不清和 EUF unsound 推导。

### 选项 2：先追查 EUF 为什么认为 `x = v1`？

**必须做，但不应该阻塞 `val()` 架构修复。**

`x = v1` 是明确的 EUF bug。即使没有 CaDiCaL 崩溃，这也会导致错误 lemma：

```text
reasons -> x = v1
```

这是 soundness 问题。

### 选项 3：在 `CadicalTheoryPropagator` 层维护赋值缓存，替代 `sat_->value()`？

**这是正确主线。**

最终实现应该是：

```text
Cadical callback receives candidate model / assignments
    -> update CadicalAssignmentView
    -> TheoryManager::check() queries AssignmentView
    -> no CaDiCaL::val() call inside callback
```

## 最终建议

按这个执行：

```text
1. 实现 CadicalAssignmentView，彻底移除 TheoryManager::check() 中的 sat_->value()。
2. TheoryManager 根据 AssignmentView 区分 Eq true / false / unknown。
3. 不采用“仅 deducedEqCache_ + lemmaDb 去重”的正式方案。
4. 立刻追查 EUF 为什么推出 x = v1。
5. getDeducedSharedEqualities() 只允许 registered shared constants 之间的 equality。
6. EUF bridge 必须编码为 v1 = app(f, x)，不能编码为 v1 = x。
```

一句话：**`val()` 问题用 propagator assignment cache 修；`x = v1` 问题用 EUF merge tracing 修。两个都要修，不能用一个掩盖另一个。**
