## 5.2 一元 algebraic 路径

这是 P1 已经部分实现的，但要从“比例检测”升级为“gcd + refine”。

目标：

```text id="xgt1e1"
sign(g(alpha))
```

其中：

```text id="qjzatv"
alpha 是 f 的某个实根。
```

算法：

```cpp id="eun8ao"
Sign signUnivariateAtAlgebraic(UniPolyId g, const AlgebraicRoot& alpha) {
    UniPolyId f = alpha.definingPoly;

    // 1. 如果 g == 0
    if (isZero(g)) return Sign::Zero;

    // 2. 判断 alpha 是否也是 g 的根
    UniPolyId d = gcd(f, g);
    if (!isConstant(d)) {
        if (rootBelongsTo(alpha, d)) {
            return Sign::Zero;
        }
    }

    // 3. alpha 不是 g 的根，则不断 refine alpha 的 isolating interval，
    //    直到 g 在该 interval 上符号不变。
    Interval I = alpha.interval();
    while (true) {
        if (!hasRootInInterval(g, I)) {
            mpq_class q = chooseRationalInside(I);
            return sign(eval(g, q));
        }
        I = refineRootInterval(f, alpha.rootIndex, I);
    }
}
```

你现在的“同定义多项式比例检测”只能覆盖：

```text id="77s6u6"
g = c * f
```

但覆盖不了：

```text id="7n8u5v"
g = f * h
```

例如：

```text id="eoa4mo"
f = x^2 - 2
g = (x^2 - 2)(x + 1)
```

`g(alpha)` 也是 0，但比例检测发现不了。必须用 gcd。

# 6. Algebraic tower 表示

CDCAC 的 sample 是按变量层级构造的，本质上是一个 tower：

```text id="plsvb3"
Q
  ⊂ Q(alpha_1)
  ⊂ Q(alpha_1, alpha_2)
  ⊂ ...
```

其中每个 `alpha_i` 是某个关于 `x_i` 的多项式根，前面变量已经固定。

所以每个 algebraic sample 最好保存：

```text id="5hx9jn"
level
mainVar
liftedDefiningPoly
specializedDefiningPoly
rootIndex
isolatingInterval
```

例如：

```text id="j0ykoj"
x = sqrt(2)
defining polynomial: x^2 - 2

y = sqrt(2)
defining polynomial: y - x
after x=sqrt(2): y - sqrt(2)
```

这个例子里，`y` 的 defining polynomial 带 algebraic coefficient，所以完整实现需要支持 algebraic coefficient polynomial。

# 7. signAt 的 tower-reduction 算法

给定：

```text id="xebifi"
p(x_1, ..., x_k)
sample = {x_1 = alpha_1, ..., x_k = alpha_k}
```

算法分两步：

```text id="9i8gux"
1. reduce p modulo defining polynomials of algebraic sample
2. 对 reduced polynomial 做 interval sign decision
```

---

## 7.1 reduce modulo tower

伪代码：

```cpp id="dngntg"
PolyId reduceModuloTower(PolyId p, const SamplePoint& sample) {
    PolyId q = p;

    for (int i = sample.level() - 1; i >= 0; --i) {
        VarId x = sample.varOrder[i];
        const RealAlg& a = sample.values[i];

        if (a.isRational()) {
            q = substituteRational(q, x, a.rational());
            continue;
        }

        const AlgebraicRoot& root = a.root();

        // f_i is the defining polynomial for alpha_i,
        // viewed as polynomial in variable x.
        PolyId f = root.liftedDefiningPoly;

        // Safety: leading coefficient of f must not vanish under prefix.
        // Otherwise this is nullification / non-regular situation.
        PolyId lc = leadingCoefficient(f, x);
        Sign lcSign = signAt(lc, sample.prefix(i));
        if (lcSign == Sign::Zero || lcSign == Sign::Unknown) {
            throw NullificationOrUnknown();
        }

        q = pseudoRemainder(q, f, x);
        q = primitivePart(q);
        q = normalize(q);
    }

    return q;
}
```

含义：

```text id="m83ujk"
如果 alpha_i 满足 f_i(alpha_i)=0，
那么 p 和 p mod f_i 在 alpha_i 处取值相同。
```

如果最终 `q` 变成零多项式，则：

```cpp id="de5t3t"
return Sign::Zero;
```

例如：

```text id="fui0o8"
p = y - x
sample:
  x = sqrt(2), f_x = x^2 - 2
  y = sqrt(2), f_y = y - x

reduce p by f_y:
  y - x mod (y - x) = 0
=> sign = Zero
```

这就解决了你现在 `mixed rational/algebraic sample` 无法判零的问题。

## 7.2 interval sign decision

如果 reduce 后不是零多项式，就用 interval refinement 判符号。

伪代码：

```cpp id="w3ixwf"
Sign signAtAlgebraic(PolyId p, const SamplePoint& sample) {
    PolyId q = reduceModuloTower(p, sample);

    if (kernel_->isZero(q)) {
        return Sign::Zero;
    }

    for (int iter = 0; iter < maxRefineIters; ++iter) {
        IntervalValue iv = intervalEvaluate(q, sample);

        if (iv.lower > 0) return Sign::Pos;
        if (iv.upper < 0) return Sign::Neg;

        refineRelevantAlgebraicIntervals(sample, q);
    }

    return Sign::Unknown;
}
```

为什么这是 sound 的？

```text id="cyfftf"
如果 q(sample) 不是 0，它是一个非零代数数。
非零代数数与 0 有正距离。
随着 isolating intervals 细化，interval evaluation 最终会排除 0。
```

所以完整逻辑是：

```text id="3l9vdu"
先用 algebraic reduction 检测 exact zero；
如果不是 zero，再用 interval refinement 决定正负。
```

不能只做 interval refinement，因为如果真实值就是 0，区间会一直包含 0，最后只能 Unknown。

## 8.3 algebraic-algebraic

```cpp id="w0pxok"
CompareResult compareAlgebraicAlgebraic(AlgebraicRoot a, AlgebraicRoot b) {
    if (a.definingPoly == b.definingPoly && a.rootIndex == b.rootIndex) {
        return Equal;
    }

    // If intervals are disjoint, compare immediately.
    while (true) {
        if (a.upper < b.lower) return Less;
        if (b.upper < a.lower) return Greater;

        // Intervals overlap.
        // Check if they may be the same algebraic number.
        UniPolyId d = gcd(a.definingPoly, b.definingPoly);
        if (!isConstant(d)) {
            if (rootBelongsTo(a, d) && rootBelongsTo(b, d)) {
                return Equal;
            }
        }

        refine(a);
        refine(b);
    }
}
```

P2 初版可以只支持：

```text id="emr25q"

rationalized roots
disjoint intervals
```

但完整版本必须支持 gcd-based equality。

```cpp id="gfsjza"
UniPolyQ specializeRational(PolyId p,
                            const SamplePoint& prefix,
                            VarId mainVar) {
    std::map<int, mpq_class> coeffByDegree;

    for (auto term : kernel_->terms(p)) {
        mpq_class coeff = term.coeff;
        int mainDegree = 0;

        for (auto [var, pow] : term.powers) {
            if (var == mainVar) {
                mainDegree += pow;
            } else if (prefix.isAssigned(var)) {
                coeff *= pow(prefix.valueOf(var).rational(), pow);
            } else {
                return NullUniPoly;
            }
        }

        coeffByDegree[mainDegree] += coeff;
    }

    return buildUniPoly(coeffByDegree);
}
```

# 11. 当前 `x*y=0, x!=0, y!=0` 的讨论

你说：

```text id="46mttl"
这个例子不需要 projection，因为每个 cell 上都有局部冲突。
```

这个判断只在非常特定的实现下成立。更准确地说：

```text id="l1brre"
它不需要完整 McCallum/Lazard projection，
但仍然需要 parent-cell generalization 的 guard。
```

变量序 `x -> y`：

```text id="122bgv"
x = 0:
  x != 0 冲突

x < 0 或 x > 0:
  xy = 0 等价于 y = 0
  但 y != 0 冲突
```

这里为什么可以把 `x=-1` 的 child conflict 推广到整个 `x<0` cell？

因为在 `x<0` 内，`x` 保持非零，`xy=0` 对 `y` 的根结构稳定，仍然只有 `y=0`。

也就是说，这个推广依赖 guard：

```text id="w34t72"
x != 0
```

而这个 guard 本质上就是 projection/root split 产生的 cell 条件。

所以你可以在 P2a 用轻量方式处理：

```text id="b6oroz"
当前变量的 roots 来自 active polynomials 的主变量根；
sector 上记录 sign-invariance guard；
child conflict 可提升到该 sector。
```

但不要在设计上说“完全不需要 projection”。严格说它需要的是：

```text id="zaxcbg"
minimal projection/generalization guard
```

而不是完整 projection engine。

不应该作为：

```text id="4ehpsd"

```

真正的功能方案应该是：

```text id="wiy3dk"
1. RootSet 阶段 canonicalize rational roots；
2. SamplePoint 支持混合 rational/algebraic values；
3. signAt 支持：
   - rational exact evaluation；
   - univariate algebraic sign via gcd + refinement；
   - multivariate algebraic sign via tower reduction + interval refinement；
4. mergeRoots / coversAllLine 使用 compareRealAlg；
5. specializeToUnivariate 支持 rational prefix 和 algebraic prefix；
6. root isolation 最终支持 algebraic coefficients；
7. nullification/leading coefficient vanishing 时不能假装继续，必须 fallback Lazard 或 Unknown。
```

一句话：**你现在可以用 rationalization 修掉当前测试，但下一步必须把 `signAt(poly, SamplePoint)` 做成 algebraic-tower evaluator。否则 CDCAC 只能算 P2a demo，不是完整 NRA solver。**
