## 1. 先在 `TheoryManager::check()` 加统一 trace

建议不要只在 LRA 里打。直接在 `TheoryManager::check()` 的每个阶段前后打印。

加一个临时 debug 开关：

```cpp
static bool noDebugEnabled() {
    const char* v = std::getenv("NLCOLVER_NO_DEBUG");
    return v && std::string(v) != "0";
}

#define NO_DBG if (!noDebugEnabled()) {} else std::cerr
```

然后准备几个 formatter：

```cpp
std::string fmtLit(SatLit lit) {
    std::ostringstream os;
    os << (lit.isNegated() ? "-" : "")
       << "v" << lit.var;
    return os.str();
}

std::string fmtClause(const std::vector<SatLit>& lits) {
    std::ostringstream os;
    os << "{ ";
    for (auto lit : lits) os << fmtLit(lit) << " ";
    os << "}";
    return os.str();
}

std::string fmtResult(const TheoryCheckResult& r) {
    switch (r.status) {
        case TheoryCheckStatus::Consistent: return "Consistent";
        case TheoryCheckStatus::Conflict: return "Conflict";
        case TheoryCheckStatus::Lemma: return "Lemma";
        case TheoryCheckStatus::Unknown: return "Unknown";
    }
    return "Invalid";
}
```

在 `TheoryManager::check()` 最外层加：

```cpp

NO_DBG << "[NO] hasImmediateConflict=" << hasImmediateConflict_ << "\n";
```

如果你有 assignment view，打印当前 assigned combination atoms：

```cpp

```

没有的话，先打印 registry 全部 shared equality atom：


## 2. 每个 solver check 后立即打印 result 和 clause

在类似这种地方：

```cpp
for (auto* solver : solvers_) {
    auto res = solver->check();
    if (!res.isConsistent()) return res;
}
```

改成：

```cpp
for (auto* solver : solvers_) {
    NO_DBG << "[NO] checking solver=" << solver->name() << "\n";

    auto res = solver->check();

    NO_DBG << "[NO] solver=" << solver->name()
           << " result=" << fmtResult(res) << "\n";

    if (res.hasLemma()) {
        NO_DBG << "[NO] solver=" << solver->name()
               << " lemma=" << fmtClause(res.lemma.lits) << "\n";
        debugDumpClauseAssignment(res.lemma.lits);
    }

    if (res.isConflict()) {
        NO_DBG << "[NO] CONFLICT from solver=" << solver->name()
               << " clause=" << fmtClause(res.lemma.lits) << "\n";
        debugDumpClauseAssignment(res.lemma.lits);
        return res;
    }

    if (res.isLemma()) {
        return res;
    }
}
```

`debugDumpClauseAssignment` 建议这样写：

```cpp
void TheoryManager::debugDumpClauseAssignment(const std::vector<SatLit>& clause) const {
    if (!noDebugEnabled() || !assignmentView_) return;

    for (auto lit : clause) {
        auto v = assignmentView_->value(lit);
        auto lvl = assignmentView_->level(lit.var); // 没有 level 就先打印 -1

        NO_DBG << "    lit " << fmtLit(lit)
               << " value=" << fmtValue(v)
               << " level=" << lvl << "\n";
    }
}
```

如果 `AssignmentView` 还没有 `level()`，建议临时加：

```cpp
int level(SatVar var) const;
```

因为你现在的问题高度依赖 level-0 / decision-level 的区别。

## 8. 特别检查空 conflict clause

你之前做了：

> translateConflict 后，只把 level > 0 的 interface equality reasons 加入冲突子句；level-0 的 interface equalities 不加入。

这个改动有风险。至少先加断言：

```cpp
if (lemma.lits.empty()) {
    NO_DBG << "[BUG] empty conflict clause generated by "
           << solver->name() << "\n";
    abort();

```

如果 test5 第一次 check 是 LRA 返回空 conflict clause，那 SAT 收到空子句，当然立即 UNSAT。

另外，从正确性角度说，**不要把“过滤 level-0 reason”作为常规修复手段**。更稳的是：

```text
完整 explanation 先生成；

绝不能把必要 reason 删成空 clause。
```

对于当前调试，建议先打印完整 conflict，再打印过滤后的 conflict：

## 11. 我最怀疑的三个具体点

### 第一怀疑：`f(x)` 和 `f(y)` 被 Purifier 或 EUF intern 成同一个 bridge/app

症状：

```text

  v1 > 0
  v1 < 0
```

直接导致第一次 LRA conflict。

检查：

```text
Purifier bridge mapping
LRA variable id of bridge_fx / bridge_fy
```

### 第二怀疑：`distinct(x,y)` polarity 反了

症状：

```text
[SEM] assertEquality x = y reason=Eq(x,y) level=0
```

而不是：

```text
[SEM] assertDisequality x != y reason=¬Eq(x,y) level=0
```

这会触发 EUF congruence：


### 第三怀疑：empty/over-minimized conflict clause

症状：

```text
[LRA] full conflict reasons={ ... }
[LRA] filtered conflict reasons={ }
[NO] CONFLICT from solver=LRA clause={}
```

这会让 CaDiCaL 直接 UNSAT。

## 最终建议

先不要继续改算法。**先把第一次 model check 的 Conflict 源头打出来。**

我预期大概率会看到以下三种之一：

```text
A. LRA conflict: same bridge variable has >0 and <0
   -> 修 Purifier / bridge cache / EUF app interning

B. SEM conflict: x,y already same class while distinct(x,y)
   -> 修 distinct polarity / stale equality / rollback

C. Empty conflict clause from LRA filtering
   -> 撤掉或修正 level-0 reason filtering
```

尤其注意：**test5 是 sat，第一次 check 正常不应该有任何理论冲突。只要第一次 Conflict 出现，就一定是状态、路由、polarity、bridge 复用或 conflict clause 生成错了。**
