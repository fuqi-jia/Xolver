建议 **Revise**，而且这次有一个比较关键的架构问题：**LIA/LRA 只接收 linearizer 生成的 cuts，但不接收原始 bounds，会导致 McCormick relaxation 基本推不动**。

这版已经修掉很多前面的问题：`NonlinearTermKey` 已经改成基于 `(var, exponent)` 的 canonical powers，不再用不稳定的 `PolyId children`；`emitAllMcCormick=true`；square nonnegativity 已经明确为 `s=x² -> s>=0` 且不需要 bounds；abstraction lemma 和 cut lemma 也已经拆开；adapter 也改成成员并维护 pending queue。整体比前一版好多了。

但还不能 approve。

---

# 1. 最大问题：LIA/LRA 没有原始 bounds，McCormick cut 难以证明 UNSAT

计划现在写：

```text
QF_NIA:
  原始 arithmetic atoms route to NIA；
  only linearizer-generated cuts route to LIA。

QF_NRA:
  原始 arithmetic atoms route to NRA；
  only linearizer-generated cuts route to LRA。
```

这个会导致一个实际问题。

以测试为例：

```smt2
0 <= x <= 10
0 <= y <= 10
x*y > 200
```

linearizer 会生成：

```text
t = x*y
t >= 201
```

以及 McCormick cuts：

```text
t >= 0
t >= 10x + 10y - 100
t <= 10x
t <= 10y
```

但如果 LIA 只收到这些 generated cuts，而没有收到：

```text
x >= 0
x <= 10
y >= 0
y <= 10
```

那么 LIA 可以令：

```text
x = 21
y = 21
t = 201
```

所有 linearizer cuts 都满足，无法证明 UNSAT。

也就是说，**guarded lemma 里的 bound reasons 只保证 cut 在这些 bounds active 时是 sound；但它不会把 bounds 作为 LIA/LRA 约束提供给 linear relaxation solver**。因此 LIA/LRA 需要同时知道这些 bounds。

必须改成下面二选一。

## 方案 A：原始 linear bounds 同时 mirror 到 LIA/LRA

在 QF_NIA 中：

```text
原始 nonlinear / polynomial atoms -> NIA
原始 linear bound atoms -> NIA + LIA mirror
linearizer-generated cuts -> LIA
```

在 QF_NRA 中：

```text
原始 nonlinear / polynomial atoms -> NRA
原始 linear bound atoms -> NRA + LRA mirror
linearizer-generated cuts -> LRA
```

可以通过 dynamic mirror lemma 做：

```text
B_NIA -> B_LIA
即 ¬B_NIA ∨ B_LIA
```

这样 LIA/LRA 才能使用 active bounds 推导 linear conflict。

## 方案 B：Linearizer 生成 bound-eliminated cuts

例如从 `x,y∈[0,10]` 和 `t=x*y` 直接生成：

```text
0 <= t <= 100
```

这样 LIA 不需要知道 `x<=10,y<=10`。但这比 McCormick envelope 弱很多，只适合 V1 补救，不适合作为最终方案。

我更建议 **方案 A**。

---

# 2. 当前 integration tests 依赖 bounds mirror，否则不成立

计划里的测试包括：

```text
NIA: xy>200, x,y∈[0,10] -> UNSAT
NRA: xy≥2, x,y∈[0,1] -> UNSAT
NIA: x²+y²≤1, x=2 -> UNSAT
```

这些都依赖 LIA/LRA 能看到原始 bounds / equality。

例如 `x²+y²≤1, x=2`：

linearizer 会生成：

```text
s_x + s_y <= 1
s_x >= 0
s_y >= 0
```

还需要根据 `x=2` 和 tangent cut 推出：

```text
s_x >= 4
```

但 tangent cut 是：

```text
s_x >= 4x - 4
```

如果 LIA 不知道 `x=2`，它不能推出 `s_x>=4`。

所以这个测试也必须依赖：

```text
x=2 被 mirror 到 LIA
```

否则测试预期不可靠。

# 3. `markEmitted(lemma)` 接口不匹配 cache API

计划里 cache API 是：

```cpp
markEmitted(key, nonlinearReason, boundValues, cutIndex)
```

但 adapter 伪代码写：

```cpp
linAdapter_->markEmitted(lemma);
```

这两个接口对不上。`TheoryLemma` 本身不包含：

```text
NonlinearTermKey
nonlinearReason
boundValues
cutIndex
```

所以 adapter 无法根据 `lemma` 标记 cache。

建议把返回类型改成：

```cpp
struct PendingLinearizationLemma {
    TheoryLemma lemma;
    std::optional<CutCacheKey> cacheKey;
};

struct LinearizationResult {
    LinearizationStatus status;
    std::vector<PendingLinearizationLemma> lemmas;
};
```

然后 adapter：

```cpp
for (auto& item : lr.lemmas) {
    if (!lemmaDb.contains(item.lemma)) {
        lemmaDb.insertIfNew(item.lemma);
        pendingLinLemmas_.push_back(std::move(item.lemma));
        if (item.cacheKey) {
            linAdapter_->markEmitted(*item.cacheKey);
        }
    }
}
```

或者更简单：只在 `IncrementalLinearizer::run()` 内部对已经进入返回列表的前 `maxLemmas` 标记 emitted，不要让 adapter 标记。

---

# 4. “No finite bounds → NoChange” 这条 red line 已经过时

计划后面已经支持：

```text
abstraction lemma: 不需要 bounds
square nonnegativity: 不需要 bounds
```

但 Correctness Red Lines 仍写：

```text
No finite bounds → NoChange
```

这会误导实现者跳过 square nonnegativity 和 abstraction lemma。

应该改成：

```text
No finite complete bounds → do not emit bounds-dependent envelope cuts.
Abstraction lemmas and square nonnegativity are still allowed.
```

---

# 5. Bound reasons 完整性还要落到 adapter 行为

计划已经补了：

```text
empty reasons does not mean unconditional
finite bound has no complete reason -> skip cut or use coarse all-active guard
```

这是对的。

但 adapter 要明确选择一种策略。建议 V1 先用：

```text
如果 bound reasons incomplete 且不是 global:
  skip bounds-dependent cut
```

不要先上 coarse all-active guard。coarse guard 需要 adapter 知道所有 active assertion literals，容易实现复杂化。

---

# 6. 可以直接给 agent 的 Revise 文本

```text
Revise before approve:

1. Fix the LIA/LRA visibility problem.
   Linearizer-generated McCormick cuts alone are not enough.
   LIA/LRA also need the original active linear bounds.

   In QF_NIA:
     original arithmetic atoms still route to NIA;
     original linear bounds must also be mirrored to LIA;
     generated linearizer cuts route to LIA.

   In QF_NRA:
     original arithmetic atoms route to NRA;
     original linear bounds must also be mirrored to LRA;
     generated cuts route to LRA.

   Otherwise tests like:
     0<=x,y<=10 and x*y>200
   will not be proved UNSAT, because LIA can set x,y outside the original bounds.

2. Add bound-mirror lemmas:
     B_NIA -> B_LIA
     encoded as ¬B_NIA ∨ B_LIA
   and similarly for NRA/LRA.
   Mirror only linear bounds/equalities that the linear relaxation needs.

3. Fix cache marking API.
   markEmitted(lemma) is not enough because the cache key needs:
     NonlinearTermKey, nonlinearReason, boundValues, cutIndex.
   Return PendingLinearizationLemma { lemma, optional<CutCacheKey> }
   or mark only items actually returned by run().

4. Update red line:
     No finite complete bounds -> do not emit bounds-dependent envelope cuts.
     Abstraction lemmas and square nonnegativity remain allowed.

5. For V1, choose one bound reason policy:
     If finite bounds have incomplete reasons and are not global, skip the cut.
   Do not silently emit unguarded cuts.
```

---

# 结论

这版已经很接近，但还不能 approve。
最大 blocker 是：

```text
linearizer cuts route to LIA/LRA，
但原始 bounds 不 route to LIA/LRA。
```

这会导致 McCormick relaxation 没有可用的变量 bounds，集成测试很可能过不了，或者只能靠很弱的 cut。修成 **bounds mirror + generated cuts route to LIA/LRA** 后，这个计划就基本可以执行。
