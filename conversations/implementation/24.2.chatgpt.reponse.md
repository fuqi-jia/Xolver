**结论：不接受当前版本。**

这版方向是对的：它已经不是再改语义，而是只做 **IncrementalEGraph 内部性能优化**；范围也很干净，明确不做 propagation、model construction、mixed fallback。文档还说明当前代码已经满足 11 条 baseline hard rules，计划只替换 signature table、增加 per-app signature rollback、用 ProofForest 加速 explanation。

但当前优化方案还不能直接实现，主要有几个新的 correctness / rollback blocker。

# 1. 最大 blocker：`RollbackSignatureTable::insertOrAssign` 的 rollback 信息不完整

当前设计：

```cpp
enum class ChangeKind { InsertNew, Overwrite, Erase };
struct Change {
    ChangeKind kind;
    AppSignature sig;
    std::optional<EufTermId> oldOwner;
};
```

问题是：`Erase` 的 rollback 需要恢复被删除的 owner，但 `oldOwner` 名字上像是 overwrite 前 owner，语义不够明确。更关键的是，`InsertNew` rollback 需要 erase，`Overwrite` rollback 需要恢复 old owner，`Erase` rollback 需要恢复 erased owner。必须写清楚每种操作的逆操作，否则实现模型容易写错。

建议改成：

```cpp
struct Change {
    ChangeKind kind;
    AppSignature sig;
    std::optional<EufTermId> previousOwner;
};
```

并明确：

```text
InsertNew:
  previousOwner = null
  rollback => erase(sig)

Overwrite:
  previousOwner = old owner
  rollback => table_[sig] = previousOwner

Erase:
  previousOwner = erased owner
  rollback => table_[sig] = previousOwner
```

`eraseIfOwner(sig, owner)` 如果 owner 不匹配，**不能 push trail**。否则 rollback 会产生幽灵状态。

---

# 2. `refreshSignature()` 在发现 congruence merge 时不能让 app 处于“未注册签名”状态

当前流程是：

```cpp
// 1. unregister old
erase old signature
setCurrentSig(app, nullopt)

// 2. compute new sig
owner = sigTable.find(sig)

// 3. if owner exists and !same(owner, app):
enqueueCongruenceMerge(owner, app)
// DO NOT register app
return
```

这有风险。`app` 此时已经从 signature table 里移除了，`currentSig_[app] = nullopt`。如果后续 merge/backtrack/queue 处理顺序出现异常，或者该 congruence merge 因 sort mismatch / rollback 被撤销，`app` 可能长期不在 signature table 中，导致后续 congruence detection 漏掉它。

更稳的做法是：**即使发现 congruent owner，也不要先破坏 app 的旧注册，或者要保证 merge 立即处理到 fixpoint。**

建议改为两阶段：

```text
1. 先 compute newSig。
2. 查 owner。
3. 如果 owner exists 且 !same(owner, app)：
     enqueue merge(owner, app)
     保留 app 的 old currentSig，不要 unregister old signature。
     return。
4. 只有确定不需要立即合并时，才 unregister old + register new。
```

或者强规则：

```text
refreshSignature() 只允许在 processToFixpoint 内调用；
一旦 enqueueCongruenceMerge，processToFixpoint 必须立即继续处理；
任何返回前都必须让所有 app 的 currentSig 状态与 sigTable 一致。
```

现在文档没有把这个 invariant 写死。

---

# 3. `ProofForest` 名字可以，但它不应该假设 active subgraph 永远是 forest

文档写：

```text
Since edges are only added when !same(u,v), the active subgraph is a forest.
```

这个通常成立，但在 rollback + congruence merge + proof edge 添加顺序下，必须保证：

```text
只有在 UF unite 真正 merged 成功之后，才 addEdge；
already-same merge 绝不能 addEdge；
sort mismatch / failed merge 绝不能 addEdge。
```

建议把规则写成硬约束：

```cpp
auto ur = uf_.unite(...);
if (ur.merged) {
    proofForest_.addEdge(lhs, rhs, reason);
}
```

不要在 `enqueueMerge` 或 merge 尝试前加 edge。
否则 proof forest 可能出现 cycle，`path()` 的“tree path”假设不再成立。

---

# 4. `ProofForest::rollback(activeEdgeCount_)` 只改 active count，但 adjacency 仍含过期边

这可以接受，但 `path()` 必须非常明确地跳过 inactive edge：

```cpp
if (edgeId >= activeEdgeCount_) continue;
```

文档说了这一点，但还缺一个重要要求：`addEdge()` 在 rollback 后如果继续添加新边，不能只 `push_back`，否则 `edges_` 会无限保留旧 inactive edges，adjacency 也不断膨胀。不是 soundness bug，但长期 CDCL 搜索可能内存膨胀。

建议：

```cpp
void addEdge(...) {
    if (activeEdgeCount_ < edges_.size()) {
        // truncate inactive suffix and rebuild or clean adjacency
    }
    ...
}
```

但 adjacency 中删除旧边麻烦。更简单的 V1 做法：

```text
允许 edges_ 追加，但 path() 必须跳过 inactive edge；
当 inactive edge 数量超过阈值时 rebuild ProofForest adjacency。
```

至少要在计划里说明，否则长时间运行会有明显内存和 BFS 性能问题。

---

# 5. `ExplainContext` 只写了 memo，不够；还需要 active-pair guard

文档说：

```text
Add ExplainContext local memo for recursive congruence explanation.
```

但 previous blocker 是递归 explanation cycle 不能返回空原因。Memo 只能复用已完成结果，不能处理“正在解释中”的重入。

必须明确：

```cpp
struct ExplainContext {
    std::unordered_map<PairKey, ExplainResult> memo;
    std::unordered_set<PairKey> activePairs;
};
```

递归时：

```cpp
if (ctx.activePairs.contains(pair)) {
    return {false, {}};
}
ctx.activePairs.insert(pair);
auto res = explainEquality(..., ctx);
ctx.activePairs.erase(pair);
ctx.memo[pair] = res;
return res;
```

否则遇到递归解释环，仍可能无限递归或错误返回。

---

# 6. `EGraphSnapshot` 缺少 `congruenceQueueSize`

当前 snapshot 扩展是：

```cpp
struct EGraphSnapshot {
    size_t ufTrailSize;
    size_t memberTrailSize;
    size_t mergeRecordSize;
    size_t mergeQueueSize;
    size_t sigTableSnap;
    size_t currentSigSnap;
    size_t proofForestSnap;
};
```

但是 state 里仍然有：

```cpp
std::deque<EufTermId> congruenceQueue_;
```

rollback order 里直接：

```cpp
congruenceQueue_.clear();
```

这通常安全，但会丢弃 rollback 前仍然有效的 pending congruence work。由于 `merge()` 要 process to fixpoint，正常情况下 rollback snapshot 不应存在未处理 queue；但这个 invariant 必须写明。

建议二选一：

```text
方案 A：
  snapshot 时要求 mergeQueue_ 和 congruenceQueue_ 都为空；
  assert(queue empty)；
  rollback 时 clear 即可。

方案 B：
  EGraphSnapshot 记录 congruenceQueueSize；
  rollback resize 到对应大小。
```

我建议方案 A，更简单：

```cpp
EGraphSnapshot snapshot() const {
    assert(mergeQueue_.empty());
    assert(congruenceQueue_.empty());
    ...
}
```

因为 baseline 已经要求 merge/process 到 fixpoint。

---

# 7. rollback order 可能不对：signature 应在 UF rollback 之前还是之后，需要说明

当前 rollback 顺序是：

```cpp
mergeQueue_.resize(...)
congruenceQueue_.clear()

proofForest_.rollback(...)
mergeRecords_.resize(...)

rollbackCurrentSig(...)
sigTable_.rollback(...)

rollbackMembers(...)
uf_.rollback(...)
```

这个顺序大体可行，因为 `currentSig_` 保存的是签名对象，不依赖当前 UF find；但要注意：`currentSig_` 里的 signature 是旧 equivalence reps。如果先 rollback UF，再 rollback currentSig/sigTable，也可能更直观。

关键不是顺序本身，而是要保证 rollback 后：

```text
currentSig_[app] == computeSignature(app) for every registered app
sigTable_ maps each currentSig to a valid owner
```

建议在 debug 模式加 invariant 检查：

```cpp
#ifndef NDEBUG
checkSignatureTableInvariant();
#endif
```

这个优化改动很容易出 rollback 细节 bug，必须用 invariant 防住。

---

# 8. `refreshSignature()` 的 stale owner 处理不够强

当前逻辑：

```cpp
if (!currentSig_[owner] || *currentSig_[owner] != sig) {
    sigTable_.insertOrAssign(sig, app);
    setCurrentSig(app, sig);
    return;
}
```

这里如果 stale owner 仍然在 table 里，`insertOrAssign(sig, app)` 会 overwrite。可以，但需要确认 rollback trail 会记录 old owner，且不会破坏 owner 自己的 `currentSig_`。此时 owner 的 `currentSig_` 可能仍然是别的 sig 或 null，这可以。

建议加一句 invariant：

```text
sigTable_ is authoritative for current signature owners;
currentSig_[t] is authoritative for whether t currently owns a signature.
A stale table owner may be overwritten without changing currentSig_[oldOwner].
```

否则实现时容易试图同时修改 oldOwner 的 currentSig，反而错。

---

# 9. 缺少对 app 被 merge 后旧 signatures 的刷新策略

优化方案依赖 `refreshSignature(app)`。但什么时候调用很关键。仅刷新 merged class 的 parents 可能不够，必须明确：

```text
当两个 classes 合并后：
  collect parents of all members in winner/loser affected classes；
  对这些 parent application 调用 refreshSignature(parent)；
  refresh 可能 enqueue congruence merge；
  processToFixpoint 直到没有新 merge。
```

文档说增加 `currentSig_` 和 `refreshSignature`，但没有明确 merge 后对 affected parents 的调用策略。建议补：

```cpp
void onClassesMerged(winner, loser) {
    for (auto t : members_[winner]) {
        for (auto p : tm_.parentsOf(t)) {
            refreshSignature(p);
        }
    }
}
```

为了效率可以只扫 smaller/changed class 的 parents，但正确性上要覆盖所有受影响 app。否则 signature table 再漂亮也不会触发必要 congruence merge。

---

# 最小修改后可以接受

把这版改成下面几条，我认为下一版就可以接受：

```text
1. RollbackSignatureTable 明确 InsertNew/Overwrite/Erase 的 rollback 逆操作。
2. refreshSignature 不得让 app 长期处于 unregistered 状态；发现 congruent owner 时要么保留旧注册，要么强制 processToFixpoint 保证一致。
3. ProofForest 只在 UF merge 真正成功后 addEdge。
4. ProofForest rollback 后 inactive edges 要么可跳过且有 compaction 策略，要么 rollback 时重建 adjacency。
5. ExplainContext 必须包含 memo + activePairs；active pair 重入返回 ok=false。
6. snapshot 时 assert mergeQueue/congruenceQueue 为空，或者记录 congruenceQueueSize。
7. 增加 debug invariant：currentSig / sigTable / UF reps 一致。
8. 明确每次 class merge 后刷新 affected parent applications，并 process to fixpoint。
```

# 最终意见

**不接受当前版本。**

不是因为方向错，而是这个优化触碰了 signature table rollback 和 explanation proof structure，属于很容易引入隐蔽 unsoundness 的部分。当前文档对 rollback 逆操作、`refreshSignature()` 一致性、ProofForest edge 生命周期、递归 explanation cycle guard、以及 parent refresh 触发时机写得还不够硬。修完上述 8 点后再实现。
