# Plan: Sparse Indexed Tableau for GeneralSimplex

## Goal

Replace the dense `std::vector<std::vector<mpq_class>> matrix_` in `GeneralSimplex` with a **sparse indexed explicit tableau** (bidirectional row/column cross-linked storage). This is the correct data structure direction for SMT LRA/LIA solvers where tableaux are typically very sparse and `mpq_class` is expensive.

## Current State

- `GeneralSimplex` uses dense `m x n` matrix, scans all rows/cols on every operation.
- `update()` scans all rows; `findEnteringVar*` scans all columns; `pivot` scans full matrix.
- 14 unit tests pass (55 assertions), including CDCL(T) LRA sat/unsat.
- Public API (`addVar`, `addConstraint`, `assertLower`, `assertUpper`, `check`, `value`, `getConflict`, `push`, `pop`, `reset`) must remain stable — `SimplexSolver` and tests depend on it.

## Design Overview

Replace the internal storage with:

```
SparseRow:   basicVar, rhs, vector<RowEntry{c,colPos}>, optional col->pos index
SparseCol:   vector<ColEntry{row,rowPos}>
VarState:    name, basicRow(-1=nonbasic), lower, upper, beta
```

Row/column entries are **cross-linked**: each `RowEntry` stores its position in the column's entry list, and each `ColEntry` stores its position in the row's entry list. This enables O(1) `eraseCoeff` via swap-pop with handle repair.

## Implementation Steps

### Step 1: Add `SparseTableau` internal struct and helpers

In `GeneralSimplex.h`, replace dense members with:

```cpp
struct RowEntry { int col; mpq_class coeff; int colPos; };
struct ColEntry { int row; int rowPos; };

struct SparseRow {
    int basicVar = -1;
    mpq_class rhs = 0;
    std::vector<RowEntry> entries;
    bool hasIndex = false;
    std::unordered_map<int, int> pos; // col -> index in entries
    int version = 0;
};

struct SparseCol {
    std::vector<ColEntry> entries;
};
```

And:

```cpp
std::vector<SparseRow> rows_;
std::vector<SparseCol> cols_;
std::vector<VarState> vars_;        // replaces vars_ + isBasic_ + lower_ + upper_ + beta_
std::vector<int> basicVars_;        // row -> basic var
std::vector<int> nonBasicVars_;     // O(1) non-basic set
std::vector<int> nonBasicPos_;      // var -> index in nonBasicVars_, -1 if basic
std::deque<int> violatedQueue_;
std::vector<bool> inViolationQueue_;
```

**No `std::vector<bool> isBasic_`**. Use `vars_[v].basicRow == -1`.

### Step 2: Implement `SparseTableau` core primitives

In `GeneralSimplex.cpp` (private section), implement:

- `findRowPos(SparseRow&, int col)` — linear scan or hash index lookup.
- `maybeBuildRowIndex(SparseRow&)` — build hash when row exceeds threshold (e.g. 32).
- `getCoeff(row, col)`, `setCoeff(row, col, value)`, `addCoeff(row, col, delta)`, `eraseCoeff(row, col)`.
- `replaceRow(row, basicVar, rhs, entries)` — clear old entries, insert new ones.

`eraseCoeff` uses swap-pop on both row and column vectors, repairing cross-handles of swapped elements. This is the most error-prone part; add `checkInvariants()` (debug-only) after every pivot.

### Step 3: Implement `rewriteToNonBasic`

When adding a new constraint after pivots have occurred, some variables in the linear form may already be basic. The new row must only contain non-basic variables. Implement:

```cpp
LinearForm rewriteToNonBasic(const LinearForm& input);
```

This substitutes each basic variable `v` with its row expression (`rhs + Σ coeff * nonbasic`). This preserves the tableau invariant and enables dynamic constraint registration.

### Step 4: Rewrite `addVar`, `addConstraint`, `recomputeBeta`, `update`

- `addVar`: create `VarState`, grow `cols_` vector (new column starts empty), update `nonBasicVars_`.
- `addConstraint`: create aux var, call `rewriteToNonBasic`, add sparse row with non-zero entries only.
- `recomputeBeta`: iterate `nonBasicVars_` for assignments, then iterate each row's `entries` (not all columns) to compute basic values.
- `update(nonBasicVar, value)`: iterate `cols_[nonBasicVar].entries` (affected rows only), update basic betas. Complexity: O(nnz in column) instead of O(m).

### Step 5: Rewrite `findEnteringVarToIncrease/Decrease` and `pivotAndUpdate`

- `findEnteringVarToIncrease`: scan `rows_[r].entries` only (non-basic columns with non-zero coefficients).
- `pivotAndUpdate`: compute theta, update betas via column snapshot of entering var, then call sparse `pivot`.
- `pivot`: copy old pivot row, build new pivot row entries, take snapshot of entering column, update all affected rows (only rows containing the entering variable), then replace pivot row. Drop exact zeros after every arithmetic operation.

### Step 6: Add violation queue

Replace `findViolatedBasicVar()` (linear scan of all basics) with:
- `refreshViolationStatus(x)` — add/remove from queue when beta changes.
- `pickViolatedBasic()` — dequeue, verify still violated (may have been fixed by previous pivot).
- `rebuildViolationQueue()` — called after `recomputeBeta()`.

### Step 7: Rewrite conflict explanation

`explainLowerConflict` and `explainUpperConflict` scan `rows_[r].entries` instead of all columns.
`explainImmediateConflict` stays the same (just two bounds on the same variable).

### Step 8: Update `push`, `pop`, `resetActiveBounds`, `reset`

Adapt to `VarState` structure. `resetActiveBounds` clears bounds and marks `betaDirty_`.

### Step 9: Preserve public API compatibility

The following must keep identical signatures and semantics:

```cpp
int addVar(const std::string& name);
int addConstraint(const std::vector<std::pair<int, mpq_class>>& terms, const mpq_class& rhs);
bool assertLower(int var, const BoundInfo& info);
bool assertUpper(int var, const BoundInfo& info);
Result check();
DeltaRational value(int var) const;
const std::vector<BoundReason>& getConflict() const;
void push();
void pop();
void resetActiveBounds();
void reset();
int numVars() const;
int numRows() const;
bool isBasic(int var) const;
```

### Step 10: Testing

1. Run existing 14 unit tests — all must pass.
2. Add a sparse-specific stress test: many variables, few constraints per variable, verify correctness against a reference dense solver or LP feasibility checker.
3. Add invariant checks (debug builds) after every pivot.
4. Run the CDCL(T) end-to-end tests.

## Risks & Mitigations

| Risk | Mitigation |
|------|------------|
| Cross-link handle corruption in `eraseCoeff` | `checkInvariants()` debug assert after every pivot; swap-pop logic reviewed carefully |
| Fill-in causing worse performance than dense for small problems | Only relevant for very small/dense problems; SMT tableaux are typically sparse. Threshold-based row index handles lookup cost. |
| `rewriteToNonBasic` introducing numerical cancellation | Exact rationals prevent rounding; cancellation is exact. Drop zero coefficients aggressively. |
| Bland's rule + sparsity causing cycling on degenerate inputs | Use exact rationals (no floating point); Bland's rule guarantees termination for LRA. |
| Regression in existing tests | Run full test suite after each step; do not proceed if tests fail. |
