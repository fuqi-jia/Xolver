**结论：不接受当前版本。**

这版已经比上一版强很多：它明确把范围收缩为 pure `QF_UF`，mixed theories 无 fallback、无 Nelson-Oppen；也把 Bool 二值语义、registered atoms propagation、lemma-level test、activePairs 递归保护都写进去了。文档里的 invariant 已经基本正确，尤其 I1–I13 很清楚。

但我仍然不建议直接实现，因为还有一个核心问题没有闭合：**Bool 二值语义和 model construction 之间没有打通**。此外，propagation 的实现顺序和 assignment 查询也需要再收紧。

# 1. 最大问题：`distinct p true` 的模型仍然可能构造不出来

文档现在说：

```text
(distinct p true) → SAT-level constraint forces p=false
```

这是对的。Bool equality / distinct 不应作为普通 EUF atom，而应命题化处理；文档已经明确要求 Bool equality / distinct 走 SAT encoding，而不是注册 EUF atom。

但是 Phase 3 的 model builder 又写：

```text
If sort is Bool:
  If class contains trueConstant → "true"
  If class contains falseConstant → "false"
  Otherwise → error
```

这会和 `distinct p true` 冲突。

例如：

```smt2
(set-logic QF_UF)
(declare-const p Bool)
(assert (distinct p true))
(check-sat)
(get-model)
```

按照 Phase 2，`distinct p true` 会在 SAT 层强制 `p` 的 Bool atom 为 false。
但在 EGraph 里，`p` 不一定会被 merge 到 `falseConstant`。如果当前实现只是把 `p=true` 的 SAT literal 赋 false 解释成 EUF disequality `p != true`，那么 EGraph 里 `p` 的 class 既不包含 `true`，也不包含 `false`。此时 Phase 3 的 buildModel 会返回 `ok=false`，但测试又要求 model 里 `p=false`。文档的测试计划也明确写了 `distinct(p,true)` 应该 SAT，model 为 `p=false`。

所以当前方案在 Bool model 上还没闭环。

## 建议修改

你需要明确增加一个机制：**Bool term 的 SAT assignment 要能反馈到 model construction。**

推荐做法是：

```text
对 Bool term atom p_bool := (= t true)：

如果 p_bool assigned true：
    model 中 t = true；
    EGraph check 阶段也可以 merge(t, true)。

如果 p_bool assigned false：
    model 中 t = false；
    EGraph check 阶段应把它视为 Bool-domain consequence t = false，而不是只记录 t != true。
```

也就是说，对 Bool formula-position atom：

```text
raw atom: (= t true)

value = true   => merge(t, true)
value = false  => merge(t, false)
```

这是 sound 的，因为 Bool 只有两个值。

这样 `distinct p true` 命题化后强制 `p_bool=false`，EUF/model 层就能得到：

```text
p = false
```

而不是只得到：

```text
p != true
```

这会直接修掉 model construction 的缺口。

如果你不想在 EGraph 中 merge Bool false，也可以让 model builder 接收 SAT assignment：

```cpp
struct BoolTermAssignment {
    ExprId boolTerm;
    bool value;
};

BuildModelResult buildModel(
    const EufTermManager& tm,
    std::span<const BoolTermAssignment> boolAssignments);
```

但我更建议第一种：**Bool formula-position atom false 直接转成 `t=false` 的 equality merge**。这更简单，也让 EGraph 状态和模型一致。

# 2. Bool equality / distinct 命题化后，要明确“Bool atom”到底是什么

文档写：

```text
Each bi is first atomized as a Bool atom (its own SAT variable).
```

这里容易歧义。对于 pure Bool variable / Bool-valued UF term，`bi` 的 SAT variable 应该仍然对应：

```text
bi_true := (= bi true)
```

也就是说：

```text
p          → SAT atom representing (= p true)
(p a)      → SAT atom representing (= (p a) true)
```

然后：

```text
(= p false)       → ¬p_true
(distinct p true) → ¬p_true
(distinct p false)→ p_true
(= p q)           → p_true ↔ q_true
(distinct p q)    → p_true XOR q_true
```

建议在文档里把这一层写死，避免实现模型把 Bool variable 直接当普通 propositional symbol，导致 EUF model 无法知道它对应哪个 term。

可以加一节：

```text
Bool atom convention:

For every Bool-valued term b, Atomizer creates or reuses a canonical SAT atom:
    lit(b) := (= b true)

This atom is also recorded as a EUF Bool-term atom, so EufSolver can reconstruct model values:
    lit(b) = true  => b = true
    lit(b) = false => b = false
```

这个约定非常关键。

---

# 3. model construction 的 Bool class error 规则要改

现在写：

```text
Bool class without true/false means Bool constraints were incomplete; return ok=false.
```

这条太强。比如：

```smt2
(set-logic QF_UF)
(declare-const p Bool)
(check-sat)
(get-model)
```

如果 `p` 没在任何 assertion 里出现，模型完全可以任选：

```text
p = false
```

所以 Bool class 不包含 true/false 不一定是错误。它可能只是 unconstrained Bool term。

建议改成：

```text
Bool class value assignment:
1. class(true)  => true
2. class(false) => false
3. class with SAT Bool assignment true  => true
4. class with SAT Bool assignment false => false
5. otherwise choose false by default
```

然后补冲突检查：

```text
If one Bool class is forced to both true and false => model/build error;
but this should already have been caught by EufSolver via true != false or SAT clauses.
```

这样 model construction 才是 total 的，而不是动不动返回 `ok=false`。

---

# 4. propagation 最好放在 conflict checks 之后

文档写 Phase 4 propagation detection 在 `closeCongruence()` 后执行。
但根据你前面的 invariant，`check()` 正确顺序是：

```text
intern terms
addTerm
assert equalities
closeCongruence
check true≠false
check disequalities
```

propagation 应该放在 **conflict checks 之后**，否则在已经 UNSAT 的情况下可能先返回一个普通 lemma，而不是直接返回 conflict。虽然返回 valid lemma 也不一定 unsound，但会降低 CDCL 行为质量，甚至可能造成重复 lemma。

建议固定为：

```text
1. assert equalities
2. closeCongruence
3. check true != false conflict
4. check asserted disequality conflict
5. then propagation over all registered equality atoms
6. consistent
```

也就是：

```cpp
auto cs = egraph_.closeCongruence(termManager_);
if (cs != CloseStatus::Ok) return unknown();

if (trueFalseConflict()) return conflict(...);

for (const auto& neq : disequalities_) {
    if (egraph_.same(neq.lhs, neq.rhs)) {
        return conflict(...);
    }
}

return tryPropagateRegisteredEqualities();
```

---

# 5. propagation 里的 SAT assignment 查询不应该是 optional

文档里写：

```cpp
// Optional: if SAT assignment query available:
//   if assigned true → skip
//   if assigned false → this is actually conflict
```

这不应该是 optional。否则可能出现两个问题：

```text
1. 对已经 assigned true 的 atom 反复返回已满足 lemma；
2. 对 assigned false 的 atom 不直接 conflict，只返回一个普通 lemma，CDCL 行为变差。
```

建议改成强制：

```cpp
auto val = assignment.value(atomLit);

if (val == LBool::True) {
    continue;
}

if (val == LBool::False) {
    // This is conflict:
    // reasons imply atomLit, but SAT assigned ¬atomLit.
    clause = negated reasons + atomLit;
    return conflict_or_lemma_as_conflict(clause);
}

if (val == LBool::Undef) {
    clause = negated reasons + atomLit;
    return propagationLemma(clause);
}
```

如果当前 TheorySolver 接口拿不到 SAT assignment，那这版不应该实现 propagation，只应保留接口 TODO。否则很容易产生重复 lemma 或弱 conflict。

# 6. propagation lemma 要加缓存，避免重复生成

假设 `a=b` 已经推出 `f(a)=f(b)`，atom `p := (= (f a) (f b))` 仍未赋值。EUF 每次 `check()` 都可能返回同一个 lemma：

```text
¬ab ∨ p
```

SAT solver 如果没有处理 repeated lemma 的保护，可能反复加同一条。

建议在 `EufSolver` 加：

```cpp
std::unordered_set<SatVar> propagatedEqAtoms_;
```

或者更精确：

```cpp
std::unordered_set<ClauseKey> emittedTheoryLemmas_;
```

最低限度：

```cpp
if (propagatedEqAtoms_.contains(rec.satVar)) continue;
...
propagatedEqAtoms_.insert(rec.satVar);
```

在 backtrack 时是否清空，取决于 SAT assignment 和 lemma 是否永久加入 SAT solver。通常 theory lemma 是永久 clause，已经加入后不需要重复生成。

# 7. Phase 2 Bool rewrite 需要明确 SAT encoding 细节

建议不要只写“via SAT encoding”，直接给出 clauses，避免实现时出错。

对于 `x ↔ y`：

```text
(¬x ∨ y) ∧ (x ∨ ¬y)
```

对于 `x XOR y`：

```text
(x ∨ y) ∧ (¬x ∨ ¬y)
```

对于 n-ary Bool equality：

```text
(= b1 b2 ... bn)
=> ∧_{i=1}^{n-1} (bi ↔ b_{i+1})
```

对于 Bool distinct：

```text
n = 0 or 1 => true
n = 2      => XOR
n >= 3     => false
```

这里要特别注意：`distinct` 的一般 EUF 规则是 pairwise disequality，但 Bool 域大小只有 2，所以 `n>=3` 直接 false，不应该生成 pairwise XOR 后交给 SAT 自己推；虽然 pairwise XOR 也能推出 UNSAT，但直接 false 更清晰。

# 8. Function default value 对 Bool result sort 不能是 fresh value

Phase 3 写：

```text
defaultValue for f is a fresh value of f's result sort.
```

如果 `f : U -> Bool`，result sort 是 Bool，default value 只能是：

```text
true or false
```

不能是：

```text
Bool!val_0
```

建议改成：

```text
defaultValue:
  result sort Bool => choose false by default
  result sort uninterpreted S => choose S!val_default_N
```

同理，function entries 的 `resultValue` 也必须遵守 result sort。

# 9. `buildModel()` 需要知道每个 class 的 sort，不能靠第一个 term 猜得太随意

Phase 3 的数据结构已经把 `sort` 放进 `EufClassValue`，这是对的。
但实现时要明确：

```text
class sort = sort of any member term;
all member terms must have same sort;
if not, this is EGraph sort bug.
```

建议加检查：

```cpp
SortId classSort = NullSort;
for (auto t : membersOrAllTermsInClass) {
    SortId s = tm.node(t).sort;
    if (classSort == NullSort) classSort = s;
    else if (classSort != s) return {false, {}, "sort-mixed equivalence class"};
}
```

如果 `EClass::members` 没维护，就用 `termClass_` 遍历所有 term，按 root 聚类。

# 10. mixed theory TODO 的匹配条件不要写得太宽

文档建议：

```cpp
} else if (logic starts with "QF_UF" or contains "UF") {
    ...
    return Result::Unknown;
}
```

这个逻辑如果只是注释和 return Unknown，问题不大。但 `contains("UF")` 可能误伤某些未来 logic 名称或自定义字符串。

建议显式列出：

```cpp
if (logic == "QF_UFLIA" ||
    logic == "QF_UFLRA" ||
    logic == "QF_UFNIA" ||
    logic == "QF_UFNRA") {
    return Result::Unknown;
}
```

`QF_UF` 则走 EUF solver。未知 logic 走统一 unsupported/unknown，不要靠 substring 判断。

# 11. 最终建议如何修改文档

我建议你把这版改成下面几个关键补丁。

## 增加 Bool atom convention

```text
For every Bool-valued term b, Atomizer creates a canonical SAT atom:
    lit(b) := (= b true)

For model construction and EUF consistency:
    lit(b) assigned true  => b = true
    lit(b) assigned false => b = false

Bool equality/distinct are encoded propositionally over lit(b).
They are not registered as ordinary EUF equality/disequality atoms.
```

## 修改 model builder 规则

```text
Bool class assignment:
  - true class => true
  - false class => false
  - class forced by Bool SAT assignment => assigned value
  - otherwise choose false
```

不要再写“Bool class without true/false is error”。

## 修改 propagation 规则

```text
Propagation runs after all conflict checks.
It requires SAT assignment query.
Already true atom: skip.
False atom: return conflict clause.
Unassigned atom: return propagation lemma.
Cache emitted propagation lemmas.
```

## 修改 function default

```text
If result sort is Bool, defaultValue = false.
If result sort is uninterpreted, defaultValue = S!val_default_N.
```
