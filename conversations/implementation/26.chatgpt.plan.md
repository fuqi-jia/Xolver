建议：**保留你现在的 `CdcacSolver` 外壳，但不要继续在这个 `check()` 里打补丁；应当重写为“CDCAC 外壳 + 精确代数内核 + covering 搜索核心”三层结构。** 你现在这版只能算 NRA 常数检查器，作为 trail/facade 可以，但 CDCAC 核心还没有开始。

CDCAC 本身适合作为 SMT 中 NRA 的 theory validation/checking 过程；cvc5 的相关系统描述也是把 cylindrical algebraic coverings 与 incremental linearization 放在 abstraction-refinement 框架里配合使用，而不是孤立使用一个纯代数过程。([理论斯坦福][1]) cvc5 官方选项里也能看到 nonlinear covering solver、projection operator、lifting mode、linear model seed、pruning、变量消元等都是独立可调模块，这说明工程上最好模块化实现，而不是把 libpoly 调用散落在 `check()` 里。([cvc5][2])

## 1. 当前代码的问题

你现在的类：

```cpp
class CdcacSolver {
    std::vector<ActiveConstraint> active_;
    std::vector<TrailEntry> trail_;
    std::optional<PendingConflict> pendingConflict_;
    std::optional<PendingUnknown> pendingUnknown_;
};
```

这个结构可以保留，但目前有几个核心问题：

1. **`check()` 对非线性约束直接 `unknown`**
   这不是 CDCAC。真正的 CDCAC 应该尝试构造 sample point，若 sample 不满足，就把当前点推广成一个不可满足 cell/interval，再递归形成 covering，最后生成 conflict clause。

2. **没有 sample/model 表示**
   CDCAC 必须维护形如 `(x1 = a1, ..., xk = ak)` 的 partial sample。`ai` 不能只用 `mpq_class`，因为可能需要代数数，即某个单变量多项式的第几个实根。

3. **没有 cell / interval / section 表示**
   CAD/CAC 的基本对象不是单个点，而是 sector、section、interval、point cell。没有 cell，就无法做“从一个失败点推广到一片区域”。

4. **没有 projection/lifting 机制**
   CDCAC 需要在变量层级之间传播失败区域。至少要有投影多项式、根隔离、符号不变区间、nullification 处理。

5. **冲突解释过弱**
   现在冲突只来自常数约束。真正 CDCAC 的冲突应来自一组 active constraints 的 covering labels，最终返回：

```cpp
TheoryConflict{ reason_1.negated(), reason_2.negated(), ... }
```

6. **`pendingUnknown_` 设计意义不大**
   `unknown` 通常是资源耗尽、代数后端失败、超出支持特性，不应该长期挂在 solver 内部。active set 变化后，旧 unknown 不一定仍然有意义。

## 2. 推荐总体架构

建议拆成 5 个模块：

```text
NraSolver
  |
  | assert literal / backtrack / check
  v
CdcacSolver                // CDCL(T) facade: trail, active constraints, result conversion
  |
  v
CdcacCore                  // recursive covering/search algorithm
  |
  +-- AlgebraKernel         // exact polynomial / root / sign operations
  +-- ProjectionEngine      // McCallum/Lazard projection, lifting guards
  +-- CoveringManager       // cells, intervals, covering union, sample selection
  +-- ReasonManager         // labels, conflict minimization, reason set covering
```

`libpoly` 不应该直接出现在 `CdcacSolver` 和 `CdcacCore` 的主要逻辑中。你可以把它封装成一个 backend：

```cpp
class AlgebraBackend {
public:
    virtual RootSet isolateRealRoots(const UniPoly& p) = 0;
    virtual Sign signAt(const PolyRef& p, const SamplePoint& s) = 0;
    virtual PolySet projectionPolys(const std::vector<PolyRef>& ps, Var x) = 0;
    virtual bool checkRootIsolation(const UniPoly& p, const RootSet& roots) = 0;
};
```

然后：

```cpp
class LibpolyBackend final : public AlgebraBackend { ... };
class NativeExactBackend final : public AlgebraBackend { ... };
class CheckedBackend final : public AlgebraBackend {
    AlgebraBackend* fast_;
    AlgebraBackend* slow_or_validator_;
};
```

这样 libpoly 出问题时不会污染主算法。原则是：**libpoly 可以做 fast oracle，但不能做唯一真相来源。**

## 3. 核心数据结构建议

### 3.1 active constraints

你现在的 `ActiveConstraint` 太薄，建议扩成：

```cpp
struct ActiveConstraint {
    ConstraintId id;
    PolyId poly;          // normalized p such that p rel 0
    Relation rel;         // Eq, Neq, Lt, Leq, Gt, Geq
    SatLit reason;        // positive SAT literal currently assigned true
    int level;
    uint64_t epoch;
};
```

注意：`reason` 存 assigned literal 本身，不要提前取反。只有生成 conflict clause 时才取反。

---

### 3.2 代数数与 sample point

```cpp
enum class RealKind {
    Rational,
    AlgebraicRoot
};

struct AlgebraicRoot {
    PolyId definingPoly;      // univariate polynomial after specialization
    int rootIndex;            // ordered real root index
    mpq_class lower;
    mpq_class upper;
    // optional: Thom encoding for robust comparison/sign
};

struct RealAlg {
    RealKind kind;
    mpq_class rational;
    AlgebraicRoot root;
};

struct SamplePoint {
    std::vector<RealAlg> values;   // x_0, ..., x_k
};
```

第一版可以先大量使用 rational sample；但如果要支持 section，即变量等于某个 projection polynomial 的根，最终必须支持 algebraic sample。

---

### 3.3 cell / covering

```cpp
enum class CellKind {
    Sector,     // open interval between roots
    Section,    // exactly one root
    Point,
    FullLine
};

struct Bound {
    enum class Kind { NegInf, PosInf, Root, Rational };
    Kind kind;
    RealAlg value;
    bool open;
};

struct Cell {
    Var var;
    CellKind kind;
    Bound lower;
    Bound upper;

    // constraints whose conjunction cannot be extended over this cell
    std::vector<SatLit> reasons;

    // projection polynomials guaranteeing sign-invariance/delineability
    std::vector<PolyId> guards;
};

struct Covering {
    Var var;
    std::vector<Cell> cells;

    bool coversAllLine() const;
    std::optional<RealAlg> chooseSampleOutside(const RealAlg* preferred) const;
};
```

CDCAC 的关键不是“发现一个点不行”，而是“把这个点推广成一个带 reason label 的 cell，然后用若干 cell 覆盖整条线或整个 cylinder”。

## 4. `check()` 应该变成什么

建议接口改成：

```cpp
TheoryCheckResult CdcacSolver::check(TheoryEffort effort,
                                     const ModelSeed* seed = nullptr);
```

其中：

```cpp
enum class TheoryEffort {
    Cheap,
    Standard,
    Full
};
```

行为建议：

```cpp
TheoryCheckResult CdcacSolver::check(TheoryEffort effort,
                                     const ModelSeed* seed) {
    auto trivial = checkConstantsAndDuplicates();
    if (trivial.isConflict()) return trivial;

    if (effort != TheoryEffort::Full) {
        return TheoryCheckResult::consistent(); // 或 defer
    }

    CdcacInput input = buildInputFromActive(seed);
    CdcacResult res = core_.solve(input);

    switch (res.status) {
        case CdcacStatus::Sat:
            model_ = res.model;
            return TheoryCheckResult::consistent();

        case CdcacStatus::Unsat: {
            std::vector<SatLit> clause;
            for (SatLit lit : res.reasons) {
                clause.push_back(lit.negated());
            }
            return TheoryCheckResult::mkConflict(TheoryConflict{clause});
        }

        case CdcacStatus::Unknown:
            return TheoryCheckResult::unknown();
    }
}
```

关键点：**不要因为存在非线性约束就 unknown。** 只有后端失败、资源限制、当前实现不支持的 algebraic case，才 unknown。

## 5. CDCAC 主算法

核心递归可以按这个形态实现：

```cpp
CdcacResult solveLevel(int k, SamplePoint& prefix) {
    if (k == numVars()) {
        return checkFullSample(prefix);
    }

    Covering cover(k);

    while (true) {
        auto sample = cover.chooseSampleOutside(preferredSample(k));
        if (!sample) {
            auto reasons = reasonMinimizer_.minimize(cover);
            return CdcacResult::unsat(reasons);
        }

        prefix.push(*sample);
        auto child = solveLevel(k + 1, prefix);
        prefix.pop();

        if (child.status == CdcacStatus::Sat) {
            return child;
        }

        if (child.status == CdcacStatus::Unknown) {
            return child;
        }

        Cell generalized = generalizeConflictCell(k, *sample, child);
        cover.add(generalized);
        cover.normalizeAndMerge();
    }
}
```

含义是：

1. 在第 `k` 个变量上维护一个 covering。
2. 找一个还没被 cover 的 sample。
3. 递归检查后续变量。
4. 如果子问题 sat，整体 sat。
5. 如果子问题 unsat，把该 sample 附近推广成一个 cell。
6. 如果所有 sample 都被 cover，则本层 unsat，并从 covering labels 提取 conflict reasons。

## 6. full sample 失败时如何生成 cell

当得到完整 sample：

```text
x1 = a1, ..., xn = an
```

检查所有 active constraints：

```cpp
bool holds(const ActiveConstraint& c, const SamplePoint& s) {
    Sign sign = algebra_.signAt(c.poly, s);
    return relationHolds(sign, c.rel);
}
```

若某个约束 `p rel 0` 不满足，则要构造一个局部 conflict cell。例子：

```text
p(x1,...,xn) < 0
```

在当前 sample 上如果 `p >= 0`，那么需要找到一个关于 `xn` 的区间或 section，使得在该 cell 内 `p < 0` 恒为假。做法：

1. 固定 prefix `(x1,...,x_{n-1})`。
2. 得到单变量多项式 `p(an-prefix, xn)`。
3. 隔离它的实根。
4. 找到包含当前 `an` 的 sector 或 section。
5. 因为该 cell 内符号不变，所以这个 cell 被当前约束排除。
6. label 为 `{c.reason}`。

伪代码：

```cpp
Cell CdcacCore::buildLeafConflictCell(const ActiveConstraint& c,
                                      const SamplePoint& s,
                                      int var) {
    UniPoly up = algebra_.specializeToUnivariate(c.poly, s.prefix(var), var);
    RootSet roots = algebra_.isolateRealRoots(up);

    Cell cell = cellContaining(s[var], roots);
    cell.reasons = {c.reason};
    cell.guards = {c.poly};

    return cell;
}
```

## 7. 从子层 conflict 推广到父层 cell

这是 CDCAC 最核心的地方。

如果在 `x_{k+1}` 层得到一个 covering，说明对于当前 `x_k = a_k`，后面的变量没有可行扩展。现在要把 `a_k` 推广成一个关于 `x_k` 的 cell，使得在这个 cell 内，子层 covering 的结构仍然有效。

需要 projection polynomials。第一版可以实现 McCallum 风格，之后加 Lazard。cvc5 里也把 projection operator 作为选项暴露，包括 McCallum、Lazard 和 Lazard-mod；lifting 也有 regular 与 Lazard 选项。([cvc5][2])

建议接口：

```cpp
Cell CdcacCore::generalizeConflictCell(int k,
                                       const RealAlg& sample,
                                       const CdcacResult& childConflict) {
    std::vector<PolyId> polys = collectGuardPolys(childConflict.cover);
    std::vector<PolyId> proj = projection_.project(polys, variable(k + 1), variable(k));

    UniRootSet roots = algebra_.rootsAtPrefix(proj, currentPrefix, variable(k));
    Cell parentCell = cellContaining(sample, roots);

    parentCell.reasons = reasonMinimizer_.minimize(childConflict.cover);
    parentCell.guards = proj;

    return parentCell;
}
```

简化理解：

```text
子层 unsat covering
        ↓ collect guards
投影到父变量
        ↓ roots split line
找到当前 sample 所在父 cell
        ↓
该父 cell 内子层 covering 稳定，因此整个 cell 不可扩展
```

## 8. relation 处理细节

所有约束统一成：

```text
p ⋈ 0
```

其中 `⋈ ∈ {=, ≠, <, ≤, >, ≥}`。

判断逻辑：

```cpp
bool relationHolds(Sign s, Relation rel) {
    switch (rel) {
        case Relation::Eq:  return s == Sign::Zero;
        case Relation::Neq: return s != Sign::Zero;
        case Relation::Lt:  return s == Sign::Neg;
        case Relation::Leq: return s == Sign::Neg || s == Sign::Zero;
        case Relation::Gt:  return s == Sign::Pos;
        case Relation::Geq: return s == Sign::Pos || s == Sign::Zero;
    }
}
```

注意几个坑：

`p != 0` 失败只发生在 section 上，也就是 `p = 0` 的根 cell。不能把它误推广成一个开区间。

`p < 0` 在根点处是 false，如果 sample 落在 root section，生成的 conflict cell 可以是 section。

`p <= 0` 失败区域是 `p > 0` 的 sector，不能把根点包括进去。

`p = 0` 失败区域通常是某个不含根的 sector；如果 sample 不在根上，整个 sector 都可排除。

## 9. ProjectionEngine 设计

建议先做一个保守版本：

```cpp
class ProjectionEngine {
public:
    ProjectionSet project(const std::vector<PolyId>& polys,
                          Var eliminate,
                          ProjectionMode mode);
};

enum class ProjectionMode {
    McCallum,
    Lazard,
    Conservative
};
```

第一版 `Conservative` 可以包含：

1. 每个多项式关于主变量的 coefficients；
2. discriminant；
3. pairwise resultants；
4. squarefree basis；
5. content / primitive part；
6. degree drop / nullification 检查用的 guard。

伪代码：

```cpp
ProjectionSet ProjectionEngine::projectConservative(const std::vector<PolyId>& ps,
                                                    Var x) {
    ProjectionSet out;

    for (PolyId p : ps) {
        out.addAll(kernel_.coefficients(p, x));
        out.add(kernel_.discriminant(p, x));
        out.addAll(kernel_.squareFreeFactors(p));
    }

    for (PolyId p : ps) {
        for (PolyId q : ps) {
            if (p < q) {
                out.add(kernel_.resultant(p, q, x));
            }
        }
    }

    out.removeConstants();
    out.normalize();
    return out;
}
```

如果你担心 libpoly 的 resultant/discriminant 有坑，第一阶段可以先支持低度多项式，并对结果做 validation。不要一上来追求全量 SMT-LIB。

## 10. libpoly 的使用策略

你说 libpoly bug 多，我建议这样处理：

### 不要让 libpoly 进入核心数据结构

不要出现这种情况：

```cpp
struct Cell {
    lp_algebraic_number_t* root;
};
```

而是只在 backend 内部用 libpoly：

```cpp
class LibpolyBackend {
    RootSet isolateRealRoots(const UniPoly& p) override {
        // call libpoly
        // convert to your RootSet
        // validate
    }
};
```

### 对 libpoly 输出做自检

每次 root isolation 后做：

1. 每个 isolating interval 内 Sturm count 必须为 1；
2. 相邻 root intervals 不重叠；
3. interval endpoint 不能是根，除非显式允许；
4. sample rational 的符号用你自己的 GMP rational evaluator 复查；
5. 若 libpoly 返回 algebraic sign，随机用 interval refinement 交叉验证；
6. 失败时 `Unknown`，不能产生 `Sat` 或 `Unsat`。

也就是说：

```cpp
RootSet CheckedBackend::isolateRealRoots(const UniPoly& p) {
    RootSet roots = fast_->isolateRealRoots(p);

    if (!validator_->checkRootIsolation(p, roots)) {
        if (options_.fallbackToNative) {
            return slow_->isolateRealRoots(p);
        }
        throw AlgebraFailure("invalid root isolation");
    }

    return roots;
}
```

### 自己先实现 univariate exact core

强烈建议你自己实现一套最小 univariate exact algebra：

```text
primitive part
content
squarefree factorization
derivative
pseudo-division
Sturm sequence
Cauchy bound
root count on interval
dyadic interval refinement
rational sign evaluation
```

这部分工作量不算夸张，但能极大降低 libpoly 依赖风险。CDCAC 很多关键失败其实都来自“根隔离/符号判断/边界比较”不可靠。

## 11. ReasonManager：冲突子句生成

CDCAC 的 conflict clause 不应该直接收集所有 active literals。应该收集真正覆盖所有区域所需的 labels。

基本版本：

```cpp
std::vector<SatLit> ReasonManager::minimize(const Covering& cover) {
    std::vector<SatLit> reasons;

    for (const Cell& c : cover.cells) {
        for (SatLit r : c.reasons) {
            addUnique(reasons, r);
        }
    }

    return reasons;
}
```

高级版本做 set-cover minimization。近期工作也专门把 CDCAC conflict generalization 中的 reason minimization 形式化为 “set covering problem with reasons”，目标是找到能覆盖 labels 的较小约束子集；这正好对应你的 `ReasonManager` 后续优化方向。([arXiv][3])

第一版建议：

```text
P1：直接 union reasons，保证正确；
P2：按出现频率 greedy reduce；
P3：用 deletion-based minimization；
P4：实现 exact/LP-based set cover。
```

最终生成冲突：

```cpp
TheoryConflict ReasonManager::toConflict(const std::vector<SatLit>& reasons) {
    std::vector<SatLit> clause;
    for (SatLit lit : reasons) {
        clause.push_back(lit.negated());
    }
    return TheoryConflict{clause};
}
```

## 12. 回溯设计

你现在的 trail 设计能用，但可以简化和增强。

建议改成：

```cpp
struct TrailEntry {
    int level;
    size_t activeSizeBefore;
    uint64_t epochBefore;
};

uint64_t activeEpoch_ = 0;
```

`assertConstraint`：

```cpp
void CdcacSolver::assertConstraint(PolyId poly,
                                   Relation rel,
                                   SatLit reason,
                                   int level) {
    trail_.push_back({level, active_.size(), activeEpoch_});

    active_.push_back({
        .id = nextConstraintId_++,
        .poly = poly,
        .rel = rel,
        .reason = reason,
        .level = level,
        .epoch = activeEpoch_++
    });

    pendingConflict_.reset();
}
```

`backtrack`：

```cpp
void CdcacSolver::backtrack(int level) {
    while (!trail_.empty() && trail_.back().level > level) {
        active_.resize(trail_.back().activeSizeBefore);
        activeEpoch_ = trail_.back().epochBefore;
        trail_.pop_back();
    }

    if (pendingConflict_ && !isStillActive(pendingConflict_->reasons)) {
        pendingConflict_.reset();
    }
}
```

`pendingUnknown_` 建议删除。`pendingConflict_` 可以保留，但里面要存 reasons，而不是只存 level：

```cpp
struct PendingConflict {
    int level;
    std::vector<SatLit> reasons;
};
```

只要这些 reasons 仍然 active，这个 conflict 在 active 增加后依然有效；回溯删掉其中某个 reason 后才失效。

## 13. 和你的 IncrementalLinearizer 配合

不要每次 `check()` 都直接跑完整 CDCAC。推荐 effort 策略：

```text
Cheap effort:
  - 常数约束
  - duplicate / contradiction check
  - 简单 bound contradiction
  - sign-known simplification

Standard effort:
  - incremental linearization
  - McCormick / tangent / square nonnegativity cuts
  - LRA model refinement
  - cheap nonlinear conflict

Full effort:
  - CDCAC complete check
```

cvc5 的系统描述里也强调了 nonlinear real arithmetic 中 covering method 与 incremental linearization 的组合，而不是单独依赖一种技术。([理论斯坦福][1]) 你的实现里可以让 LRA model 作为 CDCAC 的 preferred sample：

```cpp
struct ModelSeed {
    std::unordered_map<Var, mpq_class> rationalValues;
};
```

在 covering 里选点时优先选 LRA model 的值：

```cpp
auto sample = cover.chooseSampleOutside(seed.valueOf(var));
```

这样 CDCAC 不是盲目从 0、1、-1 开始，而是验证当前抽象模型。

## 14. 推荐分阶段实现路线

### P0：把当前代码改成稳定 facade

目标：不再把 CDCAC 核心塞进 `CdcacSolver::check()`。

完成项：

```text
CdcacSolver: active/trail/facade
CdcacCore: solve(input)
AlgebraKernel interface
Native univariate root isolation skeleton
ReasonManager
```

验收：

```text
常数冲突仍然正确；
active/backtrack 无泄漏；
非线性暂时可 unknown，但路径清晰。
```

---

### P1：完成一变量 NRA

先不要急着做多变量。做完整的一变量非常重要。

支持：

```text
x^2 - 2 = 0
x^2 + 1 = 0
x^2 < 0
x^2 >= 0
x != 0
(x - 1)(x + 1) <= 0
```

算法：

```text
收集所有 active polynomials；
隔离所有实根；
把实轴切成 sections + sectors；
每个 cell 取 sample；
检查是否有一个 cell 满足所有约束；
若没有，cells 构成 covering，返回 conflict。
```

一变量是最好的 correctness playground。

---

### P2：多变量 rational-sector CDCAC

先实现多变量，但只稳定支持 rational sector sample。遇到 algebraic section 可以保守 unknown。

目标：

```text
x^2 + y^2 + 1 = 0        unsat
x^2 + y^2 <= 0, x != 0  unsat
x*y = 1                 sat
x^2 + y^2 <= 1          sat
```

这里能打通：

```text
recursive solveLevel
leaf conflict cell
covering union
projection guards
basic conflict generation
```

---

### P3：支持 algebraic section

这是从“能跑部分例子”到“像样 CDCAC”的关键。

需要：

```text
RealAlg comparison
signAt(poly, algebraic sample)
section cell
root index / Thom encoding
nullification detection
```

遇到：

```text
x^2 - 2 = 0
y^2 - x = 0
```

这种必须能处理代数 sample。

---

### P4：Lazard / nullification fallback

McCallum projection 对 nullification 很敏感。工程上建议：

```text
默认 Conservative/McCallum
检测 nullification
触发 Lazard fallback
仍失败则 Unknown
```

不要默默继续，否则容易 unsound。

---

### P5：reason minimization 和 pruning

实现：

```text
cell merge
subsumed cell deletion
reason union 去重
greedy set-cover
deletion-based minimization
```

CDCAC 性能很大程度取决于 conflict clause 是否足够小。上面提到的 reason set-cover 方向正是这个优化点。([arXiv][3])

---

### P6：与 CDCL/NRA 主求解器深度集成

包括：

```text
从 LRA 获取 model seed；
从 linearizer 获取 candidate；
CDCAC 只在 full effort 调用；
CDCAC conflict clause 回灌 CDCL；
CDCAC model 存为 NRA model；
支持 resource budget；
支持 stats。
```

建议 stats：

```cpp
struct CdcacStats {
    uint64_t calls = 0;
    uint64_t sat = 0;
    uint64_t unsat = 0;
    uint64_t unknown = 0;
```

## 15. 你现在代码的直接修改建议

第一步先改成这个方向：

```cpp
class CdcacSolver {
public:
    explicit CdcacSolver(PolynomialKernel* kernel,
                         AlgebraBackend* algebra);

    void assertConstraint(PolyId poly, Relation rel, SatLit reason, int level);
    void backtrack(int level);

    void reset();

    CdcacInput buildInput(const ModelSeed* seed) const;
    bool isStillActive(const std::vector<SatLit>& reasons) const;

private:
    PolynomialKernel* kernel_;
    AlgebraBackend* algebra_;
    CdcacCore core_;

    std::vector<ActiveConstraint> active_;
    std::vector<TrailEntry> trail_;

    std::optional<PendingConflict> pendingConflict_;
    std::optional<SamplePoint> lastModel_;

    uint64_t activeEpoch_ = 0;
    ConstraintId nextConstraintId_ = 0;
```

`check()` 不要再是“遇到非线性 unknown”，而是：

```cpp
TheoryCheckResult CdcacSolver::check(TheoryEffort effort,
                                     const ModelSeed* seed) {
    if (pendingConflict_ && isStillActive(pendingConflict_->reasons)) {
        return TheoryCheckResult::mkConflict(
            ReasonManager::toConflict(pendingConflict_->reasons)
        );
    }

    auto trivial = checkConstantsOnly();
    if (!trivial.isConsistent()) {
        return trivial;
    }

    if (effort != TheoryEffort::Full) {
        return TheoryCheckResult::consistent();
    }

    CdcacInput input = buildInput(seed);
    CdcacResult result = core_.solve(input);

    if (result.status == CdcacStatus::Sat) {
        lastModel_ = result.model;
        return TheoryCheckResult::consistent();
    }

    if (result.status == CdcacStatus::Unsat) {
        pendingConflict_ = PendingConflict{
            .level = currentMaxLevel(),
            .reasons = result.reasons
        };
        return TheoryCheckResult::mkConflict(
            ReasonManager::toConflict(result.reasons)
        );
    }

    return TheoryCheckResult::unknown();
}
```

## 16. 最小测试集

建议你先建立这些单测：

```text
常数：
  1 = 0                       unsat
  0 = 0                       sat
  -1 < 0                      sat

一变量：
  x^2 + 1 = 0                 unsat
  x^2 - 2 = 0                 sat
  x^2 < 0                     unsat
  x^2 >= 0                    sat
  x != 0                      sat
  x = 0, x != 0               unsat

二变量：
  x^2 + y^2 + 1 = 0           unsat
  x^2 + y^2 <= 0, x != 0      unsat
  x*y = 1                     sat
  x*y = 0, x != 0, y != 0     unsat

nullification：
  x = 0, x*y + 1 = 0          unsat
  x = 0, x*y = 0              sat

边界：
  x^2 <= 0, x != 0            unsat
  x^2 < 0                    unsat
  x^2 <= 0                   sat
```

每个 unsat 测试都要检查 conflict clause 是否只包含 active literals 的 negation，不能引入内部 projection polynomial 的“假 reason”。

## 17. 最重要的工程原则

你的实现应坚持这三条：

1. **soundness first**
   后端失败、root 不确定、nullification 没处理好时，返回 unknown，不要返回 sat/unsat。

2. **algebra backend 隔离**
   libpoly 只能在 backend 内部出现。核心算法只看 `RootSet`、`Sign`、`RealAlg`、`ProjectionSet`。

3. **先做一变量完整，再扩多变量**
   一变量 CDCAC 做扎实后，多变量只是递归、projection、lifting 的工程扩展；反过来直接写多变量，很容易出现一堆不可定位的 algebra bug。

总体上，你这版 `CdcacSolver` 可以作为第 0 层 facade，但真正的实现重心应该迁到 `CdcacCore + AlgebraBackend + CoveringManager + ReasonManager`。不要继续让 `check()` 从常数判断直接跳到 `unknown`；下一步最该做的是 **一变量完整 checker + reason-labelled covering**。
