这版可以评为：**Minor Revise 后可以进入实现**。
和前几版相比，核心方向已经稳了：V1 收缩为 pure QF_UF，不碰 mixed theory；term-level proof graph、Bool predicate、true/false、n-ary normalize、assertedLit、trail/backtrack 都已经纳入方案。整体不需要推倒重来。

但还有几处必须修掉，否则实现模型容易写出错误代码。

# 1. 最大 blocker：Bool sort ID 获取方式仍然不可靠

你现在在 `FrontendAdapter::importProblem()` 里通过扫描 `ConstBool` 来找 Bool sort：

```cpp
if (e.kind == Kind::ConstBool) {
    boolSortId_ = e.sort;
    break;
}
```

这会漏掉没有 `true/false` 常量的公式，例如：

```smt2
(set-logic QF_UF)
(declare-const p Bool)
(assert p)
(check-sat)
```

这个公式里可能没有 `ConstBool`，但 `p` 仍然是 Bool sort。于是 `boolSortId_` 可能保持 `NullSort`，导致 Bool variable 不会被转成 `p = true`。

建议改成：

```text
Bool sort ID 必须来自 SortManager / parser sort table / CoreIr 内置 boolSortId，而不是扫描 ConstBool。
```

例如：

```cpp
SortId CoreIr::boolSortId() const;
```

或者在 adapter 导入 sort 时直接记录：

```cpp
if (parserSort.isBool()) {
    boolSortId_ = importedSortId;
}
```

这点必须修，否则你新增的 `euf_013_unsat_bool_variable.smt2` 测试可能失败。

# 2. EUF atom dedup 方案仍然前后不一致

前面写：

```cpp
std::unordered_set<uint64_t> eufAtomDedup_;
```

但 `unordered_set` 只能判断存在，不能返回已有 `SatLit`。后面又写：

```cpp
if (registry_) {
    return registry_->getOrCreateEufAtom(payload, originExpr);
}
```

这又回到了 Registry 负责 get-or-create，和红线里“Atomizer owns SAT var creation; Registry owns recording”矛盾。

建议彻底固定为一种设计：

```cpp
struct EufAtomKey {
    ExprId lhs;
    ExprId rhs;
    Relation rel;

    bool operator==(const EufAtomKey&) const = default;
};

std::unordered_map<EufAtomKey, SatLit, EufAtomKeyHash> eufAtomDedup_;
```

然后：

```cpp
SatLit Atomizer::getOrCreateEufAtom(EufAtomPayload payload, ExprId originExpr) {
    canonicalize(payload);

    EufAtomKey key{payload.lhsExpr, payload.rhsExpr, payload.rel};

    auto it = eufAtomDedup_.find(key);
    if (it != eufAtomDedup_.end()) {
        memo_[originExpr] = it->second;
        return it->second;
    }

    SatVar v = freshVar();
    SatLit lit = SatLit::positive(v);

    atoms_.push_back({v, originExpr, true, TheoryId::EUF});
    eufAtomDedup_[key] = lit;
    memo_[originExpr] = lit;

    if (registry_) {
        registry_->registerParsedTheoryAtom(v, originExpr, TheoryId::EUF, payload);
    }

    return lit;
}
```

然后删掉：

```cpp
registry_->getOrCreateEufAtom(...)
```

Registry 只保留：

```cpp
registerParsedTheoryAtom(...)
findBySatVar(...)
```

不要再让 Registry 创建或复用 SAT var。

# 3. 不要用 `uint64_t` 手搓 dedup key

现在的 key：

```cpp
uint64_t key = (static_cast<uint64_t>(lhs) << 32) | rhs;
key ^= static_cast<uint64_t>(payload.rel) << 48;
```

这个设计不稳。`rel << 48` 会和 `lhs << 32` 的高位区域重叠，理论上可能碰撞。即使实际 ExprId 较小，也不值得冒这个风险。

直接用结构体 key：

```cpp
struct EufAtomKey {
    ExprId lhs;
    ExprId rhs;
    Relation rel;
};
```

hash 里组合三个字段即可。这样更清晰，也避免以后 ExprId 位宽变化带来隐患。

# 4. `TheoryManager::assertTheoryLit` 的 value 语义可能是错的

你写的是：

```cpp
it->second->assertLit(atom, assignedLit.sign, level, assignedLit);
```

这里风险很大。`assignedLit.sign` 到底表示“正 literal”还是“是否取反”取决于 `SatLit` 的实现。很多 SAT 封装里 `sign=true` 反而表示 negative literal。

这里不能用字段猜，必须显式写成：

```cpp
bool value = assignedLit.var() == atom.satVar()
          && assignedLit == SatLit::positive(atom.satVar());

solver->assertLit(atom, value, level, assignedLit);
```

或者更简洁：

```cpp
bool value = assignedLit.isPositive();
```

前提是 `isPositive()` 语义明确。

这个很关键。因为 `EufSolver::check()` 里会用：

```cpp
Relation effRel = a.value ? a.rawRel
                : (a.rawRel == Relation::Eq ? Relation::Neq : Relation::Eq);
```

如果 `value` 传反，所有 negated equality / distinct 都会被解释错。

# 5. `EGraph` 缺少 term 到 class 的映射字段

当前 `EGraph` 里有：

```cpp
std::vector<EClass> classes_;
```

但没有看到：

```cpp
std::vector<EClassId> termClass_;
```

如果没有 `termClass_`，`find(EufTermId t)` 很难正确实现。建议明确补上：

```cpp
std::vector<EClassId> termClass_;
```

`addTerm(t)`：

```cpp
if (t >= termClass_.size()) {
    termClass_.resize(t + 1, NullEClass);
}
if (termClass_[t] == NullEClass) {
    EClassId c = classes_.size();
    classes_.push_back({c, c, {t}});
    termClass_[t] = c;
}
```

`find(t)`：

```cpp
return ufFind(termClass_[t]);
```

这样 `EGraph` 的接口才完整。

# 6. `getOrCreateEufAtom` 应 canonicalize payload 本身，不只是 key

你现在在 helper 里 canonicalize 了局部变量：

```cpp
ExprId lhs = payload.lhsExpr;
ExprId rhs = payload.rhsExpr;
if (lhs > rhs) std::swap(lhs, rhs);
```

但后面注册时仍然可能传原始 payload：

```cpp
registry_->registerParsedTheoryAtom(v, originExpr, TheoryId::EUF, payload);
```

建议直接修改 payload：

```cpp
if (payload.lhsExpr > payload.rhsExpr) {
    std::swap(payload.lhsExpr, payload.rhsExpr);
}
```

这样 Registry、EufSolver、diagnostics 看到的都是 canonical payload。

# 7. `pop()` 也建议使用 clamp，避免 backtrack 后 resize 回扩

你前面已经在 `backtrackToLevel()` 里 clamp 了 `scopeLimits_`，这很好。但 `pop()` 里仍然建议防御式写法：

```cpp
void EufSolver::pop(uint32_t n) {
    while (n-- && !scopeLimits_.empty()) {
        size_t lim = std::min(scopeLimits_.back(), trail_.size());
        trail_.resize(lim);
        scopeLimits_.pop_back();
    }
}
```

不要依赖所有调用顺序都完全干净。

# 8. explanation 的因果安全可以作为 V1.1 优化

你现在遇到 explanation cycle 就返回 failure：

```cpp
if (visitedPairs.count(pk)) {
    return {false, {}};
}
```

这是 sound 的，不会生成错误 clause。但可能会让一些本来可以解释的冲突返回 `Unknown`。

V1 可以接受。
但如果你想更稳，可以给 proof edge 加 `edgeId`，解释 congruence edge 时只允许使用更早的 proof edges：

```cpp
struct ProofEdge {
    EufTermId u;
    EufTermId v;
    MergeReason reason;
    uint32_t edgeId;
};
```

然后：

```cpp
ExplainResult explainPath(a, b, maxEdgeId);
```

这样 explanation 有因果顺序，不容易因为循环而失败。这个不是 V1 blocker，可以放到后续优化。

# 9. Adapter 的 `NT_UF_APPLY` 枚举名要实际核对

方案里写：

```cpp
case NODE_KIND::NT_UF_APPLY: return Kind::UFApply;
```

这个需要和 SOMTParser 真实 AST kind 对齐。有些 parser 不会专门叫 `NT_UF_APPLY`，而是统一的 `Apply` / `App` / `Oper`，再根据符号表判断是否是 uninterpreted function。

建议在实现 prompt 里加一句：

```text
Do not assume NT_UF_APPLY exists. Inspect SOMTParser node kinds and symbol metadata; map declared uninterpreted function applications to Kind::UFApply.
```

否则实现模型可能直接写一个不存在的 enum，编译不过。

# 建议最终修改清单

把这版再改成下面这样就可以实现：

```text
1. Bool sort ID 从 SortManager / parser sort table / CoreIr 获取，不通过 ConstBool 扫描。
2. Atomizer 使用 unordered_map<EufAtomKey, SatLit> 做 EUF dedup。
3. 删除 Registry::getOrCreateEufAtom，Registry 只负责 register/find。
4. EufAtomKey 使用结构体 hash，不使用 uint64_t 手工拼接。
5. TheoryManager 传 value 时不要用 assignedLit.sign，显式判断 assigned literal 是否为正 atom。
6. EGraph 增加 termClass_，明确 addTerm/find/ufFind 的关系。
7. getOrCreateEufAtom canonicalize payload 本体。
8. pop() resize 使用 min(scopeLimit, trail.size())。
9. 实现前核对 SOMTParser 的 UF application 节点类型。
```

# 总体判断

这版已经不是“架构有问题”，而是**实现细节需要收口**。

我的建议是：**Minor Revise 后进入实现**。尤其先修 1–6；这几个是实际会导致测试失败、编译失败或 negated atom 语义错误的点。其余可以边实现边调整。
