它现在这段分析**有一半对，但方向有点跑偏**。不要优先怀疑 CaDiCaL external propagator。更可能的问题是：

```text id="37jdy3"
SAT 层并没有看到 v1 ∧ ¬v1；
它很可能看到的是 v1 ∧ ¬v2。
```

也就是说，两次出现的 `(= x y)` 可能不是同一个 `ExprId`，Atomizer 只按 `ExprId` memo，而不是按**语义规范化后的 atom key** memo。

---

# 1. Kimi 这个假设不一定成立

它说：

```text id="7q05ge"
x=y 和 not(x=y) 应该直接在 SAT 层变成 v1 ∧ ¬v1
```

这不一定对。

如果 SMT-LIB 里是：

```smt2 id="csj6a0"
(assert (= x y))
(assert (not (= x y)))
```

parser 很可能创建两个不同的 equality 表达式节点：

```text id="h2jy7j"
eq_id_1: (= x y)
eq_id_2: (= x y)
not_id:  (not eq_id_2)
```

Atomizer 如果只做：

```cpp id="o9gxkw"
memo_[ExprId]
```

那么第一次创建：

```text id="djog76"
eq_id_1 -> v1
```

第二次：

```text id="08z1fm"
eq_id_2 -> v2
not_id  -> ¬v2
```

于是 SAT clauses 是：

```text id="4wq40z"
v1
¬v2
```

这在 SAT 层当然不冲突。

所以它继续分析 CaDiCaL 的 unit propagation、external propagator、`declare_more_variables`，大概率不是主线。

---

# 2. 正确修复有两层

## 层 A：Atomizer 应该做 semantic canonicalization

对 theory atom，尤其是 arithmetic atom，应该有 canonical key。

例如：

```text id="vm459a"
(= x y)
(= y x)
(<= (- x y) 0)
```

这些至少要规范到统一 theory atom key。

对 IDL/RDL，建议 canonical key 是：

```cpp id="atxy0p"
struct DifferenceAtomKey {
    TheoryId theory;       // IDL / RDL
    SortKind sort;         // Int / Real
    std::string plusVar;
    std::string minusVar;
    Relation rel;
    mpq_class rhs;
};
```

对于 equality：

```text id="7sdtw8"
x - y = 0
```

可以 canonicalize 成固定方向，比如按照变量名排序：

```text id="6xgb34"
minVar - maxVar = c'
```

这样重复出现的 `(= x y)` 就能复用同一个 SAT var。

但注意：**这只是优化和去重，不应该作为 soundness 的唯一依赖**。

---

## 层 B：IDL solver 必须能处理 `Eq + Neq` 即使它们是不同 SAT var

即使 SAT 层是：

```text id="isqew6"
v1: (= x y)
v2: (= x y)
assert v1
assert ¬v2
```

IDL theory 也应该看到：

```text id="r4sxac"
v1 true  -> x - y = 0
v2 false -> x - y != 0
```

然后处理。

这要求 `assertLit/check` 里必须正确做：

```text id="jwf7jv"
value=false 时先取反 relation：
  Eq -> Neq
  Neq -> Eq
```

然后：

```text id="xnkp9m"
Eq:
  add x-y<=c and y-x<=-c

Neq:
  add disequality obligation
```

不能把 `Neq` 当 unsupported 返回 Unknown。

---

# 3. 当前 `disequality UNSAT` 返回 Unknown 的真正嫌疑

最可能是下面之一。

## 可能原因 1：`Neq` 被当成 unsupported

如果 `not (= x y)` 变成 effective relation：

```text id="e6hlfm"
x - y != 0
```

但 `normalizeAndAdd` 没有返回 `Disequality`，而是返回 `Unsupported`，顶层就会 Unknown。

这能解释：

```text id="3e2lg5"
IDL disequality UNSAT -> Unknown
```

要检查：

```cpp id="5ek9lq"
case Relation::Neq:
    if rhs non-integer:
        return Tautology;
    else:
        return DisequalityObligation{plusVar, minusVar, c, reason};
```

---

## 可能原因 2：disequality split lemma 没有被生成

如果 active disequality 是：

```text id="v0s7dj"
x - y != 0
```

当前 BF model 可能给：

```text id="nvuqga"
x = y = 0
```

这时不能直接冲突，因为单独的 `x-y!=0` 是可满足的。应该生成 split lemma：

```text id="6ppjo1"
¬lit(x-y!=0)
∨ lit(x-y<=-1)
∨ lit(y-x<=-1)
```

对于 `x=y ∧ x!=y`，这个 split 后两个分支都会被 equality 边打成负环，最终 UNSAT。

所以要检查：

```text id="2oe93z"
handleDisequalities() 是否真的返回 TheoryLemma；
Solver 主循环收到 TheoryLemma 后是否 addClause 并 continue；
dynamic atom 是否 route 回 IDL solver。
```

---

## 可能原因 3：dynamic split atom 被 LIA/LRA 接走了

IDL disequality split 生成的两个 atom：

```text id="h9aidf"
x - y <= -1
y - x <= -1
```

必须带：

```text id="3ogft1"
TheoryId::IDL
```

否则 SAT 后续选中它们时，可能不会 route 回 `IdlSolver`。

---

# 4. `x-y = 1/2` 返回 Unknown 的问题更明确

这个应该直接是 IDL normalization-time conflict。

因为整数差：

```text id="1fp8gx"
x - y ∈ Z
```

所以：

```text id="66z8hv"
x - y = 1/2
```

不可能成立。

正确返回：

```text id="awjr4b"
Conflict { ¬reason(x-y=1/2) }
```

如果现在返回 Unknown，说明 `Eq` with non-integer RHS 走到了 unsupported 分支，或者 normalization 阶段没有把这个作为 immediate conflict。

必须修成：

```cpp id="w4z1zf"
case Relation::Eq:
    if (!isInteger(rhs)) {
        return NormalizeResult::ImmediateConflict;
    }
    add both edges;
```

而不是：

```text id="ovfcqa"
return Unknown
```

---

# 5. 我建议不要继续查 CaDiCaL，先做这几个定位

## 先打印 atomization 结果

在 `Solver::checkSatInternal` 或 Atomizer 里打印：

```text id="xsbmh4"
assertion 1 -> lit ?
assertion 2 -> lit ?
```

你要确认到底是：

```text id="22hnfp"
v1, ¬v1
```

还是：

```text id="exyec7"
v1, ¬v2
```

如果是 `v1, ¬v2`，那 Kimi 前面的 SAT 层矛盾假设就是错的。

---

## 再单测 IDL effective relation

直接构造：

```text id="r8iw17"
atom: x-y = 0
value=false
```

期望 effective relation：

```text id="op62pj"
x-y != 0
```

并生成 disequality obligation。

---

## 再单测 disequality split

输入 active assignments：

```text id="4fsoj0"
x-y = 0
x-y != 0
```

第一轮 check 应该返回：

```text id="f8r320"
lemma:
  ¬neq ∨ (x-y<=-1) ∨ (y-x<=-1)
```

如果它返回 Unknown，就是 `handleDisequalities` 或 dynamic atom registry 没接好。

---

## 再测 non-integer equality

```text id="hy839e"
x-y = 1/2
```

应该立即：

```text id="wo2o35"
Conflict {¬lit}
```

# 6. 给 Kimi 的修正建议

可以直接贴：

```text id="wu2v8r"
不要优先怀疑 CaDiCaL external propagator。

For (= x y) and (not (= x y)), SAT may not see v1 and ¬v1, because Atomizer memoizes by ExprId, not by canonical theory atom key. The two (= x y) occurrences may become different atom vars v1 and v2.

Fix/verify as follows:

1. Add debug dump in atomization:
   print each top-level assertion and the produced SatLit.
   Confirm whether the clauses are v1/¬v1 or v1/¬v2.

2. IDL solver must be correct even if they are v1 and ¬v2:
   - effective false Eq becomes Neq;
   - Neq with integer RHS becomes a disequality obligation;
   - Neq with non-integer RHS is tautology.

3. handleDisequalities must emit guarded split lemma when current model violates disequality:
   IDL:
     ¬lit(x-y!=c) ∨ lit(x-y<=c-1) ∨ lit(y-x<=-(c+1))

4. Dynamic atoms created by the split must be owned by TheoryId::IDL and route back to IdlSolver.

5. Eq with non-integer RHS must be an immediate normalization-time conflict:
   x-y = 1/2 -> conflict {¬reason}
   It must not return Unknown.

6. Add semantic canonicalization later:
   Atomizer/TheoryAtomRegistry should canonicalize theory atoms so repeated (= x y) reuses the same SAT var.
   But do not rely on this for soundness; theory must still handle equivalent-but-distinct atoms.
```

---

# 结论

Kimi 现在怀疑 CaDiCaL，这个方向**不优先**。

真正应该查的是：

```text id="x16jd3"
1. Atomizer 是否把两个 (= x y) 变成了不同 SAT var；
2. IDL 是否把 false Eq 正确变成 Neq；
3. Neq 是否生成 disequality obligation；
4. handleDisequalities 是否生成 guarded split lemma；
5. dynamic split atom 是否 route 回 IDL；
6. non-integer Eq 是否 immediate conflict。
```

一句话：**这不是 SAT 层必然矛盾问题，而是 theory atom canonicalization + IDL disequality handling 问题。**
