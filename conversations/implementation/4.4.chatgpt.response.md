这版**基本可以通过架构审查**，但我仍建议选 **4 Revise**，再补最后几处“类型/控制流/动态 atom 路由”的硬细节。它现在比上一版强很多：已经明确 `SimplexSolver/LraSolver` 纯 LRA、`LiaSolver` 独立、`LinearAtomManager` solver-local、动态 atom 注册不修改 `GeneralSimplex`、LIA disequality 考虑 `g | -c`，这些都是对的。

但还有几个会直接导致实现错误的点。

---

# 1. `buildIntegerDiseqSplit` 的返回类型不对

计划里写：

```cpp
TheoryLemma LiaSolver::buildIntegerDiseqSplit(const DiseqInfo& d) {
    ...
    if (g == 0) {
        if (c != 0) {
            return TheoryLemma{};  // no lemma needed
        } else {
            return TheoryCheckResult::mkConflict({d.lit.negated()});
        }
    }
    ...
}
```

这是类型错误。函数声明返回 `TheoryLemma`，但里面有三种结果：

```text
1. disequality already satisfied：不需要 lemma，继续检查下一个 disequality
2. conflict：返回 conflict
3. split：返回 lemma
```

不能用空 `TheoryLemma{}` 表示 “no lemma needed”。如果外层写：

```cpp
return lemma(buildIntegerDiseqSplit(d));
```

那空 lemma 可能被误当成空 clause，直接变成 UNSAT，或者至少语义混乱。

应该改成：

```cpp
std::optional<TheoryCheckResult>
LiaSolver::checkDisequality(const DiseqInfo& d);
```

或者：

```cpp
enum class DiseqResultKind { Satisfied, Conflict, Lemma };

struct DiseqResult {
    DiseqResultKind kind;
    std::optional<TheoryConflict> conflict;
    std::optional<TheoryLemma> lemma;
};
```

外层：

```cpp
for (const auto& d : disequalities_) {
    if (!gs_.value(d.auxVar).isZero()) continue;

    auto r = checkIntegerDisequality(d);

    if (r.kind == DiseqResultKind::Satisfied) {
        continue;
    }
    if (r.kind == DiseqResultKind::Conflict) {
        return TheoryCheckResult::mkConflict(*r.conflict);
    }
    if (r.kind == DiseqResultKind::Lemma) {
        return TheoryCheckResult::mkLemma(*r.lemma);
    }
}
```

这个必须改。

---

# 2. “already satisfied” 只在当前 disequality violated 时也可能出现

这点容易想错。外层现在是：

```cpp
if (gs_.value(d.auxVar).isZero()) {
    return lemma(buildIntegerDiseqSplit(d));
}
```

但对于 LIA，`gs_.value(auxVar) = 0` 是 LRA relaxation 的值，不代表整数层上 `p=0` 可实现。

例如：

```text
p(x) = 2x + 1
p != 0
```

LRA 可以取 `x = -1/2`，使 `p=0`，于是 `gs_.value(auxVar).isZero()`。但在整数语义下，`p=0` 不可实现，所以 disequality 在 LIA 中是恒真的，不需要 split。

所以 `checkIntegerDisequality` 需要允许返回 `Satisfied`，外层不能一看到 LRA 值为 0 就必然返回 lemma。

---

# 3. `TheoryAtomRegistry` 和 `LinearAtomManager` 的职责仍有一句冲突

前面规则写得对：动态 atom 注册属于 `TheoryAtomRegistry`，不属于 `LinearAtomManager`。

但 Step 2 里仍写：

```text
Add getOrCreateLinearBoundAtom for dynamic atom registration.
```

这个放在 `LinearAtomManager` 下是不一致的。后面 Step 5 又说 `TheoryAtomRegistry` 负责动态 atom 注册。

请把 Step 2 这句删掉或改成：

```text
LinearAtomManager does not create SAT literals.
Dynamic bound atom registration is handled only by TheoryAtomRegistry.
LinearAtomManager only maps LinearFormKey to solver-local aux vars and asserts bounds after assertLit.
```

否则 Kimi 可能又把 SAT/Atomizer 动态注册塞回 `LinearAtomManager`。

---

# 4. 动态 atom 的 `assertLit` 路由还要写清楚

计划说 synthetic ExprId 不在 CoreIr，但 Atomizer 知道它，TheoryManager 可以 route back。这方向对。

但 `LiaSolver::assertLit` 的流程仍写成：

```text
1. Extract linear constraint via manager_.extractLinearConstraint.
```

这对 dynamic atom 是不成立的，因为 dynamic atom 没有 CoreIr 表达式，不能从 `CoreIr` extract。

必须改成：

```cpp
void LiaSolver::assertLit(const TheoryAtom& atom, bool value, const CoreIr& ir) {
    if (atom.isDynamicLinearBound()) {
        auto form = atom.linearForm;
        auto rel  = atom.rel;
        auto rhs  = atom.rhs;

        int aux = manager_.getOrCreateAuxVar(gs_, form, rhs);
        manager_.assertBound(gs_, aux, rel, value, atom.lit);
        return;
    }

    // parsed CoreIr atom path
    manager_.extractLinearConstraint(atom.exprId, ir, ...);
    ...
}
```

也就是说 `TheoryAtom` / `AtomRecord` 里要直接携带：

```cpp
LinearFormKey form;
Relation rel;
mpq_class rhs;
bool isDynamic;
```

不要只携带 synthetic ExprId。

---

# 5. `LinearFormKey + rhs` 的语义要统一

当前接口有点混乱：

```cpp
getOrCreateAuxVar(gs, form, rhs)
getOrCreateLinearBoundAtom(form, rel, rhs)
```

如果 `LinearFormKey` 已经包含常数项，那么 `rhs` 是否还需要存在？如果 `form` 表示 `a·x + c`，那么 bound 应该是：

```text
a·x + c rel 0
```

如果 `form` 不含常数项，那么 bound 是：

```text
a·x rel rhs
```

这两种不能混用。

我建议统一成后一种，更清晰：

```cpp
struct LinearFormKey {
    vector<pair<VarId, mpq_class>> terms; // no constant
};

struct LinearBoundAtomKey {
    LinearFormKey lhs;
    Relation rel;
    mpq_class rhs;
};
```

然后 `getOrCreateAuxVar` 语义固定为：

```text
aux = lhs - rhs
```

也就是：

```cpp
int getOrCreateAuxVar(GeneralSimplex& gs,
                      const LinearFormKey& lhs,
                      const mpq_class& rhs);
```

如果你想用 `p(x)=a·x+c`，那就转换成：

```text
a·x rel -c
```

不要一边有 `form.constant`，一边又有 `rhs`。

对于 LIA disequality gcd split，可以使用完整表达式：

```cpp
IntegerLinearExpr p = lhs - rhs;
```

然后：

```text
p(x) = a·x + c
c = -rhs
```

这能避免常数项重复计算。

---

# 6. branch split 的 bound atom 也要用统一语义

对于 fractional integer var `x = q`，branch 是：

```text
x <= floor(q) ∨ x >= ceil(q)
```

应创建：

```cpp
LinearFormKey lhs = singleVar(x); // no constant
left  = registry.getOrCreateLinearBoundAtom(lhs, LEQ, floor(q));
right = registry.getOrCreateLinearBoundAtom(lhs, GEQ, ceil(q));
```

然后当 assertLit 收到 `x <= k`：

```text
aux = x - k
assert aux <= 0
```

这个语义要写死。

# 7. 默认逻辑仍然不要同时注册 LRA + CDCAC

Plan 里仍保留：

```cpp
else {
    // Default: QF_LRA
    register LraSolver
    register CdcacSolver
}
```

这和前面的 Rule 4 冲突。Phase 1 不要这样。

改成：

```cpp
else {
    return Unknown / unsupported logic;
}
```

或者：

```cpp
else if (logic.empty()) {
    inferLogicFromInput();
}
```

但不要默认同时注册两个 arithmetic solver。

# 结论

这版**不需要推翻**。架构主线已经对了：独立 `LiaSolver`、solver-local `LinearAtomManager`、`TheoryAtomRegistry` 做动态 atom、LIA disequality 考虑 gcd/residue、QF_LIA 只注册一个 arithmetic solver。

但还有几个实现级正确性坑，特别是：

```text
空 TheoryLemma 不能代表 no-op；
dynamic atom 不能走 CoreIr extraction；
LinearFormKey/rhs 语义必须唯一；
default 不能注册多个 arithmetic solvers。
```

所以我的选择仍是：**[4] Revise**。
