# Plan: QF_LIA Solver (Phase 1 — Minimal LIA)

## Goal

Add quantifier-free linear integer arithmetic (QF_LIA) support to NLColver. Phase 1 implements the minimal sound closed loop:

```
GeneralSimplex (LRA relaxation kernel)
  → disequality check (integer-strength split)
  → integral model check
  → branch-and-bound split lemma (via properly registered atoms)
  → model validation
```

**Phase 1 is sound but not necessarily complete.** Pure branch-and-bound without cuts/gcd/fallback may not terminate on some problems. A resource limit may return Unknown.

## Current State

- `GeneralSimplex` is a sparse indexed tableau (18 tests pass).
- `SimplexSolver` wraps `GeneralSimplex` for LRA. It handles CoreExpr→linear form extraction, atom registration, bound mapping, conflict translation.
- `Atomizer` creates `TheoryAtom` records mapping `SatVar → ExprId`.
- `NiaSolver` is an empty placeholder.
- `Solver.cpp` registers `SimplexSolver` (LRA) and `CdcacSolver` (NRA).
- `CoreExpr` carries `SortId`, but there is no sort name / classification table yet.

## Critical Architectural Rules

### Rule 1: `SimplexSolver` stays pure LRA. No `liaMode`.

`SimplexSolver` (or renamed `LraSolver`) must **never** know about integer variables, branch splits, or GCD reasoning. Its `id()` returns `TheoryId::LRA`.

### Rule 2: `LiaSolver` is independent. It does not contain `SimplexSolver`.

`LiaSolver` directly uses `GeneralSimplex` as its LRA relaxation kernel. It does **not** wrap `SimplexSolver`.

```text
WRONG:  LiaSolver contains SimplexSolver
RIGHT:  LiaSolver contains GeneralSimplex + LinearAtomManager
```

Reasons:
- `SimplexSolver::id()` is `LRA`. Wrapping it inside `LiaSolver` is semantically wrong.
- `SimplexSolver` has LRA-specific disequality logic. LIA disequality should be stronger (integer-strength split).
- LIA branch/cut/gcd reasoning needs direct access to GeneralSimplex state and linear forms. Going through `SimplexSolver` is awkward.
- Future proof/certificate tracking will be confused if LRA conflicts, LIA branches, and Gomory cuts are all attributed to `SimplexSolver`.

### Rule 3: Shared code is extracted, not duplicated.

`LraSolver` and `LiaSolver` share constraint extraction, atom registration, and bound mapping. This code lives in a new shared helper:

```cpp
class LinearAtomManager {
public:
    // CoreExpr -> LinearForm extraction (pure function, no state)
    bool extractLinearConstraint(ExprId eid, const CoreIr& ir,
                                  std::unordered_map<std::string, mpq_class>& coeffs,
                                  mpq_class& rhs, Relation& rel);

    // LinearFormKey canonicalization -> GeneralSimplex aux var
    // Bound to a specific GeneralSimplex instance (solver-local)
    int getOrCreateAuxVar(GeneralSimplex& gs, const LinearFormKey& form,
                          const mpq_class& rhs);

    // Relation -> bound assertion
    void assertBound(GeneralSimplex& gs, int auxVar, Relation rel,
                     bool value, SatLit reasonLit);

    // Conflict translation
    TheoryConflict translateConflict(const GeneralSimplex& gs,
                                     const std::unordered_map<int, std::pair<std::optional<SatLit>, std::optional<SatLit>>>& boundReasons);
};
```

**Important**: `LinearAtomManager` is **solver-local**. Each `LraSolver` and `LiaSolver` owns its own `LinearAtomManager` instance. `formToAux_` inside the manager is bound to a specific `GeneralSimplex` instance. Two different solvers must not share the same `formToAux_` mapping, because aux var indices are local to each `GeneralSimplex`.

### Rule 4: Exactly one arithmetic solver per logic.

```cpp
if (logic == "QF_LRA" || logic == "LRA") {
    registerSolver(std::make_unique<LraSolver>());
}
else if (logic == "QF_LIA" || logic == "LIA") {
    registerSolver(std::make_unique<LiaSolver>());
}
else if (logic == "QF_NRA" || logic == "NRA") {
    registerSolver(std::make_unique<CdcacSolver>(...));
}
```

**Do not** register multiple arithmetic solvers for the same logic. QF_LIA gets only `LiaSolver`. QF_LRA gets only `LraSolver`. No fallback "register everything and let them ignore what they don't handle."

### Rule 5: No fake ExprId or placeholder SatLit.

Branch split literals `x <= floor(q)` and `x >= ceil(q)` must be **real registered atoms**:

```cpp
// WRONG:
SatLit left = SatLit{freshSatVar_, true};  // not registered!

// RIGHT:
SatLit left = atomRegistry_.getOrCreateLinearBoundAtom(
    sat, atomizer, form, LEQ, lo);
```

Dynamic atom registration is a **separate concern** from bound assertion. It lives at the SAT/Atomizer layer:

```cpp
class TheoryAtomRegistry {
public:
    // Pure atom registration: maps a linear bound to a SatLit.
    // Does NOT create GeneralSimplex aux vars.
    // Does NOT assert any bounds.
    SatLit getOrCreateLinearBoundAtom(
        SatSolver& sat, Atomizer& atomizer,
        const LinearFormKey& form,
        Relation rel, const mpq_class& rhs);
};
```

`getOrCreateLinearBoundAtom` must:
1. Build canonical `AtomKey = (LinearFormKey, relation, rhs)`.
2. Reuse or create an `AtomRecord`.
3. Call `sat->newVar()` if no SAT var exists.
4. Register `SatVar → TheoryAtom` with the Atomizer.
5. Return a valid `SatLit`.

It must **NOT** assert any bound and must **NOT** mutate GeneralSimplex. The solver-local aux var is created later when `assertLit` receives this atom.

## Implementation Steps

### Step 1: Rename SimplexSolver → LraSolver (or keep name, but enforce purity)

Refactor `SimplexSolver` to use `LinearAtomManager` for all shared logic. Remove any integer-specific code. Keep only:
- `assertLit` → bound assertion
- `check` → GeneralSimplex check + disequality split (LRA strength: `p < 0 ∨ p > 0`)
- `push/pop/reset`

### Step 2: Create `LinearAtomManager`

Extract from current `SimplexSolver`:
- `LinearForm`, `LinearFormHash`
- `extractLinearExpr`, `extractLinearConstraint`
- `getOrCreateVar` (or move into manager)
- `registerAtom` (form → aux var mapping)
- `assertBound` (relation → lower/upper translation)
- `translateConflict`

Add `getOrCreateLinearBoundAtom` for dynamic atom registration.

### Step 3: Create `LiaSolver`

```cpp
class LiaSolver : public TheorySolver {
public:
    TheoryId id() const override { return TheoryId::LIA; }
    void assertLit(const TheoryAtom& atom, bool value, const CoreIr& ir) override;
    TheoryCheckResult check(const CoreIr& ir) override;
    void push() override;
    void pop(uint32_t n) override;
    void reset() override;

private:
    GeneralSimplex gs_;
    LinearAtomManager manager_;

    std::unordered_map<std::string, int> varToIndex_;
    std::unordered_set<int> integerVars_;  // GS var indices

    // disequality obligations
    struct DiseqInfo { int auxVar; SatLit lit; };
    std::vector<DiseqInfo> disequalities_;

    // bound reasons for conflict translation
    std::unordered_map<int, std::pair<std::optional<SatLit>, std::optional<SatLit>>> boundReasons_;

    // dynamic branch atoms
    // managed by LinearAtomManager
};
```

`assertLit`:
1. Extract linear constraint via `manager_.extractLinearConstraint`.
2. Register atom via `manager_.registerAtom` → get `auxVar`.
3. Check variable sorts; if Int, add to `integerVars_`.
4. Assert bound via `manager_.assertBound(gs_, ...)`.
5. Record disequality obligations.

`check` flow:
```cpp
auto r = gs_.check();
if (r == Unsat) return conflict(manager_.translateConflict(gs_, boundReasons_));

auto beta = gs_.model();

// 1. Disequality obligations (before branching)
for (const auto& d : disequalities_) {
    if (gs_.value(d.auxVar).isZero()) {
        // Integer-strength split if possible
        return lemma(buildIntegerDiseqSplit(d));
    }
}

// 2. Integrality check
for (int v : integerVars_) {
    auto val = gs_.value(v);
    if (val.b != 0 || val.a.get_den() != 1) {
        return lemma(buildBranchSplitLemma(v, val));
    }
}

// 3. Model validation
// Every active asserted literal must evaluate to true under beta.
// Every integer variable must have denominator 1.
// Every disequality obligation must be satisfied.
// Dynamic branch atoms must be consistent.
if (!validateModel(beta, activeAtoms_, disequalities_)) {
    return TheoryCheckResult::unknown();  // or assert(false) in debug
}

return consistent();
```

### Step 5: Dynamic atom registration via `TheoryAtomRegistry`

Dynamic atom registration is a separate layer from `LinearAtomManager`:

```cpp
class TheoryAtomRegistry {
public:
    SatLit getOrCreateLinearBoundAtom(
        SatSolver& sat, Atomizer& atomizer,
        const LinearFormKey& form,
        Relation rel, const mpq_class& rhs);
private:
    struct AtomRecord {
        LinearFormKey form;
        Relation rel;
        mpq_class rhs;
        ExprId expr;     // synthetic, not in CoreIr
        SatVar satVar;
    };
    std::vector<AtomRecord> records_;
    // map from (form, rel, rhs) -> record index
};
```

Process:
1. Build `AtomKey = (LinearFormKey, relation, rhs)`.
2. Check if an atom already exists.
3. If not:
   - Allocate synthetic `ExprId` (counter starting from a high number, distinct from CoreIr).
   - Call `sat->newVar()`.
   - Register `SatVar → TheoryAtom` with Atomizer via `atomizer.registerDynamicAtom(...)`.
   - Store in `records_`.
4. Return `SatLit`.

**The synthetic ExprId does not need to be in CoreIr.** CoreIr is for parsed expressions. Branch atoms are solver-generated. But the Atomizer must know about them, and `TheoryManager::assertLit` must be able to route them back.

When `LiaSolver::assertLit` receives a dynamic atom:
1. Look up the `LinearFormKey` from the atom record.
2. Call `manager_.getOrCreateAuxVar(gs_, form, rhs)` to get/create the solver-local aux var.
3. Call `manager_.assertBound(gs_, auxVar, rel, value, reasonLit)`.

This cleanly separates:
- **SAT/Atomizer layer**: knows about SatVar ↔ ExprId mapping.
- **Solver layer**: knows about ExprId ↔ GeneralSimplex aux/bound mapping.

### Step 6: LIA disequality with gcd strength

For `p(x) = a·x + c` where all vars are integer:

```cpp
TheoryLemma LiaSolver::buildIntegerDiseqSplit(const DiseqInfo& d) {
    auto& form = getForm(d.auxVar);  // p(x) = a·x + c
    mpz_class g = gcdAbs(form.coeffs);  // g = gcd(|a_i|)
    mpz_class c = form.constant;        // constant term

    if (g == 0) {
        if (c != 0) {
            // p(x) = c ≠ 0 always, disequality already satisfied.
            return TheoryLemma{};  // no lemma needed
        } else {
            // p(x) = 0, but disequality says p != 0 → conflict
            return TheoryCheckResult::mkConflict({d.lit.negated()});
        }
    }

    if ((-c) % g != 0) {
        // p=0 impossible in integers, disequality already satisfied.
        return TheoryLemma{};
    }

    // g | -c, so p=0 is possible. Minimum non-zero step is g.
    return TheoryLemma{{ d.lit.negated(),
        getOrCreateBoundLit(form, LEQ, -g),
        getOrCreateBoundLit(form, GEQ, g) }};
}
```

**Critical**: For `2x + 1 != 0`:
- `g = 2`, `c = 1`
- `(-c) % g = (-1) % 2 = -1 ≠ 0`
- `p=0` is impossible in integers.
- Disequality is already satisfied. **No split needed.**

If we blindly returned `¬lit ∨ (p <= -2) ∨ (p >= 2)`, it would be unsound because `x=0` gives `p=1`, which satisfies the disequality but violates the split.

### Step 8: Testing

1. **LRA regression**: All 18 existing tests still pass after `SimplexSolver` → `LraSolver` refactor.
2. **Strict sort classification**:
   ```smt2
   (set-logic QF_LIA)
   (declare-const x Int)
   (assert (= (* 2 x) 1))
   ```
   Must return UNSAT.
3. **Branch atom registration**: After branch split, SAT re-solves, new literal is asserted, next `check()` has the bound active.
4. **Basic LIA SAT with branching**:
   ```smt2
   (declare-const x Int)
   (assert (<= (* 2 x) 5))
   (assert (>= x 0))
   ```
   Branch splits to `x<=2 ∨ x>=3`; x=2 branch SAT.
5. **LIA disequality gcd strength**:
   ```smt2
   (declare-const x Int)
   (assert (not (= x 0)))
   (assert (>= x 0))
   (assert (<= x 0))
   ```
   Split uses `x <= -1 ∨ x >= 1`.
6. **QF_LIA only registers LiaSolver**: Verify no `LraSolver` or `CdcacSolver` is registered.

## Risks & Mitigations

| Risk | Mitigation |
|------|------------|
| Extracting `LinearAtomManager` breaks LRA tests | Run all 18 tests after extraction; do not proceed until green |
| Dynamic atom registration is complex | `getOrCreateLinearBoundAtom` is the only creation path; all state is in `LinearAtomManager` |
| Sort classification incomplete | Strict rule: unknown sort → return Unknown/unsupported; never silently fallback to LRA in QF_LIA |
| Branch split without cuts causes poor performance | Phase 1 documented as sound but incomplete; resource limit can return Unknown |
| SAT solver state corruption from unregistered vars | `getOrCreateLinearBoundAtom` always calls `sat->newVar()` and registers with Atomizer before returning SatLit |
