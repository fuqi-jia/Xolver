  执行方案 1 之前，有一个架构问题必须确认：

  PolynomialKernel 目前完全没有 VarId 映射机制。 它的 mkVar 接受 std::string_view，degree/getIntegerCoe
  fficients 也接受 std::string_view。整个 CDCAC 里变量全是 std::string（varOrder、Cell.var、SamplePoint
  .varOrder 等）。

  如果 substituteRational(PolyId, VarId, mpq_class) 要接受 VarId，PolynomialKernel 必须先知道 VarId 对
  应哪个内部变量。

  选项：

  A. 先在 PolynomialKernel 里加 registerVar(VarId, std::string_view) 或让 mkVar 返回 (PolyId, VarId)，
  建立映射表，然后所有按变量操作的方法（degree、leadingCoefficient、substituteRational）统一迁到 VarId
  。

  B. 先保持 std::string_view，把 substituteRational 做成 substituteRational(PolyId, std::string_view, m
  pq_class)，与现有 degree/getIntegerCoefficients 风格一致。CDCAC 的 RootOrigin.mainVar 也保持 std::str
  ing。等整个 CDCAC 准备好后再统一做 VarId 大迁移。

  C. 在 CDCAC 层维护一个 std::string → VarId 的映射（通过 SOMTParser 的 symbol table），CDCAC 内部全改
  用 VarId，但 PolynomialKernel 仍然用 std::string_view（通过变量名字符串调用）。

  您建议选哪个？这决定了 substituteRational 的签名和 CDCAC 内部变量表示的改动范围。

------------------------------------------------------
选 **A，但用渐进式 A**。

也就是：

```text
最终架构：PolynomialKernel 统一拥有 VarId 映射。
短期工程：保留现有 std::string_view 接口作为兼容 wrapper。
新增 CDCAC / NRA 代码：一律用 VarId。
```

不要选 B。B 快，但会继续把变量名字符串带进 `substituteRational / reduceModuloTower / pseudoRemainder / RootOrigin`，后面一定会返工。

不要选 C。C 会造成两套变量系统：CDCAC 一套、PolynomialKernel 一套。最危险的是两边变量顺序、变量同一性、libpoly internal variable 可能不一致。

---

# 结论

采用：

```text
A'：PolynomialKernel 维护 name <-> VarId <-> backend variable 的统一映射；
    string API 保留，但只作为 getOrCreateVar(name) 的 wrapper。
```

# 1. 先定义 VarId

建议简单一点：

```cpp
struct VarId {
    uint32_t id = UINT32_MAX;

    bool isValid() const { return id != UINT32_MAX; }

    friend bool operator==(VarId a, VarId b) {
        return a.id == b.id;
    }

    friend bool operator!=(VarId a, VarId b) {
        return a.id != b.id;
    }

    friend bool operator<(VarId a, VarId b) {
        return a.id < b.id;
    }
};

static constexpr VarId NullVar{UINT32_MAX};
```

如果你喜欢强类型，也可以：

```cpp
using VarId = uint32_t;
static constexpr VarId NullVar = UINT32_MAX;
```

但 struct 更不容易误用。

# 2. PolynomialKernel 增加变量注册表

核心接口：

```cpp
class PolynomialKernel {
public:
    virtual VarId getOrCreateVar(std::string_view name) = 0;
    virtual std::string_view varName(VarId v) const = 0;

    virtual PolyId mkVar(VarId v) = 0;

    // 兼容旧代码
    PolyId mkVar(std::string_view name) {
        return mkVar(getOrCreateVar(name));
    }

    virtual bool isValidVar(VarId v) const = 0;
};
```

内部维护：

```cpp
std::vector<std::string> varNames_;
std::unordered_map<std::string, VarId> nameToVar_;
std::vector<BackendVar> varIdToBackendVar_;
```

其中 `BackendVar` 对 libpoly 来说就是 `lp_variable_t` 或 libpoly C++ wrapper 的 variable。

关键 invariant：

```text
同一个 VarId 永远对应同一个变量名和同一个 backend variable。
同一个变量名永远映射到同一个 VarId。
CDCAC、PolynomialKernel、libpoly backend 不允许各自维护独立变量身份。
```

---

# 3. 不建议让 mkVar 返回 pair

你提到：

```text
让 mkVar 返回 (PolyId, VarId)
```

不推荐。这样会污染很多现有调用。

更好的方式：

```cpp
VarId x = kernel.getOrCreateVar("x");
PolyId px = kernel.mkVar(x);
```

旧代码继续可以写：

```cpp
PolyId px = kernel.mkVar("x");
```

但新代码应该一律写 VarId 版本。

# 4. 变量相关 API 全部新增 VarId 版本

新增正式 API：

```cpp
class PolynomialKernel {
public:
    virtual int degree(PolyId p, VarId var) const = 0;

    virtual PolyId leadingCoefficient(PolyId p, VarId var) = 0;

    virtual PolyId substituteRational(
        PolyId p,
        VarId var,
        const mpq_class& value
    ) = 0;

    virtual PolyId pseudoRemainder(
        PolyId p,
        PolyId divisor,
        VarId var
    ) = 0;

    virtual bool isUnivariateIn(PolyId p, VarId var) const = 0;

    virtual UniPolyId toUniPoly(PolyId p, VarId var) = 0;
};
```

旧 API 保留为 wrapper：

```cpp
int degree(PolyId p, std::string_view name) const {
    return degree(p, getExistingVarOrNull(name));
}

PolyId substituteRational(
    PolyId p,
    std::string_view name,
    const mpq_class& value
) {
    return substituteRational(p, getOrCreateVar(name), value);
}
```

注意：`const` 方法里如果不能 `getOrCreateVar`，就用：

```cpp
std::optional<VarId> findVar(std::string_view name) const;
```

# 7. substituteRational 的签名

正式签名：

```cpp
PolyId substituteRational(
    PolyId p,
    VarId var,
    const mpq_class& value
);
```

兼容签名：

```cpp
PolyId substituteRational(
    PolyId p,
    std::string_view varName,
    const mpq_class& value
) {
    return substituteRational(p, getOrCreateVar(varName), value);
}
```

实现第一版用 `terms()` 重构即可：

```cpp
PolyId LibPolyKernel::substituteRational(
    PolyId p,
    VarId var,
    const mpq_class& value
) {
    auto termsOpt = terms(p);
    if (!termsOpt) return NullPoly;

    std::map<std::vector<std::pair<VarId, int>>, mpq_class> acc;

    for (const auto& t : *termsOpt) {
        mpq_class coeff = t.coeff;
        std::vector<std::pair<VarId, int>> newPowers;

        for (auto [v, pow] : t.powers) {
            if (v == var) {
                coeff *= powMpq(value, pow);
            } else {
                newPowers.push_back({v, pow});
            }
        }

        normalizePowers(newPowers);
        acc[newPowers] += coeff;
    }

    return buildPolynomialFromTerms(acc);
}
```

要求：

```text
合并同类项；
删除零系数；
结果 canonicalize；
结果 hash-cons；
如果全消掉，返回 zero polynomial。
```

# 8. specializeToUnivariate 要重构为 VarId 版本

```cpp
SpecializeResult specializeToUnivariate(
    PolyId p,
    const SamplePoint& prefix,
    VarId mainVar
) {
    PolyId q = p;

    for (int i = 0; i < prefix.level(); ++i) {
        VarId v = prefix.varOrder[i];
        const RealAlg& val = prefix.values[i];

        if (v == mainVar) continue;

        if (!val.isRational()) {
            return SpecializeResult::algebraicPrefixRequired();
        }

        q = kernel_->substituteRational(q, v, val.rational);
        if (q == NullPoly) {
            return SpecializeResult::failure();
        }
    }

    if (!kernel_->isUnivariateIn(q, mainVar)) {
        return SpecializeResult::notUnivariateYet();
    }

    UniPolyId up = kernel_->toUniPoly(q, mainVar);
    if (up == NullUniPolyId) {
        return SpecializeResult::failure();
    }

    return SpecializeResult::success(up);
}
```

这比 `NullUniPolyId` 一个返回值清楚很多。

# 9. libpoly variable order 的问题要单独立 contract

这是最容易踩坑的地方。

libpoly 的很多多项式操作是按 **top variable** 做的。比如 pseudo remainder、degree、leading coefficient 可能默认作用于 top variable。

所以 `PolynomialKernel::pseudoRemainder(p, f, var)` 必须保证：

```text
要么支持任意 VarId；
要么检测 var 是否为 top variable，不满足则返回 NullPoly / Unsupported。
```

不要假装支持任意变量。

建议接口返回状态：

```cpp
struct PolyOpResult {
    enum class Status {
        Success,
        UnsupportedVarOrder,
        Failure
    };

    Status status;
    PolyId poly;
};
```

如果暂时嫌麻烦，至少在实现里写死 contract：

```cpp
if (!isTopVariable(p, var) || !isTopVariable(divisor, var)) {
    return NullPoly;
}
```

P3 的 `reduceModuloTower` 遇到 `NullPoly` 就返回 Unknown。

# 11. 为什么这是最高效完备的方法

因为它避免了两类返工：

```text
1. 现在继续用 string，P3/P4 一定要返工；
2. CDCAC 自己维护 VarId，Kernel 仍用 string，会造成变量身份双系统。
```

A' 的优点是：

```text
新代码立刻走 VarId；
旧代码不用一次性全改；
PolynomialKernel 成为唯一变量身份来源；
substituteRational / degree / leadingCoefficient / pseudoRemainder 后续接口统一；
CDCAC tower reduction 不会被变量名问题拖死。
```

---

一句话：**选 A'。PolynomialKernel 建正式 VarId registry，CDCAC 新代码全部迁到 VarId；旧 string API 只作为兼容 wrapper 保留。**
