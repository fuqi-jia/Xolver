## 1. 最可疑：`TheoryAtomRegistry` 被按值传递或被拷贝了

你要立刻检查这些签名：

```cpp
CadicalTheoryPropagator(...)
TheoryManager::setRegistry(...)
CadicalBackend::connectPropagator(...)
```

确保没有这种写法：

```cpp
CadicalTheoryPropagator(TheoryAtomRegistry registry, ...)
```

或者：

```cpp
void setRegistry(TheoryAtomRegistry registry)
```

正确写法应该是：

```cpp
CadicalTheoryPropagator(
    TheoryAtomRegistry& registry,
    TheoryManager& theoryManager,
    TheoryLemmaDatabase& lemmaDb,
    CadicalBackend& backend)
    : registry_(&registry),
      theoryManager_(&theoryManager),
      lemmaDb_(&lemmaDb),
      backend_(&backend) {}
```

以及：

```cpp
void TheoryManager::setRegistry(TheoryAtomRegistry& registry) {
    registry_ = &registry;
}
```

为了马上暴露问题，建议直接把 `TheoryAtomRegistry` 设成不可拷贝、不可移动：

```cpp
class TheoryAtomRegistry {
public:
    TheoryAtomRegistry() = default;
    ~TheoryAtomRegistry() = default;

    TheoryAtomRegistry(const TheoryAtomRegistry&) = delete;
    TheoryAtomRegistry& operator=(const TheoryAtomRegistry&) = delete;

    TheoryAtomRegistry(TheoryAtomRegistry&&) = delete;
    TheoryAtomRegistry& operator=(TheoryAtomRegistry&&) = delete;
};
```

如果这样一改编译失败，基本就能定位到哪里发生了错误拷贝。

## 2. 第二可疑：`records_` 被 clear 了，但 `sharedEqLookup_` 没 clear

你的崩溃形态是：

```cpp
it != sharedEqLookup_.end()
records_[it->second] 越界
```

这非常像 registry 曾经插入过 shared equality，然后某个地方只清了：

```cpp
records_.clear();
```

但没有同步清：

```cpp
sharedEqLookup_.clear();
```

请检查 `TheoryAtomRegistry::reset()`、`clear()`、`startNewCheck()`、`newRound()` 之类函数。必须保证这些结构一起清：

```cpp
void TheoryAtomRegistry::clear() {
    records_.clear();
    sharedEqLookup_.clear();
    eufLookup_.clear();
    linearLookup_.clear();
    satVarToRecord_.clear();
}
```

如果 `records_` 是统一存所有 atom 的 vector，那么任何 lookup map 存的 index 都依赖 `records_` 不被单独清空。

建议在 `getOrCreateSharedEqualityAtom()` 里临时加硬断言：

```cpp
auto it = sharedEqLookup_.find(key);
if (it != sharedEqLookup_.end()) {
    size_t idx = it->second;

    if (idx >= records_.size()) {
        std::cerr << "[BUG] sharedEqLookup_ stale entry\n"
                  << "  key=" << key << "\n"
                  << "  idx=" << idx << "\n"
                  << "  records_.size()=" << records_.size() << "\n"
                  << "  sharedEqLookup_.size()=" << sharedEqLookup_.size() << "\n";
        std::abort();
    }

    const auto& rec = records_.at(idx);
    assert(rec.kind == TheoryAtomKind::SharedEquality);
    assert(rec.sharedEq.a == lo && rec.sharedEq.b == hi);
    return SatLit::positive(rec.satVar);
}
```

先不要用 `operator[]`，改成 `.at()`，让错误更早、更清楚地爆出来。

## 3. 第三可疑：CaDiCaL 还持有旧 propagator 指针

虽然你说 `disconnectPropagator()` 在 `solve()` 之后调用，但要注意异常路径、early return、unsat conflict return 路径。

建议不要手写 connect/disconnect，改成 RAII：

```cpp
class PropagatorConnectionGuard {
public:
    PropagatorConnectionGuard(CadicalBackend& backend,
                              CadicalTheoryPropagator& propagator)
        : backend_(backend) {
        backend_.connectPropagator(&propagator);
    }

    ~PropagatorConnectionGuard() {
        backend_.disconnectPropagator();
    }

private:
    CadicalBackend& backend_;
};
```

然后：

```cpp
TheoryAtomRegistry registry;
TheoryLemmaDatabase lemmaDb;
TheoryManager theoryManager;

theoryManager.setRegistry(registry);

CadicalTheoryPropagator propagator(
    registry, theoryManager, lemmaDb, *cadicalBackend);

PropagatorConnectionGuard guard(*cadicalBackend, propagator);

auto result = cadicalBackend->solve();
```

这样即使中途返回，也不会留下旧 callback 指针。

## 4. `TheoryManager` 自己带一个空的 `SharedTermRegistry` 是设计问题

你提到：

> TheoryManager 自身也带了一个 SharedTermRegistry sharedTermRegistry_ 成员，而 Solver::Impl 也有一个 sharedTermRegistry_。

这个要改。**组合框架里只能有一个权威的 `SharedTermRegistry`。**

否则：

```text
Purifier 用的是 A
EUF/LRA 用的是 A
TheoryManager::solversOwning() 用的是 B
```

这会导致 owner 判断、shared term validity、lazy interning 全部不一致。

建议改成：

```cpp
class TheoryManager {
public:
    void setSharedTermRegistry(SharedTermRegistry& reg) {
        sharedTermRegistry_ = &reg;
    }

private:
    SharedTermRegistry* sharedTermRegistry_ = nullptr;
};
```

不要让 `TheoryManager` 自己默认构造一个 registry。

虽然这不一定直接导致当前 `TheoryAtomRegistry` 段错误，但它很可能导致 `getOrCreateSharedEqualityAtom()` 被错误调用，比如拿到无效的 `SharedTermId` 或错误 owner pair。

## 5. 这个 unsat 例子理论上不需要创建 shared equality atom

你的例子：

```smt2
(assert (> (f x) 0))
(assert (< (f x) 0))
```

purify 后应类似：

```text
u = f(x)      // EUF bridge
u > 0         // LRA
u < 0         // LRA
```

这个 unsat 应该由 LRA 直接发现：

```text
u > 0 ∧ u < 0
```

不需要 arrangement，也不需要 shared equality split。

所以如果 `TheoryManager::check()` 在这个例子里调用了：

```cpp
getOrCreateSharedEqualityAtom(a,b)
```

你还要检查是不是现在 `check()` 里过早枚举了 shared term pair，或者错误地为所有 shared terms 创建 equality atom。

Stage A 的 UFLRA 不应该主动枚举 arrangement literal。只应该在这些情况下创建 shared equality atom：

```text
1. 原始公式中出现 shared equality/disequality；
2. 某个 convex theory 返回 deduced shared equality；
3. 后续 Stage B 非凸 arrangement 需要 decision literal。
```

对于这个例子，正常路径应该是：

```text
LRA check -> conflict lemma -> UNSAT
```

而不是：

```text
TheoryManager check -> create shared equality atom -> crash
```

这一点也值得查。

## 最可能的根因排序

我会按这个概率排序：

1. **`TheoryAtomRegistry` 被拷贝/移动或某个 callback 持有旧对象指针。**
2. **`records_` 被 clear/reset，但 `sharedEqLookup_` 没有同步 clear。**
3. **`TheoryManager` 使用了错误的空 `SharedTermRegistry`，导致错误创建 shared equality atom。**
4. **`TheoryManager::check()` 过早创建 arrangement literals，UFLRA 不该走这条路径。**
5. **真实内存越界写坏了 unordered_map。**

哈希冲突基本可以排除。
