下面给你一版 **ICP（Interval Constraint Propagation）独立框架的详细实现计划**。核心原则先说清楚：

```text id="4s9v0p"
ICP 不应该实现成 NIA 私有模块；
它应该是 arithmetic common framework，
NIA / NRA / IncrementalLinearizer / BoundedNiaSolver / LocalSearch 都作为 client 调用。
```

ICP 在 SMT 中通常作为快速但不一定完整的 nonlinear pruning 技术：维护变量区间 box，对约束做 over-approx contraction，直到 box 为空、达到阈值、找到候选点或需要 split。这个思路也出现在 dReal / iSAT / raSAT 等 nonlinear SMT 系统中；dReal 进一步把 DPLL(T)+ICP 放在 δ-complete 框架里处理 nonlinear real problems。([arXiv][1])

---

# 0. 总体定位

ICP 的职责：

```text id="85jiid"
输入：
  一组 active nonlinear constraints；
  当前变量 domain / box；
  每个 bound 的原因 reasons；
  budget / precision / splitting 配置。

输出：
  1. Conflict：box 被 contract 到 empty；
  2. DomainUpdate：收紧若干变量区间；
  3. Candidate：给出一个候选点，交给 exact validator；
  4. SplitSuggestion：建议分裂某个变量；
  5. NoChange / UnknownBudget。
```

ICP 不负责：

```text id="mx378f"
1. 直接声称 SAT；
2. 用 sampling 证明 UNSAT；
3. 生成 fake atom；
4. 私自调用 SAT solver；
5. 私自实现 NIA/NRA 完整求解。
```

SAT 必须由外层 exact validator 确认；UNSAT 只能来自 sound interval contraction 导致的 empty box。

# 2. 区间数据结构

## 2.1 有理数区间 `IntervalQ`

用于 NRA / RDL / real relaxation：

```cpp id="qjjc03"
struct IntervalQ {
    BoundQ lo;
    BoundQ hi;

    bool loOpen = false;
    bool hiOpen = false;

    bool isEmpty() const;
    bool isPoint() const;
    bool contains(const mpq_class& v) const;
    mpq_class width() const;
};
```

`BoundQ` 支持：

```text id="r822ql"
-∞, finite rational, +∞
```

但第一版建议：

```text id="z0agjw"
ICP 只处理 finite intervals；
出现 infinity 时 NoChange 或要求先由 domain reasoner 给出 finite bound。
```

否则 contractor 会复杂很多。

---

## 2.2 整数区间 `IntervalZ`

用于 NIA / IDL：

```cpp id="h2iedd"
struct IntervalZ {
    mpz_class lo;
    mpz_class hi;

    bool isEmpty() const { return lo > hi; }
    bool isPoint() const { return lo == hi; }
    mpz_class size() const { return hi - lo + 1; }
};
```

NIA 的 contractor 必须做整数 rounding：

```text id="qejnlv"
real contraction result [a,b]
转成 integer interval:
  [ceil(a), floor(b)]
```

---

## 2.3 ReasonedInterval

所有 contraction 必须携带原因：

```cpp id="7z2l8w"
template <typename Interval>
struct ReasonedInterval {
    Interval interval;
    std::vector<SatLit> reasons;
};
```

原因来源：

```text id="c2c56a"
1. 当前 active constraint literal；
2. 原有 domain lower/upper reasons；
3. contractor 推导时使用的中间表达式 reasons；
4. split atom reasons。
```

第一版可以用 coarse reasons：

```text id="ovmpsk"
contractor conflict:
  violated constraint reason + all variable bound reasons used by this constraint
```

不要无 reason 返回 conflict。

---

## 2.4 ReasonedBox

```cpp id="99e6nq"
template <typename Interval>
class ReasonedBox {
public:
    std::optional<ReasonedInterval<Interval>>
    get(const std::string& var) const;

    void set(const std::string& var, ReasonedInterval<Interval> ri);

    NarrowResult narrow(
        const std::string& var,
        const Interval& newInterval,
        const std::vector<SatLit>& reasons
    );

    bool isEmpty() const;
    std::vector<std::string> variables() const;

private:
    std::unordered_map<std::string, ReasonedInterval<Interval>> box_;
};
```

`narrow` 语义：

```text id="pnv4kd"
old ∩ new = empty:
  return Empty with old reasons + new reasons

old ∩ new = old:
  NoChange

otherwise:
  update interval to intersection;
  merge reasons;
  return Changed
```

# 3. ICP 输入表示

ICP 不应该只吃 `PolyId`，因为 contractor 需要表达式结构。

建议同时支持两种输入：

```text id="7fvs0s"
1. Expr DAG input：用于精细 contractor；
2. Poly terms input：用于 polynomial interval evaluation fallback。
```

核心结构：

```cpp id="05soc5"
struct IcpConstraint {
    ExprId expr;              // 原始表达式 DAG，可选
    PolyId poly;              // normalized p rel 0，可选
    Relation rel;             // Leq, Lt, Geq, Gt, Eq, Neq
    SatLit reason;            // active literal
    TheoryId owner;           // NIA / NRA
};
```

统一内部形式：

```text id="7i8fee"
p rel 0
```

比如：

```text id="gh5ll2"
x^2 + y^2 <= 5
→ p = x^2 + y^2 - 5, rel = <=
```

# 4. 基础 Interval Operations

必须是 over-approx。

```cpp id="t29s3o"
IntervalQ add(IntervalQ a, IntervalQ b);
IntervalQ sub(IntervalQ a, IntervalQ b);
IntervalQ neg(IntervalQ a);
IntervalQ mul(IntervalQ a, IntervalQ b);
IntervalQ div(IntervalQ a, IntervalQ b); // if 0∈b, return wide / unsupported
IntervalQ pow(IntervalQ a, unsigned k);
IntervalQ sqrt(IntervalQ a);
```

核心规则：

```text id="ql52mo"
[a,b] + [c,d] = [a+c, b+d]
[a,b] - [c,d] = [a-d, b-c]
-[a,b] = [-b, -a]

[a,b] * [c,d]
= [min(ac,ad,bc,bd), max(ac,ad,bc,bd)]
```

平方：

```text id="znmlfa"
if 0 ∈ [l,u]:
  [l,u]^2 = [0, max(l²,u²)]
else:
  [min(l²,u²), max(l²,u²)]
```

幂：

```text id="mh6u7r"
even power:
  注意跨 0；
odd power:
  单调，端点即可。
```

红线：

```text id="6tj1p8"
不能 corner sampling；
不能 endpoint sampling 当作 over-approx；
不能用 double；
必须用 mpq/mpz。
```

# 5. Forward evaluation 与 backward contraction

ICP 的核心不是单纯 eval interval，而是：

```text id="u5pc3m"
forward:
  根据变量区间计算表达式区间；

backward:
  根据父节点目标区间反向收缩子节点区间。
```

例如：

```text id="q88wvg"
z = x + y
z ∈ [0, 5]
x ∈ [3, 10]
```

反推：

```text id="ci6zq5"
y ∈ z - x = [0,5] - [3,10] = [-10,2]
```

# 7. 关系 contractor

对于 normalized constraint：

```text id="mr002w"
p rel 0
```

先给 `p` 一个 target interval。

## 7.1 `p <= 0`

目标：

```text id="cisvx6"
p ∈ (-∞, 0]
```

如果 forward interval：

```text id="cpxoz2"
p.lo > 0
```

则 empty conflict。

如果：

```text id="g27p4x"
p.hi <= 0
```

说明当前 box 下该约束 guaranteed satisfied，但第一版不要删除约束，只返回 NoChange。

否则做 backward contraction：

```text id="ev8pna"
contract p into (-∞,0]
```

---

## 7.2 `p >= 0`

目标：

```text id="kuslgt"
p ∈ [0, +∞)
```

若 `p.hi < 0`，conflict。

---

## 7.3 `p = 0`

目标：

```text id="9hp63p"
p ∈ [0,0]
```

若 `0 ∉ p.interval`，conflict。

反向 contraction 很有用。

---

## 7.4 `p != 0`

ICP 不擅长 disequality。第一版：

```text id="yjlcwm"
if 0 ∉ p.interval:
  NoChange / satisfied

if p.interval == [0,0]:
  conflict

otherwise:
  NoChange
```

不要试图用 ICP split `p<0 ∨ p>0`，这个交给 theory solver / dynamic atom split。

# 8. Primitive contractors

## 8.1 AddContractor

表达式：

```text id="0q93n5"
z = x + y
```

Forward：

```text id="vbm0ma"
Z = X + Y
```

Backward：

```text id="g5m8p4"
X := X ∩ (Z - Y)
Y := Y ∩ (Z - X)
```

---

## 8.2 SubContractor

```text id="c3f9dz"
z = x - y
```

Backward：

```text id="y33f0k"
X := X ∩ (Z + Y)
Y := Y ∩ (X - Z)
```

---

## 8.3 MulContractor

```text id="c5pa3v"
z = x * y
```

Forward：

```text id="yuky6i"
Z = X * Y
```

Backward：

```text id="2oskon"
X := X ∩ (Z / Y)
Y := Y ∩ (Z / X)
```

Division contractor:

```text id="s8pqfh"
如果 denominator interval 含 0：
  第一版返回 no contraction；
  不要 unsound 地除。
```

可以后续支持 split：

```text id="v6xzd7"
Y contains 0:
  split Y < 0 ∨ Y = 0 ∨ Y > 0
```

但不是 V1。

---

## 8.4 SquareContractor

```text id="6bl9dx"
z = x^2
```

Forward：

```text id="22l5qv"
Z = X^2
```

Backward：

如果：

```text id="7dqovu"
Z.hi < 0
```

empty。

否则：

```text id="g9cz9y"
X := X ∩ [-sqrt(Z.hi), sqrt(Z.hi)]
```

如果 `Z.lo > 0`，更强规则是：

```text id="gr61ul"
X ∈ (-∞,-sqrt(Z.lo)] ∪ [sqrt(Z.lo),∞)
```

这不是单 interval，V1 不处理，或者产生 split suggestion。

---

## 8.5 PowContractor

V1 支持：

```text id="46pbyq"
x^2
x^3
small integer powers
```

偶次幂反向 contraction 处理上界；奇次幂可单调反推：

```text id="qs01am"
x^3 ∈ [a,b]
=> x ∈ [cuberoot(a), cuberoot(b)]
```

NIA 需要整数 floor/ceil root。

---

## 8.6 SumSquaresContractor

直接支持：

```text id="5f6dtg"
x1^2 + ... + xn^2 <= c
```

如果 `c < 0`：

```text id="3zomd8"
conflict
```

否则：

```text id="hl95fl"
xi ∈ [-sqrt(c), sqrt(c)]
```

对 NIA：

```text id="k6004y"
xi ∈ [-floor_sqrt(c), floor_sqrt(c)]
```

对 NRA：

```text id="asvfui"
xi ∈ [-sqrt(c), sqrt(c)]
```

如果不用 algebraic sqrt，V1 用 rational outer bound：

```text id="hju0jf"
r = ceil_sqrt_rational(c)
xi ∈ [-r,r]
```

必须 over-approx。

# 9. Polynomial contractor 策略

不要一开始做完整 HC4/BC3/Box consistency。建议三层：

## V1：pattern contractors

识别：

```text id="tdj9vf"
linear
single-variable polynomial
x^2
x^2 - c
sum of squares
simple product
```

## V2：expression DAG contractor

把 expression 转为 DAG：

```text id="rp1m1y"
t1 = x*y
t2 = t1 + z
t3 = t2 - 5
t3 <= 0
```

对每个中间节点维护 interval，然后 forward/backward worklist。

## V3：term-based polynomial interval

用 `PolynomialKernel::terms()`：

```text id="n3ah70"
p = Σ c_i * monomial_i
```

逐 monomial 做 interval evaluation，返回 over-approx。这个可以用于 conflict detection，但 contraction 弱。

# 10. ICP Engine

```cpp id="7w090u"
class IcpEngine {
public:
    IcpResult run(
        const std::vector<IcpConstraint>& constraints,
        ReasonedBoxQ& box,
        const IcpConfig& config
    );
};
```

主循环：

```cpp id="0xtfek"
IcpResult IcpEngine::run(...) {
    Worklist wl = buildInitialWorklist(constraints);

    int iterations = 0;

    while (!wl.empty()) {
        if (++iterations > config.maxIterations) {
            return UnknownBudget;
        }

        ContractorId cid = wl.pop();
        auto r = contractors[cid]->contract(box);

        if (r.status == Empty) {
            return Conflict(r.conflict);
        }

        if (r.status == Changed) {
            for (auto var : r.changedVars) {
                wl.pushAll(contractorsWatching(var));
            }
        }
    }

    if (box.isEmpty()) {
        return Conflict(boxConflict);
    }

    if (boxIsSmallEnough(box, config)) {
        return Candidate(samplePoint(box));
    }

    return SplitSuggestion(splitter.choose(box, constraints));
}
```

# 11. Worklist / watch 机制

每个 contractor 依赖一组变量：

```text id="oih4hl"
x^2 + y^2 <= 5
depends on x,y
```

建立 watcher：

```cpp id="dgnvyu"
std::unordered_map<Var, std::vector<ContractorId>> watchers;
```

当 `x` 区间收缩：

```text id="vsfglj"
重新调度所有依赖 x 的 contractors
```

避免每轮扫全部约束。

# 12. Splitter

当 ICP 无法继续收缩但 box 仍较大：

```cpp id="vbb3j8"
struct SplitSuggestion {
    std::string var;
    Bound splitPoint;
    SplitKind kind; // LeGe, LtGe, IntegerMid
};
```

NRA split：

```text id="2b818h"
x <= m ∨ x >= m
```

更标准要避免 overlap / strict 问题，V1 可用：

```text id="k7cpe6"
x <= m ∨ x >= m
```

但两个分支 overlap at `m`。这是 sound 但重复。

更好：

```text id="q6otah"
x <= m ∨ x > m
```

需要 strict atom。

NIA split：

```text id="54k3ze"
x <= k ∨ x >= k+1
```

这个无 overlap。

选择变量：

```text id="lkqozq"
1. 最大 width；
2. 出现在最多 unsatisfied/unknown constraints 中；
3. 过去 contraction 最少的变量；
4. 学习模块给出的 activity。
```

# 13. ICP 与 NIA/NRA 的关系

## 13.1 NRA adapter

NRA 使用 `IntervalQ`。

流程：

```text id="fj8nix"
NraSolver:
  active polynomial constraints
  existing real bounds
  -> ReasonedBoxQ
  -> IcpEngine
```

结果处理：

```text id="bh5wry"
Conflict:
  return theory conflict

DomainUpdate:
  生成 bound lemma 或直接更新 internal domain

Candidate:
  exact NRA validator / NLSAT 验证
  失败则继续 split / return Unknown

SplitSuggestion:
  dynamic atom:
    x <= m ∨ x > m
```

NRA 中 ICP 可作为：

```text id="vam7v4"
1. pre-pruning；
2. NLSAT 前置；
3. cell search 的快速 box rejection。
```

---

## 13.2 NIA adapter

NIA 使用 `IntervalZ`，或者用 `IntervalQ` contraction 后 round 到整数。

流程：

```text id="u5b9d7"
NiaSolver:
  DomainStore integer bounds
  -> ReasonedBoxZ
  -> IcpEngine with integer rounding
```

结果：

```text id="ut72s4"
Conflict:
  sound UNSAT

DomainUpdate:
  update DomainStore

Candidate:
  exact integer validator

SplitSuggestion:
  x <= k ∨ x >= k+1
```

NIA 的 ICP 与 bounded enumeration 关系：

```text id="5z4m4c"
LinearDomain
→ Square/SumSquares
→ ICP
→ if all finite and small: bounded enumeration
→ if all finite and large: B&B / split
→ local search SAT finder
```

# 14. Reason / conflict 设计

## 14.1 Bound update reason

如果 contractor 用约束 `C` 和已有 bounds `B1,B2` 推出新 bound `B3`：

```text id="1u9wbe"
reason(B3) = reason(C) ∪ reason(B1) ∪ reason(B2)
```

---

## 14.2 Empty conflict

当 narrow 导致 empty：

```text id="8qtqb8"
old interval reasons
+
new interval reasons
```

构造 clause：

```text id="bcf1fz"
¬r1 ∨ ¬r2 ∨ ... ∨ ¬rk
```

第一版可以 coarse：

```text id="5x79nt"
所有 active nonlinear constraints reasons
+
所有 involved variable bound reasons
```

但不能只给当前 constraint reason。

---

## 14.3 Propagation lemma

如果 ICP 推出：

```text id="l63u7p"
x <= c
```

可以生成：

```text id="dx2qj1"
¬r1 ∨ ... ∨ lit(x<=c)
```

但 V1 建议先只更新 internal domain，不主动生成 SAT lemma。等和 CDCL 集成稳定后再做 outward propagation。

# 16. 与 CDCL(T) 集成

ICP 的结果进入 theory solver：

```cpp id="wzhdyc"
auto r = icp_.run(constraints, box, config);

switch (r.kind) {
case Conflict:
    return TheoryCheckResult::mkConflict(r.conflict);

case DomainUpdate:
    domains_.apply(r.updates);
    return TheoryCheckResult::NoChange; // 或触发下一轮 check

case SplitSuggestion:
    if (shouldSplit) {
        return TheoryCheckResult::mkLemma(makeSplitLemma(r.split));
    }
    return UnknownBudget;

case Candidate:
    if (validator.validate(r.model)) return SatCandidate;
    return NoChange;

case NoChange:
    return NoChange;

case UnknownBudget:
    return Unknown;
}
```

注意：

```text id="9zeyb1"
ICP 不直接 addClause；
只返回 TheoryLemma / TheoryConflict；
dynamic atoms 必须走 registry。
```

# 17. ICP B&B

不要把 B&B 写在 ICP 内核里。ICP 只给 split suggestion。

NIA/NRA 的 B&B client 负责：

```text id="2lr6zc"
solveBox(box):
  r = ICP(box)
  if conflict: closed
  if candidate valid: sat
  if box small:
     exact enumeration / NLSAT
  else:
     split box
```

这样 ICP 可复用，B&B 是 client。

# 20. 红线

```text id="b51ta6"
1. ICP 所有区间必须 over-approx。
2. sampling 只能用于 candidate，不用于 UNSAT。
3. empty box 才能 UNSAT。
```
