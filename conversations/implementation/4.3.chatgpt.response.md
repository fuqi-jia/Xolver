# 1. 最大问题：`LinearAtomManager` 不能同时服务两个 `GeneralSimplex`

计划里写：

```cpp
class LinearAtomManager {
public:
    int getOrCreateAuxVar(GeneralSimplex& gs, const LinearFormKey& form,
                          const mpq_class& rhs);
    SatLit getOrCreateLinearBoundAtom(SatSolver& sat, Atomizer& atomizer,
                                      const LinearFormKey& form,
                                      Relation rel, const mpq_class& rhs);
};
```

这里有潜在错误：`LinearAtomManager` 里面如果维护 `formToAux_`，它的 aux var 是绑定到某一个 `GeneralSimplex` 实例的。`LraSolver` 和 `LiaSolver` 各自有自己的 `GeneralSimplex`，所以不能共享同一个 manager 的 `formToAux_` 状态。

必须明确：

```text
LinearAtomManager 是每个 solver 实例独立拥有的对象。
formToAux_ 属于某个 GeneralSimplex 实例，不能跨 solver 共享。
```

否则会出现：

```text
LraSolver 的 aux var index 被 LiaSolver 拿去用；
或者 dynamic atom 在 Atomizer 中注册了，但 LiaSolver 的 GeneralSimplex 里没有对应 aux var。
```

建议 plan 加：

```text
LinearAtomManager is solver-local. Each LraSolver/LiaSolver owns its own LinearAtomManager bound to its own GeneralSimplex. Atomizer/SatLit registration can be global, but formToAux_ and auxVar mappings must be per-theory-solver.
```

# 2. `getOrCreateLinearBoundAtom` 不应该“创建 GeneralSimplex aux var”

计划里写它必须：

```text
1. Reuse or create the LinearFormKey → GeneralSimplex aux mapping.
```

这个职责放错了。

动态 atom 注册是 SAT/Atomizer 层的事情，不应该在注册 SAT literal 时直接改 `GeneralSimplex`。因为：

```text
1. 这个 literal 只是被加入 clause，不代表当前已经 asserted；
2. 只有 SAT 之后把该 literal 赋 true/false，TheoryManager 才应该 assert 到 solver；
3. 如果 getOrCreateLinearBoundAtom 立刻创建 aux/bound，可能提前污染 theory state；
4. 多 solver 情况下也不知道该改哪个 GeneralSimplex。
```

正确分层应该是：

```text
getOrCreateLinearBoundAtom:
  只注册 atom 语义，让 SAT/Atomizer 知道这个 literal 代表什么；
  不 assert bound；
  不修改 GeneralSimplex 当前 bounds。

LiaSolver::assertLit:
  收到该 dynamic atom 后，
  再通过自己的 manager_.getOrCreateAuxVar(gs_, form) 创建/复用 aux；
  再 assert bound。
```

所以 plan 中 Rule 5 和 Step 5 要改成：

```text
getOrCreateLinearBoundAtom must:
  1. Build canonical AtomKey = (LinearFormKey, relation, rhs).
  2. Reuse/create AtomRecord.
  3. Call sat->newVar() if needed.
  4. Register SatVar -> TheoryAtom with Atomizer.
  5. Return valid SatLit.

It must NOT assert any bound and must NOT mutate GeneralSimplex active bounds.
The solver-local aux var is created later when assertLit receives this atom.
```

这点很重要。

# 3. `SortClass Unknown -> error` 过强，应该是 `Unknown`

计划里风险缓解写：

```text
Strict rule: unknown sort → error; never fallback to LRA in QF_LIA
```

“不能 fallback 到 LRA”是对的，但“error”不一定合适。对于 SMT solver，更专业是：

```text
Unknown sort / unsupported sort / mixed unsupported arithmetic
  => return Unknown with reason,
not internal error, unless parser/typechecker invariant is broken.
```

所以建议改成：

```text
Under QF_LIA, every arithmetic variable in a handled atom must be classified as Int.
If not, the LIA solver returns Unknown/unsupported, not SAT.
It must never silently treat unknown as Real.
```

这样更稳。

# 4. `LIA disequality` 的 gcd split 公式还不完整

计划写：

```text
Compute g = gcd(|coeffs|)
If g > 1 and all vars integer:
  return ¬lit ∨ (p <= -g) ∨ (p >= g)
Else:
  return ¬lit ∨ (p < 0) ∨ (p > 0)
```

这个还差一个关键情况：带常数项的线性形式。

如果：

```text
p(x) = a·x + c
g = gcd(a_i)
```

那么 `p(x)=0` 是否可能取决于：

```text
g | -c
```

如果 `g ∤ -c`，那么 `p=0` 不可能，`p != 0` 恒真，不需要 split。

如果 `g | -c`，则非零值距离 0 的最小步长是 `g`，可以用：

```text
p <= -g ∨ p >= g
```

所以 plan 必须补：

```text
For integer-valued p(x)=a·x+c:
  g = gcd(|a_i|)
  if g == 0:
      if c != 0, disequality is already satisfied;
      if c == 0, conflict: ¬lit
  else if (-c) % g != 0:
      p=0 impossible, disequality already satisfied;
  else:
      ¬lit ∨ (p <= -g) ∨ (p >= g)
```

否则对于：

```text
2x + 1 != 0
```

它会错误生成：

```text
2x + 1 <= -2 ∨ 2x + 1 >= 2
```

但真实情况下 `x=0` 时 `2x+1=1`，满足 disequality，却不满足上面 split。这是**unsound lemma**。这个必须修。

# 5. `check flow` 里 model validation 不能是 TODO

计划里写：

```cpp
// 3. Model validation
// TODO: validate all active atoms

return consistent();
```

这个不能是 TODO。Phase 1 目标里写了 model validation，但实现步骤里又把它 TODO 掉，这是正确性缺口。

在 LIA 中，`consistent()` 就意味着可以返回 SAT；如果不验证 active atoms 和 integer assignment，之前任何映射 bug、动态 atom bug、sort bug 都可能变成 false SAT。

必须改成：

```text
Before returning consistent/SAT, LiaSolver must validate:
  1. all active asserted literals under current model;
  2. every integer variable has denominator 1;
  3. every disequality obligation is satisfied;
  4. dynamic branch atoms are evaluated consistently.
If validation fails, return Unknown/InternalBug in debug, never SAT.
```

# 6. `branch split` 的 form 必须是变量本身，不是 auxVar

对 fractional integer var `x`，branch lemma 是：

```text
x <= floor(q) ∨ x >= ceil(q)
```

这里的 `LinearFormKey` 应该表示：

```text
p(x) = x
```

不是之前某个 constraint 的 aux form。也就是说：

```cpp
form = LinearFormKey{ terms = {{x, 1}}, constant = 0 };
left  = getOrCreateLinearBoundAtom(form, LEQ, lo);
right = getOrCreateLinearBoundAtom(form, GEQ, hi);
```

建议 plan 明确写，否则实现者容易把 `auxVar` 当作 form 去分支。

# 7. 默认 logic 注册仍然有风险

计划里 default 分支仍然注册：

```cpp
registerSolver(LraSolver)
registerSolver(CdcacSolver)
```

这和前面“Exactly one arithmetic solver per logic”的原则冲突。虽然它写在 default unknown logic 下，但仍然是风险。

建议改成：

```text
If logic is unknown or unsupported:
  either infer the minimal arithmetic solver from atoms before registration,
  or return Unknown/unsupported.
Do not default-register both LraSolver and CdcacSolver in Phase 1.
```

否则你前面刚修的“不要 register everything”又绕回来了。

# 结论

这版比上一版**架构上正确很多**：独立 `LiaSolver`、共享 `LinearAtomManager`、只注册一个 arithmetic solver、动态 atom 真实注册，这些方向都对。

但因为 **LIA disequality gcd split 可能生成 unsound lemma**，以及 **model validation 仍是 TODO**，现在不能 Approve。

我的选择：**[4] Revise**。
