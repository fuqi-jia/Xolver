选 **1：扩展 `PolynomialKernel`，添加 partial substitution 接口**。
不要把这个能力藏在 `LibpolyBackend` 里，也不要用 libpoly evaluate 替代。

`reduceModuloTower` 是 CDCAC core 的基础代数操作，不是 libpoly backend 的临时技巧。它需要一个稳定的 kernel-level API：

```cpp
PolyId substituteRational(PolyId p, VarId var, const mpq_class& value);
```

或者更通用一点：

```cpp
PolyId substituteRationals(
    PolyId p,
    const std::unordered_map<VarId, mpq_class>& subst
);
```

## 为什么选 1

因为 `reduceModuloTower` 需要的是：

```text
p(x,y,z)
  substitute x := 2
=> q(y,z)
```

它不是：

```text
p(x,y,z)
  substitute x := 2, y := 3
=> univariate in z
```

而 libpoly 暴露的 `lp_polynomial_to_univariate_m` 是后者：它返回 partial evaluation 后的一元多项式，并且要求 assignment 给除 top variable 外的所有变量赋整数值，不适合你这里“逐层代入、保留多个变量”的场景。([GitHub][1])

libpoly C API 里我看到的是：

```cpp
lp_polynomial_to_univariate_m(...)
lp_polynomial_sgn(...)
lp_polynomial_evaluate(...)
lp_polynomial_roots_isolate(...)
lp_polynomial_prem(...)
lp_polynomial_gcd(...)
```

这些都很有用，但它们不是你现在要的 “replace one variable by rational and return a multivariate polynomial” API。`lp_polynomial_roots_isolate` 可以在 prefix assignment 下隔离 top variable roots，`lp_polynomial_prem` 可以做 pseudo remainder，`lp_polynomial_gcd` 可以做 gcd，但它们不能替代 kernel-level substitution。([GitHub][1])

## 不选 2 的原因

直接在 `LibpolyBackend` 里写 substitution 会让层次反了。

`reduceModuloTower` 需要这些操作：

```text
substituteRational
leadingCoefficient
pseudoRemainder
primitivePart
normalize
isZero
degree
```

这些都应该属于 `PolynomialKernel` 或 `PolynomialAlgebra`，而不是 backend 私有函数。否则后面你会出现：

```text
CdcacCore 依赖 Backend 的私有多项式表示；
NativeBackend / CheckedBackend 无法复用；
tower reduction 和 specialization 各写一套逻辑；
测试很难单独写。
```

libpoly 可以作为 `PolynomialKernel` 的底层实现，但接口应该在 kernel 层暴露。

## 不选 3 的原因

你已经判断对了：不能用 evaluate 代替 reduction。

`evaluate` 可以用来做：

```text
signAt final sample
validation
debug oracle
```

但不能用来做：

```text
exact zero proof for tower
pseudo-remainder reduction
symbolic substitution
```

比如：

```text
x^2 - 2 = 0
y - x = 0
(y - x) != 0
```

这里要证明 `y - x` 在 tower 下精确为 0，必须靠：

```text
reduceModuloTower(y - x) = 0
```

不能靠数值 evaluate。

# 建议新增接口

不要只加一个 `substitute`，建议顺手把 P3 所需的 kernel API 一次性定义清楚。

```cpp
class PolynomialKernel {
public:
    virtual PolyId substituteRational(
        PolyId p,
        VarId var,
        const mpq_class& value
    ) = 0;

    virtual PolyId substituteRationals(
        PolyId p,
        const std::vector<std::pair<VarId, mpq_class>>& subst
    ) = 0;

    virtual PolyId leadingCoefficient(
        PolyId p,
        VarId var
    ) = 0;

    virtual int degree(
        PolyId p,
        VarId var
    ) = 0;

    virtual PolyId pseudoRemainder(
        PolyId p,
        PolyId divisor,
        VarId var
    ) = 0;

    virtual PolyId primitivePart(
        PolyId p
    ) = 0;

    virtual PolyId normalize(
        PolyId p
    ) = 0;
};
```

如果暂时不想一次做这么多，最低也要先加：

```cpp
PolyId substituteRational(PolyId p, VarId var, const mpq_class& value);
```

# substituteRational 的实现方式

第一版用 `terms()` 手工重构是可以的，但要把它做成 kernel API，而不是复制在 `specializeToUnivariate` 里。

伪代码：

```cpp
PolyId LibPolyKernel::substituteRational(
    PolyId p,
    VarId var,
    const mpq_class& value
) {
    auto termsOpt = terms(p);
    if (!termsOpt) return NullPoly;

    std::unordered_map<MonomialKey, mpq_class> acc;

    for (const auto& t : *termsOpt) {
        mpq_class coeff = t.coeff;
        MonomialKey newMono;

        for (const auto& [v, pow] : t.powers) {
            if (v == var) {
                coeff *= powMpq(value, pow);
            } else {
                newMono.add(v, pow);
            }
        }

        acc[newMono] += coeff;
    }

    return buildPolynomialFromTerms(acc);
}
```

关键点：

```text
1. 变量必须用 VarId，不要用 string。
2. value 可以是任意 mpq_class，不要只支持整数。
3. 代入后同类项必须合并。
4. 零系数项必须删除。
5. 结果必须 canonicalize/hash-cons。
6. 如果所有项消掉，返回 zero polynomial。
```

# 建议重构 specializeToUnivariate

现在 `specializeToUnivariate` 里已有一套手工 terms 代入。不要保留两套逻辑。

重构成：

```cpp
SpecializeResult specializeToUnivariate(
    PolyId p,
    const SamplePoint& prefix,
    VarId mainVar
) {
    PolyId q = p;

    for (auto [v, val] : prefix.rationalAssignments()) {
        if (v == mainVar) continue;
        q = kernel_->substituteRational(q, v, val);
        if (q == NullPoly) return failure;
    }

    if (!kernel_->isUnivariateIn(q, mainVar)) {
        return notUnivariateYet;
    }

    return kernel_->toUniPoly(q, mainVar);
}
```

这样：

```text
specializeToUnivariate
reduceModuloTower
signAtOneAlgebraic
future algebraic-prefix root isolation
```

都共享同一套 substitution。

# P3 里的 reduceModuloTower 应该这样用

```cpp
ReductionResult reduceModuloTower(PolyId p, const SamplePoint& s) {
    PolyId q = p;

    for (int i = s.level() - 1; i >= 0; --i) {
        VarId x = s.varOrder[i];
        const RealAlg& a = s.values[i];

        if (a.isRational()) {
            q = kernel_->substituteRational(q, x, a.rational);
            if (q == NullPoly) return ReductionResult::unknown();
            continue;
        }

        const auto& root = a.root;
        PolyId f = chooseLiftedDefiningPoly(root, s.prefix(i));
        if (f == NullPoly) return ReductionResult::unknown();

        PolyId lc = kernel_->leadingCoefficient(f, x);
        Sign lcSign = signAt(lc, s.prefix(i));
        if (lcSign == Sign::Zero || lcSign == Sign::Unknown)
            return ReductionResult::unknown();

        if (kernel_->degree(q, x) >= kernel_->degree(f, x)) {
            q = kernel_->pseudoRemainder(q, f, x);
            if (q == NullPoly) return ReductionResult::unknown();
            q = kernel_->primitivePart(q);
            q = kernel_->normalize(q);
        }
    }

    if (kernel_->isZero(q)) return ReductionResult::zero();
    return ReductionResult::nonzeroCandidate(q);
}
```

# libpoly 在这里怎么用

libpoly 可以用于实现 kernel API 的底层操作：

```text
lp_polynomial_prem      -> pseudoRemainder
lp_polynomial_gcd       -> gcd
lp_polynomial_pp        -> primitivePart
lp_polynomial_degree    -> degree in top variable only，需要小心
lp_polynomial_get_coefficient -> leading coefficient for top variable
```

这些 API 是存在的。`lp_polynomial_prem` 是 pseudo remainder，`lp_polynomial_gcd` 是 gcd，`lp_polynomial_pp` 是 primitive part，`lp_polynomial_get_coefficient` 可以取系数，`lp_polynomial_degree` / `lp_polynomial_top_variable` 是按 top variable 的操作。([GitHub][1])

但注意一个大坑：

```text
libpoly 很多 multivariate polynomial 操作是按 top variable 做的。
```

所以如果你调用：

```cpp
lp_polynomial_prem(A, B)
```

它默认是对当前 top variable 做 pseudo remainder，而不是任意 `VarId var`。这意味着你的 `pseudoRemainder(p, f, var)` 必须保证：

```text
var 是 p 和 f 在 libpoly context order 下的 top variable
```

否则要么：

```text
1. 重新调整 variable order；
2. 或者先用 terms-based 自己实现按 var 的 pseudo remainder；
3. 或者限制 P3 初版只在 var 是 top variable 时启用，不满足则 Unknown。
```

这个 contract 要写清楚。

# 最终建议

选：

```text
方案 1：扩展 PolynomialKernel。
```

具体执行顺序：

```text
1. 新增 PolynomialKernel::substituteRational；
2. 用 terms() 实现第一版；
3. specializeToUnivariate 改为调用 substituteRational；
4. reduceModuloTower 调用 substituteRational；
5. 后续再补 leadingCoefficient / degree / pseudoRemainder / primitivePart / normalize；
6. libpoly C API 只作为这些 kernel 方法的底层实现或 debug oracle。
```

一句话：**这是 kernel 基础设施，不是 backend hack。先把 `substituteRational` 做成正式接口，P3 才不会继续堆临时逻辑。**
