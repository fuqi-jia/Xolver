还要 **Revise**，而且这版看起来并没有真正吸收前两轮的关键修改，反而又回到了几个高风险设计上。当前不能直接交给实现模型。

# 1. 最大 blocker：仍然注册 mixed logic，且没有 Nelson-Oppen

方案一开始说支持 QF_UF 并为 QF_UFLIA/UFLRA/UFNIA/UFNRA 铺路，但 Phase 5 里明确写了：

```cpp
QF_UFLIA => register LiaSolver + EufSolver
QF_UFLRA => register LraSolver + EufSolver
QF_UFNIA/QF_UFNRA similarly
```

同时 Red Lines 又写：

```text
No Nelson-Oppen in V1.
EUF and arithmetic solvers run independently.
```

这两个是直接矛盾的。
**独立运行 EUF + arithmetic，不做 equality sharing，会导致 mixed logic unsound。**

必须改成：

```text
V1 只注册 QF_UF。
QF_UFLIA / QF_UFLRA / QF_UFNIA / QF_UFNRA 显式返回 Unknown / Unsupported。
不要注册两个 solver 假装支持 mixed theory。
```

否则会出现假 SAT。

---

# 2. explanation 又退回到了 class-level BFS，这是不能接受的

方案里 `EGraph` 仍然是：

```cpp
struct ExplainEdge {
    EClassId from;
    EClassId to;
    MergeReason reason;
};
```

并且 explanation algorithm 写的是：

```text
BFS from class(a) to class(b)
```

这个就是上一轮指出过的 blocker。
一旦 `a` 和 `b` 已经在同一个 class，`class(a) == class(b)`，BFS 可能返回空 path，最终生成错误 conflict clause。

必须改成 term-level proof graph：

```cpp
struct ProofEdge {
    EufTermId u;
    EufTermId v;
    MergeReason reason;
};
```

并且 merge 时记录的是原始 term endpoint：

```cpp
proofEdges_.push_back({a, b, reason});
```

解释时：

```cpp
explainEquality(a, b)
```

应该在 term-level proof graph 上从 term `a` BFS 到 term `b`，不是从 class root 到 class root。

---

# 3. function symbol 仍然只用 string name，签名不完整

方案里还是：

```cpp
std::unordered_map<std::string, FuncSymbolId> symbolMap_;
FuncSymbolId internSymbol(const std::string& name);
```

这不够。EUF symbol key 至少应包括：

```cpp
struct FuncSymbolKey {
    std::string name;
    std::vector<SortId> argSorts;
    SortId resultSort;
};
```

否则同名不同 arity / sort 的符号会被错误合并。即使当前 parser 不支持 overloading，内部结构也不要设计成只按 name 区分。

---

# 4. Bool-valued UF predicate 仍然没有处理

QF_UF 中常见：

```smt2
(declare-fun p (U) Bool)
(assert (p a))
(assert (not (p b)))
(assert (= a b))
```

当前方案只处理 `Eq` / `Distinct`，没有把 Bool-valued UF application 作为公式 atom 处理。

需要加入：

```text
(p a)  ==>  (= (p a) true)
(not (p a)) 由 SAT assignment false 表示
```

并且 `EufTermManager` 必须支持：

```cpp
trueConstant()
falseConstant()
```

还要在检查阶段维护：

```text
true != false
```

否则：

```smt2
(assert (= true false))
```

可能被错误认为 SAT。

---

# 5. n-ary `=` / `distinct` 仍然缺失

当前 Atomizer 直接：

```cpp
ExprId lhs = e.children[0];
ExprId rhs = e.children[1];
```

但 SMT-LIB 合法语法包括：

```smt2
(= a b c)
(distinct a b c d)
```

必须规范化：

```text
(= a b c)         ==> (= a b) ∧ (= b c)
(distinct a b c)  ==> distinct(a,b) ∧ distinct(a,c) ∧ distinct(b,c)
```

而且每个 pairwise atom 要有自己的 synthetic ExprId 或 origin metadata，不能共用原始 n-ary ExprId。

---

# 6. TheoryAtomRegistry dedup API 还是不对

现在还是：

```cpp
void registerEufAtom(SatVar satVar, ExprId exprId, const EufAtomPayload& payload);
```

这意味着 Atomizer 已经先 `freshVar()` 了。
如果 registry 发现同一个 EUF atom 已经存在，怎么办？方案没有处理。

推荐改成 get-or-create：

```cpp
SatLit getOrCreateEufAtom(
    const EufAtomPayload& payload,
    ExprId originExpr);
```

由 Registry 负责返回已有 SAT var 或创建新 SAT var。
否则重复 atom 会生成多个 SAT var，SAT 层可能给同一个 theory atom 不同 truth value。

---

# 7. `assertLit` 参数语义仍然混乱

方案里：

```cpp
void assertLit(..., SatLit reason)
```

然后：

```cpp
activeAssignments_.push_back({reason, atom.exprId, value, level});
```

这里 `reason` 实际上应该是 **当前被赋值为 true 的 SAT literal**，不是“reason”。

建议强制改名：

```cpp
SatLit assertedLit;
```

并在冲突子句中统一使用：

```cpp
clause.push_back(assertedLit.negated());
```

不要再叫 reason，否则后面 explanation 很容易写错。

---

# 8. sort mismatch 红线写了，但接口无法实现

Red Lines 写：

```text
Different sorts never merge.
```

但 `EGraph::merge` 接口是：

```cpp
void merge(EufTermId a, EufTermId b, const MergeReason& reason);
```

它没有 `EufTermManager`，无法检查：

```cpp
tm.node(a).sort == tm.node(b).sort
```

应改成：

```cpp
enum class MergeStatus {
    Merged,
    AlreadySame,
    SortMismatch
};

MergeStatus merge(EufTermId a, EufTermId b,
                  const MergeReason& reason,
                  const EufTermManager& tm);
```

sort mismatch 不能静默忽略，应该返回 Unknown 或 internal error。

---

# 9. explanation failure 不能返回空 reasons

当前 explanation 设计没有区分：

```text
真的不需要原因
```

和：

```text
解释失败 / 没找到 path
```

这很危险。解释失败后如果返回空 reasons，可能学到过强子句。

建议返回：

```cpp
struct ExplainResult {
    bool ok;
    std::vector<SatLit> reasons;
};
```

如果 `ok == false`，solver 应该返回 `Unknown` 或 internal error，而不是构造 conflict clause。

---

# 10. push/pop 仍然用整 vector 复制，不建议

当前：

```cpp
std::vector<std::vector<ActiveAssignment>> scopeStack_;
```

然后 push 时复制整个 `activeAssignments_`。这能跑，但不高效，也容易和 CDCL backtrack 产生语义混乱。

更推荐：

```cpp
std::vector<ActiveAssignment> trail_;
std::vector<size_t> scopeLimits_;
```

```cpp
void push() {
    scopeLimits_.push_back(trail_.size());
}

void pop(uint32_t n) {
    while (n-- && !scopeLimits_.empty()) {
        trail_.resize(std::min(scopeLimits_.back(), trail_.size()));
        scopeLimits_.pop_back();
    }
}
```

同时 `backtrackToLevel()` 后要同步 clamp scope limits。

---

# 11. 内部代码命名不一致，会直接影响生成实现

方案里同时出现：

```cpp
EufAtom {
    SatLit reason;
}
```

```cpp
MergeReason {
    SatLit literal;
}
```

后面又写：

```cpp
reason.literal = eq.reason;
neq.reason.negated()
```

还有 `EufAtomPayload` 用 `Relation::Neq`，但项目里现有枚举未必叫这个名字。

建议统一成：

```cpp
struct EufAtom {
    EufTermId lhs;
    EufTermId rhs;
    Relation rel;
    SatLit assertedLit;
};

struct MergeReason {
    MergeReasonKind kind;
    SatLit assertedLit;
    ...
};
```

并在 prompt 里写清楚：

```text
若现有 Relation 枚举没有 Neq，则使用项目已有的 not-equal 枚举名；若不存在再添加。
```

# 推荐收缩后的 V1

现在方案应该收缩为：

```text
V1 目标：sound QF_UF solver。

支持：
1. uninterpreted sort；
2. uninterpreted constants；
3. uninterpreted function applications；
4. equality / disequality；
5. n-ary equality / distinct normalization；
6. Bool-valued UF predicates；
7. true / false constants；
8. congruence closure；
9. term-level explanation；
10. full-rebuild check。

不支持：
1. QF_UFLIA / QF_UFLRA / QF_UFNIA / QF_UFNRA；
2. UF + arithmetic mixed terms；
3. Nelson-Oppen；
4. theory propagation；
5. rollbackable e-graph。
```

# 总体判断

这版不能实现。它保留了 EUF solver 的基本骨架，但几个核心 soundness blocker 还在：

```text
1. mixed logic 注册仍然 unsound；
2. explanation 仍然是 class-level BFS；
3. Bool-valued predicate 缺失；
4. n-ary equality / distinct 缺失；
5. symbol signature 不完整；
6. dedup API 不闭合；
7. sort mismatch 只写了红线，接口上做不到。
```

建议不要在这版基础上继续小修小补，而是直接把 V1 重新定义为 **pure QF_UF baseline**，先实现一个正确的小闭环。mixed UF + arithmetic 等 Nelson-Oppen 方案单独开 V2。
