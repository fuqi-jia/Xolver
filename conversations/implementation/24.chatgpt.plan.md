下面给你一版 **incremental e-graph 中 signature table rollback 与 explanation speed 优化的完整高效计划**。目标是从之前的稳妥版：

```text
backtrack 后标记 sigTableDirty
下一次 check / merge 前全量 rebuild signature table
explanation 时临时扫描 mergeRecords 建图 + BFS
```

升级成真正高性能版本：

```text
signature table 支持 rollback，不需要 backtrack 后全量 rebuild；
explanation 使用 proof forest / explanation forest，不再每次全量 BFS；
congruence merge、propagation、conflict explanation 都能快速生成原因。
```

这两个优化是 EUF solver 从“正确可用”走向“能跑大 benchmark”的关键。

---

# 0. 最终目标

优化后应该达到：

```text
1. assert equality 增量更新 congruence closure；
2. backtrack 只回滚受影响状态，不 rebuild 全表；
3. signature table 无 stale bug；
4. congruent apps 能快速发现；
5. equality explanation 不再扫描全部 mergeRecords；
6. disequality conflict 能快速返回 compact reasons；
7. theory propagation 能快速 explain entailed equality；
8. 所有结构都支持 decision-level rollback。
```

最终架构：

```text
EufTermManager          // 永久 term DAG，不回滚
RollbackUnionFind       // 回滚等价类
RollbackSignatureTable  // 回滚签名表
UseList / Parents       // 永久 parent use-list
ProofForest             // 回滚 explanation forest
IncrementalEGraph       // 调度 merge / congruence / explanation
```

# 1. Baseline 的瓶颈

之前推荐的稳妥方案是：

```text
UF / members / mergeRecords 可回滚；
signature table 在 backtrack 后 dirty；
下次使用时全量 rebuild signature table；
explanation 时从 active mergeRecords 临时建图 BFS。
```

这个 correctness 好，但性能问题明显。

## 1.1 signature table rebuild 的问题

若 terms 很多：

```text
term 数量 = N
function application 数量 = A
```

每次 backtrack 后 rebuild：

```text
O(A * arity)
```

CDCL 中 backtrack 非常频繁，会反复扫描所有 applications。

---

## 1.2 explanation BFS 的问题

每次冲突：

```text
explain(a=b)
```

如果临时从所有 mergeRecords 建图：

```text
O(#merges)
```

冲突多时非常慢。

而 EUF explanation 通常只需要沿着 proof forest 上的一条路径，不应该扫描全部历史。

---

# 2. Signature table 的正确设计

signature table 的职责：

```text
对每个 function application t = f(args)，维护当前 signature：

  sig(t) = f(rep(arg1), ..., rep(argn))

如果有两个 app 拥有相同 signature，它们必须 merge。
```

核心难点：

```text
rep(arg) 会随着 class merge 改变；
所以 app 的 signature 会改变；
backtrack 时 signature 还要恢复。
```

---

# 3. 推荐方案：currentSig + owner-checked rollback table

这是我建议的生产级设计。

维护：

```cpp
std::vector<std::optional<AppSignature>> currentSig_;
RollbackSignatureTable sigTable_;
```

语义：

```text
currentSig_[app] = app 当前注册在 signature table 中的 signature；
sigTable_[signature] = 某个 owner app。
```

每次某个 application 需要重新签名时：

```text
1. unregister old signature；
2. compute new signature；
3. lookup new signature；
4. 如果已有 congruent app，则 enqueue merge；
5. 否则 register new signature。
```

# 4. AppSignature

```cpp
struct AppSignature {
    EufSymbolId symbol;
    std::vector<EufClassId> argReps;

    bool operator==(const AppSignature& o) const {
        return symbol == o.symbol && argReps == o.argReps;
    }
};

struct AppSignatureHash {
    size_t operator()(const AppSignature& s) const {
        size_t h = hashSymbol(s.symbol);
        for (auto r : s.argReps) {
            h = hashCombine(h, std::hash<EufClassId>{}(r));
        }
        return h;
    }
};
```

`argReps` 必须使用当前 `uf.find(arg)`。

# 5. RollbackSignatureTable

## 5.1 基本接口

```cpp
class RollbackSignatureTable {
public:
    std::optional<EufTermId> find(const AppSignature& sig) const;

    void insertOrAssign(const AppSignature& sig, EufTermId owner);

    void eraseIfOwner(const AppSignature& sig, EufTermId owner);

    size_t snapshot() const;
    void rollback(size_t snap);

    void clear();

private:
    std::unordered_map<AppSignature, EufTermId, AppSignatureHash> table_;

    struct Change {
        enum class Kind {
            InsertNew,
            Overwrite,
            Erase
        };

        Kind kind;
        AppSignature sig;
        std::optional<EufTermId> oldOwner;
    };

    std::vector<Change> trail_;
};
```

---

## 5.2 `insertOrAssign`

```cpp
void insertOrAssign(const AppSignature& sig, EufTermId owner) {
    auto it = table_.find(sig);

    if (it == table_.end()) {
        trail_.push_back(Change{
            .kind = Change::Kind::InsertNew,
            .sig = sig,
            .oldOwner = std::nullopt
        });
        table_.emplace(sig, owner);
        return;
    }

    if (it->second == owner) {
        return;
    }

    trail_.push_back(Change{
        .kind = Change::Kind::Overwrite,
        .sig = sig,
        .oldOwner = it->second
    });

    it->second = owner;
}
```

---

## 5.3 `eraseIfOwner`

这个非常关键，不能普通 erase。

```cpp
void eraseIfOwner(const AppSignature& sig, EufTermId owner) {
    auto it = table_.find(sig);
    if (it == table_.end()) return;

    if (it->second != owner) {
        return; // 不能删掉别人的 owner
    }

    trail_.push_back(Change{
        .kind = Change::Kind::Erase,
        .sig = sig,
        .oldOwner = it->second
    });

    table_.erase(it);
}
```

为什么必须 owner check？

因为可能出现：

```text
currentSig_[app1] = sig
currentSig_[app2] = sig
sigTable_[sig] = app2
```

如果 app1 refresh 时直接 erase(sig)，就会把 app2 的有效 entry 删除，导致漏 congruence。

---

## 5.4 rollback

```cpp
void rollback(size_t snap) {
    while (trail_.size() > snap) {
        auto ch = trail_.back();
        trail_.pop_back();

        switch (ch.kind) {
        case Change::Kind::InsertNew:
            table_.erase(ch.sig);
            break;

        case Change::Kind::Overwrite:
            table_[ch.sig] = *ch.oldOwner;
            break;

        case Change::Kind::Erase:
            table_[ch.sig] = *ch.oldOwner;
            break;
        }
    }
}
```

# 6. currentSig rollback

signature table 回滚还不够，`currentSig_` 也必须回滚。

```cpp
struct CurrentSigChange {
    EufTermId app;
    std::optional<AppSignature> oldSig;
};

std::vector<std::optional<AppSignature>> currentSig_;
std::vector<CurrentSigChange> currentSigTrail_;
```

设置 current signature：

```cpp
void setCurrentSig(EufTermId app, std::optional<AppSignature> sig) {
    if (currentSig_[app] == sig) return;

    currentSigTrail_.push_back(CurrentSigChange{
        .app = app,
        .oldSig = currentSig_[app]
    });

    currentSig_[app] = std::move(sig);
}
```

rollback：

```cpp
void rollbackCurrentSig(size_t snap) {
    while (currentSigTrail_.size() > snap) {
        auto ch = currentSigTrail_.back();
        currentSigTrail_.pop_back();
        currentSig_[ch.app] = std::move(ch.oldSig);
    }
}
```

# 7. Snapshot 结构

```cpp
struct EGraphSnapshot {
    size_t ufSnap;
    size_t memberSnap;
    size_t sigTableSnap;
    size_t currentSigSnap;
    size_t mergeRecordSize;
    size_t proofForestSnap;
    size_t diseqSnap;
    size_t mergeQueueSize;
    size_t congruenceQueueSize;
};
```

rollback：

```cpp
void IncrementalEGraph::rollback(EGraphSnapshot s) {
    mergeQueue_.resize(s.mergeQueueSize);
    congruenceQueue_.resize(s.congruenceQueueSize);

    proofForest_.rollback(s.proofForestSnap);
    mergeRecords_.resize(s.mergeRecordSize);

    rollbackCurrentSig(s.currentSigSnap);
    sigTable_.rollback(s.sigTableSnap);

    rollbackMembers(s.memberSnap);
    uf_.rollback(s.ufSnap);
}
```

顺序建议：

```text
先回滚依赖高层结构；
最后回滚 UF。
```

这样如果 rollback 中 debug 检查签名，仍能看到较新 UF；不过一般不在 rollback 中算 signature。

# 8. refreshSignature：核心函数

```cpp
void IncrementalEGraph::refreshSignature(EufTermId app) {
    if (!isApplication(app)) return;

    // 1. unregister old signature
    if (currentSig_[app]) {
        sigTable_.eraseIfOwner(*currentSig_[app], app);
        setCurrentSig(app, std::nullopt);
    }

    // 2. compute new signature
    AppSignature sig = computeSignature(app);

    // 3. find existing owner
    auto otherOpt = sigTable_.find(sig);

    if (!otherOpt) {
        sigTable_.insertOrAssign(sig, app);
        setCurrentSig(app, sig);
        return;
    }

    EufTermId other = *otherOpt;

    // 4. owner might be stale if currentSig_[other] != sig
    if (!currentSig_[other] || *currentSig_[other] != sig) {
        sigTable_.insertOrAssign(sig, app);
        setCurrentSig(app, sig);
        return;
    }

    // 5. if other and app are not same, congruence merge needed
    if (!same(other, app)) {
        MergeReason reason = MergeReason::congruence(
            other,
            app,
            buildArgPairs(other, app)
        );

        enqueueMerge(other, app, reason);

        // Do not register app yet. After merge, classes change.
        return;
    }

    // 6. already same class; can register either owner.
    sigTable_.insertOrAssign(sig, other);
    setCurrentSig(app, sig);
}
```

关键点：

```text
不要在发现 congruence merge 时强行 register app；
因为 merge 后 signature 状态又会变化。
```

# 9. 什么时候 refresh affected applications？

当 merge 两个 classes 时，只有这些 applications 受影响：

```text
parents of members in class A
parents of members in class B
```

流程：

```cpp
void IncrementalEGraph::performMerge(EufTermId a, EufTermId b, MergeReason reason) {
    EufClassId ra = rep(a);
    EufClassId rb = rep(b);
    if (ra == rb) return;

    auto affected = collectParents(ra);
    append(affected, collectParents(rb));

    MergeId mid = recordMerge(a, b, reason, ra, rb);
    EufClassId newRoot = uf_.unite(ra, rb, mid);

    mergeMembers(newRoot, oldRoot);

    for (auto app : affected) {
        congruenceQueue_.push_back(app);
    }
}
```

然后：

```cpp
void processCongruenceQueue() {
    while (!congruenceQueue_.empty()) {
        auto app = congruenceQueue_.front();
        congruenceQueue_.pop_front();
        refreshSignature(app);
    }
}
```

# 12. Signature table 性能优化

## 12.1 Signature hash 缓存

计算 signature 每次要 `find(arg)`，频繁发生。

可以给每个 app 缓存：

```cpp
struct SignatureCacheEntry {
    AppSignature sig;
    uint64_t version;
};
```

但由于 UF rollback 和 merge 会改变 reps，版本管理复杂。V1 优化不必做。

---

## 12.2 affected parent 去重

同一个 parent 可能被多次加入队列。

加 stamp：

```cpp
std::vector<uint64_t> queuedStamp_;
uint64_t currentQueueStamp_ = 1;
```

enqueue：

```cpp
void enqueueCongruence(EufTermId app) {
    if (queuedStamp_[app] == currentQueueStamp_) return;
    queuedStamp_[app] = currentQueueStamp_;
    congruenceQueue_.push_back(app);
}
```

每次 process 前：

```cpp
++currentQueueStamp_;
```

需要注意 rollback 不一定要回滚 stamp，因为 stamp 只是临时去重结构，不参与语义。

---

## 12.3 class parent list

`collectParents(root)` 如果每次遍历 class members 再查 parents，可能慢。

可维护每个 class 的 parent applications：

```cpp
std::vector<std::vector<EufTermId>> classParents_;
```

merge class 时合并 parent list。

但 parent list 会有 duplicates / stale，需去重。
建议第一版先用 `members -> parentsOf`，等 profiling 后再加 classParents。

# 13. Explanation speed 优化：从 BFS 到 proof forest

之前 explanation：

```text
根据 mergeRecords 临时建图；
BFS 找 a 到 b 的路径；
递归展开 congruence reason。
```

优化目标：

```text
每次 merge 就把 explanation forest 维护好；
explain(a,b) 只在森林上找路径；
避免扫描所有 mergeRecords。
```

# 14. ProofForest 设计

每个真正的 merge 对应一条 proof edge：

```text
rootA --reason--> rootB
```

但 union-find root 会变，所以 proof forest 不应该依赖当前 root 作为唯一节点。更稳的是以 **term id / class node id** 作为 proof nodes。

推荐：

```text
proof node = EufTermId
proof edge = asserted/congruence merge connecting two terms
```

每次 merge(a,b,reason)，在 proof forest 中添加一条无向边：

```text
a --reason--> b
```

因为 equality explanation 只需要证明 terms 相等，term-level proof graph 足够。

---

## 14.1 Rollbackable proof graph

```cpp
struct ProofEdge {
    EufTermId u;
    EufTermId v;
    MergeReason reason;
};

class ProofForest {
public:
    void addEdge(EufTermId u, EufTermId v, MergeReason reason);

    size_t snapshot() const;
    void rollback(size_t snap);

    std::vector<ProofEdgeId> path(EufTermId u, EufTermId v) const;

private:
    std::vector<ProofEdge> edges_;
    std::vector<std::vector<std::pair<EufTermId, ProofEdgeId>>> adj_;
};
```

Rollback：

```cpp
void rollback(size_t snap) {
    while (edges_.size() > snap) {
        auto id = edges_.size() - 1;
        auto e = edges_.back();
        edges_.pop_back();

        removeLastAdj(e.u, e.v, id);
        removeLastAdj(e.v, e.u, id);
    }
}
```

如果 adjacency removal 成本麻烦，可以 adj list append-only + active edge count：

```cpp
size_t activeEdgeCount_;
```

path 查找时忽略 `edgeId >= activeEdgeCount_`。

更简单：

```cpp
void rollback(size_t snap) {
    activeEdgeCount_ = snap;
}
```

adj list 不删，BFS 时检查 active。

# 15. Path 查询优化

Proof graph 是森林吗？

如果每次 union 只在两个不同 UF class 之间加边，那么 proof graph 对 asserted/congruence merges 是森林。因为只在 `same(a,b)==false` 时加 edge。

所以 active proof graph 是 forest。

在 forest 上解释 `a=b` 等价于找树路径。

---

## 15.1 V1：BFS on proof forest adjacency

相比扫描所有 mergeRecords 建图，已经快很多，因为 adjacency 已经存在。

```cpp
std::vector<ProofEdgeId> ProofForest::path(EufTermId src, EufTermId dst) {
    queue.clear();
    parentTerm.assign(numTerms, INVALID);
    parentEdge.assign(numTerms, INVALID);

    queue.push(src);
    parentTerm[src] = src;

    while (!queue.empty()) {
        auto u = queue.pop();
        if (u == dst) break;

        for (auto [v, eid] : adj_[u]) {
            if (eid >= activeEdgeCount_) continue;
            if (parentTerm[v] != INVALID) continue;

            parentTerm[v] = u;
            parentEdge[v] = eid;
            queue.push(v);
        }
    }

    reconstruct path;
}
```

复杂度：

```text
O(size of connected proof tree)
```

比扫描所有 mergeRecords 好。

---

## 15.2 V2：dynamic LCA / union tree explanation

更高效的是维护 union tree：

```text
每次 merge 两个 components，创建一个 explanation node E；
E 的 children 是两个 component roots；
edge reason 是 merge reason。
```

然后 explain(a,b) 通过 LCA 找路径。

但支持 rollback 和 congruence recursive explanation 更复杂。
建议先做 BFS proof forest，再优化。

# 17. Recursive explanation optimization

`MergeReason::Congruence` 解释时会递归解释 arg equalities。

```text
explain f(a,b)=f(c,d)
  explain a=c
  explain b=d
```

这些子解释可能重复。使用 per-call memo：

```cpp
std::unordered_map<PairTerm, std::vector<SatLit>> localMemo;
```

接口：

```cpp
std::vector<SatLit> explainEquality(EufTermId a, EufTermId b) {
    ExplainContext ctx;
    return explainEqualityRec(a,b,ctx);
}
```

```cpp
std::vector<SatLit> explainEqualityRec(a,b,ctx) {
    auto key = canonicalPair(a,b);

    if (auto it = ctx.memo.find(key); it != ctx.memo.end()) {
        return it->second;
    }

    if (a == b) return {};

    auto path = proofForest_.path(a,b);

    std::vector<SatLit> out;
    for (auto eid : path) {
        auto rs = explainReason(edge[eid].reason, ctx);
        appendUnique(out, rs);
    }

    ctx.memo[key] = out;
    return out;
}
```

# 19. Explanation minimization

基础 explanation 可能包含冗余 equality。后续可以做轻量 minimization：

```text
对于 reason set R：
  尝试删除某个 literal r；

```

这个昂贵，不建议 V1 做。

# 20. ProofForest + congruence explanation 正确性

每个 proof edge 的 reason 必须满足：

```text
reason entails u = v
```

两类：

```text
AssertedEquality:
  SAT literal l means u=v

Congruence:
  app1=f(a1..an), app2=f(b1..bn)
  premises are a1=b1,...,an=bn
```

因此递归解释能最终落到 asserted equality literals。

Conflict:

```text
explain(lhs=rhs) + diseq literal
```

sound。

# 21. Congruence reason 的 argPairs 必须冻结

当创建 congruence merge 时，必须保存：

```cpp
std::vector<std::pair<EufTermId,EufTermId>> argPairs;
```

不要之后再从 current terms / reps 推断，因为 backtrack / later merges 会改变 reps，导致 explanation 错。

创建时：

```cpp
MergeReason makeCongruenceReason(EufTermId app1, EufTermId app2) {
    const auto& n1 = tm_.node(app1);
    const auto& n2 = tm_.node(app2);

    assert(n1.symbol == n2.symbol);
    assert(n1.args.size() == n2.args.size());

    std::vector<Pair> pairs;
    for (i) {
        pairs.push_back({n1.args[i], n2.args[i]});
    }

    return MergeReason::congruence(app1, app2, pairs);
}
```

    auto ra = rep(a);

## 22.3 refresh application

```cpp
void refreshSignature(app) {
    unregisterCurrentSig(app);

    auto sig = computeSignature(app);
    auto owner = sigTable_.find(sig);

    if (!owner) {
        registerSig(app,sig);
        return;
    }

    if (!ownerIsFresh(*owner, sig)) {
        sigTable_.insertOrAssign(sig, app);
        setCurrentSig(app, sig);
        return;
    }

    if (!same(app,*owner)) {
        enqueueMerge(app,*owner, makeCongruenceReason(app,*owner));
        return;
    }

    registerSig(app,sig);
}
```

# 23. Rollback optimized workflow

```cpp
void rollback(snapshot s) {
    mergeQueue_.resize(s.mergeQueueSize);
    congruenceQueue_.resize(s.congruenceQueueSize);

    explainCache_.clear();

    proofForest_.rollback(s.proofForestSnap);
    mergeRecords_.resize(s.mergeRecordSize);

    rollbackCurrentSig(s.currentSigSnap);
    sigTable_.rollback(s.sigTableSnap);

    rollbackMembers(s.memberSnap);
    uf_.rollback(s.ufSnap);
}
```

No rebuild.

# 28. 最终建议

最完整高效的方案是：

```text
Signature table:
  rollbackable table
  + currentSig_[app]
  + eraseIfOwner
  + stale owner check
  + affected-parent refresh

Explanation:
  proof forest
  + rollback active edge count
  + path BFS
  + recursive congruence explanation
  + local memo
  + reason dedup
```

不要一开始就做复杂动态 LCA。
先把 proof forest BFS 做好，它已经比“每次扫描所有 mergeRecords”强很多，而且 correctness 更容易保证。

后续如果 explanation 仍然是瓶颈，再做：

```text
dynamic LCA / union tree explanation
proof compression
reason minimization
```

第一轮最应该实现的是：

```text
RollbackSignatureTable + currentSig rollback + ProofForest path explanation
```

这三件事完成后，incremental e-graph 才算真正从“能增量”变成“高效可回滚”。
