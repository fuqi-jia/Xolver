建议 **再 Revise 一小轮**。这版已经明显比上一版稳很多，之前几个大问题基本修掉了：现在 ICP 被定位成 theory-agnostic framework；`IcpConstraint` 已经加入 `std::optional<ExprId>`，并明确 V1 只做 `PolyId` forward evaluation + square/sum-of-squares pattern contractors；Candidate 也已经从 V1 移除；`ContractorZ/ContractorQ` 分开，避免 NIA 的 Z→Q→Z 转换；旧的 `SquareBoundReasoner / SumOfSquaresBoundReasoner` 也不再立即删除，而是并行保留。整体方向是对的。

但还有几个执行级问题需要修。

---

# 1. `IcpResult` / `BoundUpdate` 类型还不完整

现在写的是：

```cpp
struct BoundUpdate {
    std::string var;
    Interval newInterval;
    std::vector<SatLit> reasons;
};
```

但你现在明确有两套 box：

```text
ReasonedBoxZ -> IntervalZ
ReasonedBoxQ -> IntervalQ
```

所以这里的 `Interval` 是未定义/不清楚的。建议二选一。

最稳的是 V1 直接定义两套结果：

```cpp
struct BoundUpdateZ {
    std::string var;
    IntervalZ newInterval;
    std::vector<SatLit> reasons;
};

struct BoundUpdateQ {
    std::string var;
    IntervalQ newInterval;
    std::vector<SatLit> reasons;
};

struct IcpResultZ {
    IcpStatus status;
    std::optional<TheoryConflict> conflict;
    std::vector<BoundUpdateZ> updates;
    std::optional<SplitSuggestion> split;
};

struct IcpResultQ {
    IcpStatus status;
    std::optional<TheoryConflict> conflict;
    std::vector<BoundUpdateQ> updates;
    std::optional<SplitSuggestion> split;
};
```

或者用模板，但既然你已经选择避免模板膨胀，那就用 `IcpResultZ / IcpResultQ`。

---

# 2. `IcpEngine` 也要区分 Z/Q

现在计划里仍然写一个统一的：

```cpp
IcpResult r = icpEngine.run(...);
```

但前面 contractor 已经分成：

```cpp
ContractorZ
ContractorQ
```

所以 engine 也应该明确：

```cpp
class IcpEngineZ {
public:
    IcpResultZ run(
        const std::vector<IcpConstraint>& constraints,
        ReasonedBoxZ& box,
        const IcpConfig& config);
};

class IcpEngineQ {
public:
    IcpResultQ run(
        const std::vector<IcpConstraint>& constraints,
        ReasonedBoxQ& box,
        const IcpConfig& config);
};
```

这样 `NiaIcpAdapter` 调 `IcpEngineZ`，`NraIcpAdapter` 调 `IcpEngineQ`。否则 agent 实现时会卡在 “一个 engine 怎么同时持有 `ContractorZ` 和 `ContractorQ`”。

---

# 3. `UnknownBudget` 不应该直接导致顶层 Unknown

当前 CDCL(T) integration 写：

```cpp
case UnknownBudget:
    return Unknown;
```

这对 ICP 来说太强了。ICP 是 pruning/advisor，不是完整求解器。如果 ICP budget 用完，仍然可以继续：

```text
bounded enumeration
local search
branch lemma
NRA CDCAC/NLSAT
NIA algebraic reasoning
```

所以建议改成：

```cpp
case UnknownBudget:
    return NoChange;  // 或 IcpSkipped，继续后续 pipeline
```

只有当某个模式明确“必须依赖 ICP 完成”时，才返回 top-level Unknown。对当前 NLColver，ICP 应该是可选增强，因此 budget exhausted 应该是：

```text
不剪枝，但继续求解。
```

红线应该改成：

```text
Budget exhaustion → NoChange / UnknownBudget to caller; caller must not report UNSAT.
NIA/NRA adapter should normally treat ICP UnknownBudget as NoChange and continue pipeline.
```

---

# 4. `SquareContractor` 的 V1 语义要写成“直接 pattern rule”，不要写成泛化 `z=x²`

当前写：

```text
SquareContractor — forward + backward; Z.hi < 0 → Empty
```

但 V1 没有 expression DAG / intermediate variable，所以不要写得像处理：

```text
z = x^2
```

更准确的 V1 应该是直接识别 normalized pattern：

```text
x^2 - c <= 0
x^2 - c = 0
x^2 - c != 0
```

以及：

```text
Σ xi^2 - c <= 0
Σ xi^2 - c = 0
```

然后更新变量 bound。

建议改成：

```text
V1 SquareContractor:
  input: normalized p rel 0
  recognize p = x^2 - c
  produce bound updates / conflict

V1 SumSquaresContractor:
  recognize p = Σ xi^2 - c
  produce bound updates / conflict

General z=x^2 contractor is V2 after ExprId DAG / temporary expression nodes exist.
```

否则实现者可能会开始引入中间节点 `z`，这会把 ICP V1 范围扩大。

---

# 5. `PowContractor` 建议从 V1 删除或降级

V1 已经有：

```text
SquareContractor
SumSquaresContractor
Relation forward conflict
```

这已经足够。`PowContractor` 很容易引入奇偶幂、负数区间、整数 rounding、root 计算等边界问题。

建议 V1 只做：

```text
x^2
Σxi^2
single-var forward interval evaluation
```

`PowContractor` 放 V2。否则 “small integer powers only” 也需要非常严格的 root / rounding 设计。

---

# 6. Strict relation 的策略还需要更具体

现在写：

```text
NIA strict normalized to non-strict before ICP.
NRA strict either handled with open intervals or returns NoChange in V1.
```

这可以，但建议再明确一条：

```text
If NRA strict is not fully implemented, RelationContractor must return NoChange for Lt/Gt, not approximate them as Leq/Geq.
```

不能把：

```text
p < 0
```

粗暴当成：

```text
p <= 0
```

用于 conflict/pruning。对 over-approx 来说，方向稍微写错就会产生 false UNSAT。

---

# 7. `DomainUpdate -> return NoChange` 后要确保后续 pipeline 使用更新后的 domain

现在写：

```cpp
case DomainUpdate:
    domainStore.update(...);
    return NoChange;
```

这可以，但要明确：

```text
更新 DomainStore 后，本轮 NiaSolver::check() 必须继续使用更新后的 domain。
如果 update 导致 empty domain，应立即返回 Conflict。
如果 update 改变了 domain，最好重新运行 bounded/local-search 前的相关步骤，或者至少 bounded solver 读取最新 DomainStore。
```

建议加：

```cpp
case DomainUpdate:
    bool changed = false;
    for (...) {
        auto ur = domainStore.update(...);
        if (ur.conflict) return mkConflict(ur.conflict);
        changed |= ur.changed;
    }
    if (changed) markDomainChanged();
    return NoChange;
```

否则可能出现 ICP 更新了区间，但后续 bounded solver 没重新拿最新 domain 的问题。

---

# 8. SplitSuggestion 放 Phase 6 可以，但不要让 ICP 直接生成 lemma

你已经写了 client owns split/B&B，这是对的。

但 integration 里仍写：

```cpp
case SplitSuggestion:
    return TheoryCheckResult::mkLemma(makeSplitLemma(r.split));
```

这个应该放在 `NiaIcpAdapter / NraIcpAdapter`，不是 `IcpEngine`。建议写成：

```text
IcpEngine returns SplitSuggestion only.
NiaIcpAdapter decides whether to turn it into x <= k ∨ x >= k+1 lemma.
NraIcpAdapter decides whether strict split is supported; otherwise ignore/NoChange.
```

因为 NRA split 需要 strict atom support，当前 V1 未必有。

# 结论

这版已经接近可执行，但还不建议直接 approve。主要 blocker 是：

```text
IcpResult / IcpEngine 的 Z/Q 类型没有彻底分开；
UnknownBudget 不能直接顶层 Unknown；
SquareContractor 语义仍像泛化 z=x²；
PowContractor V1 范围偏大。
```

修完这些以后就可以执行。整体架构方向是正确的：ICP 独立、over-approx、reason tracking、NIA/NRA adapter、B&B/split 由 client 负责，这些原则都可以保留。
