建议选 **[4] Revise**，不是 Reject。整体方向是对的：它已经不是“小补丁”，而是把 NIA-Core 按 `Normalizer + Validator + DomainStore + Univariate + AlgebraicReasoner + BoundedSolver + LocalSearch` 搭起来，目标也明确要求 `x^2=4 -> sat`、`x^2=2 -> unsat`、bounded NIA 完备搜索，这个方向符合你要的求解器内核。

但这版还有几个**正确性硬伤**，如果直接执行，可能会产生 false SAT / false UNSAT。

---

# 1. `StubPolyKernel::evalInteger()` 不能返回 0

计划里写：

```text
Stub in StubPolyKernel: Return 0 for evalInteger.
```

这个绝对不行。`evalInteger()` 会被 `IntegerModelValidator`、`BoundedNiaSolver`、`LocalSearch` 用来判断 SAT。如果 stub 返回 0，会导致错误 SAT 或错误 UNSAT。

必须改成：

```text
StubPolyKernel::evalInteger must not return fake 0.
It should either:
  1. implement exact evaluation, or
  2. throw / return unsupported through an optional-result API, causing Unknown.
```

更好的接口是：

```cpp
virtual std::optional<mpz_class> evalInteger(
    PolyId a,
    const IntegerModel& sample
) const = 0;
```

如果不能求值，返回 `nullopt`，上层返回 `Unknown`。

---

# 2. RRT 枚举被 cap 后，不能推出 UNSAT

计划里风险写：

```text
RRT divisor enumeration too large for large a0
Cap divisor enumeration; fallback to modular reasoning
```

但前面的算法又写：

```text
If no roots → UNSAT conflict
```

这两个结合起来会有 soundness 风险。
如果因为 divisor 太多只枚举了一部分，那么“没找到根”不等于“没有整数根”。

必须把结果类型改成三态：

```cpp
enum class IntegerRootStatus {
    Complete,
    Incomplete
};

struct IntegerRootResult {
    IntegerRootStatus status;
    std::set<mpz_class> roots;
    bool allIntegers = false;
};
```

规则：

```text
RRT 完整枚举所有候选且 roots empty:
  p(x)=0 -> UNSAT

RRT 被 cap / coefficient extraction incomplete:
  不能返回 UNSAT，只能返回 no-result 或 Unknown
```

---

# 3. 零多项式必须单独处理

一元等式里必须处理：

```text
p(x) ≡ 0
```

情况：

```text
0 = 0      tautology
0 != 0     UNSAT
0 <= 0     tautology
0 >= 0     tautology
```

不能把零多项式丢给 “constant term divisors” 或 RRT。

补规则：

```cpp
if (kernel.isZero(poly)) {
    switch (rel) {
        case Eq:
        case Leq:
        case Geq:
            return noResult; // satisfied
        case Neq:
            return conflict(reason);
    }
}
```

---

# 4. Interval `pow` 不能只看 endpoints

计划里写：

```text
For pow: compute bounds by evaluating at extremes.
```

这对偶次幂是错的。例如：

```text
[-2, 3]^2
```

端点值是 `4, 9`，但真实区间是：

```text
[0, 9]
```

如果 interval 下界算成 4，后续可能错误剪枝，导致 false UNSAT。

必须改成保守 interval arithmetic：

```text
x^k:
  k odd:
    [l^k, u^k]
  k even:
    if 0 ∈ [l,u]:
      [0, max(l^k,u^k)]
    else:
      [min(l^k,u^k), max(l^k,u^k)]
```

乘法也必须用四个端点：

```text
[a,b] * [c,d] =
[min(ac,ad,bc,bd), max(ac,ad,bc,bd)]
```

Interval pruning 必须是 over-approximation。否则 bounded solver 的 UNSAT 不可靠。

---

# 5. `UnivariateIntegerReasoner::run()` 不能用 `optional<TheoryCheckResult>` 表达 domain 更新

计划中：

```cpp
if (auto r = univariate_.run(*normalized, domains_, lemmaDb)) return *r;
```

但 univariate reasoning 的常见结果是：

```text
更新 domains_，继续后续 bounded solver
```

这不是 `TheoryCheckResult::Consistent`。如果 `run()` 返回 `Consistent`，`NiaSolver::check()` 会提前结束，跳过 bounded solver，`x^2=4` 可能仍然不能 SAT。

建议定义专门结果：

```cpp
enum class NiaReasoningKind {
    NoChange,
    DomainUpdated,
    Conflict,
    Lemma,
    Unknown
};

struct NiaReasoningResult {
    NiaReasoningKind kind;
    std::optional<TheoryConflict> conflict;
    std::optional<TheoryLemma> lemma;
};
```

然后流程应是：

```cpp
auto ur = univariate_.run(...);

if (ur.kind == Conflict) return mkConflict(...);
if (ur.kind == Lemma) return mkLemma(...);
if (ur.kind == Unknown) return unknown();
// NoChange / DomainUpdated: continue
```

`AlgebraicIntegerReasoner`、`BoundInferer` 也应如此。

---

# 6. DomainStore 需要 reason，否则 domain-empty conflict 不够严谨

计划里写：

```text
Conflict detection: If any domain becomes empty → TheoryConflict.
```

但 `IntDomain` 只保存 bounds / finite values / excluded values，没有保存这些限制来自哪些 active literals。

如果 domain empty，要么：

```text
返回所有 active reasons 的 conflict
```

这是粗但 sound。

要么更好：

```cpp
struct DomainUpdate {
    SatLit reason;
    ...
};
```

每个 bound / finite set / exclusion 都保存 reason，empty 时返回相关 reasons。

最低要求写清楚：

```text
For Phase NIA-Core, if DomainStore detects empty domain and precise reasons are unavailable,
build conflict from all active constraint reasons. This is coarse but sound.
Do not build a conflict from an incomplete reason set.
```

---

# 7. `getIntegerCoefficients()` 不能依赖不完整 degree / interpolation

计划写：

```text
Evaluate p(0), p(1), ..., p(n) where n = degree
Use Newton interpolation
```

这里需要先能 exact 得到 degree。否则插值点数不够会恢复出错误系数。

必须补：

```cpp
virtual std::optional<int> degree(PolyId p, VarId x) const = 0;
virtual std::optional<std::vector<mpz_class>>
getIntegerCoefficients(PolyId p, VarId x) const = 0;
```

如果 degree 或 coefficients 无法 exact 获取：

```text
UnivariateIntegerReasoner must return no-result / Unknown,
not an UNSAT proof.
```

---

# 8. Coefficient GCD equality 公式要写准

计划写：

```text
g = gcd(|coeff_i|)
constant term c % g != 0 -> UNSAT
```

这里必须明确：

对：

```text
p(x) = Σ a_i m_i + c = 0
```

其中 `m_i` 是非 constant monomials，应该取：

```text
g = gcd(|a_i|)
```

然后检查：

```text
g ∤ (-c)
```

如果把 constant coefficient 也放进 gcd，规则会变弱；如果符号写错，容易实现混乱。

建议写成：

```text
Let p = q + c, where q has no constant term.
Let g = gcd(coefficients of q).
If g != 0 and (-c) mod g != 0, then p=0 is UNSAT over integers.
```

---

# 9. Bounded solver 有预算时，预算耗尽只能 Unknown

计划里 bounded solver 看起来会做 complete enumeration / interval B&B。这个方向对。但如果加任何预算：

```text
max nodes
max time
max productSize
```

预算耗尽不能返回 UNSAT。

必须写：

```text
Finite-domain UNSAT is returned only after exhaustive search of the whole finite domain.
If enumeration/B&B is interrupted by budget, return Unknown.
```

---

# 10. Dynamic polynomial atom 支持可以做，但不要影响 NIA-Core 主路径

计划把 `getOrCreatePolynomialAtom()` 放在基础设施里，这是对的。
但 factor split / branch split 产生 lemma 时必须：

```text
1. lemmaDb 去重；
2. conclusion atom 注册到 TheoryAtomRegistry；
3. addObservedVar；
4. 不使用 fake ExprId 路由；
5. guarded lemma 的 premise 是当前 active reason。
```

这一点计划基本有，但建议在实现约束里再写死。

# 结论

这版计划的**方向可以保留**，但还不能 Approve。
我建议 **[4] Revise**。

核心原因是：目前有几处会直接影响正确性，尤其是：

```text
Stub evalInteger 返回 0；
RRT cap 后仍可能误判无根；
interval pow 非保守；
domain update 和 check result 混在一起；
finite-domain 搜索预算与 UNSAT 语义没分清。
```

这些补完后，这个 NIA-Core 计划就可以执行。
