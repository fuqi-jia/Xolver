这版已经**接近可实现**了，但还要 **Revise**。
不是大方向问题，而是几个实现细节如果不改，仍然会出现 **假 SAT / explanation 不可靠 / API 接不上**。

# 总体评价

比上一版明显好很多，主要 blocker 已经修掉：

```text
1. V1 收缩为 pure QF_UF；
2. mixed logic 显式 unsupported；
3. 不再 fallback 到 LRA；
4. term-level proof graph；
5. Bool predicate；
6. true / false 常量；
7. full signature function symbol；
8. get-or-create dedup；
9. assertedLit；
10. trail push/pop；
11. explanation failure 不再静默当空原因。
```

但当前还不能直接交给实现模型，因为有几个新的关键问题。

---

# 1. 最大 blocker：`true != false` 检查位置错了

你现在在 `check()` 里：

```cpp
// 2b. Built-in: true and false are distinct
if (egraph_.same(true, false)) {
    ...
}

// 3. Assert equalities
for (const auto& eq : equalities_) {
    egraph_.merge(...)
}
```

这会漏掉：

```smt2
(set-logic QF_UF)
(assert (= true false))
(check-sat)
```

因为检查 `true` 和 `false` 是否同类时，还没有 assert equality。之后合并了 `true=false`，但没有再次检查，所以会错误返回 SAT。

必须改成：

```text
1. intern terms
2. add all terms
3. assert equalities
4. close congruence
5. check true != false
6. check asserted disequalities
```

也就是 `true != false` 要放在 equality merge 和 congruence closure 之后。

建议改成：

```cpp
// after closeCongruence()
if (termManager_.trueConstant() != NullEufTerm &&
    termManager_.falseConstant() != NullEufTerm &&
    egraph_.same(termManager_.trueConstant(), termManager_.falseConstant())) {
    auto er = egraph_.explainEquality(
        termManager_.trueConstant(),
        termManager_.falseConstant(),
        termManager_);
    if (!er.ok) return TheoryCheckResult::unknown();

    std::vector<SatLit> clause;
    for (auto lit : er.reasons) {
        clause.push_back(lit.negated());
    }
    return TheoryCheckResult::mkConflict(TheoryConflict{clause});
}
```

这个必须修，否则你自己加的 `true-false conflict` 测试会失败。

---

# 2. Bool-valued UF predicate 还不够：Bool 变量也要处理

你现在只处理：

```cpp
case Kind::UFApply:
    if (e.sort == boolSortId_) {
        // (= this_expr true)
    }
```

但 QF_UF 里还有合法的 Bool 常量 / Bool 变量：

```smt2
(set-logic QF_UF)
(declare-const p Bool)
(assert p)
(assert (= p false))
(check-sat)
```

这个应该 UNSAT。

如果 `p` 在公式位置被 Atomizer 当成普通 SAT variable，而 `(= p false)` 被 EUF 处理，那么 EUF 看不到 `p = true`，会错误 SAT。

所以在 QF_UF 下，**所有 Bool-valued term 在公式位置都应转成 `term = true`**，至少包括：

```text
Kind::UFApply with Bool sort
Kind::Variable with Bool sort
0-ary UF Bool constant
```

建议抽一个 helper：

```cpp
SatLit Atomizer::atomizeBoolTermAsEufPredicate(ExprId eid) {
    EufAtomPayload payload{
        .lhsExpr = eid,
        .rhsExpr = TrueSentinelExpr,
        .rel = Relation::Eq
    };
    return registry_->getOrCreateEufAtom(payload, eid);
}
```

然后在 `atomizeRec` 中：

```cpp
if (defaultTheory_ == TheoryId::EUF &&
    isBoolSort(e.sort) &&
    isTermKind(e.kind)) {
    return atomizeBoolTermAsEufPredicate(eid);
}
```

其中 `isTermKind` 至少包括：

```text
Variable
UFApply
ConstBool
```

`ConstBool` 可以特殊处理，`true` 直接返回 SAT true literal，`false` 直接返回 SAT false literal；但 Bool variable / Bool UFApply 必须进 EUF。

# 3. `boolSortId_ = 0` 不能写死

方案里写：

```cpp
atomizer.setBoolSortId(0);
```

这个风险很大。即使当前实现里 Bool 经常是 0，也不要把这个当作不变量写进 solver。

建议：

```text
CoreIr / SortManager / FrontendAdapter 明确提供 boolSortId()
```

例如：

```cpp
SortId CoreIr::boolSortId() const;
```

然后：

```cpp
atomizer.setBoolSortId(ir->boolSortId());
```

如果现在没有 sort manager，至少在 adapter 里记录：

```cpp
SortId FrontendAdapter::getBoolSortId() const;
```

不要用“假设 Bool 是 0”。

---

# 4. n-ary normalize 的代码和后面的 get-or-create 设计不一致

Phase 2 里仍然写：

```cpp
SatVar pv = freshVar();
SatLit plit = SatLit::positive(pv);

registry_->registerParsedTheoryAtom(
    pv, synthExpr, TheoryId::EUF,
    EufAtomPayload{lhs, rhs, Relation::Eq});
```

但 Phase 4 又说要改成：

```cpp
SatLit lit = registry_->getOrCreateEufAtom(payload, eid);
```

这两个设计不能同时存在。最终方案里要删掉前面的 `freshVar + registerParsedTheoryAtom` 版本，否则实现模型很容易照着旧代码写。

n-ary helper 应统一写成：

```cpp
EufAtomPayload payload{lhs, rhs, Relation::Eq};
SatLit plit = registry_->getOrCreateEufAtom(payload, synthExpr);
pairwiseLits.push_back(plit);
```

`distinct` 同理。

# 5. `TheoryAtomRegistry` 调用 `atomizer_->registerDynamicAtom` 可能形成循环依赖

你写的是：

```text
registry internally calls atomizer_->registerDynamicAtom(...)
```

这在架构上不太干净。现在是 Atomizer 持有 Registry；如果 Registry 反过来持有 Atomizer，就会形成循环依赖，而且实现模型很可能接不上现有代码。

更稳的方案有两个。

## 方案 A：get-or-create 放在 Atomizer 侧

```cpp
SatLit Atomizer::getOrCreateEufAtom(
    const EufAtomPayload& payload,
    ExprId originExpr);
```

Atomizer 自己可以 `freshVar()`，然后调用 Registry 注册。

## 方案 B：Registry 接收 fresh callback

```cpp
SatLit TheoryAtomRegistry::getOrCreateEufAtom(
    const EufAtomPayload& payload,
    ExprId originExpr,
    std::function<SatVar()> freshVar);
```

调用时：

```cpp
SatLit lit = registry_->getOrCreateEufAtom(
    payload, originExpr,
    [&]() { return freshVar(); });
```

我更建议方案 A：**Atomizer 负责 SAT var 创建，Registry 负责记录和 dedup**。这样职责更清楚。

---

# 6. explanation 里的 cycle handling 仍然有隐患

你现在写：

```cpp
if (visitedPairs.count(pk)) return {true, {}};
```

这比“失败返回空原因”好了一点，但仍然危险。
如果某个 equality 的 explanation 还没有真正解释完，就因为 cycle 返回空 reasons，最终 conflict clause 仍可能缺少必要前提。

更安全的 V1 做法是：

```cpp
if (visitedPairs.count(pk)) {
    return {false, {}}; // explanation cycle; fallback unknown
}
```

也就是说，**遇到解释环，不要假装无需原因**。

更完善的做法是给 proof edge 加时间戳：

```cpp
struct ProofEdge {
    EufTermId u;
    EufTermId v;
    MergeReason reason;
    uint32_t edgeId;
};
```

解释某条 congruence edge 的参数 equality 时，只允许使用更早的 proof edges：

```cpp
explainPath(ai, bi, maxEdgeId = currentEdgeId)
```

这样 explanation graph 是因果有序的，不会用“当前结论”解释“当前结论”。

V1 可以先用简单策略：

```text
visited pair 重入 => explanation failure => Unknown
```

不要返回 `{true,{}}`。

# 7. `closeCongruence()` 不能是 `void`

现在：

```cpp
void EGraph::closeCongruence(const EufTermManager& tm)
```

但内部可能出现：

```cpp
status == MergeStatus::SortMismatch
```

你只写了注释：

```cpp
// This should never happen
```

如果真的发生，`check()` 不知道，可能继续跑。

建议改成：

```cpp
enum class CloseStatus {
    Ok,
    SortMismatch,
    InternalError
};

CloseStatus closeCongruence(const EufTermManager& tm);
```

然后：

```cpp
auto cs = egraph_.closeCongruence(termManager_);
if (cs != CloseStatus::Ok) {
    return TheoryCheckResult::unknown();
}
```

# 8. sentinel 定义不要重复放两处

方案里：

```text
src/expr/types.h — add SyntheticExprBase, TrueSentinelExpr, FalseSentinelExpr
```

但 `EufTypes.h` 里又定义了：

```cpp
constexpr ExprId TrueSentinelExpr  = ...
constexpr ExprId FalseSentinelExpr = ...
```

这会导致重复定义或命名冲突。

建议只放一个地方。更合理的是放在：

```cpp
expr/types.h
```

因为 Atomizer 和 EufTermManager 都要用。

例如：

```cpp
constexpr ExprId TrueSentinelExpr  = std::numeric_limits<ExprId>::max() - 1;
constexpr ExprId FalseSentinelExpr = std::numeric_limits<ExprId>::max() - 2;
constexpr ExprId FirstSyntheticExpr = std::numeric_limits<ExprId>::max() - 1024;
```

不过更稳的是不要用固定区间冒充 ExprId，而是显式区分：

```cpp
struct AtomOrigin {
    ExprId originExpr;
    uint32_t syntheticIndex;
    bool isSynthetic;
};
```

如果为了少改代码，使用 sentinel 也可以，但必须集中定义。

# 9. `SyntheticExprBase = 0x40000000` 不够稳

```cpp
constexpr ExprId SyntheticExprBase = 0x40000000;
```

这个假设真实 `ExprId` 永远不会达到这个区间。一般不会，但工程上不是硬保证。

更安全的选择：

```cpp
class SyntheticExprIdAllocator {
public:
    ExprId next() { return nextId_--; }
private:
    ExprId nextId_ = std::numeric_limits<ExprId>::max() - 100;
};
```

并避开：

```text
TrueSentinelExpr
FalseSentinelExpr
```

例如：

```cpp
TrueSentinelExpr  = max - 1
FalseSentinelExpr = max - 2
Synthetic start   = max - 100
```

或者最好如上所述，不把 synthetic origin 伪装成真实 `ExprId`。

# 10. `EufTermManager::trueConstant()` / `falseConstant()` 要明确 lazy create

现在接口是：

```cpp
EufTermId trueConstant() const { return trueConstant_; }
EufTermId falseConstant() const { return falseConstant_; }
```

但 interning logic 又写：

```text
TrueSentinelExpr → trueConstant()
FalseSentinelExpr → falseConstant()
```

如果 `trueConstant_` 还没创建，直接返回的是 `NullEufTerm`。

建议明确：

```cpp
EufTermId EufTermManager::intern(ExprId exprId, const CoreIr& ir) {
    if (exprId == TrueSentinelExpr) return internTrueConstant();
    if (exprId == FalseSentinelExpr) return internFalseConstant();
    ...
}
```

也就是说：

```text
public getter 只读；
intern sentinel 时调用 internTrueConstant / internFalseConstant。
```

`clear()` 也要把二者重置为 `NullEufTerm`。

# 11. `assertLit` 签名修改会影响所有 solver

你在 `EufSolver` 里写：

```cpp
void assertLit(const TheoryAtomRecord& atom, bool value, int level, SatLit assertedLit) override;
```

如果 `TheorySolver` 原来的虚函数不是这个签名，那么所有现有 solver 都要改：

```text
LiaSolver
LraSolver
NiaSolver
NraSolver
IdlSolver
RdlSolver
...
```

这不是问题，但需要在文件清单里明确加上：

```text
TheorySolver interface changed; update all derived solvers and TheoryManager dispatch.
```

否则实现模型可能只改 EUF，导致编译不过。

如果不想大改，也可以给 base class 提供兼容 overload，但为了长期清晰，我支持把 `assertedLit` 作为统一接口传下去。

# 12. 还需要补两个测试

你已有测试不错，但还缺两个很关键的。

## Bool variable 与 EUF equality 联动

```smt2
(set-logic QF_UF)
(declare-const p Bool)
(assert p)
(assert (= p false))
(check-sat)
; UNSAT
```

这个能检测“Bool variable 是否也被转成 p=true”。

## Bool variable congruence

```smt2
(set-logic QF_UF)
(declare-sort U 0)
(declare-fun p (U) Bool)
(declare-const a U)
(declare-const b U)
(assert (= a b))
(assert (= (p a) true))
(assert (= (p b) false))
(check-sat)
; UNSAT
```

这个能检测 predicate、true/false、congruence 三者是否联动正确。

# 建议修改后的核心 check 流程

把 `EufSolver::check()` 固定成这个顺序：

```text
1. clear termManager / egraph / caches
2. 从 trail 构造 effective equalities / disequalities
3. intern all terms
4. add all terms to egraph
5. merge asserted equalities
6. close congruence
7. check built-in true != false
8. check asserted disequalities
9. consistent
```

对应伪代码：

```cpp
// 5. assert equalities
for (const auto& eq : equalities_) {
    auto status = egraph_.merge(eq.lhs, eq.rhs, assertedReason(eq), termManager_);
    if (status != MergeStatus::Merged &&
        status != MergeStatus::AlreadySame) {
        return TheoryCheckResult::unknown();
    }
}

// 6. congruence closure
auto closeStatus = egraph_.closeCongruence(termManager_);
if (closeStatus != CloseStatus::Ok) {
    return TheoryCheckResult::unknown();
}

// 7. built-in true != false
if (both true and false exist && egraph_.same(trueTerm, falseTerm)) {
    return conflict(explainEquality(trueTerm, falseTerm));
}

// 8. asserted disequalities
for (const auto& neq : disequalities_) {
    if (egraph_.same(neq.lhs, neq.rhs)) {
        return conflict(explainEquality(neq.lhs, neq.rhs) + ¬neq.assertedLit);
    }
}
```

# 总体判断

这版可以评价为：

```text
架构基本正确；
V1 范围正确；
主要 soundness 红线已经补上；
但仍需修正若干实现级 blocker。
```

必须改的点是：

```text
1. true != false 检查必须放到 equality merge + congruence closure 之后；
2. Bool 变量在公式位置也要转成 (= term true)，不能只处理 UFApply；
3. 不要写死 boolSortId = 0；
4. n-ary helper 统一使用 getOrCreateEufAtom，删掉旧的 freshVar + registerParsedTheoryAtom；
5. Registry 不要反向依赖 Atomizer，避免循环 API；
6. explanation cycle 不能返回 ok + empty；
7. closeCongruence 需要返回状态；
8. sentinel / synthetic ExprId 定义要集中且避开冲突；
9. assertLit 接口变化要同步更新所有现有 theory solver。
```

改完这些后，这版就可以进入实现阶段了。
