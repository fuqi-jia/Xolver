结论：**需要 Revise 后再实现**。大方向是对的，尤其是“V1 full-rebuild EUF，不急着做 rollbackable e-graph”这个选择很稳；但当前方案里有几个 blocker，不修会导致 **SAT/UNSAT 错误、冲突子句不可靠、混合逻辑 unsound**。

## 1. 最大 blocker：不要在 V1 注册 QF_UFLIA/UFLRA/UFNIA/UFNRA

方案里说 mixed theory 先让 arithmetic solver 和 EUF solver 独立跑，同时“不做 Nelson-Oppen”。这会直接不可靠。

例如：

```smt2
(set-logic QF_UFLIA)
(declare-fun f (Int) Int)
(declare-const x Int)
(declare-const y Int)
(assert (<= x y))
(assert (<= y x))
(assert (distinct (f x) (f y)))
(check-sat)
```

LIA 推出 `x = y`，EUF 应该推出 `f(x)=f(y)`，所以整体 UNSAT。
但如果 LIA 和 EUF 独立运行，EUF 看不到 `x=y`，可能认为 SAT。这是 **soundness bug**，不是“不完整”这么简单。

建议 V1 明确改成：

```text
P0: 只支持 QF_UF。
P1: 支持 UF + 显式共享等式的 very limited fragment，但默认 mixed logic 返回 unknown。
P2: 实现 Nelson-Oppen / equality sharing / arrangement 后再声明支持 QF_UFLIA 等。
```

`Solver.cpp` 里不要先注册 mixed logics，否则以后 benchmark 一跑会出现假 SAT。

---

## 2. explanation 设计有严重问题：不能用“当前 class root”做 BFS

方案写的是：

> Build temporary undirected graph from explainEdges_，BFS from class(a) to class(b)

这在 union-find 里会出错。因为一旦 `a` 和 `b` 已经同类，`class(a) == class(b)`，BFS 起点终点相同，可能直接返回空原因。

例如：

```smt2
(assert (= a b))
(assert (= b c))
(assert (distinct a c))
```

正确冲突子句应是：

```text
¬(a=b) ∨ ¬(b=c) ∨ ¬(a≠c)
```

但如果 explanation 返回空，可能学到：

```text
¬(a≠c)
```

这是不合法的，因为 `a≠c` 本身并不总是矛盾。

建议改成 **term-level proof graph**，不要用当前 class root 做解释图节点：

```cpp
struct ProofEdge {
    EufTermId u;
    EufTermId v;
    MergeReason reason;
};

std::vector<ProofEdge> proofEdges_;
std::vector<std::vector<EdgeId>> proofAdjByTerm_;
```

每次真正 merge 时：

```cpp
if (!same(a, b)) {
    addProofEdge(a, b, reason);  // 注意是原始 term endpoint，不是 class root
    unite(find(a), find(b));
}
```

解释 `explainEquality(a,b)` 时：

1. 在 proof graph 上从 term `a` BFS 到 term `b`；
2. 遇到 asserted equality edge，收集对应 SAT literal；
3. 遇到 congruence edge，递归解释每对参数相等；
4. 用 `visitedPairs` / memo 防止重复递归。

这个改动是必须的，否则 learned clause 不可信。

---

## 3. Bool-valued UF / predicate application 没处理，QF_UF benchmark 会大量缺失

QF_UF 里常见：

```smt2
(declare-fun p (U) Bool)
(assert (p a))
(assert (not (p b)))
(assert (= a b))
```

这应该 UNSAT，因为 `a=b` 推出 `p(a)=p(b)`。

当前方案只处理：

```smt2
(= t1 t2)
(distinct t1 t2)
```

但没有处理 Boolean-valued UF application 在公式位置出现的情况。

建议 Atomizer 增加一类转换：

```text
Bool-valued UF term t appearing as formula atom
=> EUF atom: t = true
```

也就是说：

```smt2
(assert (p a))
```

内部看成：

```smt2
(assert (= (p a) true))
```

而：

```smt2
(assert (not (p a)))
```

通过 SAT literal 的 false assignment 变成 effective disequality：

```text
p(a) != true
```

这样可以和 congruence closure 配合。

如果暂时不做 predicate support，就不要声称完整支持 QF_UF，只能说支持 **equality-only QF_UF fragment**。

---

## 4. `=` 和 `distinct` 不是二元的，需要先规范化

SMT-LIB 里：

```smt2
(= a b c)
(distinct a b c d)
```

都是合法的。

当前方案默认 `children[0]` 和 `children[1]`，会漏掉 n-ary 情况。

建议在 Atomizer 前或 Atomizer 内做 normalize：

```text
(= t1 t2 ... tn)
=> (= t1 t2) ∧ (= t2 t3) ∧ ... ∧ (= t_{n-1} tn)

(distinct t1 ... tn)
=> ∧_{i<j} (distinct ti tj)
```

注意 `distinct` 在否定、析取里出现时不能简单原地拆，需要走已有 Boolean/Tseitin atomization 流程。最干净的方式是增加一个 pre-rewrite，把 n-ary equality/distinct 改写成 Boolean 组合表达式，再交给 Atomizer。

---

## 5. function symbol 不能只用 name，至少要包含 arity 和 sort signature

方案中：

```cpp
name string -> FuncSymbolId
```

风险较大。更稳的是：

```cpp
struct FuncSymbolKey {
    std::string name;
    std::vector<SortId> argSorts;
    SortId resultSort;
};
```

否则不同 sort、不同 arity、同名符号可能被错误合并。即使当前 parser 不允许重载，也建议内部 key 设计成完整 signature，避免以后混合理论或 SMT-LIB 边界情况出问题。

---

## 6. `assertLit` 里不要把参数叫 reason，必须明确它是“当前为真的 SAT literal”

当前方案里：

```cpp
void assertLit(..., SatLit reason)
```

然后 conflict clause 用：

```cpp
clause.push_back(lit.negated());
```

这里必须确认 `reason` 到底是不是“当前赋值为 true 的 literal”。如果它其实是 assignment reason，冲突子句会错。

建议统一改名：

```cpp
SatLit assertedLit;
```

并保证：

```text
raw atom variable p assigned true  => assertedLit = p
raw atom variable p assigned false => assertedLit = ¬p
```

冲突子句永远是：

```text
∨ ¬assertedLit_i
```

这样最不容易错。

## 7. EGraph::merge 需要能检查 sort，当前接口不够

方案红线写了：

> Different sorts never merge.

但接口是：

```cpp
void merge(EufTermId a, EufTermId b, const MergeReason& reason);
```

`EGraph` 自己拿不到 term sort，除非它持有 `EufTermManager`，否则没法检查。

建议改成：

```cpp
bool merge(EufTermId a, EufTermId b,
           const MergeReason& reason,
           const EufTermManager& tm);
```

内部强制：

```cpp
if (tm.node(a).sort != tm.node(b).sort) {
    markInternalErrorOrUnsupported();
}
```

更好的是在 Atomizer 阶段就拒绝 sort-mismatched equality，但 EGraph 也要防御。

---

## 8. push/pop 建议不要复制整个 activeAssignments_

当前方案：

```cpp
scopeStack_.push_back(activeAssignments_);
```

能跑，但容易变成 O(n²)。

V1 更推荐：

```cpp
std::vector<ActiveAssignment> trail_;
std::vector<size_t> scopeLimits_;
```

实现：

```cpp
void push() {
    scopeLimits_.push_back(trail_.size());
}

void pop(uint32_t n) {
    while (n-- && !scopeLimits_.empty()) {
        trail_.resize(scopeLimits_.back());
        scopeLimits_.pop_back();
    }
}
```

CDCL backtrack 继续按 decision level 删除：

```cpp
erase assignments with level > target
```

或者更好地维护按 level 的 trail limit。

## 建议修改后的 V1 范围

我建议把 V1 收紧成这个版本：

```text
目标：实现 sound 的 QF_UF equality solver。

支持：
1. uninterpreted sort constants；
2. uninterpreted function applications；
3. equality / disequality；
4. congruence closure；
5. negated equality；
6. n-ary equality / distinct 规范化；
7. 可选支持 Bool-valued UF predicate，建议支持。

暂不支持：
1. QF_UFLIA/UFLRA/UFNIA/UFNRA；
2. arithmetic terms inside UF arguments；
3. Nelson-Oppen；
4. incremental e-graph rollback；
5. theory propagation。
```

## 总体评价

这份方案的架构拆分是合理的，**最值得保留的是 full-rebuild EUF + 独立 EufTermManager + EGraph + TheoryAtomPayload 这条路线**。

但现在不能直接实现，因为有三个核心风险：

```text
1. mixed logic 注册会导致 unsound SAT；
2. explanation graph 设计会生成错误 conflict clause；
3. Bool-valued UF / n-ary = / distinct 缺失，会导致 QF_UF 覆盖不完整。
```

改完这三类问题后，这个 V1 就可以作为一个稳健的 EUF baseline 开始实现。
