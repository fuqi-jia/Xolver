## 结论：**有条件接受，但还要 Revise 两个关键点**

这一版已经比上一版成熟很多。几个上轮 blocker 都修掉了：arrangement 不再靠 `Eq(a,b) ∨ ¬Eq(a,b)`，而是走 SAT decision；只有 SAT-assigned equality 才进入 `SharedEqualityManager`；NIA/NRA 不再从 sampling/model validation 产生 lemma；`check()` 也改成了 CDCL(T) 单轮调用风格。

所以整体架构可以接受，尤其 **Stage A: EUF + LRA** 可以作为第一阶段实现主线。

但还不能完全批准，因为还有两个会影响 correctness/completeness 的关键问题。

---

## 1. 最大问题：purification 后的共享等式不能都归给单一 theory

计划里现在强调：

> after purification, every atom has exactly one owner theory.
> Bridge atom owner is correct; no atom has multiple owners. 

这个原则对普通 theory atom 是对的，但对 **共享等式** 不对。

例如计划里的测试：

```smt2
(= (f (+ x 1)) y)
```

你现在写成：

```text
t = x + 1    // LIA
f(t) = y     // EUF
```

这里有问题。更安全的 purification 应该是：

```text
t = x + 1    // LIA bridge
u = f(t)     // EUF bridge
u = y        // Combination shared equality atom
```

原因是 `u = y` 是两个共享常量之间的 interface equality，不应该只送给 EUF。它应该注册成 `TheoryId::Combination` 的 shared equality atom，然后由 SAT assignment 决定，再通过 `assertInterfaceEquality(u,y)` 发给相关 owner theories。

建议把 invariant 改成：

```text
After purification:
1. Every non-interface theory atom has exactly one owner theory.
2. Equality between shared constants is represented as a Combination atom.
3. Bridge definitions are routed to the theory in whose signature the RHS is pure.
4. No mixed compound term reaches any theory solver.
```

也就是说，`PurifiedAtom.owner` 允许：

```cpp
TheoryId::EUF
TheoryId::LRA
TheoryId::LIA
TheoryId::NRA
TheoryId::NIA
TheoryId::Combination   // only for shared equalities/disequalities
```

这个必须改，否则后面 `SharedEqualityManager` 和 shared equality atom registry 的设计虽然对，但 purification 前端不会正确产生这些 atom。

---

## 2. Stage B 的 SAT decision fallback 不能用 unit clause

计划里 Stage B-2 写：

> If no native decision hook, fallback: emit unit clause `Eq(a,b)` or `¬Eq(a,b)` under guard. But prefer native decision hook. 

这个不接受。

如果 fallback 直接发：

```text
Eq(a,b)
```

或者：

```text
¬Eq(a,b)
```

它不是 split，而是强行选了一边。除非你有额外机制保证另一边之后也会被系统探索，否则会造成 incompleteness。

这里应该改成硬要求：

```text
QF_UFLIA is enabled only if arrangement literals can be made SAT-decision-visible without constraining one branch.
```

更具体：

```text
B-2 acceptance:
1. getOrCreateSharedEqualityAtom(a,b) creates a real SAT variable.
2. The SAT backend can branch on this variable via decision callback / user decision hook / assumption-based controlled split.
3. No arbitrary unit clause Eq(a,b) or ¬Eq(a,b) may be emitted as arrangement fallback.
4. If no valid decision mechanism exists, QF_UFLIA remains disabled.
```

否则 Stage B 不能声称 functional。

---

## 3. EUF 的 deduced equality 测试表述要改

计划里写：

```text
EUF: assert x=y interface → same(f(x), f(y)) true; getDeduced reports Eq(f(x), f(y))
```

这个不总是对。

Nelson-Oppen 传播的是 **shared constants** 之间的 equality，不是任意 EUF 内部复合项之间的 equality。`f(x)` 和 `f(y)` 只有在 purification 后作为共享常量出现时，才需要作为 shared equality 传播。

如果 `not (= (f x) (f y))` 是一个纯 EUF atom，那么 EUF 在收到 `x=y` 后直接内部 conflict 就行，不需要向组合层报告 `Eq(f(x),f(y))`。

建议改成：

```text
EUF:
1. interface x=y causes e-graph merge.
2. if EUF has asserted disequality f(x)≠f(y), then conflict with congruence explanation.
3. getDeducedSharedEqualities() only reports equalities between registered SharedTermId constants.
```

这个是概念问题，不一定是实现 blocker，但测试写错会误导实现。

---

## 4. EUF model 不应该强行给 numeric concrete values

计划中写：

```text
EufSolver::getModel(): return equivalence class representatives + concrete values.
```

对 EUF 来说，模型主要是等价类/Herbrand 结构。对于 Int/Real sort 的 shared constant，如果 LRA/LIA 给了数值，EUF 只需要检查等价类一致性；如果某个 numeric shared constant 只出现在 EUF 侧，没有 arithmetic 约束，可以由组合层补一个 arbitrary value。

建议改成：

```cpp
struct EufTheoryModel {
    // equality partition over shared terms
    Partition sharedPartition;

    // disequality requirements between classes
    std::vector<std::pair<ClassId, ClassId>> disequalities;

    // uninterpreted function interpretation, optional/lazy
};
```

而不是让 EUF 直接返回 rational values。

---

## 5. pending queue 的 rollback 要写进 Phase A-5

你已经在 acceptance criteria 里写了 pending queues 要 rollback，但 Phase A-5 的伪代码里是：

```cpp
pendingSharedEqEvents_.push_back(...)
...
pendingSharedEqEvents_.clear()
```

这里还缺少 decision level 语义。否则可能出现：

```text
level 3 assign Eq(x,y)
pending queue 里记录事件
还没 check
SAT backtrack 到 level 2
pending event 仍在
下一次 check 错误处理过期 equality
```

建议事件结构加 level：

```cpp
struct PendingSharedEqEvent {
    SharedTermId a, b;
    bool isEquality;
    SatLit reasonLit;
    int decisionLevel;
};
```

并且 backtrack 时清掉高层事件：

```cpp
void TheoryManager::backtrackTo(int level) {
    erase pending events with event.decisionLevel > level;
    sharedEqMgr_.rollback(...);
    arrangementMgr_.rollback(...);
    for solver: solver->backtrackTo(level);
}
```

## 6. Stage A 可以先接受，Stage B/C 不要同时承诺

我建议最终执行口径改成：

```text
Accepted implementation target:
Stage A only: QF_UFLRA / QF_UFRDL.

Stage B:
Design approved only after SAT decision hook is verified.

Stage C:
Skeleton hooks allowed, but QF_UFNRA/QF_UFNIA should remain experimental/disabled unless final validator can safely return Unknown instead of Sat.
```

当前总时间线写成 5–6 周可以保留，但不要让实现者误以为三阶段都同等确定。Stage A 是工程实现；Stage B 取决于 CaDiCaL decision hook；Stage C 是保守接入，不是完整 NO completion。

---

## 最终建议

这版可以改成：

**“Stage A 接受；整体计划有条件接受；Stage B/C 需要补充前置条件。”**

必须修改的点：

1. purification 后的 shared equality/disequality 归 `TheoryId::Combination`，不要强行归单一 owner。
2. 删除 Stage B-2 的 unit-clause fallback；没有合法 decision hook 就不启用 `QF_UFLIA`。
3. EUF 只传播 registered shared constants 的 equality，不传播任意内部复合项 equality。
4. EUF model 返回 equality partition，不强行返回 numeric concrete values。
5. pending shared equality events 必须按 decision level 回滚。

改完这些，Stage A 就可以正式执行。
