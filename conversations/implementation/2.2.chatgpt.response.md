现在这版**已经可以作为实现计划继续推进**，比上一版成熟很多：它已经把 V1 定义成 semi-incremental，而不是伪装成完全增量；修正了 tableau 行符号；补了 `BoundValue/BoundInfo/reason`；也把 `!=` 放到 wrapper 层做 split lemma，不塞进 GeneralSimplex 内核。文档里现在明确写了 tableau/basis 持久、active bounds 每个 SAT model 重建、V2 再做 SAT decision-level 全增量，这个定位是合理的。

但我建议你在 Kimi 那里**选 4 Revise**，再加几个很关键的小修正；不是大改，但不加容易实现出 bug。

# 1. `check()` 里不能直接访问 `lower_.bound.value`

现在伪代码写的是：

```cpp
if (beta_[xi] < lower_[xi].bound.value)
...
else if (beta_[xi] > upper_[xi].bound.value)
```

但 `lower_` 可能是 `NegInf`，`upper_` 可能是 `PosInf`。此时 `.value` 没有语义。虽然 `BoundValue` 有 `NegInf/Finite/PosInf` 三种 kind，但 `check()` 还在把它当成永远 finite 的值用。

应该改成：

```cpp
bool violatesLower(int x) {
    return lower_[x].bound.isFinite()
        && beta_[x] < lower_[x].bound.value;
}

bool violatesUpper(int x) {
    return upper_[x].bound.isFinite()
        && beta_[x] > upper_[x].bound.value;
}
```

然后：

```cpp
if (violatesLower(xi)) {
    ...
    pivotAndUpdate(xi, xj, lower_[xi].bound.value);
} else if (violatesUpper(xi)) {
    ...
    pivotAndUpdate(xi, xj, upper_[xi].bound.value);
}
```

`findEnteringVarToIncrease/Decrease` 也一样，不能直接写：

```cpp
beta_[xj] < upper_[xj].bound
```

而要写：

```cpp
canIncrease(xj) = !upper finite || beta[xj] < upper.value
canDecrease(xj) = !lower finite || beta[xj] > lower.value
```

这是当前最需要修的 correctness bug。

# 2. `BoundInfo.reason` 应该是 optional

现在写：

```cpp
struct BoundInfo {
    BoundValue bound;
    SatLit reason;
};
```

但 `-inf/+inf` 没有 reason。`resetActiveBounds()` 里：

```cpp
for (auto& l : lower_) l = {BoundValue::negInf()};
for (auto& u : upper_) u = {BoundValue::posInf()};
```

这在结构上其实构造不完整。

建议改成：

```cpp
struct BoundInfo {
    BoundValue bound;
    std::optional<SatLit> reason;
};
```

或者：

```cpp
SatLit reason = SatLit::invalid();
```

但我更推荐 optional。冲突解释时必须 assert：

```cpp
assert(lower_[xi].bound.isFinite());
assert(lower_[xi].reason.has_value());
```

无限界绝不能进入 conflictLits。

# 3. `resetActiveBounds()` 和 `trail_` 要说清楚，否则会污染 push/pop

现在 V1 是每个 SAT model 开始：

```cpp
resetActiveBounds();
assert all current theory literals;
check();
```

这时 `assertUpper/assertLower` 又会把旧 bound 压进 `trail_.back()`。如果每个 SAT model 都这么做，而不清理 active trail frame，那么 trail 会不断积累上一轮 SAT model 的临时 bound entries。

建议 V1 直接规定：

```cpp
resetActiveBoundsForSatModel() {
    clear active bounds;
    clear active bound trail frame;
    betaDirty_ = true;
}
```

更干净的分层是：

```text
userTrail_      管用户 push/pop 的公式作用域
activeTrail_    管当前 SAT model 的 bound assertion
```

V1 最简单可以先不把 `push/pop` 和 per-model active bounds 混在一个 `trail_` 里。否则后面会出现：SAT model A 的临时 bound 被 SAT model B 的 pop/restore 逻辑错误恢复。

可以在 plan 里加一句：

```text
In V1, resetActiveBounds() also clears the active bound trail frame. User push/pop and SAT-model active bounds must not share the same trail entries.
```

# 4. `AtomKey` 最好拆成 `LinearFormKey` 和 `AtomKey`

现在 plan 写：

```cpp
struct AtomKey {
    terms;
    rhs;
    Relation rel;
};
unordered_map<AtomKey, int> atomToAux_;
```

这能工作，但不够理想。因为 auxiliary variable `s` 本质对应的是：

```text
s = p(x) - c
```

不是对应某个 relation。

例如：

```text
x <= 1
x < 1
x >= 1
x = 1
```

都可以共用同一个 auxiliary var：

```text
s = x - 1
```

relation 只决定 assertLit 时给 `s` 加什么 bound。

所以更好的结构是：

```cpp
struct LinearFormKey {
    vector<pair<VarId, mpq_class>> terms;
    mpq_class constant;  // p(x) - c
};

struct AtomKey {
    LinearFormKey form;
    Relation rel;
};

unordered_map<LinearFormKey, int, LinearFormKeyHash> formToAux_;
unordered_map<AtomKey, AtomInfo, AtomKeyHash> atomInfo_;
```

这样既避免重复 tableau row，也更接近 simplex 的语义。

这不是 soundness bug，但会影响复用、性能和 conflict 解释的清晰度。

# 5. `disequality split` 要明确返回 SAT lemma，而不是 theory conflict

目前写：

```text
If current model has s = 0, return a split lemma: s < 0 ∨ s > 0
```

这个方向对。但要强调：这不是 UNSAT conflict，而是一个 theory-valid split lemma。

也就是：

```text
s != 0  ⇒  s < 0 ∨ s > 0
```

在当前 assignment 里，如果 literal `s != 0` 为 true，而 simplex model 恰好 `s = 0`，wrapper 应返回：

```text
¬lit(s != 0) ∨ lit(s < 0) ∨ lit(s > 0)
```

或者如果内部不想显式建 `s<0/s>0` 两个 atom，也要动态注册两个 strict bound atoms：

```text
s <= -δ
s >= δ
```

建议 plan 里补一句：

```text
Disequality split is emitted as a theory lemma guarded by the original disequality literal:
    ¬(s != 0) ∨ (s < 0) ∨ (s > 0).
It is not an immediate conflict unless both branches are already contradicted by current bounds.
```

否则实现者容易把 `s != 0` 当前 model 等于 0 误当成直接 UNSAT。

---

# 6. `assertUpper` 需要显式要求输入 bound 是 finite

现在：

```cpp
AssertResult assertUpper(int var, BoundInfo info)
```

建议在开头加：

```cpp
assert(info.bound.isFinite());
assert(info.reason.has_value());
```

因为 `assertUpper/assertLower` 只应该用于真实 finite bound。无限界只用于初始化/reset，不应该被 assert 进来。

---

# 7. conflict explanation 中 `== upper/lower` 不要用裸等号

现在写：

```text
N+ = { xj | a_ij > 0, beta[xj] == upper[xj].bound }
N- = { xj | a_ij < 0, beta[xj] == lower[xj].bound }
```

数学上可以，但工程上建议用“不能再往需要方向移动”的谓词：

```cpp
atUpper(x) = upper[x].finite && beta[x] == upper[x].value
atLower(x) = lower[x].finite && beta[x] == lower[x].value
```

并在解释时 assert reason 存在。

否则如果上界是 `+inf`，裸比较容易误用。
