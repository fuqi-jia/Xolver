**结论：不接受当前版本。**

这版方向比前一版清楚，但不能整体进入实现。原因不是 EUF V1 基线不行，而是这个 enhancement plan 里新增的 **model construction、theory propagation、mixed fallback** 仍然有 soundness 风险。文档自己也说明当前版本缺 model construction 和 theory propagation，并准备加入 mixed-theory fallback；这些正是现在最需要谨慎的部分。

下面是详细修改建议。

# 1. 先修 Bool 语义，否则会有假 SAT

当前 Phase 2 的 model construction 是：

```text
每个 equivalence class 分配 ufval_N
```

这对 uninterpreted sort 没问题，但对 Bool sort 不对。Bool 只有两个值：`true` 和 `false`。如果把 Bool class 当作普通 EUF class 分配 `ufval_0, ufval_1, ...`，会导致模型不合法，更严重的是可能掩盖 solver 的假 SAT。

典型反例：

```smt2
(set-logic QF_UF)
(declare-const p Bool)
(declare-const q Bool)
(assert (distinct p true))
(assert (distinct q true))
(assert (distinct p q))
(check-sat)
```

这个公式应该是 **UNSAT**。
因为 `p != true` 推出 `p = false`，`q != true` 推出 `q = false`，所以 `p != q` 不可能成立。

如果 EUF 把 Bool 当成普通无限 uninterpreted sort，就可能认为：

```text
p = ufval_0
q = ufval_1
true = true
```

然后错误返回 SAT。

## 建议改法

不要让 Bool equality / distinct 走普通 EUF 无限域语义。建议在 Atomizer 层把 Bool equality / distinct 改写为命题公式。

### Bool term 在公式位置

保留现有设计：

```text
p          ==> (= p true)
(p a)      ==> (= (p a) true)
(not p)    ==> SAT 层 literal 取反
```

### Bool equality

```text
(= b1 b2)       ==>  b1 ↔ b2
(= b1 b2 b3)    ==>  (b1 ↔ b2) ∧ (b2 ↔ b3)
```

其中每个 `bi` 先 atomize 成 Bool atom。

### Bool distinct

```text
(distinct b1)          ==> true
(distinct b1 b2)       ==> b1 XOR b2
(distinct b1 b2 b3)    ==> false
```

因为 Bool 域只有两个值，三个 Bool term 不可能 pairwise distinct。

### 必加测试

```smt2
(set-logic QF_UF)
(declare-const p Bool)
(assert p)
(assert (= p false))
(check-sat)
; UNSAT
```

```smt2
(set-logic QF_UF)
(declare-const p Bool)
(declare-const q Bool)
(assert (distinct p true))
(assert (distinct q true))
(assert (distinct p q))
(check-sat)
; UNSAT
```

```smt2
(set-logic QF_UF)
(declare-const p Bool)
(declare-const q Bool)
(declare-const r Bool)
(assert (distinct p q r))
(check-sat)
; UNSAT
```

```smt2
(set-logic QF_UF)
(declare-const p Bool)
(assert (distinct p true))
(check-sat)
(get-model)
; SAT, and p must be false
```

这点是当前版本最大 blocker。

# 2. Model construction 要区分 sort，不能全局 `ufval_N`

当前模型设计是：

```cpp
std::unordered_map<EClassId, std::string> classValues;
```

这太粗。EUF model 至少要按 sort 区分值域。否则不同 sort 的 class 可能都被打印成同一种 `ufval_N`，模型输出不清晰，也不利于后续检查。

建议改成：

```cpp
struct EufClassValue {
    SortId sort;
    std::string value;
};

struct EufModel {
    std::unordered_map<EClassId, EufClassValue> classValues;
    std::unordered_map<FuncSymbolId, EufFunctionInterp> functions;
};
```

生成 class value 时：

```text
Bool sort:
  class(true)  => "true"
  class(false) => "false"
  其他 Bool class 必须根据约束选择 true/false；
  如果无法选择，说明前面 Bool 约束没有处理完整。

Uninterpreted sort S:
  class C => "S!val_N"
```

函数解释也要 sort-correct：

```text
f : S1 × ... × Sn -> R
defaultValue 必须属于 R 的域。
如果 R 是 Bool，则 defaultValue 只能是 true 或 false。
如果 R 是 uninterpreted sort，则可以是 R!val_N。
```

也就是说，`defaultValue = "ufval_0"` 这种全局字符串不够严谨。

---

# 3. Theory propagation 的候选 atom 来源错了

当前计划在 `EufSolver::check()` 里从 `trail_` 收集 propagation candidates：

```cpp
for (const auto& a : trail_) {
    ...
    candidateProps_.push_back(atom);
}
```

这不对。`trail_` 里只有 **已经被 SAT 赋值过的 atom**。
但 theory propagation 要找的是：

```text
尚未赋值，但已经被当前 EUF 等价类蕴含的 equality atom
```

例如 SAT 里有 atom：

```text
p := (= (f a) (f b))
```

当前 theory 已经通过 `a=b` 推出 `f(a)=f(b)`，但如果 `p` 还没有被 SAT 赋值，你就应该传播：

```text
¬(a=b) ∨ (= (f a) (f b))
```

如果只看 `trail_`，你永远看不到这个 unassigned atom。

## 建议改法

必须从 **所有注册过的 EUF atom** 中找候选，而不是从 trail 里找。

可以新增：

```cpp
std::vector<TheoryAtomRecord> TheoryAtomRegistry::recordsForTheory(TheoryId id) const;
```

或者在 Atomizer 完成后，把所有 EUF atom 交给 EufSolver：

```cpp
void EufSolver::setAllRegisteredAtoms(std::vector<TheoryAtomRecord> atoms);
```

propagation loop 应该是：

```cpp
for (const auto& rec : allEufAtoms_) {
    const auto& p = std::get<EufAtomPayload>(rec.payload);

    if (p.rel != Relation::Eq) continue;

    EufTermId lhs = termManager_.intern(p.lhsExpr, ir);
    EufTermId rhs = termManager_.intern(p.rhsExpr, ir);

    if (!egraph_.same(lhs, rhs)) continue;

    SatLit atomLit = SatLit::positive(rec.satVar);

    // 如果能查询 SAT assignment：
    // assigned true  -> skip
    // assigned false -> this is conflict
    // unassigned     -> propagation lemma

    auto er = egraph_.explainEquality(lhs, rhs, termManager_);
    if (!er.ok) continue;

    std::vector<SatLit> clause;
    for (SatLit r : er.reasons) {
        clause.push_back(r.negated());
    }
    clause.push_back(atomLit);

    return TheoryCheckResult::mkLemma(TheoryLemma{std::move(clause)});
}
```

注意这里最后加入的是：

```cpp
atomLit
```

不是：

```cpp
atom.assertedLit
```

`assertedLit` 只适用于 trail 里“当前已经被赋值为 true 的 literal”。对 unassigned atom 来说，这个名字语义是错的。

# 4. 当前 propagation acceptance test 不合格

计划里的测试是：

```smt2
(set-logic QF_UF)
(declare-sort U 0)
(declare-const a U) (declare-const b U)
(declare-fun f (U) U)
(assert (= a b))
(assert (not (= (f a) (f b))))
(check-sat)
```

这个测试即使没有 theory propagation，也可以通过 disequality conflict 检出 UNSAT。它不能证明 propagation 真的生效。

## 建议改成 unit test

应该直接检查 EUF 是否生成 lemma：

```text
给定：
  asserted: (= a b)

registered but unassigned atom:
  p := (= (f a) (f b))

期望 EUF 生成 lemma:
  ¬(= a b) ∨ p
```

也就是测试 `TheoryCheckResult` 里确实出现：

```cpp
{ not lit_ab, lit_fafb }
```

如果现有测试框架拿不到 lemma，那就先不要声称完成 theory propagation。可以把 Phase 3 降级为：

```text
Phase 3: add infrastructure only; no acceptance until lemma-level unit test exists.
```

# 5. Mixed-theory fallback 暂时不要实现

文档前面说：

```text
QF_UFLIA/UFLRA/UFNIA/UFNRA remain Unknown
```

但 Phase 4 又准备做 mixed-theory fallback。

这个容易出问题。尤其是这一句：

```text
If only UF with no arithmetic ops → fall back to pure QF_UF path.
```

不够安全。

例如：

```smt2
(set-logic QF_UFLIA)
(declare-const x Int)
(assert (= x x))
(check-sat)
```

它没有 arithmetic operator，但有 `Int` sort。
如果你把它扔给 QF_UF，会把 `Int` 当 uninterpreted sort 处理。对于 SAT/UNSAT 可能暂时没错，但 model 输出就会不合法，因为 `x` 应该是整数值，不应该是 `ufval_0`。

再比如：

```smt2
(set-logic QF_UFLIA)
(declare-fun f (Int) Int)
(declare-const x Int)
(assert (distinct (f x) (f x)))
(check-sat)
```

这里没有显式 arithmetic op，但 UF 的 domain/codomain 是 Int。纯 EUF 可以判断 UNSAT，但 model 和 theory boundary 都不干净。以后加 Nelson-Oppen 时还容易留下历史包袱。

## 建议

当前版本不要做 fallback，只保留设计注释：

```cpp
// TODO(Combination-V2):
// Mixed UF + arithmetic needs purification, shared equality exchange,
// arrangement lemmas, and Nelson-Oppen style combination.
// Until then QF_UFLIA/QF_UFLRA/QF_UFNIA/QF_UFNRA return Unknown.
```

如果一定要做 fallback，必须非常保守：

```text
pure arithmetic fallback:
  - no UFApply
  - no uninterpreted sort
  - all non-Bool terms are arithmetic terms
  - all predicates are arithmetic predicates or Boolean structure

pure QF_UF fallback:
  - no arithmetic operators
  - no arithmetic sort terms
  - all non-Bool sorts are uninterpreted sorts
  - all function symbols are uninterpreted over uninterpreted/Bool sorts
```

只要出现 `Int` / `Real` sort，就不要走 QF_UF fallback。

我的建议是：**本轮直接删除 Phase 4 fallback，只保留 TODO 注释。**

# 6. Explanation BFS 修复方向对，但递归保护不能全删

Phase 1 说要把 `visitedPairs` 完全删除，改成标准 BFS visited-node。这个对普通无向图 BFS 是正确的。文档也正确指出，当前 bug 是 undirected explanation graph 里 benign revisit 被错误当成 failure。

但是要注意：
**BFS 内部的 visited-node** 和 **congruence explanation 递归层的 cycle guard** 不是一回事。

建议结构改成两层：

```cpp
ExplainResult explainPath(from, to, maxEdgeId, activePairs)
```

内部 BFS：

```text
visitedTerms:
  只用于 BFS，遇到已访问 term 就 continue，不失败。
```

递归保护：

```text
activePairs:
  用于 explainMergeReason -> explainPath 的递归调用。
  如果同一个 pair 正在解释中再次进入，返回 failure。
```

更稳的做法是给每条 proof edge 加时间戳：

```cpp
struct ProofEdge {
    EufTermId u;
    EufTermId v;
    MergeReason reason;
    uint32_t edgeId;
};
```

解释冲突时：

```cpp
explainPath(a, b, maxEdgeId = proofEdges_.size())
```

当路径用到某条 congruence edge `e` 时，解释它的参数相等只能使用更早的 proof edges：

```cpp
explainPath(argA, argB, maxEdgeId = e.edgeId)
```

这样 explanation 有因果顺序，不会用“当前结论”解释“当前结论”。

如果你不想现在做 edgeId 因果约束，至少保留：

```cpp
activePairs.contains(pair) => return {false, {}}
```

不要把所有递归保护都删掉。

---

# 7. Phase 2 model 不应该阻塞 SAT 正确性

模型输出可以作为生产化功能，但不能影响 `check()` 的 SAT/UNSAT 结果。建议：

```cpp
auto result = consistencyCheck();
if (result is conflict/lemma/unknown) return result;

if (produceModels) {
    auto m = buildModel(...);
    if (!m.ok) {
        // 不要把已确认 SAT 改成 UNSAT
        // 可以返回 SAT with no model 或 diagnostic unknown，取决于 API 设计
    }
}
return sat;
```

更推荐 model builder 返回：

```cpp
struct BuildModelResult {
    bool ok;
    EufModel model;
    std::string error;
};
```

不要让 model construction 里的 assert 直接导致 solver 崩溃。尤其是 Phase 2 里写了 “assert no collision”，这在 debug 可以，但 release 里应该变成错误返回。

# 最终意见

**不接受当前版本。**

原因是当前版本仍有三个核心 correctness 风险：

```text
1. Bool 被当成普通 EUF 域处理，可能导致假 SAT 和非法模型。
2. Propagation 候选来自 trail，无法发现 unassigned entailed equality。
3. Mixed fallback 条件过宽，可能把 arithmetic sort 当作 uninterpreted sort 处理。
```

修完这些后，这个 enhancement plan 才可以进入实现。
