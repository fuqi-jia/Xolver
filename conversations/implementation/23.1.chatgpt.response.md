**结论：不接受当前版本。**

这版不能直接交给实现模型。它的方向是有价值的：在已有 full-rebuild EUF solver 的基础上，改成 rollback union-find + incremental congruence closure，确实是下一步性能优化方向。文档也说明当前已有 full-rebuild EUF solver、20 个 unit tests 和 21 个 regression tests 通过，新计划是替换为 incremental e-graph。

但当前版本仍有几个 correctness blocker。尤其是：**Bool 二值语义没有闭合、true/false 冲突没有检查、propagation 候选来源错误、model construction 不区分 sort、rollback 对 pending conflict 和 signature/members 的细节不够严谨**。

# 1. 最大 blocker：Bool 语义又退回成占位符

`assertLit()` 里只写了：

```cpp
// handle bool special case (= t true) with value=false → t = false
...
```

这不够。这里不能留省略号，因为这是 soundness 关键点。Bool 是二值域，不是普通 uninterpreted sort。

必须明确写成：

```text
For BoolTermAsFormula atom:
  lit(b) := (= b true)

If lit(b) assigned true:
  merge(b, true)

If lit(b) assigned false:
  merge(b, false)
```

并且 `lit(true)` / `lit(false)` 应该是 SAT 常量：

```text
lit(true)  = satTrueLit()
lit(false) = satFalseLit()
```

否则下面这个仍然可能假 SAT：

```smt2
(set-logic QF_UF)
(declare-const p Bool)
(declare-const q Bool)
(assert (distinct p true))
(assert (distinct q true))
(assert (distinct p q))
(check-sat)
```

正确结果是 `unsat`，因为 `p=false`、`q=false`。

---

# 2. `true != false` 冲突没有明确处理

当前 incremental `check()` 只检查：

```text
pendingConflict
disequalities
optional propagation
```

但没有看到内置检查：

```text
true and false must never be in the same e-class
```

在 full-rebuild 版本里，这个检查必须在 equality merge / congruence closure 之后做。Incremental 版本里更直接：一旦 `merge(true,false, reason)` 发生，就应该立即形成 pending conflict。

建议明确加入：

```cpp
if (egraph_.same(trueTerm, falseTerm)) {
    auto reasons = egraph_.explainEquality(trueTerm, falseTerm);
    pendingConflict_ = makeConflict(negateAll(reasons));
}
```

更好的是把 `true != false` 当成一个内置 disequality invariant：

```text
after every merge, if rep(true) == rep(false), create conflict
```

否则：

```smt2
(assert (= true false))
(check-sat)
```

可能漏掉。

---

# 3. `pendingConflict_` 需要随 backtrack 回滚

计划里 `assertLit()` 可能设置：

```cpp
pendingConflict_
```

但 `backtrackToLevel()` 没有说明如何清理它。

这会导致一个明显问题：

```text
level 1: assert a=b
level 2: assert distinct a b  -> pendingConflict_
backtrack to level 1
check()
```

如果 `pendingConflict_` 不清掉，就会错误继续返回 conflict。

建议：

```cpp
struct PendingConflict {
    int level;
    TheoryConflict conflict;
};
```

然后：

```cpp
if (pendingConflict_ && pendingConflict_->level > targetLevel) {
    pendingConflict_.reset();
}
```

或者更稳：backtrack 后清空 `pendingConflict_`，让 `check()` 重新扫描当前有效 disequalities。

---

# 4. Propagation 候选来源仍然错了

Phase 6 写：

```text
Maintain registeredAtoms_ — all Eq atoms that have ever been seen by assertLit
```

这不对。`assertLit` 只能看到已经被 SAT 赋值过的 atom。Theory propagation 要传播的是：

```text
已注册但尚未赋值、且被当前 EUF 等价类蕴含的 equality atom
```

所以候选必须来自 **TheoryAtomRegistry / Atomizer 注册过的所有 EUF atoms**，不是 `assertLit` 见过的 atoms。

应该改成：

```cpp
for (const auto& rec : registry.recordsForTheory(TheoryId::EUF)) {
    if (rec.payload.rel != Relation::Eq) continue;

    SatLit atomLit = SatLit::positive(rec.satVar);
    auto val = satAssignment.value(atomLit);

    if (val == True) continue;
    if (val == False) return conflict(negReasons + atomLit);
    if (val == Undef && egraph.same(lhs, rhs)) {
        return lemma(negReasons + atomLit);
    }
}
```

如果当前拿不到 SAT assignment query，那么本版就不要实现 propagation，只保留 TODO。现在文档虽然说“没有接口就 skip”，但前面又设计了 `registeredAtoms_`，容易让实现模型写出半成品。

---

# 5. Model construction 又是 naive `ufval_N`

Phase 7 里：

```text
Each class gets a distinct UF value index
```

这对 uninterpreted sort 可以，但对 Bool sort 不行，对不同 uninterpreted sorts 也不够严谨。

必须改成 sort-correct model：

```text
Bool sort:
  class(true)  -> true
  class(false) -> false
  other Bool class -> false by default

Uninterpreted sort S:
  class C -> S!val_N
```

函数默认值也必须按 result sort：

```text
result sort Bool -> false
result sort S    -> S!val_default_N
```

不能所有 class 统一分配 `ufval_N`。否则 `p : U -> Bool` 的默认值可能被构造成非法的 `ufval_0`。

---

# 6. Signature table rollback 方案还不够完整

你写了：

```text
After rollback(): set sigTableDirty_ = true.
Next merge() or check() calls rebuildSignatureTable().
```

方向可以，但要明确两点。

第一，`same()`、`rep()` 不需要 rebuild；但是任何可能依赖 signature table 的操作都必须先保证 fresh：

```cpp
void ensureSignatureTableFresh();
```

在这些地方调用：

```text
merge()
processCongruenceQueue()
check() if propagation/model relies on congruence closure completeness
```

第二，rollback 后 `congruenceQueue_` 被恢复到 snapshot，但 signature table dirty 后，旧 queue 里的元素可能对应过期状态。最稳的做法是：

```text
rollback 后清空 congruenceQueue_
rebuildSignatureTable 后，从当前所有 application terms 重新检测 congruence 或等待下一次 merge 的 parent-driven propagation
```

否则可能出现 stale queue + stale signature 混合行为。

---

# 7. `members_` rollback 需要写清楚，否则很容易错

文档说：

```text
Class members must rollback — members_ trail restores old state.
```

但没有给出具体 trail 结构。这里很容易写错，因为 union-by-size merge members 时，至少要恢复：

```text
winner class members
loser class members
```

建议记录：

```cpp
struct MemberChange {
    EufClassId dstRoot;
    EufClassId srcRoot;
    size_t oldDstSize;
};
```

merge members 时：

```cpp
memberTrail_.push_back({dst, src, members_[dst].size()});
members_[dst].insert(... members_[src] ...);
```

rollback 时：

```cpp
members_[dst].resize(oldDstSize);
```

如果你还清空了 `members_[src]`，也要 trail 它；更简单是不要清空 loser members，只保证只有 root 的 members 被使用。

# 8. `EufSolver::pop()` 说得太 vague

现在只写：

```text
pop(n) restores trail size and rolls back egraph accordingly.
```

但这正是容易出 bug 的地方。建议给出明确伪代码：

```cpp
void EufSolver::pop(uint32_t n) {
    while (n-- && !scopeLimits_.empty()) {
        size_t lim = std::min(scopeLimits_.back(), trail_.size());
        int targetLevel = trail_.empty() ? 0 : trail_[lim].level; // 需要按你实际 level 设计修正
        trail_.resize(lim);
        scopeLimits_.pop_back();

        rollbackEGraphToTrailLimit(lim);
        removeDisequalitiesAfterTrailLimit(lim);
        clearOrRecomputePendingConflict();
    }
}
```

更推荐维护：

```cpp
std::vector<EGraphSnapshot> scopeSnapshots_;
```

`push()` 同时存：

```cpp
scopeLimits_.push_back(trail_.size());
scopeSnapshots_.push_back(egraph_.snapshot());
```

`pop()` 直接：

```cpp
trail_.resize(limit);
egraph_.rollback(snapshot);
```

不要从 decision level 间接推断。

# 9. `levelSnapshots_` 设计不够稳

`assertLit()` 里：

```cpp
ensureSnapshotForLevel(level);
```

`backtrackToLevel()` 里：

```cpp
if (level < levelSnapshots_.size()) {
    egraph_.rollback(levelSnapshots_[level]);
}
```

这个设计要求 decision level 连续且 snapshot 与 level 一一对应。CDCL 里通常是连续的，但还要处理：

```text
level 0
push/pop scope
backtrack to non-existing or already popped level
multiple assignments at same level
```

建议改成 assignment trail limit，而不是直接按 level 数组：

```cpp
struct LevelSnapshot {
    int level;
    size_t trailLimit;
    EGraphSnapshot egraphSnap;
};
```

或者维护：

```cpp
std::vector<EGraphSnapshot> decisionLevelSnapshots_;
```

并明确：

```text
snapshot for level L is taken before the first assignment at level L.
rollbackToLevel(L) restores to the snapshot after all assignments with level > L are removed.
```

这需要写清楚，不然实现模型很容易 off-by-one。

# 10. 增量 merge 后必须立即处理 congruence 到 fixpoint

`merge()` 的接口说会 enqueue + process。这里必须明确：

```text
merge() returns only after mergeQueue and congruenceQueue reach fixpoint.
```

否则 `assertLit(a=b)` 后马上 assert `distinct f(a) f(b)`，如果 congruence 还没处理，就会漏掉 immediate conflict。

建议写成硬规则：

```cpp
MergeStatus IncrementalEGraph::merge(...) {
    enqueueMerge(...);
    return processToFixpoint();
}
```

不要让 `processCongruenceQueue()` 延迟到 `check()`，除非你明确所有 conflict 检查都只在 `check()` 中做。

# 11. Explanation 需要返回 `ExplainResult`，不能只返回 vector

现在接口是：

```cpp
std::vector<SatLit> explainEquality(...)
```

但前几版已经反复确认：explanation failure 不能伪装成空原因。

应该改成：

```cpp
struct ExplainResult {
    bool ok;
    std::vector<SatLit> reasons;
};
```

如果 BFS 找不到 path，或者 recursive activePairs 检测到 cycle：

```cpp
return {false, {}};
```

然后 caller：

```cpp
if (!er.ok) return TheoryCheckResult::unknown();
```

否则一旦解释失败返回空 vector，可能学到错误 conflict clause。

# 12. 本版范围过大：incremental e-graph + propagation + model 一起做风险太高

这个版本同时做：

```text
rollback union-find
incremental congruence closure
rollback explanation
incremental EufSolver integration
theory propagation
model construction
performance validation
```

太大了。当前已有 full-rebuild solver 能跑，建议这版只做 **incremental e-graph 替换**，不要同时做 propagation 和 model。

建议 scope 改成：

```text
Revision A:
  - RollbackUnionFind
  - IncrementalEGraph
  - incremental assert/backtrack/check
  - conflict explanation
  - disequality conflict
  - all existing tests pass

Out of scope:
  - model construction
  - theory propagation
```

等 incremental baseline 稳定后，再开：

```text
Revision B: sort-correct model construction
Revision C: theory propagation
```

# 必须修改后的最小可接受版本

请把方案改成下面这样：

```text
1. 本版只做 incremental e-graph，不做 model，不做 propagation。
2. Bool formula-position atom 必须明确：
   lit(b)=true  => merge(b,true)
   lit(b)=false => merge(b,false)
3. true/false conflict 必须作为内置 invariant 检查。
4. pendingConflict_ 必须随 backtrack 清理或重算。
5. propagation 删除，只保留 TODO；候选未来必须来自 all registered EUF atoms。
6. model construction 删除，只保留 TODO；未来必须 sort-correct。
7. IncrementalEGraph::merge 必须 process to fixpoint。
8. Explanation 返回 ExplainResult{ok,reasons}。
9. rollback 明确恢复：
   UF trail、members trail、mergeRecords、queues、sigTableDirty。
10. push/pop 使用 scopeSnapshots_，不要只靠 trail size。
```

# 最终意见

**不接受当前版本。**

不是因为 incremental e-graph 方向错，而是这个方案把太多高风险功能混在一起，而且几个关键 soundness 点没有写实：Bool 二值语义、true/false conflict、pending conflict rollback、propagation 来源、model sort-correctness、explanation failure。

建议先收缩成 **Incremental EUF baseline only**。修完上面这些后，下一版可以再审。
