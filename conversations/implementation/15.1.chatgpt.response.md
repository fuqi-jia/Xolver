建议选 **[4] Revise**。这版方向比之前更符合你的架构判断：不把 Interval B&B 私有塞进 NIA，而是做 **NIA 代数增强 + 公共 interval 框架**。但现在不能直接 Approve，因为里面有几处会造成 soundness 或实现边界混乱的问题，尤其是 **factor lemma、modular reasoning、interval reason 表达**。

---

# 1. Factor split lemma 必须 guarded，不能只发 `p=0 ∨ q=0`

计划里 B2 写：

```text
p*q = 0 → emit lemma: p = 0 ∨ q = 0
```

这在 CDCL(T) 里不完整。必须是 guarded lemma：

```text
¬lit(p*q = 0) ∨ lit(p = 0) ∨ lit(q = 0)
```

否则这个 lemma 会在原约束没被 SAT assignment 启用时也强制 `p=0 ∨ q=0`，这是错误的。

同理：

```text
p*q != 0
```

应该生成：

```text
¬lit(p*q != 0) ∨ lit(p != 0)
¬lit(p*q != 0) ∨ lit(q != 0)
```

并且所有新 atom 都必须走 `TheoryAtomRegistry::getOrCreatePolynomialAtom`。

---

# 2. Factorization 不要直接用 square-free factorization 当作 syntactic product rule

计划说：

```text
try lp_polynomial_factor_square_free via C API
```

这个要谨慎。你原先说的是 **syntactic product**，即只有明确知道原始约束来自 `mul(a,b)` 才发 factor lemma。现在改成 libpoly factorization，语义更强，也更复杂。

理论上，若 factorization 是 exact 的，`p=0 ⇒ f1=0 ∨ ... ∨ fn=0` 是 sound。但实现上需要处理：

```text
constant factor
unit factor
repeated factors
多个 factors
factorization failure
factor over Q vs integer normalization
```

建议 Phase 5 第一版只做：

```text
syntactic product detection
```

如果没有 construction history / AST metadata，就先跳过。不要先上 libpoly factorization。否则一旦 factor API 使用错，会直接生成 unsound lemma。

可以改成：

```text
Factor rules Phase 5.1:
  only for syntactic product nodes preserved by PolynomialConverter.
  If syntactic product metadata is unavailable, return NoChange.
  Algebraic factorization via libpoly is a later phase.
```

---

# 3. Modular reasoning 不要生成 domain restriction lemmas

计划里 B3 写：

```text
Emit domain restriction lemmas, e.g. x ≡ 1 (mod 2) as x = 2k+1.
For now: restrict to simple parity/odd constraints via DomainStore::excludeValue or restrictToFiniteSet.
```

这不合适。`DomainStore` 目前表达的是 interval / finite set / excluded concrete values，不能表达无限同余类：

```text
x ≡ 1 mod 2
```

不能把它误写成有限集合，也不能用 `excludeValue` 表达“排除所有偶数”。

Phase 5 的 modular reasoning 应该先只做 **conflict detection**：

```text
给定 active equalities p1=0,...,pk=0；
对小模数 m 枚举 residues；
如果没有 residue assignment 同时满足所有 equality；
返回 conflict: ¬reason(p1) ∨ ... ∨ ¬reason(pk)。
```

不要生成 parity/domain restriction lemma。那需要新的 congruence atom 或 modular domain abstraction，之后再做。

---

# 4. `x^2 + y^2 = 2` 不是 unsat 测试

计划测试里写：

```text
x^2 + y^2 = 2 → sat with parity constraints
```

这个是 SAT，例如：

```text
x = 1, y = 1
```

所以它不能作为 modular conflict 测试。可以作为 “modular reasoning 不应误判 UNSAT” 的 soundness regression。

真正的 modular UNSAT 测试应该是：

```text
x^2 + y^2 = 3
```

mod 4 下平方只能是 `0` 或 `1`，两个平方和只能是 `0,1,2`，不能是 `3`，所以 UNSAT。

---

# 5. `IntervalReason` 不能只有一个 `optional<SatLit>`

计划里：

```cpp
struct IntervalReason {
    IntervalReasonKind kind;
    std::optional<SatLit> lit;
};
```

这不够。区间 reason 经常来自多个 literal，例如：

```text
x lower bound reason: x >= 0
x upper bound reason: x <= 100
finite-set restriction reason: x²=4
```

一个 interval 的 reason 应该是 `vector<SatLit>`，而不是单个 `optional<SatLit>`。

建议改成：

```cpp
struct IntervalReason {
    IntervalReasonKind kind;
    std::vector<SatLit> lits;
};
```

或者更简单：

```cpp
struct ReasonedInterval {
    IntervalZ interval;
    std::vector<SatLit> reasons;
};
```

第一版不要过度设计 `DomainBound / Constraint / Propagation`，直接保留完整 SAT reasons 更实用。

---

# 6. `ReasonedBox::buildConflict()` 不能泛泛“从所有 reasons 构造”

`ReasonedBox` 本身不知道哪个 constraint 被 interval 证明 violated。更正确的做法是：

```text
IntervalEvaluator 返回：
  violated constraint reason
  + used variable interval reasons
```

由 NIA adapter 构造 conflict。

也就是说，公共 interval 框架可以返回：

```cpp
struct IntervalEvalResult {
    IntervalStatus status;
    std::vector<SatLit> usedReasons;
};
```

但不要让 `ReasonedBox::buildConflict()` 随便构造 theory conflict。否则很容易把不相关 reason 混进来，虽然 coarse 可能 sound，但组件职责不清。

---

# 7. `terms()` 接口要先作为可选能力，不要强依赖 C API 成功

计划把 `PolynomialKernel::terms()` 放在 Implementation Order 第 1 步，这对后续 GCD、interval、modular 都很关键。但 libpoly C API 细节可能不稳定。这里要加硬规则：

```text
If terms() is unavailable or incomplete, return std::nullopt.
Any reasoning depending on terms() must return NoChange, never UNSAT.
```

另外，`MonomialTerm::coefficient` 用 `mpz_class`，这意味着只支持整数系数。NIA 里应该可以，但如果 kernel 里出现 rational coefficients，必须先清分母，或者 `terms()` 返回 `nullopt`。不要强转。

---

# 8. 执行范围还是偏大，建议拆成两批

这版同时做：

```text
PolynomialKernel::terms()
AlgebraicIntegerReasoner multivar GCD
factor lemmas
enhanced modular reasoning
common interval framework
multivar interval evaluator
NiaSolver integration
```

范围很大，而且每块都有 correctness 风险。

我建议本轮只执行：

```text
Part A + B1 + C skeleton
```

即：

```text
1. PolynomialKernel::terms() optional interface;
2. multivar GCD conflict;
3. common interval skeleton: IntervalZ, ReasonedBox, IntervalOperations;
4. 不做 factor lemmas；
5. 不做 modular domain restriction；
6. 不接 full multivar IntervalEvaluator 到 NiaSolver。
```

下一轮再做：

```text
factor lemmas + modular conflict detection + interval evaluator integration
```

如果你确实想一次做大一点，至少把 factor 和 modular 限制成 very conservative。

# 结论

这版方向是对的，但不能直接 Approve。主要 blocker 是：

```text
factor lemma 没明确 guarded；
modular reasoning 试图生成同余 domain restriction；
ReasonedInterval reason 表达不够；
执行范围过大。
```

我的建议：**选 [4] Revise**。修完上面这些点后，可以批准一个更稳的版本。
