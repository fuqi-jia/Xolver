这版方向是对的，**不建议 Reject**。但我也不建议直接 Approve；我会选 **4 Revise**，因为还有几个稀疏 tableau 实现里很容易导致严重隐蔽 bug 的点需要补上。当前计划已经明确要用双向 row/column cross-linked storage 替换 dense matrix，并且把 `update`、`findEntering`、`pivot` 都改成只访问相关稀疏项，这是正确方向。

最需要补的是下面这些。

---

# 1. `addConstraint` 里 aux var 必须从 non-basic 集合移除

计划里写：

```text
addVar: create VarState, grow cols_, update nonBasicVars_
addConstraint: create aux var, call rewriteToNonBasic, add sparse row
```

但这里少了关键一步：**新建的 aux var 初始会被 `addVar` 放进 `nonBasicVars_`，随后它作为新 row 的 basic variable，必须立即从 `nonBasicVars_` 删除，并设置 `vars_[aux].basicRow = row`。**

否则 aux 会同时是 basic 和 non-basic，后面 `recomputeBeta`、`findEntering`、`pivot` 全都会错。

建议补：

```cpp
int aux = addVar(auxName);        // initially non-basic
LinearForm f = rewriteToNonBasic(raw);

int row = rows_.size();
rows_.push_back(...);

makeBasicWithoutPivot(aux, row);  // remove aux from nonBasicVars_, set basicRow
```

并明确：

```cpp
void makeBasicWithoutPivot(int v, int row) {
    assert(vars_[v].basicRow == -1);
    removeFromNonBasic(v);
    vars_[v].basicRow = row;
    basicVars_[row] = v;
    rows_[row].basicVar = v;
}
```

这是**必须修**。

---

# 2. `rewriteToNonBasic` 必须在插入 aux row 之前做

计划已经提到 `rewriteToNonBasic`，这是对的：动态注册新约束时，如果某个变量已经 basic，新 row 必须先把它替换成当前 tableau 中的 non-basic 表达式。

但要补一句：

```text
rewriteToNonBasic(raw) 必须在 aux var 变成 basic 之前执行，并且 raw 不应包含 aux 本身。
```

否则如果 aux 已经被错误加入 basis，再做 rewrite，有可能把新 row 自己也卷进去。

---

# 3. column invariant 要写死：basic 变量的 column 必须为空

当前设计说 row entries 只包含 non-basic vars，column 是 rows containing variable。这个隐含了一个重要不变量：

```text
如果 vars_[v].basicRow != -1，那么 cols_[v].entries 必须为空。
```

因为 basic var 不应该出现在任何 row 的 RHS entries 中。

pivot 后必须满足：

```text
entering becomes basic  => cols_[entering] is empty
leaving becomes nonbasic => cols_[leaving] may be nonempty
```

建议在 `checkInvariants()` 里加：

```cpp
for (int v = 0; v < vars_.size(); ++v) {
    if (vars_[v].basicRow != -1) {
        assert(cols_[v].entries.empty());
    }
}
```

这能直接抓住 80% 的 sparse pivot 错误。

---

# 4. violation queue 不要写“add/remove”，应改成 lazy queue

计划写：

```text
refreshViolationStatus(x) — add/remove from queue when beta changes
```

如果用 `std::deque<int>`，你没有 O(1) remove。不要真的做 remove。

建议改成：

```text
refreshViolationStatus(x):
  if x is basic and currently violated and not inViolationQueue[x]:
      push_back(x)
      inViolationQueue[x] = true

pickViolatedBasic():
  pop_front
  inViolationQueue[x] = false
  re-check whether still violated
```

也就是**只 lazy enqueue，不主动 remove**。`rebuildViolationQueue()` 只在 `recomputeBeta()` 后做。

---

# 5. `replaceRow` / `eraseCoeff` 必须是唯一修改 row/col 的入口

计划已经说 `eraseCoeff` 是最危险部分，需要 debug invariant，这是对的。 但还要加一条工程纪律：

```text
严禁直接修改 rows_[r].entries 或 cols_[c].entries。
所有系数改动只能通过 setCoeff/addCoeff/eraseCoeff/replaceRow。
```

否则 cross-link 很快就坏。

可以在 plan 里明确：

```cpp
// Only these functions may mutate row/column structure:
setCoeff(...)
addCoeff(...)
eraseCoeff(...)
replaceRow(...)
```

---

# 6. `SparseRow::pos` 在 swap-pop 后必须同步修复

计划里有 optional col->pos index，但没有明确 swap-pop 后如何维护。这个要补。

当 `eraseCoeff` 删除 row entry 时，如果最后一个 entry 被移动到 `rowPos`，必须：

```cpp
if (row.hasIndex) {
    row.pos[moved.col] = rowPos;
    row.pos.erase(deletedCol);
}
```

当 `setCoeff` 新增 entry 时，如果 `hasIndex`：

```cpp
row.pos[col] = newRowPos;
```

当 `replaceRow` 清空 row 时：

```cpp
row.pos.clear();
row.hasIndex = false; // 或重建
```

我建议简单一点：

```text
replaceRow 清空后直接 hasIndex=false; pos.clear();
后续 maybeBuildRowIndex 再重建。
```

---

# 7. `pivot` 需要明确 affected column snapshot 的顺序

计划写了“take snapshot of entering column”，这是必要的。 但要补顺序：

```text
1. copy old pivot row
2. build new pivot row
3. copy affected = cols_[entering].entries
4. for affected rows except pivot row:
      eraseCoeff(row, entering)
      add c * newPivotRow
5. replace pivot row
6. update basis metadata
7. assert cols_[entering].entries.empty()
```

关键是：**不能边遍历 `cols_[entering].entries` 边 eraseCoeff**，必须 snapshot。

---

# 8. public API 不变，但语义必须确认 `addConstraint` 的 rhs 方向

计划说保留：

```cpp
int addConstraint(const std::vector<std::pair<int, mpq_class>>& terms,
                  const mpq_class& rhs);
```

但没有写 row 语义。必须明确：

```text
addConstraint(terms, rhs) creates aux s such that:
    s = rhs + Σ coeff_i * x_i
```

如果旧实现语义是：

```text
s = Σ coeff_i*x_i - rhs
```

那所有上层调用都会符号反掉。

所以 plan 里必须补：

```text
Public API unchanged, but its exact semantic contract must be:
  addConstraint(terms, rhs) returns aux s with row:
      s = rhs + Σ terms[i].coeff * terms[i].var
```

或者如果当前 SimplexSolver 已按 `s = p(x)-c` 调用，那就要保证它传入的是：

```cpp
rhs = -c;
terms = coefficients of p
```

这个点不写清楚，测试少的时候很容易“碰巧过”。

---

# 9. 稀疏 stress test 不够，要加 pivot invariant 随机测试

当前测试计划包括 existing tests、sparse stress、invariant checks、CDCL(T) end-to-end。 还应加一个专门的随机 pivot 测试：

```text
随机生成稀疏 tableau；
随机选择 legal pivot；
每次 pivot 后检查：
  1. row/col cross-link invariant；
  2. basic column empty；
  3. row 只含 non-basic vars；
  4. 对随机 beta，pivot 前后所有原方程等价；
  5. checkInvariants() 通过。
```

普通 sat/unsat tests 不一定能覆盖 pivot cross-link 错误。

---

# 结论

这版方案**总体正确，值得继续**，但现在不要直接 Approve。选 **4 Revise**，把上面 9 条补进去。尤其是：

```text
aux var 从 nonBasicVars_ 移除；
basic variable column 必须为空；
violation queue lazy；
addConstraint 语义写死；
pivot 使用 entering column snapshot；
row/col 只能由统一接口修改。
```

这些补完后就可以 Approve 让它写代码。
