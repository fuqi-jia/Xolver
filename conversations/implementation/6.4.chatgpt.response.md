我建议：**可以接近 Approve，但我仍会选 [4] Revise，补最后几条非常关键的边界条件**。
这版已经比上一版成熟很多：统一一条 CDCL(T) 路径、删除 legacy scan path、`connectPropagator` 在 `addObservedVar` 之前、parsed/dynamic atoms 都进入 registry、NRA/CdcacSolver 也走新接口、`Unknown/duplicate` 不再无 clause `return false`，这些关键点都已经修正了。

但还有几处如果不写清楚，执行时容易变成 unsound 或卡住。

---

# 1. `currentLevel_` 没有定义和维护

`notify_assignment` 里用了：

```cpp
tm_.assertTheoryLit(*atom, SatLit{var, sign}, currentLevel_);
```

但 `CadicalTheoryPropagator` 的 private 字段里没有 `currentLevel_`，也没有说明 `notify_new_decision_level()` 如何更新它。计划里有 `notify_new_decision_level()`、`notify_backtrack()`，但 level 状态没有闭环。

必须补：

```cpp
int currentLevel_ = 0;

void notify_new_decision_level() override {
    ++currentLevel_;
}

void notify_backtrack(size_t new_level) override {
    currentLevel_ = static_cast<int>(new_level);
    tm_.backtrackToLevel(currentLevel_);
}
```

如果 CaDiCaL 回调提供的是 target decision level，就按它同步；如果只通知 backtrack 而没有精确 level，要以 vendored API 为准。总之：**理论 trail 的 level 必须和 CaDiCaL decision level 对齐**。

---

# 2. unsupported theory atom 不能“静默跳过”

当前 atomize 里写：

```cpp
else {
    // Unsupported theory atom; will be handled by returning Unknown
}
```

但如果这里只是注释，没有实际状态，那么这个 atom 仍然被 SAT 赋值，却没有 registry record，theory solver 完全看不到，可能错误返回 SAT。这个是 correctness bug。

必须补机制，二选一：

```cpp
unsupportedTheorySeen_ = true;
```

然后 `checkSatInternal()` 在 atomize 后直接：

```cpp
if (registry.hasUnsupportedTheoryAtom()) return Result::Unknown;
```

或者注册一个 payload：

```cpp
struct UnsupportedTheoryPayload {
    ExprId exprId;
};

TheoryAtomRecord{..., payload = UnsupportedTheoryPayload{eid}}
```

然后任意 solver 收到 unsupported payload 返回 Unknown。

更简单、更安全的是第一种：**只要有 unsupported theory atom，就直接 Unknown，不能继续 SAT**。

---

# 3. `Eq` / `Distinct` 不能一律当 arithmetic theory atom

计划里：

```cpp
bool isTheory = (Eq || Distinct || Lt || Leq || Gt || Geq)
```

`Lt/Leq/Gt/Geq` 基本是算术关系，但 `Eq/Distinct` 可能是：

```smt2
(= a b)        ; Boolean / UF / algebraic object
(distinct u v) ; UF objects
```

如果 extract 失败后只是 “will be handled by Unknown” 但没有真的 Unknown，就会出错。这里要写成：

```text
Eq/Distinct only become arithmetic theory atoms if extraction to LinearAtomPayload or PolynomialAtomPayload succeeds.
Otherwise:
  - if the solver supports Boolean equality / UF later, route to that theory;
  - otherwise mark unsupported and return Unknown.
```

不要让 non-arithmetic equality 静默变成普通 SAT atom，除非你已经有对应语义编码。

---

# 4. `setPendingClause` 目前只接收 `TheoryLemma`，但 conflict 用的是 clause

计划里：

```cpp
setPendingClause(tr.conflict.clause);
```

但函数定义是：

```cpp
void setPendingClause(const TheoryLemma& lemma);
```

这个类型不一致。要统一：

```cpp
void setPendingClause(const std::vector<SatLit>& lits);
void setPendingClause(const TheoryLemma& lemma) {
    setPendingClause(lemma.lits);
}
```

然后 conflict 和 lemma 都走同一个 clause buffer。

---

# 5. `addObservedVar` 必须保证去重

parsed atom 和 dynamic atom 都会调用：

```cpp
sat_->addObservedVar(v);
```

如果同一个 dynamic atom 被 `getOrCreateLinearBoundAtom` 复用，不应该重复 observed。`getOrCreate...` 的 existing path 应直接返回，不再 observe；`registerParsedTheoryAtom` 也要防止重复注册同一 SatVar。

建议加：

```cpp
std::unordered_set<SatVar> observedVars_;
```

```cpp
void TheoryAtomRegistry::observeIfNeeded(SatVar v) {
    if (observedVars_.insert(v).second) {
        sat_->addObservedVar(v);
    }
}
```

---

# 6. `CdcacSolver` 的 false-literal relation 需要明确 `Neq` / strict 支持策略

计划已经说 false literal 要做 `negateRelation`，这对；也写了如果 CDCAC 不支持 disequality/strict，则 `check()` 返回 Unknown 或生成 split lemma。

这里建议再强一点：

```text
Phase 1 implementation:
  if effective relation is unsupported by CdcacSolver, return Unknown.
Do not silently drop the atom.
Do not approximate strict as non-strict.
```

否则 NRA 很容易 unsound。

---

# 7. `Unknown` 终止后 backend 状态要明确

计划里：

```cpp
abortWithUnknown_ = true;
terminateSolve();
return true;
```

并说 `CadicalBackend::solve()` 检查 `terminateRequested_` 后映射为 Unknown。这个方向可以，但必须写清楚优先级：

```cpp
int res = solver_->solve();

if (terminateRequested_) {
    terminateRequested_ = false;
    return SolveResult::Unknown;
}

return mapCadicalResult(res);
```

也就是说，即使 CaDiCaL 返回 SAT，只要外部 propagator 期间请求了 unknown termination，最终也必须返回 Unknown。
