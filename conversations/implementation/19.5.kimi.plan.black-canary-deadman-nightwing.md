# ICP (Interval Constraint Propagation) Framework Implementation Plan

## Goal

Transform the existing NIA-private interval evaluation into a **theory-agnostic ICP common framework** under `src/theory/arith/`. NIA, NRA, IncrementalLinearizer, BoundedNiaSolver, and LocalSearchAdvisor will all be clients.

## Design Principles

1. **Over-approx only**: every interval operation must be sound over-approximation.
2. **No direct SAT claims**: ICP never returns `Sat` directly; candidates go through exact validator.
3. **Reason tracking**: every contraction carries `SatLit` reasons; empty box conflicts are explainable.
4. **Budget bounded**: `maxIterations` / `maxContractors` limits; budget exhaustion → `UnknownBudget`.
5. **Client owns split/B&B**: ICP returns `SplitSuggestion`; recursive search stays in NIA/NRA.

## Current State

Already exists under `src/theory/arith/interval/`:
- `IntervalZ` — integer interval `[lo, hi]`
- `IntervalOperations` — add, sub, neg, mul, pow (all over-approx)
- `ReasonedInterval` — interval + `std::vector<SatLit>` reasons
- `ReasonedBox` — per-variable map of `ReasonedInterval`
- `IntervalEvaluator` — single-var polynomial forward eval (multi-var → `NoChange`)

NIA already uses: `IntervalEvaluator`, `SquareBoundReasoner`, `SumOfSquaresBoundReasoner`.

## Key Data Structures

### IntervalQ
```cpp
struct IntervalQ {
    mpq_class lo, hi;
    bool loOpen = false, hiOpen = false;
    bool isEmpty() const;
    bool contains(const mpq_class& v) const;
    mpq_class width() const;
};
```
V1 only handles **finite intervals**; infinity → `NoChange` or require domain reasoner to give finite bounds first.

### IcpConstraint
```cpp
struct IcpConstraint {
    std::optional<ExprId> expr; // original expression DAG (optional, for V2 backward contractors)
    PolyId poly;                // normalized p rel 0 (fallback for V1 forward eval)
    Relation rel;               // Eq, Neq, Lt, Leq, Gt, Geq
    SatLit reason;              // active literal
    TheoryId owner;             // NIA / NRA
};
```

**V1 scope**: `PolyId` forward evaluation + pattern contractors (square, sum-of-squares).  
**V2 scope**: Add `ExprId` for expression-DAG backward contractors (Add, Sub, Mul, etc.).

### IcpResult (split for Z/Q)

Since `ContractorZ` / `ContractorQ` are concrete, the result types must also be concrete:

```cpp
enum class IcpStatus {
    Conflict, DomainUpdate, SplitSuggestion, NoChange, UnknownBudget
};

struct BoundUpdateZ {
    std::string var;
    IntervalZ newInterval;
    std::vector<SatLit> reasons; // must include constraint reason + used bound reasons
};

struct BoundUpdateQ {
    std::string var;
    IntervalQ newInterval;
    std::vector<SatLit> reasons;
};

struct IcpResultZ {
    IcpStatus status;
    std::optional<TheoryConflict> conflict;
    std::vector<BoundUpdateZ> updates;
    std::optional<SplitSuggestion> split;
};

struct IcpResultQ {
    IcpStatus status;
    std::optional<TheoryConflict> conflict;
    std::vector<BoundUpdateQ> updates;
    std::optional<SplitSuggestion> split;
};
```

**Candidate model generation is reserved for future (LocalSearchAdvisor / B&B client), not part of ICP core V1.**

**V1 contractor list (explicit)**:
- `RelationContractorZ` / `RelationContractorQ`
- `SquareContractorZ` / `SquareContractorQ`
- `SumSquaresContractorZ` / `SumSquaresContractorQ`

Add/Sub/Mul/Pow/general DAG contractors deferred to V2.

### Contractor Interface
ICP must serve both NIA (`ReasonedBoxZ`) and NRA (`ReasonedBoxQ`).  
**Phase-1 choice: define two concrete interfaces and two concrete result types.**

```cpp
struct ContractorResultZ {
    IcpStatus status;
    std::optional<TheoryConflict> conflict;
    std::vector<BoundUpdateZ> updates;
};

struct ContractorResultQ {
    IcpStatus status;
    std::optional<TheoryConflict> conflict;
    std::vector<BoundUpdateQ> updates;
};

class ContractorZ {
public:
    virtual ContractorResultZ contract(ReasonedBoxZ& box) = 0;
    virtual std::vector<std::string> vars() const = 0;
    virtual SatLit reason() const = 0;
};

class ContractorQ {
public:
    virtual ContractorResultQ contract(ReasonedBoxQ& box) = 0;
    virtual std::vector<std::string> vars() const = 0;
    virtual SatLit reason() const = 0;
};
```

This keeps cpp files simple and avoids Z→Q→Z conversion overhead in the NIA hot path.

## Implementation Order

### Phase 1: Interval Foundation (1-2 days)
1. `IntervalQ.h/.cpp` + `IntervalQOps.h` — rational interval ops with mpq
2. `ReasonedBoxQ` alongside existing `ReasonedBoxZ` (rename current to ReasonedBoxZ)
3. Unit tests for IntervalQ ops (add, sub, mul, div-by-zero handling, pow, sqrt)

### Phase 2: Relation Forward Conflict (1 day)
4. `RelationContractorZ` / `RelationContractorQ` — forward eval only, with **multivar safeguard**:
   - If `poly` is single-var and `IntervalEvaluator` can soundly over-approx → apply rules below.
   - If `poly` is multivar and no specialized pattern matches → `NoChange` (do not assume conflict).
   - Rules:
     - `p <= 0`: if `p.lo > 0` → `Empty` conflict
     - `p >= 0`: if `p.hi < 0` → `Empty` conflict
     - `p = 0`: if `0 ∉ p.interval` → `Empty` conflict
     - `p != 0`: if `p.interval == [0,0]` → `Empty`; otherwise `NoChange`
   - **Strict relations**:
     - NIA: strict (`Lt`, `Gt`) normalized to non-strict **before** ICP (`p < 0` → `p <= -1`), **only after integer-coefficient normalization** (`clearDenominators`). If integer normalization fails, ICP returns `NoChange`.
     - NRA: if open-interval strict handling is **not** implemented, `Lt`/`Gt` must return `NoChange`. **Never approximate strict as non-strict for conflict/pruning.** Regression: `x ∈ [0,0], x < 0` must return `NoChange` in V1.
5. `IcpEngineZ` / `IcpEngineQ` skeleton with full worklist loop
6. `Worklist` + watcher map (per-engine, since contractor sets differ)

### Phase 3: Pattern Contractors (1-2 days)
**V1 scope (this phase)**: only pattern-based contractors over normalized `p rel 0`. No expression DAG / intermediate variables.

7. `SquareContractorZ` / `SquareContractorQ` — recognizes normalized pattern `x^2 - c rel 0`.
   **NIA (`SquareContractorZ`)**:
   - `x^2 <= c`: `c < 0` → `Empty`; `c >= 0` → `r = floor_sqrt(c)`, bound update `x ∈ [-r, r]`.
   - `x^2 = c`: `c < 0` → `Empty`; `c` not perfect square → `Empty`; `c = r^2` → `NoChange` in V1 (ICP V1 only returns interval updates; exact finite-set refinement `{-r,r}` stays in old `SquareBoundReasoner`).
   - `x^2 >= c`: `c < 0` → `NoChange` (tautology); `c >= 0` → `NoChange` in V1.
   - `x^2 != c`: `c < 0` → `NoChange` (tautology); otherwise `NoChange` in V1.
   **NRA (`SquareContractorQ`)**:
   - `x^2 <= c`: `c < 0` → `Empty`; `c >= 0` → `x ∈ [-sqrt_bound(c), sqrt_bound(c)]` (rational over-approx).
   - `x^2 = c`: `c < 0` → `Empty`; `c >= 0` → `NoChange` in V1 (real equality not an interval).
   - `x^2 >= c`: `c < 0` → `NoChange`; otherwise `NoChange` in V1.
   - If exact rational/algebraic sqrt bounds are not implemented, return `NoChange` rather than unsafe approximation.
   - **V1 does NOT implement general `z = x^2` DAG node.**

8. `SumSquaresContractorZ` / `SumSquaresContractorQ` — recognizes `Σ xi^2 - c rel 0`.
   **NIA (`SumSquaresContractorZ`)**:
   - `Σxi^2 <= c`: `c < 0` → `Empty`; `c >= 0` → `r = floor_sqrt(c)`, each `xi ∈ [-r, r]`.
   - `Σxi^2 = c`: `c < 0` → `Empty`; otherwise `NoChange` in V1.
   - `Σxi^2 >= c`: `c < 0` → `NoChange` (tautology); otherwise `NoChange` in V1.
   **NRA (`SumSquaresContractorQ`)**:
   - `Σxi^2 <= c`: `c < 0` → `Empty`; `c >= 0` → each `xi ∈ [-sqrt_bound(c), sqrt_bound(c)]` (rational over-approx).
   - `Σxi^2 = c`: `c < 0` → `Empty`; otherwise `NoChange` in V1.
   - `Σxi^2 >= c`: `c < 0` → `NoChange`; otherwise `NoChange` in V1.
   - If exact rational sqrt bounds not implemented, return `NoChange`.

9. `ContractorFactory` — pattern match constraints to V1 contractors

**V2 scope (deferred)**:
- Add/Sub/Mul backward contractors (require `ExprId` expression-DAG support)
- General `z = x^2` DAG node contractor
- PowContractor (odd/even powers beyond square)

### Phase 4: NIA Adapter (1-2 days)
13. `NiaIcpAdapter`
    - Build `ReasonedBoxZ` from `DomainStore`
    - Convert active constraints to `IcpConstraint`
    - Run `IcpEngineZ`
    - Translate results back to `DomainStore` updates / conflicts
    - **Critical**: every `BoundUpdateZ.reasons` must be stored into `DomainStore` lower/upper reasons. If reasons are incomplete, do not generate a theory conflict from that update.
14. Wire into `NiaSolver::check()` pipeline **in parallel** with existing logic:
    ```
    linearDomain → squareBound(old) → sumSquares(old) → algebraic → ICP(new) → bounded
    ```
    Do **not** remove old `SquareBoundReasoner` / `SumOfSquaresBoundReasoner` yet.
    Do **not** modify `SquareBoundReasoner.cpp` to delegate to ICP contractors in V1.
    - After ICP `DomainUpdate`, check if `DomainStore` became empty → return Conflict immediately.
    - Ensure `BoundedNiaSolver` reads the **latest** `DomainStore` after ICP updates.
15. Only after ICP passes identical regression tests, consider migrating/deleting old direct logic.

### Phase 5: NRA Adapter (1 day, optional for V1)
16. `NraIcpAdapter` — real bounds → `ReasonedBoxQ`
17. Wire as pre-pruning stage in `NraSolver`

### Phase 6: Splitter & Integration (1-2 days)
18. `Splitter` — choose variable by max width / activity
19. `SplitSuggestion` integration:
    - `IcpEngineZ/Q` returns `SplitSuggestion` only (advisory).
    - `NiaIcpAdapter` decides whether to turn it into `x <= k ∨ x >= k+1` lemma.
    - `NraIcpAdapter` decides whether strict split is supported; otherwise ignore / `NoChange`.
    - **IcpEngine never directly generates lemmas or dynamic atoms.**
20. `IcpConfig` tuning (max iterations, min width, budget)

### Phase 7: Testing (2-3 days)
21. Unit tests for each contractor in isolation
22. Integration tests: NIA ICP on benchmark subset
23. Regression tests: ensure existing NIA verified cases still pass
24. Soundness checks: no false UNSAT on known-SAT instances

## Integration Points

### NIA Solver Pipeline (parallel ICP)
```
assertLit → normalize → linear → squareBound(old) → sumSquares(old) → algebraic → ICP(new) → bounded → local search → split
```

### CDCL(T) TheoryCheckResult (NIA adapter example)
```cpp
IcpResultZ r = icpEngineZ.run(...);
switch (r.status) {
  case Conflict:
      return TheoryCheckResult::mkConflict(r.conflict);

  case DomainUpdate: {
      bool changed = false;
      for (auto& u : r.updates) {
          auto ur = domainStore.update(u.var, u.newInterval, u.reasons);
          if (ur.conflict) return TheoryCheckResult::mkConflict(ur.conflict);
          changed |= ur.changed;
      }
      if (changed) markDomainChanged(); // ensure bounded solver reads latest domain
      return NoChange;
  }

  case SplitSuggestion:
      if (adapter.canSplit(r.split)) {
          return TheoryCheckResult::mkLemma(adapter.makeSplitLemma(r.split));
      }
      return NoChange; // ignore split if not supported

  case UnknownBudget:
      return NoChange; // ICP is advisory; continue with remaining pipeline
}
```

### IncrementalLinearizer Relationship
IncrementalLinearizer should depend **only** on `ReasonedBox` / `IntervalOps` / bound reasons. It should **not** depend on `IcpEngine` worklist or contractors. This keeps the two agents decoupled.

## Red Lines

1. All interval ops over-approx; no corner sampling for UNSAT.
2. Division by interval containing zero → `NoChange`, never contract.
3. NIA contractor results rounded to integers: `[ceil(a), floor(b)]`.
4. Every conflict includes all used bound reasons.
5. Budget exhaustion → `UnknownBudget`, never `Unsat`.
6. Candidate model must pass exact validator before returned as `Sat`.
7. ICP core has **zero dependency** on NIA/NRA types (only generic `TheoryId`).
