我建议选 **[4] Revise**，不能直接 Approve。方向是对的：trail-based CDCL(T)、`TheoryBridge`、`SatVar -> TheoryAtomRecord`、`TheoryManager::check()` 不再扫 atoms，这些都是最终架构应该走的方向。Plan 也明确当前 scan-all-atoms 的问题：每个 SAT model 都扫 `atomizer.atoms()`、`solver->reset()`、重新 assert theory literals。

但这版还有几个**正确性级别的硬伤**，尤其是“parsed atoms 怎么进入 `TheoryAtomRegistry`”和“bound trail undo 顺序”。

---

# 1. 最大问题：parsed theory atoms 也必须进 `TheoryAtomRegistry`

Plan 里 `TheoryBridge::onAssign()` 是这样：

```cpp
const auto* atom = registry_.findBySatVar(lit.var());
if (!atom) return;
tm_.assertTheoryLit(*atom, lit, level);
```

这意味着：**只有能在 `TheoryAtomRegistry` 中通过 SatVar 查到的 atom，才会被送给 theory solver。**

但 Plan 只明确了：

```text
getOrCreateLinearBoundAtom populates both maps
```

这主要覆盖 dynamic atoms，比如 branch/cut/tightening 产生的新 atom。它没有明确说明：**SMT 输入中原始 parsed theory atoms 在 `atomizer.atomize(assertion, ir)` 时，也必须同步注册到 `TheoryAtomRegistry`。**

否则会出现严重问题：

```text
(assert (<= (* 2 x) 5))
(assert (>= x 0))
```

这两个原始 atom 被 SAT 赋值后，`TheoryBridge` 查不到 `SatVar -> TheoryAtomRecord`，于是直接 `return`，`LiaSolver` 完全收不到原始约束。这样 solver 可能错误返回 SAT，或者 theory state 为空。

必须补：

```text
All theory atoms, parsed or dynamic, must be registered in TheoryAtomRegistry.

Atomizer::atomize() for parsed CoreIr theory atoms must call:
    registry.registerParsedTheoryAtom(satVar, exprId, theory, normalized metadata)

Dynamic atoms still use:
    getOrCreateLinearBoundAtom(...)

TheoryBridge must rely only on TheoryAtomRegistry::findBySatVar().
No separate Atomizer::atoms() route remains.
```

对于 parsed LIA/LRA atoms，`TheoryAtomRecord` 里可以先只存：

```cpp
exprId = original expr
isDynamic = false
```

然后 `LiaSolver::assertLit()` 用 `exprId` 从 CoreIr extract；但更好的最终版本是 parsed atom 注册时也填好 `lhs/rel/rhs`，避免 assert 时再解析。

---

# 2. `LiaSolver::assertLit` 里 old bound 记录顺序是错的

Plan 里写：

```cpp
manager_.assertBound(gs_, aux, atom.rel, value, reason);
// Record old bound for undo
bool isLower = ...;
BoundInfo old = isLower ? gs_.getLower(aux) : gs_.getUpper(aux);
trail_.push_back({level, TrailEntry::Kind::Bound, aux, isLower, old});
```

这个顺序错误。你先 assert 了新 bound，再读取 `old`，读到的是**新 bound**，不是旧 bound。回溯时无法恢复。

必须改成：

```cpp
auto boundUpdate = manager_.computeBoundUpdate(atom.rel, value);
bool isLower = boundUpdate.isLower;

BoundInfo old = isLower ? gs_.getLower(aux) : gs_.getUpper(aux);

auto ar = manager_.assertBound(gs_, aux, atom.rel, value, reason, level);
if (ar.conflict) return/store conflict;

trail_.push_back({level, TrailEntry::Kind::Bound, aux, isLower, old});
```

更干净的是：**不要 LiaSolver 自己恢复 GeneralSimplex bound**，让 `GeneralSimplex` 自己通过 `backtrackToLevel()` 管理 bound trail。这样 `LiaSolver::TrailEntry::Bound` 就不需要保存 `oldBound`，避免双重回滚。

建议最终改成：

```text
GeneralSimplex owns bound trail.
LiaSolver owns activeAtoms_ and disequalities_ trail.
LiaSolver::backtrackToLevel(level):
    pop activeAtoms_/disequalities_ trail
    gs_.backtrackToLevel(level)
```

不要同时让 LiaSolver 和 GeneralSimplex 都恢复同一个 bound。

---

# 3. `onBacktrack(0) + replay all vars` 不是最终实现，不能说 complexity 只是“moves into SAT wrapper”

Plan 说 CaDiCaL 没有 fine-grained callback，所以 `CadicalBackend::solve()` 通过 model delta simulation，并且简化为：

```text
listener_->onBacktrack(0)
then replay every true/false var
```

这本质上还是**每次 SAT model 后全量重放所有变量**，只是把扫描从 `TheoryManager` 移到了 `CadicalBackend`。这不是用户要求的最终 solver 形态。Plan 里也承认仍是 `O(#vars)` per solve。

可以作为临时兼容层，但不能作为最终架构的“完成版本”。必须写清楚：

```text
CadicalBackend model replay is a temporary compatibility adapter, not the final implementation.

Final backend must provide assignment trail events:
    onAssign(lit, decisionLevel)
    onBacktrack(level)

The code architecture must isolate replay backend behind SatAssignmentListener, and tests must include a fake callback-capable backend to validate true incremental behavior.
```

如果你坚持“不要分调试和正经 solver”，那更严格的要求是：**不要把 model-delta simulation 当主路径**。至少要实现一个内部 SAT wrapper 的 assignment trail 记录，或者换/封装支持 trail 事件的 backend。否则 performance 目标并没有真正实现。

---

# 4. duplicate lemma 时 `continue` 会导致无限循环

Plan 里写：

```cpp
if (tr.kind == Lemma) {
    if (lemmaDb.insertIfNew(*tr.lemma)) {
        sat->addClause(tr.lemma->lits);
    }
    // Even if duplicate, we continue
    continue;
}
```

这个逻辑有风险。如果 theory solver 反复返回同一个 duplicate lemma，而 `sat->addClause` 没有发生，那么下一轮 `sat->solve()` 可能返回同一个 model，theory 再返回同一个 duplicate lemma，进入死循环。

必须改成：

```text
Theory solver should not return duplicate lemmas.

If Solver layer detects duplicate lemma, it must either:
  1. ask the same theory solver to continue to another reasoning source in the same check; or
  2. treat duplicate lemma as a theory-solver bug and return Unknown/internal error; or
  3. call theoryManager.blockLemmaAndContinue(key), then re-run theory check without SAT solve.

Do not simply continue to SAT solve without adding a clause.
```

更好的最终设计：

```cpp
auto tr = theoryManager_.check(lemmaDb);
```

让 `LiaSolver` 在内部：

```cpp
if (lemmaDb.contains(candidate)) {
    skip candidate;
    continue to next reasoning source;
}
return lemma(candidate);
```

所以要改成：

```text
TheoryLemmaDatabase must be visible to TheoryManager/TheorySolver during check(), not only used after the lemma is returned.
```

---

# 5. `emittedLemmas_` 不应该在 `LiaSolver::reset()` 中无脑清空，至少要明确作用域

Plan 里 Phase 2.5 写：

```text
reset() clears ... emittedLemmas_
```

如果 `reset()` 只在 fresh `check-sat` call 前调用，可以接受。
但如果 `reset()` 仍被某些 CDCL loop、`onBacktrack(0)`、legacy path 调用，那就会再次导致重复 lemma。

必须写死：

```text
TheorySolver::reset() is only called once per fresh check-sat initialization, never between SAT models.

Lemma database belongs to Solver/checkSatInternal, not LiaSolver reset/rebuild loop.
It must survive SAT solve iterations and theory backtracks during the same check-sat.
```

---

# 6. `SatLit` 的 value 不能只用 `lit.isPositive()`

Plan 里 `TheoryBridge` 调：

```cpp
tm_.assertTheoryLit(*atom, lit, level);
```

然后：

```cpp
bool value = assignedLit.isPositive();
```

这只有在 `lit` 表示“该 SatVar 被赋值为 true/false”的 signed literal 时才成立。你要确保语义是：

```text
onAssign(SatLit(var, positive)) means SatVar var is assigned true.
onAssign(SatLit(var, negative)) means SatVar var is assigned false.
```

这可以，但必须在 `SatAssignmentListener` 注释里写死。否则容易被误解成“assigned literal 本身为 true”，那 negative literal 的含义会混乱。

建议写：

```cpp
// onAssign receives the signed assignment:
//   positive(var) means variable var is assigned true;
//   negative(var) means variable var is assigned false.
```

---

# 7. ActiveAtom / Disequality undo 假设 LIFO pop，必须保证 trail 顺序一致

Plan 里 undo：

```text
Disequality: pop from disequalities_
ActiveAtom: pop from activeAtoms_
```

这要求所有 push 顺序严格 LIFO，而且每个 `TrailEntry` 对应最后一个元素。可行，但要在结构里保存 index 更稳：

```cpp
struct TrailEntry {
    int level;
    Kind kind;
    size_t vectorSizeBefore;
    ...
};
```

undo 时：

```cpp
activeAtoms_.resize(entry.vectorSizeBefore);
disequalities_.resize(entry.vectorSizeBefore);
```

不要只 `pop_back()`，因为一个 `assertLit` 可能 push 多个 entries，后续如果某个 assertBound 冲突或提前返回，LIFO 很容易乱。

---

# 8. CdcacSolver “stub backtrackToLevel” 不能在注册时被使用

Plan 说 CdcacSolver 可以 stub 新接口，full migration deferred。可以，但前提是：

```text
在 QF_LIA / QF_LRA 路径下不注册 CdcacSolver。
在 QF_NRA 路径下，如果 CdcacSolver 仍是 scan-based，而新的 loop 不传 atoms，它就不能工作。
```

所以 Phase 6 里不能简单“stub backtrackToLevel”。要么：

```text
QF_NRA 暂时走 legacy scan-based path；
QF_LIA/QF_LRA 走 trail-based path。
```

要么同时迁移 CdcacSolver 的 assertLit。否则 NRA tests 会坏。

Plan 的风险表说“Keep fallback scan mode if needed”，这个要变成明确设计，不然 execution 会卡住。

# 结论

这版**方向正确，但不能 Approve**。核心架构是对的，但有几处会直接导致：

```text
原始 atoms 不进入 theory solver；
bound 回溯恢复错误；
duplicate lemma 死循环；
“模拟 trail”仍然全量回放；
NRA solver 新旧路径混乱。
```

我的选择：**[4] Revise**。
