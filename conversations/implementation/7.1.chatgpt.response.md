建议选 **[4] Revise**，不要直接 Approve。

这版 Phase NRA-1 的方向是对的：目标只是打通 QF_NRA 的 polynomial payload、让 `CdcacSolver` 能精确处理常量约束，非常量先返回 `Unknown`，这符合“先 sound 后 complete”的路线。计划里也明确说 Phase NRA-1 不做 local search、root isolation、CAD/CDCAC、projection 等高级内容，这个边界是合理的。

但现在有几个正确性/接口问题必须修。

---

# 1. QF_NRA 中 linear atom 也必须进 CdcacSolver

计划里写：

```text
Step 1: Try extractLinearConstraint first (unchanged).
Step 2: If it fails and polyKernel_ != nullptr, extract polynomial.
```

这个在 QF_NRA 下有风险。因为 QF_NRA 里的线性约束也是 NRA 约束，应该交给 `CdcacSolver`。例如：

```smt2
(set-logic QF_NRA)
(declare-const x Real)
(assert (>= x 0))
(assert (= (* x x) 1))
```

如果 `x >= 0` 被 `extractLinearConstraint` 抽成 `LinearAtomPayload`，但当前 QF_NRA 只注册了 `CdcacSolver`，那这个 atom 可能不会被正确处理。

Phase NRA-1 应该明确：

```text
Under QF_NRA, all arithmetic comparison atoms, including linear ones,
must be registered as PolynomialAtomPayload and routed to TheoryId::NRA.
```

也就是说，在 `defaultTheory_ == TheoryId::NRA` 时，不应该优先注册成 linear atom。可以这样写：

```cpp
if (defaultTheory_ == TheoryId::NRA) {
    // Convert both linear and nonlinear arithmetic expressions to polynomial payload.
    if (extractPolynomialConstraint(eid, ir, polyPayload.poly,
                                    polyPayload.rhs, polyPayload.rel)) {
        registry_->registerParsedTheoryAtom(
            v, eid, TheoryId::NRA, polyPayload);
    } else {
        registry_->setUnsupportedTheorySeen();
    }
}
else {
    // Existing LIA/LRA linear path.
    if (extractLinearConstraint(...)) { ... }
}
```

这个是必须修的，否则 QF_NRA 会漏掉线性约束。

---

# 2. `PolynomialKernel` 不能有两个不共享的实例

计划里代码是：

```cpp
auto polyKernel = createPolynomialKernel();
atomizer.setPolynomialKernel(polyKernel.get());
theoryManager.registerSolver(
    std::make_unique<CdcacSolver>(std::move(polyKernel)));
```

这个写法本身可以成立，因为 `atomizer` 拿的是同一个 kernel 对象的 raw pointer，所有权转给 `CdcacSolver`。但 Risks 里写“两份 `PolynomialKernel` 实例也可以接受”，这个说法不对。

`PolynomialId` 只能在创建它的 `PolynomialKernel` 里解释。如果 Atomizer 用 A kernel 创建 `PolynomialId`，CdcacSolver 用 B kernel 读取这个 id，就会错。

必须改成：

```text
Atomizer and CdcacSolver must share the same PolynomialKernel instance
for a check-sat call. PolynomialId is kernel-local and cannot cross kernel instances.
```

更稳的实现方式：

```cpp
auto polyKernel = std::shared_ptr<PolynomialKernel>(createPolynomialKernel());
atomizer.setPolynomialKernel(polyKernel.get());
theoryManager.registerSolver(
    std::make_unique<CdcacSolver>(polyKernel));
```

或者保留 `unique_ptr`，但风险表里必须删除“两份 kernel 可接受”的说法。

---

# 3. 非整数有理常数不能暂时忽略

计划风险里写：

```text
LibPolyKernel::mkConst drops denominators (known TODO)
Only affects non-integer constants; Phase NRA-1 tests use integer constants
```

这个对“正确性优先”的 solver 不可接受。即使测试只用整数常量，用户输入可以是：

```smt2
(assert (= (/ 1 2) 0))
(assert (> (/ 1 2) 0))
(assert (= x (/ 1 2)))
```

如果 `mkConst` 丢分母，NRA 结果可能直接 unsound。

Phase NRA-1 必须二选一：

```text
Option A: Fix LibPolyKernel::mkConst(mpq_class) to preserve rational constants exactly.

Option B: Until fixed, PolynomialConverter rejects non-integer rational constants and marks unsupportedTheorySeen_, returning Unknown.
```

不能说“只影响非整数常量，所以暂时接受”。

---

# 4. `pendingConflict_` 类型不一致

计划里 `CdcacSolver.h` 写：

```cpp
std::optional<TheoryConflict> pendingConflict_;
```

但 `backtrackToLevel()` 里写：

```cpp
if (pendingConflict_ && pendingConflict_->level > level) {
    pendingConflict_.reset();
}
```

这说明它其实需要 level。应该统一为：

```cpp
struct PendingConflict {
    int level;
    TheoryConflict conflict;
};

std::optional<PendingConflict> pendingConflict_;
```

然后：

```cpp
if (pendingConflict_) {
    return TheoryCheckResult::mkConflict(pendingConflict_->conflict);
}
```

否则代码本身就不一致。

---

# 5. payload mismatch 不能生成 theory conflict

计划里：

```cpp
const auto* payload = std::get_if<PolynomialAtomPayload>(&atom.payload);
if (!payload) {
    pendingConflict_ = TheoryConflict{{reason.negated()}};
    return;
}
```

这不严谨。payload mismatch 是内部路由错误，不是一个合法的 NRA theory conflict。你不能从“这个 atom 不是 polynomial payload”推出 `¬reason` 是 theory-valid clause。

应该改成：

```cpp
if (!payload) {
    pendingUnknown_ = true;  // or pendingInternalError_
    return;
}
```

然后 `check()` 返回 `Unknown`。
只有在确实证明某个 theory literal 在 NRA 中不可满足时，才能返回 conflict clause。

---

# 6. `Distinct` 的 polynomial extraction 要谨慎

计划把：

```text
Distinct -> Neq
```

这对二元 arithmetic distinct 是对的：

```smt2
(distinct p q)  ==>  p - q != 0
```

但 SMT-LIB 的 `distinct` 可以有多个参数：

```smt2
(distinct a b c)
```

多元 distinct 语义是 pairwise disequality。Phase NRA-1 如果只支持二元 comparison，必须写清楚：

```text
For Distinct, support only arity 2 in Phase NRA-1.
If arity > 2, mark unsupported and return Unknown,
or expand into pairwise disequalities at atomization.
```

计划中只写了 `e.children.size() == 2`，这对多元 distinct 会 unsupported，没问题，但要明确这是 intentional。

---

# 7. `CdcacSolver::check()` 返回 constant conflict 是 sound 的，但测试还不够

当前三个测试：

```text
constant unsat
constant sat
non-constant unknown
```

是最低限度。还应该加：

```smt2
(set-logic QF_NRA)
(assert (not (> 1 0)))
(check-sat)
```

应为 `unsat`，测试 false literal negation。

再加：

```smt2
(set-logic QF_NRA)
(assert (distinct 1 1))
(check-sat)
```

应为 `unsat`。

再加：

```smt2
(set-logic QF_NRA)
(assert (not (= 1 1)))
(check-sat)
```

应为 `unsat`。

这可以验证 `Eq/Neq` 和 false literal 的 effective relation 路径。

# 结论

这版计划**方向正确，但还不能 Approve**。
我的选择是 **[4] Revise**。

主要卡点不是 CDCAC 算法，而是 Phase NRA-1 的工程语义：**QF_NRA 下 linear atoms 也要进 polynomial path、PolynomialId 不能跨 kernel、rational constants 不能错、payload mismatch 不能伪造 conflict**。这些补上后，Phase NRA-1 可以 Approve。
