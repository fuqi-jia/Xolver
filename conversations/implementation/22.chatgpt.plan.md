下面给你一版 **EUF（Equality with Uninterpreted Functions）算法与工程实现详细计划**。目标不是写一个玩具 congruence closure，而是写成可以长期接入 NLColver / CDCL(T) / 未来 MCSAT 的 **独立 EUF theory solver**。

核心目标：

```text
支持 QF_UF / QF_UFLIA / QF_UFLRA / QF_UFNIA / QF_UFNRA
支持 equality / disequality
支持 congruence closure
支持 theory conflict explanation
支持 theory propagation
支持 model construction
支持与算术 theory 的 equality sharing
```

---

# 0. EUF 求解的本质

EUF 处理的是：

```smt2
a = b
a != b
f(a) = f(b)
g(f(a), c) != g(f(b), c)
```

核心规则是 **congruence**：

```text
a1 = b1, ..., an = bn
--------------------------------
f(a1,...,an) = f(b1,...,bn)
```

EUF 可满足性判断本质上是维护等价类：

```text
term equivalence classes
```

然后检查：

```text
如果某个 disequality 的两端落入同一个等价类，则 UNSAT。
```

例如：

```smt2
(assert (= a b))
(assert (not (= (f a) (f b))))
```

因为 `a=b` 推出 `f(a)=f(b)`，所以与 `f(a)!=f(b)` 冲突。

## 2.2 与 arithmetic solver 并存


但如果要做真正 `QF_UFLIA`，需要 **EqualityManager**，后面会详细说。

# 3. Term 表示

EUF 需要自己的 term DAG，但应从 Core IR / SOMTParser 转换而来。

## 3.1 EufTermId

```cpp
using EufTermId = uint32_t;
using EClassId = uint32_t;
using FunctionSymbolId = uint32_t;
```

---

## 3.2 ENode

每个 EUF term 是一个 node：

```cpp
struct ENode {
    EufTermId id;

    // 对变量/常量/函数应用统一表示
    FunctionSymbolId symbol;

    std::vector<EufTermId> args;

    SortId sort;

    // 原始 Core IR term，便于 explanation / model / debug
    ExprId origin = INVALID_EXPR;
};
```

例子：

```text
a                  -> symbol = a, args = []
b                  -> symbol = b, args = []
f(a)               -> symbol = f, args = [a]
g(f(a), b)         -> symbol = g, args = [f(a), b]
```

常量也可以看作 0 元函数。

---

## 3.3 Canonical App Key

Congruence closure 需要根据：

```text
symbol + representatives(args)
```

判断两个函数应用是否应该合并。

```cpp
struct AppKey {
    FunctionSymbolId symbol;
    std::vector<EClassId> argClasses;

    bool operator==(const AppKey&) const;
};
```

hash：

```cpp
struct AppKeyHash {
    std::size_t operator()(const AppKey& k) const;
};
```

例如：

```text
a = b

f(a) key = f(rep(a))
f(b) key = f(rep(b))

如果 rep(a)=rep(b)，则 f(a) 和 f(b) key 相同，需要 merge。
```

# 4. EGraph 数据结构

## 4.1 Union-Find

```cpp
class EufUnionFind {
public:
    EClassId find(EClassId x);
    bool same(EClassId a, EClassId b) const;

    // merge 返回新的 representative
    EClassId unite(EClassId a, EClassId b, MergeReason reason);
```

如果第一版采用 rebuild 模式，可以不做 rollback：

```text

```

我建议 V1 先用 full rebuild，和你 IDL/RDL V1 类似：


这样 correctness 最稳。

## 4.2 Class members

每个 equivalence class 维护成员：

```cpp
struct EClass {
    EClassId id;
    EClassId parent;

    std::vector<EufTermId> members;

    // 该类中函数应用节点的 use list
    std::vector<EufTermId> uses;

```

## 4.3 Use lists

当两个 class merge 后，可能导致 parent applications congruent。

例如：

```text
merge(a,b)
```

要检查所有使用 `a` 或 `b` 的函数应用：

```text
f(a), g(a,c), h(d,a)
f(b), g(b,c), h(d,b)
```

因此 term manager 要维护：

```cpp
std::unordered_map<EufTermId, std::vector<EufTermId>> parentsOf;
```

也就是：

```text
term -> applications containing term as argument
```

---

## 4.4 Congruence table

```cpp
std::unordered_map<AppKey, EufTermId, AppKeyHash> appTable_;
```

当 class merge 后，要重新 canonicalize affected applications：

```text
key = symbol + current representatives of arguments
```

如果 key 已存在另一个 term，则 merge 两个 application terms。

# 5. Atom 提取

EUF solver 只处理 equality / disequality atom。

## 5.1 Supported atoms

```text
t1 = t2
distinct(t1,t2)
not (= t1 t2) 通过 effective relation 变成 disequality
```

内部统一为：

```cpp
struct EufAtom {
    EufTermId lhs;
    EufTermId rhs;
    Relation rel;      // Eq or Neq
    SatLit reason;
};
```

---

## 5.2 Effective literal handling

和 IDL/RDL 一样，必须先根据 SAT assignment 取 effective relation：

```text
literal value=true:
  原 relation

literal value=false:
  Eq  -> Neq
  Neq -> Eq
```

例如：

```smt2
(assert (not (= a b)))
```

如果 parser 保留为 `Not(Eq)`，atomizer 可能给 `Eq` 一个 literal，然后 assignment false。

如果 parser 转成 `distinct`，则是 `Neq true`。

EUF solver 不应该依赖 parser 的形式，统一处理：

```cpp
Relation effectiveRelation(Relation rel, bool value) {
    if (value) return rel;
    if (rel == Relation::Eq) return Relation::Neq;
    if (rel == Relation::Neq) return Relation::Eq;
    unsupported;
}
```

# 6. 第一版算法：Full Rebuild Congruence Closure

V1 建议不要做 incremental e-graph。先做 full rebuild：

```cpp
TheoryCheckResult EufSolver::check() {
    egraph_.clear();
    disequalities_.clear();

    // 1. register all terms appearing in active atoms
    for (auto& a : activeAssignments_) {
        auto atom = extractEffectiveAtom(a);
        registerTermRec(atom.lhsExpr);
        registerTermRec(atom.rhsExpr);
    }

    // 2. assert equalities
    for (auto& atom : atoms) {
        if (atom.rel == Eq) {
            egraph_.merge(atom.lhs, atom.rhs, atom.reason);
            propagateCongruence();
        } else {
            disequalities_.push_back(atom);
        }
    }

    // 3. check disequalities
    for (auto& d : disequalities_) {
        if (egraph_.same(d.lhs, d.rhs)) {
            return conflict(explain(d.lhs, d.rhs, d.reason));
        }
    }

    // 4. optional propagation
    if (auto prop = findPropagation()) {
        return lemma(prop);
    }

    // 5. model
    buildModel();
    return consistent;
}
```

# 7. Congruence closure 核心流程

## 7.1 Merge

```cpp
void EGraph::merge(EufTermId a, EufTermId b, SatLit reason) {
    auto ra = find(a);
    auto rb = find(b);

    if (ra == rb) return;

    auto r = unionClasses(ra, rb, reason);

    // 检查 affected uses
    enqueueUses(ra);
    enqueueUses(rb);

    processCongruenceQueue();
}
```

---

## 7.2 Process congruence queue

```cpp
void EGraph::processCongruenceQueue() {
    while (!queue.empty()) {
        EufTermId t = queue.pop();

        if (!isApplication(t)) continue;

        AppKey key = canonicalKey(t);

        auto it = appTable_.find(key);
        if (it == appTable_.end()) {
            appTable_[key] = t;
            continue;
        }

        EufTermId existing = it->second;

        if (!same(t, existing)) {
            MergeReason reason = MergeReason::congruence(t, existing);
            merge(t, existing, reason);
        }
    }
}
```

注意：每次 class merge 后，旧 `appTable_` 可能有 stale key。V1 简单做法：

```text
每轮 merge 后重建 appTable_。
```

更简单但慢：

```cpp
bool changed = true;
while (changed) {
    changed = false;
    rebuild appTable_;

    for each application t:
        key = canonicalKey(t);
        if key already exists as u and !same(t,u):
            merge(t,u);
            changed = true;
}
```

复杂度稍高，但 V1 correctness 好。

建议 V1 就用这个：

```text
Full rebuild + repeated appTable scan
```

后续再优化为 use-list incremental。

# 8. Explanation 设计

EUF 最关键的是 conflict explanation。

例如：

```smt2
(assert (= a b))                         ; l1
(assert (= b c))                         ; l2
(assert (not (= (f a) (f c))))           ; l3
```

冲突来自：

```text
a = b
b = c
=> a = c
=> f(a) = f(c)
contradicts f(a) != f(c)
```

Conflict clause：

```text
¬l1 ∨ ¬l2 ∨ ¬l3
```

---

## 8.1 Merge reason

每次 merge 记录 reason：

```cpp
enum class MergeReasonKind {
    AssertedEquality,
    Congruence
};

struct MergeReason {
    MergeReasonKind kind;

    // asserted equality
    SatLit literal;

    // congruence merge: f(a1,..,an) = f(b1,..,bn)
    EufTermId lhsApp;
    EufTermId rhsApp;
    std::vector<std::pair<EufTermId, EufTermId>> argEqualities;
};
```

---

## 8.2 Explanation forest

Union-Find 本身压缩路径后不适合解释。建议维护一个 explanation graph：

```cpp
struct ExplainEdge {
    EClassId from;
    EClassId to;
    MergeReason reason;
};
```

每次 merge 两个 class，添加一条 explanation edge：

```text
classA --reason--> classB
```

要解释 `a=b`，在 explanation graph 上找从 `class(a)` 到 `class(b)` 的路径，收集 reasons。

V1 简单实现：

```text
记录所有 merge events；
要 explain(a,b) 时，用这些 merge events 构建临时无向图；
BFS 找路径；
递归展开 congruence reason。
```

---

## 8.3 Recursive explanation

如果 merge reason 是 asserted equality：

```text
return {literal}
```

如果 merge reason 是 congruence：

```text
f(a1,...,an) = f(b1,...,bn)
```

则需要解释每个：

```text
a_i = b_i
```

递归调用：

```cpp
explainEquality(ai, bi)
```

最终收集所有 asserted literals。

伪代码：

```cpp
std::vector<SatLit> explainEquality(EufTermId a, EufTermId b) {
    if (a == b) return {};

    auto path = findMergePath(a, b);

    std::vector<SatLit> reasons;
    for (auto& edge : path) {
        auto rs = explainMergeReason(edge.reason);
        appendUnique(reasons, rs);
    }
    return reasons;
}

std::vector<SatLit> explainMergeReason(MergeReason r) {
    if (r.kind == AssertedEquality) {
        return {r.literal};
    }

    if (r.kind == Congruence) {
        std::vector<SatLit> out;
        for (auto [ai, bi] : r.argEqualities) {
            appendUnique(out, explainEquality(ai, bi));
        }
        return out;
    }
}
```

---

## 8.4 Disequality conflict

若 active disequality：

```text
d: lhs != rhs
```

但 `same(lhs,rhs)`。

原因：

```text
explainEquality(lhs,rhs) + d.reason
```

Conflict clause：

```cpp
std::vector<SatLit> clause;
for (auto lit : equalityReasons) {
    clause.push_back(lit.negated());
}
clause.push_back(d.reason.negated());
```

# 9. Theory propagation

EUF 可以主动传播 registered equality atom。

如果有 atom：

```text
A: s = t
```

当前 egraph 已经推出：

```text
s = t
```

且 SAT 中 `A` 未赋值，则可以 propagate `A=true`：

```text
reason(s=t) => A
```

lemma：

```text
¬r1 ∨ ... ∨ ¬rk ∨ A
```

如果当前 egraph 推出：

```text
s != t
```

EUF 本身通常不会推出 disequality，除非有 active disequality 或 disjoint sorts。
V1 不做 negative propagation。

---

## 9.1 Propagation source

EufSolver 应该维护所有 registered equality atoms：

```cpp
std::vector<EufAtomRecord> registeredAtoms_;
```

在 check 后遍历：

```cpp
for atom in registeredAtoms:
    if atom.rel == Eq && unassigned(atom.lit):
        if egraph.same(atom.lhs, atom.rhs):
            reasons = explainEquality(atom.lhs, atom.rhs)
            return propagation lemma
```

V1 如果没有 SAT assignment 查询接口，可以先不做 propagation，只做 conflict。

但我建议至少实现 propagation lemma，因为对 EUF 很有用。

---

# 10. Disequality handling

EUF disequality 是直接检查，不需要 split。

```text
a != b
```

如果 `a,b` 当前不同 class：

```text
NoChange
```

如果后续 merge 成同 class：

```text
Conflict
```

不同于 LRA/LIA disequality，EUF 不需要：

```text
a < b ∨ a > b
```

因为 EUF domain 没有 order。

# 11. Model construction

如果无冲突，EUF model 需要给每个 equivalence class 一个值。

## 11.1 Class values

```cpp
struct EufModel {
    std::unordered_map<EClassId, ModelValue> classValue;
    std::unordered_map<FunctionSymbolId, FunctionInterpretation> functions;
};
```

每个 equivalence class 分配：

```text
ufval_0
ufval_1
...
```

---

## 11.2 Function interpretation

对于每个 function application：

```text
f(a1,...,an) = r
```

模型中记录：

```text
f(class(a1),...,class(an)) -> class(r)
```

例如：

```text
term f(a)
args classes: [class(a)]
result class: class(f(a))
```

函数解释：

```text
f:
  (class(a)) -> class(f(a))
  default -> fresh value
```

---

## 11.3 Consistency

如果两个 applications 的 argument classes 相同，congruence closure 已经保证 result classes 相同。

因此 model construction 只要遍历所有 app nodes：

```cpp
for app in applications:
    key = canonicalKey(app)
    result = find(app)
    functions[symbol].entries[key.args] = result;
```

如果发现同一个 key 对应不同 result class，说明 egraph bug。

# 12. Sort handling

EUF 必须尊重 sort。

## 12.1 Equality sort check

```text
t1 = t2
```

只有当：

```text
sort(t1) == sort(t2)
```

才合法。

如果 Core IR 已经 sort check，可以 trust；否则 EUF extractor 必须防御：

```text
sort mismatch -> Unsupported / InternalError
```

不要把不同 sort merge。

---

## 12.2 Function symbol signature

```cpp
struct FunctionSymbol {
    std::string name;
    std::vector<SortId> domainSorts;
    SortId rangeSort;
    bool interpreted = false;
};
```

EUF 只处理 uninterpreted function。
如果遇到 interpreted arithmetic op：

```text
+，-，*，<=
```

不归 EUF。

但混合项中可能有：

```text
f(x+1)
```

需要 purification，后面讲。

# 13. 与 arithmetic 的组合

如果只做纯 `QF_UF`，EUF 简单。
但你要支持：

```text
EUF + LIA/LRA/NIA/NRA
```

那就需要 **purification + equality sharing**。

---

## 13.1 Purification

例如：

```smt2
(assert (= (f (+ x 1)) y))
```

混合了 UF 和 arithmetic term `(+ x 1)`。

Purify：

```text
t = x + 1       // arithmetic owns
f(t) = y        // EUF owns
```

引入 fresh term `t`。

对于：

```smt2
(assert (= (+ (f a) 1) x))
```

Purify：

```text
u = f(a)        // EUF owns
u + 1 = x       // arithmetic owns
```

---

## 13.2 Shared terms

Shared terms 是多个 theory 都关心的 term。

```cpp
struct SharedTerm {
    TermId coreTerm;
    SortId sort;
    std::vector<TheoryId> owners;
};
```

EqualityManager 维护 shared equality：

```text

```

并广播给 EUF / arithmetic。

# 14. Atomizer / Registry 设计

EUF equality atom 也必须走 unified atom registry，不能重复创建。

## 14.1 EufAtomKey

```cpp
struct EufAtomKey {
    EufTermId lhs;
    EufTermId rhs;
    Relation rel; // Eq / Neq
};
```

Equality 要 canonicalize：

```text
a = b 与 b = a 是同一个 atom
```

Disequality 也 canonicalize：

```text
a != b 与 b != a 是同一个 atom
```

可以按 term id 排序：

```cpp
if (lhs > rhs) std::swap(lhs, rhs);
```

---

## 14.2 注意：term id 要 canonical

如果 `(= (f a) (f b))` 多次出现，必须复用 term nodes。

Term manager 应该通过 key：

```text
symbol + children term ids
```

```cpp
EufTermId intern(FunctionSymbolId symbol, std::vector<EufTermId> args);
```

这样相同 term DAG 不重复。

# 17. Rebuild check 详细伪代码

```cpp
TheoryCheckResult EufSolver::check() {
    egraph_.clear();
    disequalities_.clear();
    atomRecords_.clear();

    // 1. Convert active assignments to effective EUF atoms
    for (const auto& a : activeAssignments_) {
        auto raw = extractor_.extract(a.atomExpr, ir_);
        if (!raw) {
            return TheoryCheckResult::unknown("unsupported EUF atom");
        }

        Relation rel = effectiveRelation(raw.rel, a.value);

        EufAtom atom{
            .lhs = termManager_.intern(raw.lhs),
            .rhs = termManager_.intern(raw.rhs),
            .rel = rel,
            .reason = a.lit
        };

        if (rel == Relation::Eq) {
            equalities_.push_back(atom);
        } else {
            disequalities_.push_back(atom);
        }
    }

    // 2. Register all terms in graph
    for each term in termManager:
        egraph_.addTerm(term)

    // 3. Assert equalities
    for (auto& eq : equalities_) {
        egraph_.merge(eq.lhs, eq.rhs,
                      MergeReason::asserted(eq.reason));
        egraph_.closeCongruence();
    }

    // 4. Check disequalities
    for (auto& neq : disequalities_) {
        if (egraph_.same(neq.lhs, neq.rhs)) {
            auto rs = explainer_.explainEquality(neq.lhs, neq.rhs);
            rs.push_back(neq.reason);

            return TheoryCheckResult::mkConflict(
                makeConflictClause(rs)
            );
        }
    }

    // 5. Optional propagation
    if (enablePropagation_) {
        if (auto lemma = findEufPropagation()) {
            return TheoryCheckResult::mkLemma(*lemma);
        }
    }

    buildModel();
    return TheoryCheckResult::consistent();
}
```

# 18. Explanation example

输入：

```smt2
(assert (= a b))                 ; l1
(assert (= b c))                 ; l2
(assert (distinct (f a) (f c)))  ; l3
```

流程：

```text
merge(a,b) reason l1
merge(b,c) reason l2

close congruence:
  f(a) and f(c) have same arg class
  merge(f(a), f(c)) reason congruence

disequality f(a)!=f(c) violated
```

explain:

```text
explain(f(a)=f(c))
```

## 20.2 Efficient congruence closure

用 use-list + signature table。


## 20.3 Lazy explanation

不要每次 merge 都展开 explanation。只在 conflict / propagation 时递归解释。

对 large UF benchmarks，使用：

```text
symbol -> applications
class -> parent applications
```

避免全表扫描。

# 24. 关键红线

```text
1. 不同 sort 不能 merge。
2. disequality conflict 必须包含完整 equality explanation。
3. congruence merge 的 explanation 必须能递归追到 asserted equality。
4. parser 把 not (= a b) 转成 distinct 或保留 not 都必须正确。
5. EUF 不处理 interpreted arithmetic ops。
6. 混合 UF/arithmetic 必须 purification。
7. SAT 前必须 validate EUF model。
8. V1 可以 full rebuild，先保正确。
9. propagation lemma 必须 guarded by explanation reasons。
10. function model 必须尊重 congruence closure。
```

# 总结

EUF 第一版最重要的是：

```text
term intern
+ congruence closure
+ disequality conflict
+ explanation
+ model validation
```

不要一开始就做复杂 incremental e-graph 或 Nelson-Oppen。先用 full rebuild 保证 correctness，然后再优化。

最终 EUF 在 NLColver 中的定位应该是：

```text
EufSolver:
  pure congruence closure engine

EqualityManager:
  future shared equality propagation for EUF + arithmetic

Purifier:
  future mixed-term normalization

ModelValidator:
  final SAT correctness guard
```

这样结构最稳，也最容易后续接 MCSAT、Proof 和 mixed theories。
