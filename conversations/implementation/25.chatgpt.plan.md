下面给你一版 **完整、高效、可工程落地的 Nelson-Oppen 理论组合实现计划**。我会按照你现在 NLColver 的方向来设计：已有/计划中的 EUF、LIA/LRA、NIA/NRA、IDL/RDL、ICP、IncrementalLinearizer 都要能接入，而不是写一个孤立的 textbook Nelson-Oppen。

---

# 0. 先明确 Nelson-Oppen 的适用边界

经典 Nelson-Oppen 组合适用于：

```text
1. quantifier-free formulas；
2. 各 theory signature disjoint，只共享 equality 和 shared variables；
3. shared sorts 上 stably infinite；
4. 各子 theory solver 能判断自身纯公式可满足性；
5. 组合通过 shared variables 上的 equality / disequality arrangement 完成。
```

但你现在要支持的理论里有两类：

```text
Convex theories:
  EUF
  LRA
  RDL

Non-convex theories:
  LIA
  IDL
  NRA
  NIA
```

区别非常重要。

---

## 0.1 Convex theory

如果某个 theory 推出：

```text
x = y ∨ u = v
```

convex theory 必须能推出其中某一个：

```text
x = y
```

或：

```text
u = v
```

典型 convex：

```text
EUF
LRA
RDL
```

组合时可以靠 equality propagation。

---

## 0.2 Non-convex theory

Non-convex theory 可能只能推出一个等式析取，但不能推出单个等式。

例如整数算术中：

```text
x ∈ {0,1}
y ∈ {0,1}
x + y = 1
```

可推出：

```text
x ≠ y
```

或者某些结构会产生 case split。

NRA 也 non-convex，例如：

```text
x² = 1
```

推出：

```text
x = 1 ∨ x = -1
```

但不能推出其中一个。

所以完整组合不能只做 equality propagation。对 non-convex theory，需要支持：

```text
arrangement split
case split lemma
delayed equality decision
```

也就是：

```text
x = y ∨ x ≠ y
```

由 SAT/CDCL 层搜索 arrangement。

# 2. Purification：组合的第一步

Nelson-Oppen 要求每个 theory 只看到自己的 pure atoms。混合项必须拆开。

---

## 2.1 什么是 alien subterm

如果一个 theory atom 中出现了另一个 theory 的 term，就是 alien subterm。

例如：

```smt2
(assert (> (f x) 0))
```

`>` 是 arithmetic atom，但 `(f x)` 是 EUF term。

需要 purification：

```text
u = f(x)      // EUF owns
u > 0         // arithmetic owns
```

再如：

```smt2
(assert (= (f (+ x 1)) y))
```

`f` 是 EUF，参数 `(+ x 1)` 是 arithmetic term。

purify：

```text
t = x + 1     // arithmetic owns
f(t) = y      // EUF owns
```

---

## 2.2 Purification 输出

```cpp
struct PurifiedFormula {
    std::vector<PureTheoryAtom> pureAtoms;
    std::vector<BridgeDefinition> bridges;
    std::vector<SharedTermId> sharedTerms;
};
```

`BridgeDefinition`：

```cpp
struct BridgeDefinition {
    SharedTermId fresh;
    TermId originalTerm;
    TheoryId ownerTheory;
    SatLit definitionLit;
};
```

例如：

```text
u = f(x)
```

是 EUF bridge。

```text
t = x + 1
```

是 arithmetic bridge。

---

## 2.3 Fresh shared variable

每个 alien subterm 被替换成一个 fresh shared variable：

```text
__sh_0
__sh_1
...
```

这些 shared variables 要：

```text
1. 有 sort；
2. 注册到 SharedTermRegistry；
3. 在所有 owner theories 中可见；
4. 不暴露到 public model，或在 model 输出时可解释回原 term。
```

# 4. Shared equality atom

组合层需要把 shared terms 之间的 equality 变成 SAT-visible atom。

```cpp
struct SharedEqualityKey {
    SharedTermId a;
    SharedTermId b;
    SortId sort;
};
```

Canonicalization：

```text
a = b 与 b = a 是同一个 key。
```

Registry API：

```cpp
SatLit getOrCreateSharedEqualityAtom(
    SharedTermId a,
    SharedTermId b
);

SatLit getOrCreateSharedDisequalityAtom(
    SharedTermId a,
    SharedTermId b
);
```

实际上 disequality 可以用 equality atom 的 negation：

```text
a ≠ b  等价于  ¬Eq(a,b)
```

推荐只建一个 equality atom：

```text
Eq(a,b)
```

然后：

```text
a=b  -> Eq(a,b)
a≠b -> ¬Eq(a,b)
```

这样 arrangement split 很自然：

```text
Eq(a,b) ∨ ¬Eq(a,b)
```

SAT 自己决定。

# 5. SharedEqualityManager

这是组合层核心。它维护当前已知的 shared equality / disequality。

```cpp
class SharedEqualityManager {
public:
    void assertEquality(
        SharedTermId a,
        SharedTermId b,
        SatLit reason);

    void assertDisequality(
        SharedTermId a,
        SharedTermId b,
        SatLit reason);

    bool same(SharedTermId a, SharedTermId b) const;

    bool diseqKnown(SharedTermId a, SharedTermId b) const;

    std::optional<TheoryConflict> checkDisequalityConflict();

    std::vector<SatLit> explainEquality(
        SharedTermId a,
        SharedTermId b) const;

    std::vector<SatLit> explainDisequality(
        SharedTermId a,
        SharedTermId b) const;

    Snapshot snapshot() const;
    void rollback(Snapshot s);
};
```

内部用 rollback union-find + disequality set。

---

## 5.1 Shared equality source

Shared equality 可能来自：

```text
1. SAT 显式 assignment：
   Eq(x,y) = true

2. 子 theory propagation：
   LRA 推出 x=y

3. EUF 推出 shared terms 相等：
   a=b -> f(a)=f(b)

4. arrangement decision：
   SAT 选择 x=y 或 x≠y
```

每个 equality 都必须有 reason。

```cpp
struct SharedEqualityReason {
    enum Kind {
        SatLiteral,
        TheoryPropagation,
        ArrangementDecision
    };

    SatLit lit;
    TheoryId sourceTheory;
    std::vector<SatLit> premises;
};
```

---

## 5.2 Disequality conflict

如果：

```text
x = y
```

和：

```text
x ≠ y
```

同时存在，则产生 combination conflict：

```text
reasons(x=y) + reason(x≠y)
```

clause：

```text
¬r1 ∨ ¬r2 ∨ ... ∨ ¬d
```

其中 `d` 是 disequality literal，或者 `¬Eq(x,y)` 的 assignment reason。

# 6. 子 theory 的组合接口

每个 theory solver 需要暴露组合能力。

```cpp
class CombinationTheory {
public:
    virtual TheoryId id() const = 0;

    // 当前 theory 涉及哪些 shared terms
    virtual std::vector<SharedTermId> sharedTerms() const = 0;

    // 组合层告诉 theory：shared equality 已成立
    virtual TheoryCheckResult assertSharedEquality(
        SharedTermId a,
        SharedTermId b,
        SatLit reason) = 0;

    // 组合层告诉 theory：shared disequality 已成立
    virtual TheoryCheckResult assertSharedDisequality(
        SharedTermId a,
        SharedTermId b,
        SatLit reason) = 0;

    // theory 自己推出 shared equality
    virtual std::vector<SharedEqualityPropagation>
    getPropagatedSharedEqualities() = 0;

    // non-convex theory 请求 arrangement split
    virtual std::vector<ArrangementRequest>
    getArrangementRequests() = 0;
};
```

不过工程上不一定拆出新 base class，可以在现有 `TheorySolver` 加 optional hooks。

# 7. Convex theory 的组合方式

Convex theory 可以主动传播 implied equality。

例如 LRA：

```text
x <= y
y <= x
```

推出：

```text
x = y
```

返回：

```cpp
SharedEqualityPropagation {
    SharedTermId x;
    SharedTermId y;
    std::vector<SatLit> reasons;
    TheoryId source = LRA;
}
```

组合层生成 lemma：

```text
¬r1 ∨ ... ∨ Eq(x,y)
```

然后 SAT 层赋 `Eq(x,y)=true`，所有其他 theories 都会看到这个 shared equality。

---

## 7.1 EUF equality propagation

EUF 推出：

```text
a = b
```

如果 `a,b` 是 shared terms，则传播：

```text
Eq(a,b)
```

例如：

```text
x = y
```

EUF 中导致：

```text
f(x) = f(y)
```

如果 `f(x)` 和 `f(y)` 都被 arithmetic 使用并成为 shared terms，那么 EUF 可以向组合层传播它们相等。

# 8. Non-convex theory 的 arrangement split

对 non-convex theories，完整组合需要考虑 shared variables 的 arrangement。

对于 shared terms：

```text
s1, s2, ..., sn
```

理论组合需要知道每一对是否相等：

```text
si = sj
或
si ≠ sj
```

naive 做法是枚举所有 equivalence relations，指数级。实际实现要 lazy。

---

## 8.1 ArrangementManager

```cpp
class ArrangementManager {
public:
    std::optional<TheoryLemma> requestArrangement(
        SharedTermId a,
        SharedTermId b,
        ArrangementReason reason);

    bool isDecided(SharedTermId a, SharedTermId b) const;

    void noteDecision(SharedTermId a, SharedTermId b, bool equal);

private:
    std::unordered_set<Pair> requestedPairs_;
};
```

当某 theory 需要知道 `a` 和 `b` 的关系时，返回 split lemma：

```text
Eq(a,b) ∨ ¬Eq(a,b)
```

这是 tautological split，但让 SAT 决定 arrangement。

---

## 8.2 Guarded arrangement request

有时 split 只在某些 theory literals active 时才需要。可以加 guard：

```text
¬r1 ∨ ... ∨ Eq(a,b) ∨ ¬Eq(a,b)
```

但 `Eq∨¬Eq` 本身永真，guard 没意义。真正需要的是防止过早创建 split。工程上：

```text
只在当前 check 中 theory request 时创建 pair atom，
交给 SAT 后续分支。
```

---

## 8.3 Non-convex theory API

```cpp
struct ArrangementRequest {
    SharedTermId a;
    SharedTermId b;
    TheoryId sourceTheory;
    std::string debug;
};
```

例如 LIA solver 在某些情形下需要知道：

```text
x = y ?
```

则 request arrangement。

组合层创建 equality atom `Eq(x,y)`，如果未赋值，则返回 decision lemma / split request。

# 9. Nelson-Oppen 主循环

在 CDCL(T) 架构中，组合层作为一个 theory manager 子模块运行。

伪代码：

```cpp
TheoryCheckResult NelsonOppenManager::check() {
    bool changed = true;

    while (changed) {
        changed = false;

        // 1. 从 SharedEqualityManager 检查 equality/disequality 冲突
        if (auto c = sharedEq_.checkDisequalityConflict()) {
            return conflict(*c);
        }

        // 2. 把已知 shared equalities/disequalities 发送给各 theory
        for (auto& eq : newlyKnownSharedEqualities_) {
            for (auto* th : theoriesOwning(eq)) {
                auto r = th->assertSharedEquality(eq.a, eq.b, eq.reasonLit);
                if (r.isConflictOrLemma()) return r;
            }
        }

        for (auto& neq : newlyKnownSharedDisequalities_) {
            for (auto* th : theoriesOwning(neq)) {
                auto r = th->assertSharedDisequality(neq.a, neq.b, neq.reasonLit);
                if (r.isConflictOrLemma()) return r;
            }
        }

        // 3. 收集 theory 推出的 shared equalities
        for (auto* th : theories_) {
            for (auto& prop : th->getPropagatedSharedEqualities()) {
                SatLit eqLit = registry_.getOrCreateSharedEqualityAtom(prop.a, prop.b);

                TheoryLemma lemma;
                for (auto r : prop.reasons) {
                    lemma.lits.push_back(r.negated());
                }
                lemma.lits.push_back(eqLit);

                return lemma;
            }
        }

        // 4. 收集 non-convex arrangement requests
        for (auto* th : theories_) {
            for (auto& req : th->getArrangementRequests()) {
                auto eqLit = registry_.getOrCreateSharedEqualityAtom(req.a, req.b);

                if (!sat_.isAssigned(eqLit)) {
                    return arrangementSplit(eqLit);
                }
            }
        }
    }

    return NoChange;
}
```

# 10. Equality atom 的 route

Shared equality atom 必须能 route 到：

```text
1. SharedEqualityManager；
2. 所有 owning theories。
```

例如：

```text
Eq(x,y)
```

其中 `x,y` 属于 EUF 和 LIA，则 assignment true 时：

```text
SharedEqualityManager.assertEquality(x,y, EqLit)
EUF.assertSharedEquality(x,y, EqLit)
LIA.assertSharedEquality(x,y, EqLit)
```

assignment false 时：

```text
SharedEqualityManager.assertDisequality(x,y, ¬EqLit)
EUF.assertSharedDisequality(x,y, ¬EqLit)
LIA.assertSharedDisequality(x,y, ¬EqLit)
```

# 12. Model combination

If all pure theories are SAT under shared arrangement, build combined model.

## 12.1 Shared equality consistency

All theories must agree on shared equivalence classes.

```text
SharedEqualityManager gives final partition of shared terms.
```

Each theory model must respect:

```text
if sharedEq.same(a,b), then model_T(a)=model_T(b)
if diseqKnown(a,b), then model_T(a)≠model_T(b)
```

---

## 12.2 Model merge

```cpp
struct CombinedModel {
    std::vector<TheoryModel> theoryModels;
    SharedEqualityModel shared;
};
```

For purification variables:

```text
u = f(a)
```

combined model must satisfy:

```text
model(u) = model(f(a))
```

For public getModel, hide internal shared variables unless needed for getValue.

---

## 12.3 Final validation

Before returning SAT:

```text
FullModelValidator validates original unpurified formula.
```

This is mandatory because purification, relaxation, and combination errors are otherwise dangerous.

# 13. Explanation and lemmas

Combination lemmas must be SAT clauses.

## 13.1 Theory equality propagation

If theory T proves:

```text
x = y
```

from reasons `r1...rk`:

```text
¬r1 ∨ ... ∨ ¬rk ∨ Eq(x,y)
```

---

## 13.2 Combination disequality conflict

If:

```text
SharedEqualityManager has x=y from reasons R
and x≠y from reason d
```

conflict:

```text
∨_{r in R} ¬r  ∨  ¬d
```

---

## 13.3 Theory conflict after shared equality

If arithmetic becomes unsat after EUF propagated equality:

```text
EUF reasons -> Eq(x,y)
Arithmetic uses Eq(x,y) and other constraints -> conflict
```

CDCL will combine through learned clauses:

```text
EUF lemma:
  ¬r_euf ∨ Eq(x,y)

Arithmetic conflict:
  ¬Eq(x,y) ∨ ¬r_arith1 ∨ ...
```

Resolution yields global conflict.

This is exactly the right CDCL(T) style.

# 14. Efficient arrangement strategy

Naive arrangement over `n` shared variables requires deciding `O(n²)` equalities.

Efficient strategy:

```text
1. only create Eq(a,b) atom when needed;
2. only split pair if some theory requests it;
3. use equivalence class representatives to avoid redundant pairs;
4. prefer pairs that appear together in some theory constraint;
5. cache decided pairs.
```

---

## 14.1 Relevant pair generation

For each theory, compute relevant shared pairs:

```text
pairs that occur together in one pure atom
pairs in same UF congruence context
pairs in arithmetic equality/disequality constraints
```

Do not generate all pairs upfront.

---

## 14.2 Arrangement split priority

```text
1. pair requested by non-convex theory conflict candidate;
2. pair occurring in disequality;
3. pair with high activity in clauses;
4. pair connecting large shared classes.
```

# 15. Convex vs non-convex theory behavior

## 15.1 EUF

```text
Convex.
Can propagate equality.
Disequality conflict direct.
```

## 15.2 LRA / RDL

```text
Convex.
Can propagate equality if bounds imply x-y=0.
```

Example:

```text
x-y <= 0
y-x <= 0
=> x=y
```

## 15.3 LIA / IDL

```text
Non-convex.
Can still propagate some equalities, but not complete by propagation alone.
May request arrangement split.
```

## 15.4 NRA / NIA

```text
Non-convex.
Do not rely on equality propagation for completeness.
Use arrangement splitting lazily.
```

# 17. Backtracking

Combination layer must be rollbackable.

Snapshots:

```cpp
struct CombinationSnapshot {
    SharedEqualitySnapshot sharedEqSnap;
    ArrangementSnapshot arrangementSnap;
    size_t pendingEqSize;
    size_t pendingDiseqSize;
};
```

At decision level:

```text
snapshot all:
  SharedEqualityManager
  ArrangementManager
  theory solvers
```

Backtrack:

```text
rollback shared equalities/disequalities；
clear pending propagations above level；
rollback arrangement requests if needed；
theory solvers rollback their own state。
```

# 20. Efficiency design

## 20.1 Shared equality propagation cache

Avoid generating same lemma repeatedly:

```cpp
struct SharedEqPropKey {
    TheoryId source;
    SharedTermId a;
    SharedTermId b;
    std::vector<SatLit> reasons; // or reason hash
};
```

---

## 20.2 Arrangement request cache

```cpp
std::unordered_set<Pair<SharedTermId,SharedTermId>> requestedPairs_;
```

Do not repeatedly split same pair.

---

## 20.3 Shared pair indexing

```text
SharedTermId -> theories owning it
TheoryId -> shared terms
TheoryId -> relevant pairs
```

Avoid scanning all shared terms.

---

## 20.4 Incremental propagation

Only process newly added shared equalities/disequalities.

```cpp
std::deque<SharedEqualityEvent> pendingEqEvents_;
std::deque<SharedDisequalityEvent> pendingDiseqEvents_;
```

# 21. Common pitfalls

## 21.1 Forgetting purification

If EUF sees arithmetic term internally or arithmetic sees UF term internally, Nelson-Oppen assumptions break.

---

## 21.2 Sharing non-variable terms directly

Classical Nelson-Oppen shares variables, not arbitrary terms. Always purify alien terms into shared variables.

---

## 21.3 Over-eager arrangement

Generating all `O(n²)` equality atoms upfront will kill performance.

---

## 21.4 Treating LIA/NIA as convex

They are not convex. Equality propagation alone is incomplete.

---

## 21.5 Missing explanation

Every propagated equality must have a reason clause. Otherwise CDCL cannot learn correctly.

---

## 21.6 Model mismatch

Theories may produce individually valid models that disagree on shared variables. Combined model must respect SharedEqualityManager partition.

---

# 22. Red lines

```text
1. Purification mandatory for mixed terms.
2. Shared equality atoms must be canonical.
3. Every propagated shared equality needs a lemma.
4. Every combination conflict needs full reasons.
5. Non-convex theories require lazy arrangement splits for completeness.
6. Do not enumerate all arrangements upfront.
7. Final SAT must validate original unpurified formula.
8. Internal purification variables hidden from public model.
9. Backtracking must rollback shared equality state.
10. Original theory atoms stay with owner theory; shared equality atoms go to combination layer and relevant owners.
```

# 23. 最终推荐实现顺序

```text
1. Purifier
2. SharedTermRegistry
3. SharedEqualityManager
4. Shared equality atom registry
5. EUF + LRA convex combination
6. Explanation and model validation
7. Lazy arrangement manager
8. EUF + LIA / NIA / NRA non-convex combination
9. Efficiency optimization
10. Proof trace integration
```

# 总结

完整高效的 Nelson-Oppen 在你的 solver 里应该长这样：

```text
Purifier:
  mixed formula -> pure theory atoms + shared vars

SharedTermRegistry:
  records which terms are shared by which theories

SharedEqualityManager:
  maintains shared equality/disequality partition with reasons

NelsonOppenManager:
  propagates shared equalities between theories
  generates combination lemmas
  requests lazy arrangement splits for non-convex theories

Theory solvers:
  solve pure constraints
  accept shared equalities/disequalities
  propagate implied shared equalities
  request arrangement when needed

CDCL:
  searches Boolean structure + arrangement choices
```

最关键的设计判断是：

```text
Convex theories:
  equality propagation suffices.

Non-convex theories:
  need lazy arrangement splitting.

All theories:
  purification + shared equality explanation + final model validation are mandatory.
```

这样实现出来的组合层既符合 Nelson-Oppen 的理论边界，也能和你现在的 EUF、LIA/LRA、NIA/NRA、incremental linearizer、ICP 架构自然衔接。
