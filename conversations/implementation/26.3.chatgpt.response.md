审阅结论：**比上一版更完整，但仍然建议 Revise，不能直接执行。** 这版路线图清楚了，P0–P6 的工程拆分也合理；但还有几个关键 blocker，尤其是 **P1 单变量代数根支持、AlgebraBackend 接口、P2/P3 边界、Reason minimization soundness**。

## 总体判断

**接受整体架构，不接受当前执行细节。**

架构方向是对的：

```text
NraSolver
  -> CdcacSolver
    -> CdcacCore
      -> AlgebraBackend / ProjectionEngine / CoveringManager / ReasonManager
```

libpoly 只封装在 `LibpolyBackend`，这点也正确。Soundness 红线写得比较好，尤其是“后端失败返回 Unknown”“Sat 必须验证”“conflict 只能含 active literals”。

但这版仍然有几个硬问题。

---

## 1. P1 仍然自相矛盾：必须提前支持一元代数根

P1 写的是：

```text
每个 cell 取有理 sample
```

但 P1 测试集要求：

```text
x^2 - 2 = 0 -> sat
```

这个没有有理解。你如果只取 rational sample，就不可能返回 Sat model。

所以 P1 必须改成：

```text
P1 支持一元代数根 sample；
P3 再支持多变量代数 section。
```

也就是 P1 就要有最小版：

```cpp
struct AlgebraicRoot {
    PolyId definingPoly;
    int rootIndex;
    mpq_class lower;
    mpq_class upper;
};

struct RealAlg {
    enum class Kind { Rational, AlgebraicRoot };
    Kind kind;
    mpq_class rational;
    AlgebraicRoot root;
};
```

P1 不需要完整多变量代数点求值，但必须支持：

```text
univariate polynomial at univariate algebraic root
```

否则 `x^2 - 2 = 0` 这个验收条件不成立。

---

## 2. `AlgebraBackend` 接口还不合格

当前接口还是：

```cpp
virtual RootSet isolateRealRoots(const std::vector<mpz_class>& coeffs) = 0;
virtual Sign signAt(PolyId p, const std::unordered_map<std::string, mpq_class>& sample) = 0;
virtual std::vector<std::vector<mpz_class>> projectionPolys(...) = 0;
```

这个接口后面一定会推翻。问题是：

```text
1. 只支持整数系数，不支持有理系数；
2. projectionPolys 无法表达多变量多项式；
3. sample 用 string map，性能和一致性都差；
4. signAt 只支持 rational sample；
5. 表格里提到 specializeToUnivariate，但接口代码里没有；
6. RootSet / UniPolyId / PolyId 的边界不清楚。
```

建议直接改为：

```cpp
class AlgebraBackend {
public:
    virtual ~AlgebraBackend() = default;

    virtual RootSet isolateRealRoots(UniPolyId p) = 0;

    virtual Sign signAt(PolyId p, const SamplePoint& sample) = 0;

    virtual UniPolyId specializeToUnivariate(
        PolyId p,
        const SamplePoint& prefix,
        VarId mainVar
    ) = 0;

    virtual std::vector<PolyId> projectionPolys(
        const std::vector<PolyId>& polys,
        VarId eliminateVar,
        ProjectionMode mode
    ) = 0;

    virtual bool validateRootIsolation(UniPolyId p, const RootSet& roots) = 0;
};
```

内部统一用：

```text
PolyId / UniPolyId / VarId / SamplePoint
```

不要用 `std::string` 和裸 `vector<mpz_class>` 做主接口。

---

## 3. `CdcacCore` 不应依赖 `ActiveConstraint`

你在 `CdcacCore` skeleton 里写：

```cpp
Cell buildLeafConflictCell(const ActiveConstraint& c, ...);
```

但 `ActiveConstraint` 是 `CdcacSolver` facade 的内部 active/trail 类型。Core 不应该依赖 Solver 的 private 数据结构。

建议新增公共类型：

```cpp
struct CdcacConstraint {
    ConstraintId id;
    PolyId poly;
    Relation rel;
    SatLit reason;
    int level;
};

struct CdcacInput {
    std::vector<CdcacConstraint> constraints;
    std::vector<VarId> varOrder;
    std::optional<ModelSeed> seed;
};
```

`CdcacSolver` 负责从 `active_` 构造 `CdcacInput`，`CdcacCore` 只看 `CdcacConstraint`。

---

## 4. P2 的目标太大，建议拆开

P2 现在同时做：

```text
ProjectionEngine
recursive solveLevel
parent-cell generalization
multi-variable covering
rational-sector handling
```

这一步太重，容易一次性爆炸。

建议拆成：

```text
P2a：多变量递归框架 + rational sample + leaf conflict
P2b：ProjectionEngine + generalizeConflictCell
P2c：rational section 支持 + parent-cell validation
```

尤其是 P2 测试集里的：

```text
x^2 + y^2 <= 0 && x != 0 -> unsat
x = 0 && x*y = 0 -> sat
```

需要明确 **rational section** 在 P2 就支持。否则只做 sector 会处理不稳。

所以 P2 应明确：

```text
支持 rational sector；
支持 rational section，例如 x = 0；
不支持 non-rational algebraic section；
遇到 non-rational algebraic section 返回 Unknown。
```

---

## 5. P3 估时仍然偏乐观

你写 P3 代数数 section 支持 4–5 天，这还是偏乐观。

真正难的是：

```text
signAt(poly, SamplePoint containing algebraic values)
```

这不是简单 interval refinement 就一定稳。你至少要处理：

```text
多变量 specialization；
区间求值不确定；
重复 refinement；
零判断；
根重合；
代数数比较；
projection root 与 sample root 的一致性；
sign unknown 时 fallback。
```

建议拆成：

```text
P3a：一元代数根表示与比较；
P3b：一元代数根上的 signAt；
P3c：多变量代数 sample 的 specialization；
P3d：section lifting；
P3e：validator。
```

这样比“P3 4–5 天”更可信。

---

## 6. P4 不能只写 Lazard fallback，要先写 nullification contract

P4 写得太短。需要明确：

```text
什么叫 nullification；
在哪个阶段检测；
检测到后当前 partial cell 是否必须作废；
fallback 后哪些 projection guards 需要重建；
失败时是否保留已有 covering；
如何避免用 McCallum 的不安全 cell 产生 conflict。
```

建议加一个 contract：

```text
只要检测到 projection polynomial 在当前 prefix 上恒为 0，
且当前 projection mode 不能证明 delineability，
则当前 branch 不得返回 Unsat，只能：
1. 切换 Lazard；
2. 重建 guards；
3. 若仍失败，返回 Unknown。
```

这条很关键。

---

## 7. Reason minimization 仍有 soundness 风险

P5 写：

```text
Greedy set-cover
Deletion-based minimization
```

这个方向对，但不能只看 reason 集合覆盖 cell。

正确要求是：

```text
删除 reason 后，每个 cell 的 local refutation 仍然必须有效。
```

否则可能出现：

```text
covering 形式上仍覆盖全空间，
但某个 cell 的不可满足性已经没有原约束支撑。
```

建议 `Cell` 不要只有 flat reasons：

```cpp
struct Cell {
    ...
    std::vector<ReasonSet> alternativeReasonSets;
};
```

最小化时只能删除不会破坏所有 cell local proof 的 reason。

P5 第一版建议只做：

```text
cell merge
subsumed cell deletion
reason union 去重
```

先不要急着做 set-cover minimization。

---

## 8. P6 的集成顺序应该提前一点

现在 `NraLinearizationAdapter` 在 P0 里写“可选”，P6 才深度集成。这个会导致 CDCAC 单独跑，效率差，也不好调试。

建议从 P1 开始就设计接口，但不一定启用全部功能：

```text
P1：允许 ModelSeed 传入；
P2：chooseSampleOutside 优先使用 seed；
P6：再完整接入 LRA / linearizer / cuts。
```

这样 CDCAC 的 sample selection 一开始就不会写死。
