## libpoly 有没有 partial evaluation API？

有。libpoly 头文件里有：

```cpp
lp_upolynomial_t* lp_polynomial_to_univariate_m(
    const lp_polynomial_t* A,
    const lp_assignment_t* m
);
```

说明是：返回多项式 `A` 在 assignment `m` 下部分求值得到的一元多项式；`m` 必须给 `A` 中除 top variable 以外的所有变量赋整数值。libpoly 同一段接口还提供 `lp_polynomial_is_univariate_m`、`lp_polynomial_sgn`、`lp_polynomial_evaluate` 等函数。([GitHub][1])

所以你可以用它来处理：

```text
x*y - 1
prefix: x = 1
main variable: y
```

前提是：

```text
1. libpoly 的 variable order 里 y 是 top variable；
2. assignment m 里给 x 赋整数 1；
3. 当前 prefix 全是整数值。
```

注意限制：它要求 assignment 是 integer value。对你现在 P2a 的 rational sample，如果以后取到 `x = 1/2`，这个接口就不够用了。对于 P2a 当前测试 `x=1` 是可以用的；但长期仍然建议保留你自己的 `specializeToUnivariate`，或者实现 rational assignment 版本。

建议策略：

```text
P2a 当前：
  优先修自己的 terms-based specialize；
  同时用 lp_polynomial_to_univariate_m 作为 debug oracle，仅在 prefix 全整数时对比结果。

长期：
  自己的 PolynomialKernel specialization 作为主路径；
  libpoly partial evaluation 作为 cross-check / fallback。
```

## `terms()` 的实际返回值怎么确认？

我现在不能直接确认 `PolynomialKernel::terms()` 对 `mul(x,y)` 的返回结构，因为你没有贴 `PolynomialKernel::terms()` / `MonomialTerm` 的实现。你应该不要猜，直接加测试：

```cpp
TEST(PolynomialKernelTerms, ProductXY) {
    auto x = kernel.var("x");
    auto y = kernel.var("y");
    auto p = kernel.sub(kernel.mul(x, y), kernel.constant(1));

    auto terms = kernel.terms(p);
    ASSERT_TRUE(terms.has_value());

    for (auto& t : *terms) {
        dump(t);
    }

    // 期望：
    // coeff 1, powers[x]=1, powers[y]=1
    // coeff -1, powers empty
}
```

如果这个测试失败，说明问题在 `terms()` 或多项式 canonicalization。

如果这个测试通过，那问题在你的 `specializeToUnivariate`，常见错误有：

```text
1. prefix 的变量标识和 terms() 的变量标识不一致；
2. mainVar 判断错，把 y 当成“未赋值的其他变量”；
3. powers 遍历时没有处理 x^1*y^1 这种多变量 monomial；
4. 代入后 coefficient 没乘 prefix^power；
5. 清分母后系数顺序反了；
6. 构造 UniPolyId 时 coeffs 顺序和 isolateRealRoots 期望相反；
7. 零系数 / leading zero 没 trim，导致 degree 错。
```

尤其第 6 个很危险。你要统一：

```text
coeffs = high-degree -> low-degree
```

还是：

```text
coeffs = low-degree -> high-degree
```

不要一个地方 `[1, -1]` 表示 `y - 1`，另一个地方表示 `1 - y`。

## 阻塞 2：`x*y=0, x!=0, y!=0` 返回 Unknown

这里有两个问题。

### 2.1 `mergeRoots` 必须去重

必须去重。`-y` 和 `y` 的根都是 0。如果 merge 后有两个 root 0，你会生成类似：

```text
(-inf,0), [0], (0,0), [0], (0,+inf)
```

或者两个相邻 section `[0] [0]`。这会直接破坏：

```text
cell generation
section ordering
coversAllLine
reason collection
```

所以 root 去重应该在 `mergeRoots` 层面做，而不是让 `coversAllLine` 容忍重复根。

P1 的核心流程本来就是“merge all roots, cut real line into cells”；如果 merge 不是集合语义，后面的 cell decomposition 就不是合法分解。

你应该实现：

```cpp
std::vector<RealAlg> mergeRoots(std::vector<RootRef> roots) {
    sort by real value;
    unique by same real value;
}
```

P2a 初期可以支持这几类相等：

```text
rational 0 == algebraic root with lower=upper=0
same definingPoly + same rootIndex
same rational value
```

后面 P3 再做 general algebraic equality。

---

### 2.2 但这个例子不一定只靠去重就能 Unsat

`x*y=0, x!=0, y!=0` 的 unsat 有两种证明方式：

#### 方式 A：因式分解预处理

最简单、最稳：

```text
x*y = 0  =>  x = 0 OR y = 0
```

结合：

```text
x != 0
y != 0
```

直接冲突。

这更像 NRA 预处理/因式推理，不应该强行靠 CDCAC P2a 证明。

#### 方式 B：CDCAC covering

如果变量顺序是 `x -> y`：

```text
x = 0:
  x != 0 失败

x != 0:
  x*y = 0 推出 y = 0
  但 y != 0 失败
```

要把这个证明成整条 x 轴都不可行，父层需要两个 cell：

```text
section x=0
sector x<0 / x>0 或 x≠0 的区域
```

这其实已经需要某种 parent-cell generalization。没有 projection/generalization 的 P2a 很难完整证明所有 x 都不行。

你之前路线图里 P2a 写“`x*y=0, x!=0, y!=0 -> unsat` 是 leaf 冲突，无需 projection”，这个判断偏乐观。
除非你加 factor reasoning，否则它不应该作为 P2a 的必过测试。

我的建议：

```text
短期：
  1. mergeRoots 必须去重；
  2. 把该测试移到 P2b/P2c；
  3. 或者加一个 very cheap factor rule，专门处理 product = 0 与 factors != 0。

长期：
  factor reasoning 应该进 NRA 预处理；
  CDCAC 不要承担所有显然代数分解。
```

### 第一步：确认 `specializeToUnivariate` 输出

### 第二步：对照 libpoly oracle

```cpp
auto u1 = yourSpecialize(p, prefix, mainVar);
auto u2 = libpolyToUnivariateM(p, assignment);

```

libpoly 的 `lp_polynomial_to_univariate_m` 正好适合这个检查，但注意它要求除 top variable 外都被整数 assignment 赋值。([GitHub][1])

### 第三步：修 `mergeRoots`

加入测试：

```text
roots(y) = {0}

mergeRoots = {0}
```

再测：

```text
roots(y^2) = {0}
roots(y) = {0}
mergeRoots = {0}
```

### 第四步：调整 P2a 测试预期

如果没有 factor reasoning：

```text
x*y=0, x!=0, y!=0
```

P2a 返回 Unknown 是可以接受的。

如果你想 P2a 就过，那就加：

```text
product-zero conflict:
  p*q = 0
  p != 0
  q != 0
  => conflict
```

但这个要有 reason：

```text
not(p*q=0) OR not(p!=0) OR not(q!=0)
```
