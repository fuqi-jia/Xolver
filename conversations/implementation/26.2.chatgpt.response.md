审阅结论：**整体方向是对的，可以作为实现任务单，但现在还不能直接交给 coding agent 执行。主要问题不是架构，而是若干关键接口和阶段边界不够严谨，尤其是 P1/P3 的代数数支持划分有冲突。**

这份计划的优点很明显：

1. **架构分层正确**：`CdcacSolver / CdcacCore / AlgebraBackend / ProjectionEngine / CoveringManager / ReasonManager` 这个拆分是合理的。

3. **Soundness 红线写得好**：后端失败返回 Unknown、nullification 未处理返回 Unknown、Sat 必须验证、conflict 只能含 active literals，这些都是关键红线。

## 1. 最大问题：P1 和 P3 的代数数支持边界冲突

你在 P0 里写：

```cpp
struct RealAlg {
    bool isRational;
    mpq_class rational;
    // P0 只支持 rational； algebraic root 字段留占位
};
```

这没问题。

但 P1 测试集里有：

```text
x^2 - 2 = 0 → sat
```

这个例子没有有理数解，必须返回代数数 `±sqrt(2)`。如果 P1 仍然只支持 rational sample，就无法正确返回 Sat。

所以这里必须改：

### 方案 A：P1 就实现最小代数数

P1 不需要完整多变量 algebraic sign evaluator，但必须支持 **一元代数根 sample**：

```cpp
struct AlgebraicRoot {
    PolyId definingPoly;
    int rootIndex;
    mpq_class lower;
    mpq_class upper;
};

struct RealAlg {
    enum class Kind { Rational, AlgebraicRoot };
    Kind kind;
    mpq_class rational;
    AlgebraicRoot root;
};
```

P1 的 `signAt` 只需要支持：

```text
univariate polynomial at univariate algebraic root
```

这比 P3 的多变量代数点求值简单得多。

### 方案 B：P1 不测 `x^2 - 2 = 0`

但这样 P1 的价值大幅下降，不建议。

**建议采用方案 A：P1 就支持一元 algebraic root；P3 再扩展为多变量 algebraic section。**

## 2. `AlgebraBackend` 接口目前太粗，会卡住后续实现

现在接口是：

```cpp
virtual RootSet isolateRealRoots(const std::vector<mpz_class>& coeffs) = 0;
virtual Sign signAt(PolyId p, const std::unordered_map<std::string, mpq_class>& sample) = 0;
virtual std::vector<std::vector<mpz_class>> projectionPolys(...) = 0;
```

问题：

1. 只支持 `mpz_class` 系数，不支持有理系数。
2. `projectionPolys` 用 `vector<vector<mpz_class>>` 表示多项式，无法表达多变量多项式。
3. `signAt` 只支持 rational sample，P3 会推翻这个接口。
4. 用 `string` 表示变量，后续性能和 canonicalization 都会很差。
5. `specializeToUnivariate` 在接口说明里提到了，但代码接口里没有。

建议改成：

```cpp
class AlgebraBackend {
public:
    virtual ~AlgebraBackend() = default;

    virtual RootSet isolateRealRoots(UniPolyId p) = 0;

    virtual Sign signAt(PolyId p, const SamplePoint& sample) = 0;

    virtual UniPolyId specializeToUnivariate(
        PolyId p,
        const SamplePoint& prefix,
        VarId mainVar
    ) = 0;

    virtual std::vector<PolyId> projectionPolys(
        const std::vector<PolyId>& polys,
        VarId eliminateVar,
        ProjectionMode mode
    ) = 0;

    virtual bool validateRootIsolation(UniPolyId p, const RootSet& roots) = 0;
};
```

也就是说：**内部统一用 `PolyId / VarId / SamplePoint`，不要用 `string` 和裸 coefficient vector 作为主接口。**

## 3. `ActiveConstraint` 不应该只定义在 `CdcacSolver`

你在 P0 里说扩展 `ActiveConstraint`，但 `CdcacCore` 的 skeleton 又直接使用：

```cpp
Cell buildLeafConflictCell(const ActiveConstraint& c, ...);
```

如果 `ActiveConstraint` 是 `CdcacSolver` 的 private struct，`CdcacCore` 就不应该依赖它。

建议抽一个公共类型：

```cpp
struct CdcacConstraint {
    ConstraintId id;
    PolyId poly;
    Relation rel;
    SatLit reason;
    int level;
};
```

然后：

```cpp
struct CdcacInput {
    std::vector<CdcacConstraint> constraints;
    std::vector<VarId> varOrder;
    std::optional<ModelSeed> seed;
};
```

`CdcacSolver::ActiveConstraint` 可以直接删掉，或者只作为 facade 内部 trail item，但不要让 core 依赖 facade 的 private 类型。

## 4. P2 的 “rational-sector CDCAC” 表述需要更精确

P2 写“遇到需要代数 section 的情况返回 Unknown”，这个原则对 soundness 是安全的。但测试集中有些例子会逼近 section 或需要根边界判断：

```text
x^2 + y^2 <= 0 && x != 0 → unsat
x = 0 && x*y = 0         → sat
```

这些涉及 `x = 0` section。虽然 0 是 rational section，不是 algebraic section，但你的 `CellKind::Section` 必须在 P2 就支持 **rational root section**。

因此 P2 应该明确：

```text
P2 支持 rational section；
P2 不支持 non-rational algebraic section；
遇到 non-rational algebraic section 返回 Unknown。
```

否则实现者可能误以为 P2 只做 open sector，导致 `x = 0` 这类基础例子处理不稳。

```cpp
class RootTable {
```

P2 再把它扩展成：


这样 P1 和 P2 之间不会出现大规模重写。

## 6. P5 的 Reason Minimization 有一个 soundness 隐患

你写：

```text
Greedy set-cover：从 covering labels 中贪心选择最小 reason set
Deletion-based minimization：逐个尝试删除 reason，检查 covering 是否仍完整
```

这里要补一句：**不能只检查 reason 集合是否覆盖 cell labels，还要检查删除 reason 后对应 cell 的不可满足性证明是否仍然成立。**

否则可能出现：

```text
cell 仍在 covering 里，但支撑这个 cell 的原始约束 reason 被删掉了
```

这会生成过强 conflict clause。

建议把 P5 改成：

```text
Reason minimization 只能删除“不会破坏每个 cell 的 local refutation label”的 reason；
每次删除 reason 后，需要重新验证 covering 中所有 cell 仍有至少一个合法 local reason set。
```

更安全的数据结构是：

```cpp
struct Cell {
    Interval region;
    std::vector<ReasonSet> alternativeReasons;
};
```

而不是单个 flat `std::vector<SatLit> reasons`。

## 7. `pendingConflict_` 缓存需要记录 active epoch

你写了：

```text
回溯后 pendingConflict_ 若 reason 不再 active 则自动失效
```

这是对的，但还不够。因为 active 集合增加后，旧 conflict 仍然 sound；active 集合减少后，如果 reasons 都还在，也仍然 sound；真正要检查的是：

```text
pendingConflict 的所有 reason literals 是否仍然 assigned true。
```

建议结构改为：

```cpp
struct PendingConflict {
    std::vector<SatLit> reasons;
    uint64_t createdEpoch;
};
```

并实现：

```cpp
bool isStillActive(const std::vector<SatLit>& reasons) const {
    for (SatLit lit : reasons) {
        if (!activeReasonSet_.contains(lit)) return false;
    }
    return true;
}
```

不要只靠 level 判断。
