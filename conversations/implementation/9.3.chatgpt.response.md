建议 **[4] Revise**，还差几处会影响实现正确性的细节。整体方向已经对：NIA-Core 不再是小补丁，而是统一做 univariate root、DomainStore、bounded search、GCD/modular/factor、exact validation；并且 `StubPolyKernel::evalInteger` 主体已经改成 `nullopt`，这点修正是对的。

但我不建议现在 Approve，原因如下。

---

# 1. `PolynomialKernel.h` 文件计划漏了 `degree()`

前面 Univariate 部分要求：

```cpp
virtual std::optional<int> degree(PolyId a, std::string_view var) const = 0;
virtual std::optional<std::vector<mpz_class>>
getIntegerCoefficients(PolyId a, std::string_view var) const = 0;
```

但 File-by-File 里 Phase A 只写：

```text
PolynomialKernel.h Add evalInteger(), getIntegerCoefficients()
```

漏了 `degree()`。这会导致后续 `getIntegerCoefficients()` 插值方案没有可靠 degree，RRT 不能 sound 地运行。

必须改成：

```text
PolynomialKernel.h:
  add evalInteger()
  add degree()
  add getIntegerCoefficients()
```

---

# 2. `DomainStore` 的接口没有携带 reason

计划说 DomainStore 空域时要“from all reasons that contributed to the emptiness”，但接口还是：

```text
restrictToFiniteSet(var, values)
addLowerBound(var, lb)
addUpperBound(var, ub)
excludeValue(var, v)
```

没有 reason 参数。这样实现时无法知道哪个约束贡献了 domain restriction。

要改成：

```cpp
restrictToFiniteSet(var, values, SatLit reason);
addLowerBound(var, lb, SatLit reason);
addUpperBound(var, ub, SatLit reason);
excludeValue(var, v, SatLit reason);
```

如果暂时不做精确 reason，也要明确：

```text
Phase NIA-Core 可用 all active reasons 构造 empty-domain conflict，保证 sound。
但不要声称 DomainStore 能精确追踪 contributing reasons。
```

---

# 3. Reasoner 的 `Unknown` 不能被滥用为“不适用”

当前 pipeline 是：

```cpp
auto ur = univariate_.run(...);
if (ur.kind == Unknown) return unknown();
```

这意味着：如果 univariate coefficient extraction 不可用、RRT divisor cap 超限、某条约束不是一元，整个 NIA solver 直接 `Unknown`，不会继续 modular / bounded / local search。

这会破坏你的目标。比如 RRT cap 了，但 bounded domain 可能已经足够求解；或者 univariate 不适用，但 modular reasoning 可以证明 unsat。

必须明确：

```text
Unknown 只表示 fatal unsupported / unsafe state，必须终止。
“不适用”“信息不足”“RRT incomplete”“modular skipped”应该返回 NoChange，而不是 Unknown。
```

建议改名：

```cpp
enum class NiaReasoningKind {
    NoChange,
    DomainUpdated,
    Conflict,
    Lemma,
    FatalUnknown
};
```

或者保留 `Unknown`，但写清楚：

```text
RRT Incomplete -> NoChange
coefficients unavailable -> NoChange
modular skipped due too many vars -> NoChange
factor pattern not recognized -> NoChange
only unsafe unsupported backend -> Unknown
```

---

# 4. Risk 表还有一句容易误导

风险表最后写：

```text
evalInteger stub returns fake 0    Must return nullopt / Unknown
```

作为风险项可以，但最好改成：

```text
Accidental fake evalInteger implementation    Unit tests require StubPolyKernel returns nullopt
```

否则虽然主文已修正，执行者看到“evalInteger stub returns fake 0”仍可能困惑。

---

# 5. Bounded solver 的 interval B&B 必须有“完整搜索”状态

计划已经说预算耗尽返回 `Unknown`，这是对的。但 `bounded_.solve()` 的返回类型需要体现：

```text
Sat(model)
Unsat(exhaustive)
Unknown(budget exhausted / eval unsupported)
```

不要只返回 `TheoryCheckResult` 而不区分 “完整穷尽” 和 “预算中断”。

这点建议写进 `BoundedNiaSolver` 接口。

---
