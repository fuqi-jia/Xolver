LIA 的最小集要比 LRA 大很多。原因是：**LRA 主要是增量 Simplex 问题，LIA 是 LRA relaxation + branch-and-cut + integer propagation + SAT/CDCL 协同搜索**。Z3 官方也明确说，LIA 使用 integer programming 风格的 **cuts + branch-and-bound**；Z3 算术实现还包含 Gomory / Chvátal-Gomory / Hermite 相关 cuts。([GitHub微软][1])

当前 SMT-COMP 2025 QF_LIA Parallel Track 里 SMTS 是主要赢家，所以如果说“彻底超过 SOTA”，不能只做一个普通 Branch-and-bound，而要做一个**分层、组合、可回溯、可解释、可特化的 SMT-LIA solver**。([SMT-COMP][2])

一句话版本：

> **Incremental simplex relaxation + branch-and-cut + Gomory/MIR/CG cuts + integer bound propagation + congruence/modular reasoning + PB extraction + CDCL(T) explanation + backtrackable state + LRA-to-LIA repair + theory combination。**

## 按模块看

### 1. LRA relaxation 内核

必须有：

```text
12, 13, 14, 92
```

即：

```text
12. Simplex-based LIA
13. Dual Simplex
14. Incremental Simplex
92. LIA-LRA relaxation
```

LIA 的底层仍然强依赖 LRA relaxation。现代 SMT 算术求解器普遍围绕 SMT 场景下的 Simplex 变体构建，并使用精确数值表示保证 soundness。([CVC4][3])

---

### 2. Branch-and-cut 主框架

必须有：

```text
1, 2, 3, 4, 5, 6, 11, 15, 53, 58, 59
```

对应：

```text
1. Branch-and-bound
2. Branch-and-cut
3. Cutting planes

5. Mixed-integer rounding cuts
6. Chvátal–Gomory cuts

```

这里不要只做裸 Branch-and-bound。裸 B&B 很容易爆炸。要超过 SOTA，必须是 **cut first / branch later / explanation-aware** 的结构。

### 3. 整数传播与同余推理

必须有：


```text

27. Linear congruence solving
```

这部分是 LIA 和 LRA 的根本区别。很多 LIA 实例不需要深 branch，只要 GCD、divisibility、congruence、implied bound 做得好，就能在 SAT 层展开前直接杀掉。

对应：

LIA 很吃输入形态。很多 benchmark 的关键不是主算法，而是能不能识别出：

```text

pseudo-Boolean constraints
cardinality-like constraints
small-domain integer variables
```

如果不做这些，branch-and-cut 再强也会被简单特例拖死。

### 5. PB / finite-domain 特化

最小集里我保留：

```text
73, 74
```

对应：

```text

```

这两个很重要。很多 LIA benchmark 实际上带有 PB / scheduling / counting 结构。要进一步冲 SMT-COMP，最好再加：

```text
60, 61, 62, 63, 70, 71, 72, 75
```

也就是：

```text

61. Boolean variable detection
62. Finite-domain propagation
63. Order encoding
70. Knapsack constraint detection
```

但这些我没有放进“绝对最小集”，因为会显著扩大工程量。

---

### 6. 冲突解释与 lemma 生成

必须有：

```text
47, 48, 51, 52, 89
```

对应：

```text
47. Conflict analysis
48. Farkas explanation
51. Minimal conflict extraction
52. Theory lemma generation
89. Explanation cache
```

这是 SMT-LIA 能否超过普通 MILP/LIA solver 的关键。不是只要判定 UNSAT，而是要给 SAT core 返回短 lemma、可复用 conflict、可传播 bound。

Griggio 的 LA(Z) 经典工作也强调，实用 SMT(LIA) 不是靠单一完备技术，而是靠多层技术和启发式组合。([es-static.fbk.eu][4])

---

### 7. 增量、回溯、缓存

必须有：

```text
82, 83, 84, 85, 86, 88, 89
```

对应：

```text
82. Propagation queue
83. Watch-based bound propagation
84. Backtrackable state restoration
85. Trail-based bound updates
86. Basis restoration
88. Bound cache
89. Explanation cache
```

LIA 在 SMT 里不是一次性求解，而是反复被 SAT solver 调用。没有 trail-based backtracking 和 cache，就会输给成熟求解器的工程实现。

---

### 8. 模型构造与整数修复

必须有：

```text
45, 46, 93, 94
```

对应：

```text
45. Model validation
46. Integer model reconstruction
93. LRA-to-LIA integrality repair
94. SAT-guided integer search
```

SAT 实例上不能只靠 branch-and-cut。要有 repair：LRA model 出来后，优先尝试 rounding / integer reconstruction / SAT-guided search，而不是马上切分。

---

### 9. CDCL(T) 与组合理论接口

必须有：

```text
95, 96, 97
```

对应：

```text
95. CDCL(T) integration
96. Nelson-Oppen equality propagation
97. Theory combination interface
```

如果只做纯 QF_LIA benchmark，可以弱化 96/97；但如果目标是“SMT 求解器整体超过 SOTA”，这三个必须做。

## 最短结论

LIA 的最小超越路线不是：

> Branch-and-bound + Gomory cuts。

而是：

> **Incremental LRA relaxation + branch-and-cut + integer propagation + congruence/divisibility reasoning + PB extraction + explanation-aware CDCL(T) integration + backtrackable state + integer model repair。**

所以最终建议锁定这 58 项：
