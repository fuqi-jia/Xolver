下面按 SMT 求解器实现角度写。这里的 **general-simplex** 指 SMT(LRA/LIA) 中常用的 Dutertre–de Moura 风格 general simplex / dual simplex theory solver；**cube test** 指 LIA/Mixed LIRA 中的 **largest cube test / unit cube test**。cube test 是 **sound but incomplete** 的整数解快速发现算法，不是完整判定过程。general-simplex 是 LRA 的完整过程；对 LIA 需要 branch-and-bound / cuts / cube test 等补充。Dutertre–de Moura 的 simplex 设计目标就是适配 DPLL(T)：支持断言、检查、回溯和 theory propagation。

---

# 1. General-Simplex：SMT(LRA) 的核心算法

## 1.1 输入形式

原始 SMT arithmetic atoms：

```text
a1*x1 + ... + an*xn <= c
a1*x1 + ... + an*xn >= c
a1*x1 + ... + an*xn = c
a1*x1 + ... + an*xn < c
a1*x1 + ... + an*xn != c
```

先统一转成 **bounded variable + linear equations** 的形式：

```text
A x = 0
l_i <= x_i <= u_i
```

做法是给每个线性项引入一个变量。例如：

```text
2x + 3y <= 5
```

引入：

```text
s = 2x + 3y
s <= 5
```

标准化成：

```text
s - 2x - 3y = 0
-∞ <= s <= 5
```

所以 general-simplex 内部不直接维护所有不等式，而是维护：

```text
1. tableau: basic variables expressed by non-basic variables
2. lower/upper bounds: l_i, u_i
3. current assignment: β
```

---

## 1.2 核心状态

设变量分为：

```text
B: basic variables
N: non-basic variables
```

tableau 每一行形如：

```text
x_i = Σ_{x_j ∈ N} a_ij x_j      where x_i ∈ B
```

求解器维护两个不变量：

```text
Invariant 1:
β 总是满足 tableau 中的所有等式。

Invariant 2:
所有 non-basic variables 都满足自己的上下界：
l_j <= β(x_j) <= u_j, for x_j ∈ N.
```

basic variables 可以暂时违反上下界。general-simplex 的任务就是通过 pivot，把 basic variables 也修到满足 bounds。Dutertre–de Moura 论文明确把状态设计为 tableau、变量上下界和 assignment β，并维护 non-basic bounds 与 Ax=0 两个不变量。

---

## 1.3 两个基础操作

### 操作一：update

当 non-basic variable `x_j` 的值被改成 `v` 时，所有 basic variables 要同步更新，保证 tableau 等式仍成立。

若：

```text
x_i = Σ a_ij x_j
```

则：

```text
β(x_i) := β(x_i) + a_ij * (v - β(x_j))
β(x_j) := v
```

伪代码：

```cpp
void update(Var xj, Rational v) {
    Rational delta = v - beta[xj];

    for (Var xi : basicVars) {
        beta[xi] += tableau.coeff(xi, xj) * delta;
    }

    beta[xj] = v;
}
```

### 操作二：pivotAndUpdate

当 basic variable `x_i` 违反 bounds 时，找一个 non-basic variable `x_j` 进入基，使 `x_i` 离开基。

伪代码：

```cpp
void pivotAndUpdate(Var xi, Var xj, Rational target) {
    // 目标：把 xi 调整到 target，例如 l_i 或 u_i
    Rational aij = tableau.coeff(xi, xj);
    Rational theta = (target - beta[xi]) / aij;

    // x_j 的新值
    Rational new_xj = beta[xj] + theta;

    // 先更新 assignment
    beta[xi] = target;
    beta[xj] = new_xj;

    for (Var xk : basicVars) {
        if (xk == xi) continue;
        beta[xk] += tableau.coeff(xk, xj) * theta;
    }

    // 再做 tableau pivot：xi 出基，xj 入基
    tableau.pivot(xi, xj);
}
```

---

## 1.4 主算法 Check

核心逻辑：

1. 找到一个违反上下界的 basic variable `x_i`。
2. 如果 `β(x_i) < l_i`，说明 `x_i` 太小，需要增大。
3. 在 `x_i = Σ a_ij x_j` 这一行里找一个可以通过改变 `x_j` 来增大 `x_i` 的 non-basic variable。
4. 如果找不到，说明 `x_i` 已经无法再增大，冲突。
5. 如果 `β(x_i) > u_i`，对称处理，需要减小 `x_i`。
6. 如果所有 basic variables 都满足 bounds，则 SAT。

伪代码如下：

```cpp
SimplexResult check() {
    while (true) {
        Var xi = findSmallestViolatedBasicVar();

        if (xi == NONE) {
            return SAT;
        }

        if (beta[xi] < lower[xi]) {
            Var xj = NONE;

            for (Var x : nonBasicVarsInRow(xi)) {
                Rational a = tableau.coeff(xi, x);

                bool canIncreaseXi =
                    (a > 0 && beta[x] < upper[x]) ||
                    (a < 0 && beta[x] > lower[x]);

                if (canIncreaseXi) {
                    xj = x;
                    break; // Bland rule: choose smallest eligible variable
                }
            }

            if (xj == NONE) {
                return UNSAT_WITH_EXPLANATION(explainLowerConflict(xi));
            }

            pivotAndUpdate(xi, xj, lower[xi]);
        }

        else if (beta[xi] > upper[xi]) {
            Var xj = NONE;

            for (Var x : nonBasicVarsInRow(xi)) {
                Rational a = tableau.coeff(xi, x);

                bool canDecreaseXi =
                    (a < 0 && beta[x] < upper[x]) ||
                    (a > 0 && beta[x] > lower[x]);

                if (canDecreaseXi) {
                    xj = x;
                    break;
                }
            }

            if (xj == NONE) {
                return UNSAT_WITH_EXPLANATION(explainUpperConflict(xi));
            }

            pivotAndUpdate(xi, xj, upper[xi]);
        }
    }
}
```

这正是经典 SMT simplex 的核心结构：选择一个 violated basic variable；若它低于下界，则寻找一个能补偿 gap 的 non-basic variable；若找不到，则 UNSAT；高于上界时对称处理。论文中也指出使用 Bland’s rule 保证终止。

# 2. General-Simplex 的冲突解释

SMT theory solver 不能只说 UNSAT，还要返回 conflict clause 给 SAT solver。

假设 basic variable `x_i` 违反下界：

```text
β(x_i) < l_i
```

tableau 行为：

```text
x_i = Σ a_ij x_j
```

为了增大 `x_i`：

```text
a_ij > 0 时，需要增大 x_j
a_ij < 0 时，需要减小 x_j
```

如果找不到可 pivot 的 `x_j`，说明：

```text
a_ij > 0 的变量都已经在上界
a_ij < 0 的变量都已经在下界
```

也就是说 `x_i` 已经达到最大可能值，但仍然小于 `l_i`，因此冲突。

定义：

```text
N+ = { x_j ∈ N | a_ij > 0 }
N- = { x_j ∈ N | a_ij < 0 }
```

冲突解释为：

```text
Γ =
  { x_j <= u_j | x_j ∈ N+ }
∪ { x_j >= l_j | x_j ∈ N- }
∪ { x_i >= l_i }
```

对应 conflict clause：

```text
¬(x_i >= l_i) ∨
∨_{x_j ∈ N+} ¬(x_j <= u_j) ∨
∨_{x_j ∈ N-} ¬(x_j >= l_j)
```

对于 `β(x_i) > u_i` 的上界冲突，对称处理。

论文中给出的解释机制正是从 violated row 中抽取达到边界的 non-basic bounds，加上 violated basic bound，形成 theory conflict。

# 3. SMT 中如何 assert / backtrack

## 3.1 assert upper bound

断言：

```text
x_i <= c
```

伪代码：

```cpp
AssertResult assertUpper(Var xi, Rational c) {
    if (c >= upper[xi]) {
        return OK; // bound 没有变强
    }

    if (c < lower[xi]) {
        return CONFLICT({xi <= c, xi >= lower[xi]});
    }

    saveOldUpperOnTrail(xi, upper[xi]);
    upper[xi] = c;

    if (isNonBasic(xi) && beta[xi] > c) {
        update(xi, c);
    }

    return OK;
}
```

## 3.2 assert lower bound

```cpp
AssertResult assertLower(Var xi, Rational c) {
    if (c <= lower[xi]) {
        return OK;
    }

    if (c > upper[xi]) {
        return CONFLICT({xi >= c, xi <= upper[xi]});
    }

    saveOldLowerOnTrail(xi, lower[xi]);
    lower[xi] = c;

    if (isNonBasic(xi) && beta[xi] < c) {
        update(xi, c);
    }

    return OK;
}
```

Dutertre–de Moura 的设计里，assert bound 只需要更新上下界；如果被断言的是 non-basic variable，则通过 update 恢复 non-basic bound invariant。回溯时主要恢复 bound 栈，不需要保存每一次 β 和 tableau，因此适合 DPLL(T) 高频 backtracking。

# 4. 严格不等式怎么处理

严格不等式：

```text
x > c
x < c
```

可以用符号无穷小 `δ > 0` 转成非严格不等式：

```text
x > c   ⇒   x >= c + δ
x < c   ⇒   x <= c - δ
```

内部数值从 `Q` 扩展为：

```text
Q_δ = { a + bδ | a,b ∈ Q }
```

比较规则：

```text
a + bδ < c + dδ
iff
a < c
or
a = c and b < d
```

所以：

```cpp
struct DeltaRational {
    Rational real;   // a
    Rational eps;    // b

    int compare(const DeltaRational& rhs) const {
        if (real < rhs.real) return -1;
        if (real > rhs.real) return 1;
        return compare(eps, rhs.eps);
    }
};
```

例如：

```text
x > 0    ⇒    x >= 0 + 1δ
x < 3    ⇒    x <= 3 - 1δ
```

论文中的 strict inequality 处理也是这一思路：用符号 infinitesimal δ，把严格不等式转成非严格 bounds，并在 `Q_δ` 上运行同一个 simplex。

# 5. LIA：general-simplex 为什么不够

对 LRA：

```text
linear constraints over rationals/reals
```

general-simplex 是完整的。

但对 LIA：

```text
linear constraints over integers
```

simplex 只能先解 relaxation：

```text
integer variables temporarily treated as rationals
```

如果得到：

```text
x = 3/2
```

那么 LRA sat，但 LIA 不一定 sat。

所以 LIA 需要：

```text
general-simplex
  + branch-and-bound
  + cutting planes
  + cube test
  + integer model validation
```

Dutertre–de Moura 也明确说明，若部分或全部变量要求为整数，simplex 本身不完整，需要 branch-and-cut，包括 branch-and-bound 和 mixed integer Gomory cuts。

# 6. Cube Test：LIA 的快速整数解发现算法

cube test 的目标不是证明 UNSAT，而是快速证明：

```text
这个 rational polyhedron 中一定存在 integer point。
```

也就是说：

```text
如果 cube test 成功：LIA SAT
如果 cube test 失败：UNKNOWN，继续 branch-and-bound / cuts
```

Bromberger 等提出的 cube tests 面向 SMT solver，核心包括 largest cube test 和 unit cube test；它们寻找 polyhedron 内部的 axis-parallel hypercube，而传统 branch-and-bound/simplex 更偏向在 polyhedron surface 上找点。([Springer Link][1])

---

# 7. Cube Test 的数学基础

给定 LIA 约束：

```text
A x <= b
x ∈ Z^n
```

对应 rational polyhedron：

```text
P = { x ∈ Q^n | A x <= b }
```

定义中心为 `z`、边长为 `e` 的 axis-parallel cube：

```text
C_e(z) = { x ∈ Q^n | |x_j - z_j| <= e/2, for all j }
```

如果：

```text
C_1(z) ⊆ P
```

那么 `P` 中一定有整数点。

为什么？因为对任意中心 `z`，四舍五入到最近整数：

```text
round(z)
```

满足：

```text
|round(z_j) - z_j| <= 1/2
```

所以：

```text
round(z) ∈ C_1(z)
```

而如果：

```text
C_1(z) ⊆ P
```

则：

```text
round(z) ∈ P
```

于是找到 integer solution。

unit cube test 就是利用这个性质。文章也指出，边长 1 是保证任意中心 cube 内含整数点的最小边长。([Springer Link][1])

---

# 8. Linear Cube Transformation

关键问题：

```text
如何判断 C_e(z) ⊆ { x | A x <= b }？
```

考虑第 `i` 个不等式：

```text
a_i · x <= b_i
```

在 cube `C_e(z)` 内，`a_i · x` 的最大值为：

```text
a_i · z + (e/2) * ||a_i||_1
```

因此：

```text
C_e(z) ⊆ P
```

等价于对所有 row `i`：

```text
a_i · z + (e/2) * ||a_i||_1 <= b_i
```

这就是 linear cube transformation。它把“cube containment”转成一个线性约束系统。

# 9. Largest Cube Test

## 9.1 目标

找一个包含在 polyhedron 中的最大 axis-parallel cube：

```text
maximize e
subject to:
A z + (e/2) * norm1(A_row) <= b
e >= 0
```

具体形式：

```text
maximize e

for each row i:
a_i · z + (e/2) ||a_i||_1 <= b_i

e >= 0
```

如果最大 `e` 很大，说明 polyhedron 内部空间大，有很大概率能 round 出整数解。

## 9.2 算法

```cpp
CubeResult largestCubeTest(Matrix A, Vector b) {
    // Build LP:
    // variables: z_1,...,z_n,e
    // constraints: a_i*z + (e/2)*||a_i||_1 <= b_i
    // objective: maximize e

    LpResult r = simplexMaximizeEdgeLength(A, b);

    if (r.UNSAT) {
        return NO_RATIONAL_POINT; // 原 polyhedron 也为空
    }

    RationalVector z = r.center;
    Rational e = r.edgeLength;

    IntegerVector candidate = roundNearest(z);

    if (satisfies(A, candidate, b)) {
        return INTEGER_SAT(candidate);
    }

    return UNKNOWN;
}
```

## 9.3 结果语义

```text
LP infeasible:
  原 rational polyhedron 为空，因此 LIA 也 UNSAT。

LP feasible + rounded center satisfies:
  找到 integer model，返回 SAT。

LP feasible + rounded center 不满足:
  UNKNOWN。
```

注意：largest cube test 失败不能说明无整数解。论文中也给了例子：最大 cube 的 center rounding 可能错过唯一整数解。([Springer Link][1])

# 10. Unit Cube Test

largest cube test 需要 optimization，不一定适合 SMT solver 的 incremental simplex。因此 unit cube test 固定：

```text
e = 1
```

于是约束变成：

```text
a_i · z + (1/2) ||a_i||_1 <= b_i
```

等价于：

```text
a_i · z <= b_i - (1/2)||a_i||_1
```

也就是只需要把每个 row bound 从：

```text
b_i
```

改成：

```text
b'_i = b_i - (1/2)||a_i||_1
```

然后跑一次普通 rational simplex。

Bromberger 等特别强调，unit cube test 不需要优化，只需要改变 bounds，因此很适合集成到 SMT solver 的 simplex framework 中；测试结束后恢复原 bounds 即可，仍能保持 incremental 结构。([Springer Link][1])

## 10.1 算法

```cpp
CubeResult unitCubeTest(Matrix A, Vector b) {
    Vector bPrime;

    for (int i = 0; i < A.rows(); ++i) {
        Rational rowNorm = l1Norm(A.row(i));
        bPrime[i] = b[i] - rowNorm / 2;
    }

    // Solve rational system:
    // A z <= b'
    SimplexResult r = rationalSimplex(A, bPrime);

    if (r.UNSAT) {
        return UNKNOWN; 
        // 注意：不是原问题 UNSAT，只是不存在 unit cube
    }

    RationalVector z = r.model;
    IntegerVector candidate = roundNearest(z);

    // 理论上一定满足；工程上仍建议 validate
    if (satisfies(A, candidate, b)) {
        return INTEGER_SAT(candidate);
    }

    return BUG_OR_NUMERICAL_ERROR;
}
```

## 10.2 为什么一定 sound

若 simplex 找到：

```text
A z <= b'
```

也就是：

```text
a_i · z <= b_i - 1/2 ||a_i||_1
```

则：

```text
a_i · z + 1/2 ||a_i||_1 <= b_i
```

所以：

```text
C_1(z) ⊆ P
```

而：

```text
round(z) ∈ C_1(z)
```

因此：

```text
round(z) ∈ P
```

所以返回的整数点是 sound 的。

# 11. Mixed Integer / Rational 情况

如果变量分成：

```text
x^Z: integer variables
x^Q: rational variables
```

约束：

```text
S x^Z + R x^Q <= b
```

cube 只需要在 integer variables 方向有边长，在 rational variables 方向可以是 flat 的。

largest cube 形式：

```text
S z^Z + R z^Q + (e/2)||s_i||_1 <= b_i
```

unit cube 形式：

```text
S z^Z + R z^Q <= b_i - (1/2)||s_i||_1
```

其中 `s_i` 是第 `i` 行中 integer variables 的系数。

算法：

```cpp
MixedCubeResult mixedUnitCubeTest(Matrix S, Matrix R, Vector b) {
    for each row i:
        bPrime[i] = b[i] - l1Norm(S.row(i)) / 2;

    // solve over rationals:
    // S zZ + R zQ <= bPrime
    SimplexResult r = rationalSimplex([S R], bPrime);

    if (r.UNSAT) {
        return UNKNOWN;
    }

    IntegerVector intCandidate = roundNearest(r.zZ);
    RationalVector ratCandidate = r.zQ;

    if (satisfies(S, R, intCandidate, ratCandidate, b)) {
        return SAT(intCandidate, ratCandidate);
    }

    return BUG_OR_NUMERICAL_ERROR;
}
```

# 12. Cube Test 在 SMT solver 中怎么放

建议流程：

```text
LIA theory solver check:

1. Run rational simplex.
   - 若 rational UNSAT：返回 theory conflict，LIA UNSAT。
   - 若 rational SAT：得到 rational model β。

2. 检查 integer variables 是否已经都是整数。
   - 若是：返回 SAT model。
   - 若不是：继续。

3. 判断是否适合 cube test。
   - 若存在大量 Boolean 0/1 变量，通常跳过。
   - 若有很多 implied equalities，先做 equality elimination。
   - 若 bounds 很紧，cube test 可能收益低。

4. Run unit cube test。
   - 若成功：返回 integer model。
   - 若失败：UNKNOWN。

5. Run branch-and-bound / Gomory cuts / CutSat-style procedure。
```

Bromberger 等也指出，对很多有限 lattice width 的 SMT-LIB 问题，普通 rational simplex 找到的 rational solution 本身就可能已经是 integer solution，因此最好先跑 rational simplex，再决定是否做 unit cube test；有 Boolean variables 时 unit cube test 往往容易失败，建议跳过。([Springer Link][1])

# 13. Cube Test 的局限

## 13.1 不完整

失败只表示：

```text
没有找到符合条件的 cube
```

不表示：

```text
没有整数解
```

例如 polyhedron 很薄，但刚好穿过一个 lattice point，这时 unit cube 不存在，但整数解存在。

## 13.2 implied equalities 会破坏 interior

如果约束隐含一个 equality，比如：

```text
x + y = 3
```

那么 feasible region 只有低维表面，没有 full-dimensional interior，自然放不下 unit cube。

Bromberger 等指出，若 polyhedron implies equality，就没有 interior，cube test 会失败；他们还进一步用 strict-system unsat 来检测 implied equalities。([Springer Link][1])

## 13.3 对 Boolean/integer bounded variables 不友好

如果变量是：

```text
x ∈ {0,1}
```

那么可行区域本来就很窄，不太可能容纳 unit cube。此时 branch-and-bound 或 bit-blasting 更合适。

# 14. 两个算法的关系

| 项目     | General-Simplex                            | Cube Test                      |
| ------ | ------------------------------------------ | ------------------------------ |
| 主要理论   | LRA，也作为 LIA relaxation                     | LIA / mixed LIRA               |
| 完整性    | 对 LRA 完整                                   | 不完整                            |
| 成功时结论  | SAT 或 UNSAT                                | SAT，或 rational UNSAT 时可得 UNSAT |
| 失败时结论  | 若 LRA check 失败可给 UNSAT                     | UNKNOWN                        |
| 适合位置   | theory solver 主干                           | LIA 快速 sat test                |
| 主要搜索位置 | polyhedron surface / bounds                | polyhedron interior            |
| SMT 集成 | bound assertion、pivot、conflict explanation | 临时替换 bounds，跑 simplex          |
| 回溯支持   | 很强                                         | 很强，尤其 unit cube test 只改 bounds |

```cpp
TheoryResult LiaSolver::check() {
    auto r = simplex.check();

    if (r.unsat()) {
        return conflict(simplex.explainConflict());
    }

    Model beta = simplex.model();

    if (integerVarsAllIntegral(beta)) {
        return sat(beta);
    }

    if (shouldRunCubeTest()) {
        auto cr = cube.unitCubeTest(currentLinearSystem());

        if (cr.integerSat()) {
            return sat(cr.model());
        }
    }

    return branchAndBoundOrCuts(beta);
}
```

# 16. 一个小例子

约束：

```text
x + y <= 10
-x <= -2      即 x >= 2
-y <= -3      即 y >= 3
x, y ∈ Z
```

矩阵：

```text
[ 1  1 ] [x] <= 10
[-1  0 ] [y] <= -2
[ 0 -1 ]     <= -3
```

unit cube test 修改 bounds：

第一行：

```text
||[1,1]||_1 = 2
b'_1 = 10 - 1 = 9
```

第二行：

```text
||[-1,0]||_1 = 1
b'_2 = -2 - 1/2 = -2.5
```

第三行：

```text
b'_3 = -3.5
```

求：

```text
x + y <= 9
-x <= -2.5     即 x >= 2.5
-y <= -3.5     即 y >= 3.5
```

一个 rational center：

```text
z = (3,4)
```

round：

```text
round(z) = (3,4)
```

原约束验证：

```text
3 + 4 <= 10
3 >= 2
4 >= 3
```

所以 LIA SAT。

# 17. 对你的 SMT 求解器的建议

如果你要实现：

```text
general-simplex + cube test
```

我建议顺序是：

```text
1. 实现 exact rational GeneralSimplex。
2. 支持 Qδ，处理 strict inequalities。
3. 支持 assertLower/assertUpper/push/pop/conflict explanation。
4. 用 GeneralSimplex 做 LRA theory solver。
5. LIA 先做 relaxation。
6. relaxation model 若 integral，直接 SAT。
7. 否则先跑 unit cube test。
8. cube test 失败，再进入 branch-and-bound / Gomory cuts。
9. 所有 SAT model 必须 exact validate。
10. 所有 UNSAT 必须能导出 bounds explanation。
```

关键点一句话概括：

```text
General-simplex 是 LRA 的主判定器；
cube test 是 LIA 中快速找整数点的 interior test；
二者不是替代关系，而是 cube test 调用 simplex 来做一个临时变换后的 rational feasibility check。
```

[1]: https://link.springer.com/article/10.1007/s10703-017-0278-7 "New techniques for linear arithmetic: cubes and equalities | Formal Methods in System Design | Springer Nature Link"


可以。对 **LIA** 来说，`general-simplex` 只是求解 **有理松弛问题**：

```text
A x <= b,  x ∈ Q^n
```

但 LIA 真正要求：

```text
A x <= b,  x ∈ Z^n
```

所以 SMT(LIA) theory solver 通常是：

```text
LRA simplex
  ↓
integer model check
  ↓
cube test / rounding / divisibility propagation
  ↓
cuts
  ↓
branch-and-bound
```

其中，simplex 是主引擎；cube test 是快速找整数解；cuts 是切掉当前 fractional rational model；branch-and-bound 是最后的搜索兜底。Dutertre–de Moura 的 simplex 设计本来就是为了和 DPLL(T) 集成，支持快速回溯、理论传播和严格不等式处理；但对整数变量，论文也明确需要 branch-and-cut 类补充。([ResearchGate][1])

# 1. SMT(LIA) 的基本流程

当前 SAT solver 给出一个 Boolean assignment，例如：

```text
b1 = true  →  x + 2y <= 7
b2 = true  →  3x - y >= 1
b3 = true  →  x <= 10
```

LIA theory solver 收到的是一组当前为真的线性整数约束：

```text
Γ = {
  x + 2y <= 7,
  3x - y >= 1,
  x <= 10,
  x, y ∈ Z
}
```

内部先把整数条件忘掉，求有理松弛：

```text
Γ_Q = {
  x + 2y <= 7,
  3x - y >= 1,
  x <= 10,
  x, y ∈ Q
}
```

然后分四种情况：

```text
1. Γ_Q unsat
   ⇒ Γ_Z 一定 unsat
   ⇒ 返回 theory conflict。

2. Γ_Q sat，simplex model 中所有 integer vars 都是整数
   ⇒ Γ_Z sat
   ⇒ 返回 model。

3. Γ_Q sat，但有 fractional integer vars
   ⇒ 尝试 cube test / rounding / divisibility / cuts。

4. 还解决不了
   ⇒ branch-and-bound / branch-and-cut。
```

总伪代码：

```cpp
LiaResult LiaSolver::check() {
    auto lra = simplex.check();

    if (lra.unsat()) {
        return conflict(simplex.explainConflict());
    }

    Model beta = simplex.model();

    if (allIntegerVarsIntegral(beta)) {
        return sat(beta);
    }

    if (tryCheapIntegerPropagation()) {
        return check(); // bounds/cuts changed
    }

    auto cube = unitCubeTest(currentLinearSystem());
    if (cube.sat()) {
        return sat(cube.integerModel());
    }

    auto cut = generateCut(beta);
    if (cut.generated()) {
        addTheoryLemma(cut.lemma());
        return lemma(cut.lemma());
    }

    auto br = makeBranch(beta);
    addTheorySplit(br.leftOrRightLemma());
    return branch(br);
}
```

# 2. Branch-and-Bound：分裂整数变量的取值域

## 2.1 核心思想

如果 LRA simplex 给出：

```text
x = 7/3
```

但 `x` 是整数变量，那么当前 rational model 不能作为 LIA model。

因为整数 `x` 不可能满足：

```text
2 < x < 3
```

所以可以加入一个有效的整数分裂：

```text
x <= floor(7/3)  OR  x >= ceil(7/3)
```

也就是：

```text
x <= 2  OR  x >= 3
```

这个 split 对所有整数 `x` 都恒真：

```text
∀x ∈ Z.  x <= 2 ∨ x >= 3
```

但它会排除当前 fractional model `x = 7/3`。

在 SMT 中，这个 split 通常作为 theory lemma 返回给 SAT solver：

```text
(x <= 2) ∨ (x >= 3)
```

SAT solver 之后会选择其中一个分支。

---

## 2.2 Branch-and-bound 的 SMT 版本

传统 MIP 里 branch-and-bound 常有目标函数；SMT(LIA) 里通常是可满足性问题，没有 objective，所以更准确地说是 **integer domain splitting**。

算法：

```cpp
Branch makeBranch(Model beta) {
    Var x = chooseFractionalIntegerVar(beta);
    Rational v = beta[x];

    Integer lo = floor(v);
    Integer hi = ceil(v);

    Atom left  = mkLeq(x, lo); // x <= floor(v)
    Atom right = mkGeq(x, hi); // x >= ceil(v)

    return Branch(left, right);
}
```

完整流程：

```cpp
LiaResult branchAndBound(Model beta) {
    Var x = chooseFractionalIntegerVar(beta);
    Rational v = beta[x];

    Atom left  = (x <= floor(v));
    Atom right = (x >= ceil(v));

    // Valid for all integer x
    Clause split = { left, right };

    return theoryLemma(split);
}
```

SMT 层看到：

```text
theory lemma: (x <= 2) ∨ (x >= 3)
```

然后 CDCL 可能先走：

```text
x <= 2
```

如果冲突，再回溯走：

```text
x >= 3
```

---

## 2.3 例子

约束：

```text
2x <= 5
x >= 0
x ∈ Z
```

LRA relaxation：

```text
0 <= x <= 2.5
```

simplex 可能给：

```text
x = 2.5
```

整数变量 `x` fractional，于是 branch：

```text
x <= 2  OR  x >= 3
```

右分支：

```text
x >= 3
```

与：

```text
2x <= 5
```

冲突，因为：

```text
x >= 3 ⇒ 2x >= 6 > 5
```

左分支：

```text
x <= 2
```

可行，得到整数解：

```text
x = 2
```

---

## 2.4 Branch variable 选择策略

不要随便选 fractional variable。常见策略：

```text
1. 最大 fractional distance
   选离整数最远的变量，比如 3.5。

2. 最小 domain size
   选当前上下界最紧的变量。

3. 最大约束影响
   选出现在最多约束中的变量。

4. pseudo-cost
   记录过去分裂某变量后带来的冲突/传播收益。

5. activity-based
   类似 SAT VSIDS，冲突中频繁出现的整数变量优先。

6. learned advisor
   让 GNN/RL 给 branching priority，但只作为 heuristic。
```

实现接口：

```cpp
class BranchSelector {
public:
    Var choose(const Model& beta,
               const LinearSystem& sys,
               const BranchStats& stats);
};
```

## 2.5 Branching lemma 的 proof

branching lemma：

```text
x <= floor(v) ∨ x >= ceil(v)
```

不是由当前约束推出的，而是由 `x ∈ Z` 的语义推出的。

proof obligation：

```text
x integer
⇒ x <= k ∨ x >= k+1
```

所以它是一个 **theory-valid lemma**。

内部记录：

```cpp
struct IntegerSplitProof {
    Var x;
    Rational value;
    Integer floorValue;
    Integer ceilValue;
};
```

## 2.6 Branch-and-bound 的问题

纯 branch-and-bound 有三个问题：

### 问题一：可能分支爆炸

例如：

```text
0 <= x1,...,xn <= 1000
```

每个变量都 fractional，朴素分裂会形成巨大搜索树。

### 问题二：无界变量可能导致非终止

例如变量没有上下界：

```text
x ∈ Z
x >= 0
```

如果策略很差，可能不断往一个方向分。

### 问题三：很多 fractional model 可以被一个 cut 一次性排除

所以实际 SMT(LIA) 一般不会只用 branch-and-bound，而是结合 cuts。CUTSAT/CUTSAT++ 这类工作就是试图把 SAT 风格搜索、integer reasoning 和 quantifier-elimination 思想组合起来；Bromberger 等指出原始 CUTSAT sound 但一般不保证终止，而 CUTSAT++ 通过规则和策略改进保证终止。([arXiv][2])

---

# 3. Cuts：加入“所有整数解都满足，但当前 rational model 不满足”的约束

## 3.1 cuts 的本质

当前 LRA model 是：

```text
β
```

它满足所有线性约束，但某些 integer variable 是 fractional。

cut 的目标是生成一个新约束 `C`，满足：

```text
1. 所有整数解都满足 C。
2. 当前 rational model β 不满足 C。
```

于是加入 `C` 之后：

```text
Γ ∧ C
```

不会删掉任何真正的 LIA 解，但会排除当前 fractional solution。

在 SMT 里，cut 是 theory lemma。

---

# 4. 最简单的 cut：整数舍入 / bound tightening

## 4.1 单变量舍入

如果有：

```text
2x <= 11
x ∈ Z
```

那么：

```text
x <= 5.5
```

由于 `x` 是整数，所以加强为：

```text
x <= 5
```

也可以写成：

```text
2x <= 10
```

这就是最简单的 cutting plane。Gomory cuts 的教学材料也用类似例子说明：对整数变量，可以从 `2x ≤ 11` 推出更强的 `x ≤ 5`，而不是只做 branching。([RWTH Aachen University][3])

伪代码：

```cpp
optional<Cut> tightenSingleVarBound(Constraint c) {
    // a*x <= b, x integer
    if (!isSingleVar(c)) return none;

    Integer a = c.coeff(x);
    Rational b = c.rhs;

    if (a > 0) {
        // x <= floor(b/a)
        return mkLeq(x, floor(b / a));
    } else {
        // x >= ceil(b/a)
        return mkGeq(x, ceil(b / a));
    }
}
```

---

## 4.2 gcd tightening

如果：

```text
6x + 10y <= 23
x,y ∈ Z
```

左边一定是 `gcd(6,10)=2` 的倍数。

所以：

```text
6x + 10y <= 23
```

可以加强为：

```text
6x + 10y <= 22
```

一般形式：

```text
a1x1 + ... + anxn <= b
g = gcd(a1,...,an)
```

左边一定是 `g` 的倍数，因此：

```text
a1x1 + ... + anxn <= g * floor(b/g)
```

伪代码：

```cpp
Cut gcdTighten(LinearConstraint c) {
    Integer g = gcdAbs(c.coeffs);

    if (g == 0 || g == 1) return none;

    Rational b = c.rhs;
    Integer newRhs = g * floor(b / g);

    if (newRhs < b) {
        return LinearConstraint(c.coeffs, LEQ, newRhs);
    }

    return none;
}
```

对于下界：

```text
a·x >= b
```

加强为：

```text
a·x >= g * ceil(b/g)
```

# 5. Gomory Fractional Cut

## 5.1 从 tableau 行生成 cut

simplex 最终 tableau 里有一行：

```text
x_B = b + a1 y1 + ... + ak yk
```

假设：

```text
x_B, y1,...,yk 都是整数变量
y_j >= 0
b 不是整数
```

记：

```text
f0 = frac(b)
fj = frac(aj)
```

因为左边 `x_B` 必须是整数，所以右边也必须是整数。

当前 simplex model 通常令 non-basic variables `y_j = 0`，于是：

```text
x_B = b
```

如果 `b` fractional，当前 model 非整数。

Gomory cut 的典型形式是：

```text
Σ frac(aj) * yj >= frac(b)
```

它满足两个性质：

```text
1. 所有整数解都满足。
2. 当前 yj = 0 的 fractional simplex model 不满足。
```

所以可以切掉当前 fractional model。

教学材料中给了相同思路：从 final tableau 中 fractional 的 basic variable 出发，重写等式，让左边为 integer-valued，再推出一个当前解违反、但所有整数解满足的新约束。([RWTH Aachen University][3])

---

## 5.2 例子

假设 simplex tableau 有：

```text
x3 = 1.75 + 0.5 y1 + 2.5 y2
```

其中：

```text
x3, y1, y2 ∈ Z
y1, y2 >= 0
```

当前 model：

```text
y1 = 0
y2 = 0
x3 = 1.75
```

显然 `x3` fractional。

取小数部分：

```text
frac(1.75) = 0.75
frac(0.5) = 0.5
frac(2.5) = 0.5
```

得到 Gomory cut：

```text
0.5 y1 + 0.5 y2 >= 0.75
```

乘 4：

```text
2y1 + 2y2 >= 3
```

当前 model 下：

```text
0 >= 3
```

违反，所以当前 fractional model 被切掉。

但任何整数解都必须满足这个 cut。

---

## 5.3 实现伪代码

```cpp
optional<Cut> generateGomoryCut(const Tableau& tab,
                                const Model& beta) {
    for (Row row : tab.rows()) {
        Var xB = row.basicVar();

        if (!isIntegerVar(xB)) continue;
        if (isInteger(beta[xB])) continue;

        Rational b = row.constant();
        Rational f0 = frac(b);

        LinearExpr lhs;

        for (auto [y, a] : row.nonBasicTerms()) {
            if (!isIntegerVar(y)) return none;
            if (!hasLowerBoundZero(y)) return none;

            Rational fj = frac(a);
            if (fj != 0) {
                lhs += fj * y;
            }
        }

        // lhs >= f0
        return Cut(lhs, GEQ, f0);
    }

    return none;
}
```

实际系统里变量不一定都是 `>=0`，所以要先做变量平移：

```text
x = l + y,  y >= 0
```

或使用带上下界的 mixed-integer Gomory cut。

# 6. Mixed Integer Gomory Cut / Bound-aware Gomory Cut

SMT(LIA/LIRA) 中常见情况是：

```text
integer variables + rational variables
variables have lower/upper bounds
non-basic variables may be at lower bound or upper bound
```

所以要用更一般的 bound-aware cut。

从 tableau 行：

```text
x_i = Σ a_ij x_j
```

当前 assignment 为 `α`。

构造：

```text
T = x_i - Σ_{original integer nonbasic j} a_ij x_j
```

若 `T` 是 integer-valued，但当前：

```text
α(T) fractional
```

就能推出一个 cut。

Gomory cut 的一般条件包括：

```text
1. 至少一个 basic/original variable 的当前值是 fractional。
2. non-basic variables 要么是 additional variables，要么其系数满足整数性要求。
3. 当前 fractional part 0 < f < 1。
```

教学材料中也明确列出：从 general simplex 的 final tableau 中选择 fractional 行，要求相关 non-basic/original coefficients 满足整数性条件，然后根据 non-basic variables 当前在 lower/upper bound 的情况生成 cut。([RWTH Aachen University][3])

---

## 6.1 lower/upper bound 情况

把 non-basic additional variables 分成：

```text
L = 当前在 lower bound 的变量
U = 当前在 upper bound 的变量
```

对于：

```text
j ∈ L
```

非负量是：

```text
x_j - l_j >= 0
```

对于：

```text
j ∈ U
```

非负量是：

```text
u_j - x_j >= 0
```

Gomory/MIG cut 就是由这些非负量组合成一个当前 assignment 下左边为 0、但整数性要求它至少为 1 的约束。材料中也正是先把 non-basic variables 分成 lower-bound 和 upper-bound 两组，再按系数正负推导两个 case，最终得到排除当前 assignment 的 cut。([RWTH Aachen University][3])

---

## 6.2 工程简化版

如果你是第一版 solver，不建议一上来实现完整 MIG cut。建议先实现三层：

```text
第一层：gcd tightening / bound tightening
第二层：pure Gomory fractional cut，要求所有变量 integer，且已转成 y >= 0
第三层：bound-aware Gomory cut
第四层：mixed integer Gomory / MIR cut
```

伪代码：

```cpp
optional<Cut> CutGenerator::generate(const SimplexState& st) {
    if (auto c = generateBoundTightening(st)) return c;

    if (auto c = generateGcdTightening(st)) return c;

    if (auto c = generatePureGomory(st)) return c;

    if (auto c = generateBoundAwareGomory(st)) return c;

    if (auto c = generateMIR(st)) return c;

    return none;
}
```

# 7. MIR Cut：Mixed Integer Rounding

MIR cut 适合 mixed integer system。基本直觉是：

```text
integer 部分可以舍入；
real 部分不能随便舍入。
```

例如约束：

```text
x + y <= 3.5
x ∈ Z
y ∈ Q, y >= 0
```

不能简单变成：

```text
x + y <= 3
```

因为 `y` 是 rational。

但可以根据 real variable 的 bounds 和系数生成更安全的 mixed rounding cut。

第一版实现可以不做 MIR，先做：

```text
integer-only Gomory + gcd tightening + branch-and-bound
```

等基础稳定后再加 MIR。

---

# 8. Cube Test 在这个组合里的位置

cube test 和 cuts/branching 的目标相反：

```text
cuts/branching：排除当前 fractional model
cube test：尝试直接找到 integer model
```

cube test 的理论基础是：如果 rational polyhedron 里面包含一个边长为 1 的 axis-parallel hypercube，那么这个 hypercube 一定包含整数点。Bromberger–Weidenbach 的工作专门提出 largest cube test 和 unit cube test；largest cube test 找最大内接 hypercube，unit cube test 检查是否存在边长 1 的 hypercube，这个边长是保证存在整数点的最小边长。([Springer Link][4])

## 8.1 推荐顺序

```text
LRA relaxation sat
  ↓
如果 rational model 已经 integral：SAT
  ↓
尝试 unit cube test
  ↓
成功：SAT
失败：不要返回 UNSAT
  ↓
cuts
  ↓
branching
```

原因：

```text
cube test 成功很便宜，可以直接结束；
cube test 失败没有负面结论；
cuts/branching 会改变搜索状态，成本更高。
```

---

# 9. Cut 和 Branch 在 CDCL(T) 里的表达方式

## 9.1 cut 作为 theory lemma

如果生成 cut：

```text
2y1 + 2y2 >= 3
```

SMT solver 需要把它注册成一个新的 arithmetic atom：

```text
c_cut := (2y1 + 2y2 >= 3)
```

然后加 lemma：

```text
c_cut
```

也就是 unit clause。

因为 cut 是当前整数理论下的有效后果：

```text
Γ ⇒ c_cut
```

所以可以直接加入。

如果你做 proof-aware solver，应记录：

```cpp
struct CutProof {
    CutKind kind;       // GCD, Gomory, MIR
    RowId tableauRow;
    vector<BoundReason> bounds;
    vector<IntegerVar> integralityAssumptions;
};
```

---

## 9.2 branch 作为 split lemma

branch 不是当前约束推出的，而是整数理论永真：

```text
x <= k ∨ x >= k+1
```

所以加到 SAT solver 的是：

```text
(x <= k) ∨ (x >= k+1)
```

这个 lemma 没有依赖当前 Γ，只依赖 `x ∈ Z`。

---

## 9.3 cuts 和 branches 的区别

| 项目              | Cut                   | Branch   |
| --------------- | --------------------- | -------- |
| 形式              | 单个新约束                 | 析取 split |
| 作用              | 切掉当前 fractional model | 分裂搜索空间   |
| 是否增加 Boolean 分支 | 通常不增加                 | 会增加      |
| 是否保留所有整数解       | 是                     | 是        |
| 是否可能很多          | 是                     | 是        |
| 优先级             | 通常先于 branch           | 通常兜底     |

# 10. 完整 LIA Solver 的建议架构

```cpp

private:
    GeneralSimplex simplex;
    IntegerVarSet integerVars;

    BoundTightener boundTightener;
    DivisibilityReasoner divisibility;
    CubeTester cubeTester;
```

主 check：

```cpp
TheoryCheckResult LiaSolver::check(TheoryCheckLevel level) {
    auto r = simplex.check();

    if (r.unsat()) {
        return conflict(simplex.explainConflict());
    }

    Model beta = simplex.getModel();

    if (allIntegerVarsIntegral(beta)) {
        if (validator.validate(beta)) {
            return sat(beta);
        }
    }

    if (auto prop = boundTightener.tryTighten(simplex)) {
        return lemma(prop->asLemma());
    }

    if (auto div = divisibility.tryReason(simplex)) {
        return lemmaOrConflict(div);
    }

    if (level >= TheoryCheckLevel::FULL_EFFORT) {
        auto cube = cubeTester.unitCubeTest(simplex.currentSystem());
        if (cube.sat()) {
            return sat(cube.model());
        }
    }

    if (auto cut = cutGenerator.generate(simplex, beta)) {
        return lemma(cut->asLemma());
    }

    Branch br = branchSelector.makeBranch(beta);
    return split(br.asLemma());
}
```

# 11. Divisibility reasoning：LIA 里非常重要

除了 cuts 和 branching，还有一个很重要的整数专用推理：**整除性**。

## 11.1 等式 gcd 检查

如果有：

```text
6x + 10y = 7
x,y ∈ Z
```

左边一定是：

```text
gcd(6,10)=2
```

的倍数，但右边 `7` 不是 2 的倍数，所以 UNSAT。

一般形式：

```text
a1x1 + ... + anxn = c
```

若：

```text
g = gcd(a1,...,an)
g ∤ c
```

则无整数解。

伪代码：

```cpp
optional<Conflict> checkDiophantineEq(LinearEquality eq) {
    Integer g = gcdAbs(eq.coeffs);

    if (eq.rhs % g != 0) {
        return conflict(eq.reason);
    }

    return none;
}
```

## 11.2 等式归约

如果：

```text
6x + 10y = 8
```

可以除以 gcd 2：

```text
3x + 5y = 4
```

这能降低系数，改善 simplex 和 cuts。

## 11.3 与 branch 的关系

如果某个变量可以通过等式表达：

```text
3x + 5y = 4
```

就不要盲目对 `x` branch。可以先做：

```text
x = (4 - 5y)/3
```

然后要求：

```text
4 - 5y ≡ 0 mod 3
```

这能生成 modular constraint，减少分支。

# 12. 实际组合策略

我建议你的 LIA solver 用这个优先级：

```text
1. LRA simplex
   - 如果 LRA unsat，直接 conflict。

2. integer model check
   - 如果 rational model 已经 integral，直接 sat。

3. cheap tightening
   - 单变量舍入
   - gcd tightening
   - equality gcd conflict
   - equality normalization

4. cube test
   - 尤其在变量多、约束不太紧、没有大量 Boolean 0/1 变量时。

5. cuts
   - pure Gomory
   - bound-aware Gomory
   - MIR later

6. branch-and-bound
   - split fractional integer variable。

7. restart / activity update
   - 如果 cuts 太多或 branch 太深，切换策略。
```

伪代码：

```cpp
while (true) {
    LraResult r = simplex.check();

    if (r.unsat()) {
        return UNSAT(simplex.explain());
    }

    Model beta = simplex.model();

    if (isIntegral(beta)) {
        return SAT(beta);
    }

    if (auto c = cheapIntegerLemma(beta)) {
        addLemma(c);
        continue;
    }

    if (shouldTryCube(beta)) {
        auto cube = cubeTest();
        if (cube.sat()) return SAT(cube.model());
    }

    if (auto cut = cutGenerator(beta)) {
        addLemma(cut);
        continue;
    }

    auto split = branch(beta);
    addLemma(split);
    return NEED_SAT_BRANCHING;
}
```

# 13. 工程注意点

## 13.1 cuts 不能无限加

否则会出现：

```text
大量弱 cuts
simplex tableau 膨胀
性能下降
proof 变大
```

需要限制：

```text
max_cuts_per_check
max_cut_size
max_cut_coefficient
min_violation
cut_activity_decay
```

## 13.2 branch 也不能太早

如果过早 branch，会把 SAT solver 拖进巨大整数搜索树。

建议：

```text
先 cheap cuts，再 branch。
但如果 cut 连续无效，立即 branch。
```

## 13.3 每个 SAT model 必须重新验证

尤其 cuts/cube/local search 都混进来后，必须做：

```cpp
bool validateLiaModel(Model m) {
    for (constraint c : inputConstraints) {
        if (!evalExact(c, m)) return false;
    }
    for (Var x : integerVars) {
        if (!isInteger(m[x])) return false;
    }
    return true;
}
```

## 13.4 所有 lemma 要有 proof object

建议记录：

```text
LRA conflict:
  Farkas coefficients

GCD tightening:
  gcd certificate

Gomory cut:
  tableau row + fractional part + integrality assumptions

Branch split:
  integer trichotomy proof

Cube test:
  sat model only；失败不产生 conflict
```

# 14. 最终推荐实现版本

第一版不要做太复杂，按这个顺序：

```text
V1:
  general-simplex for LRA
  LIA relaxation
  integer model validation
  branch split

V2:
  single-variable rounding
  gcd tightening
  equality gcd conflict

V3:
  unit cube test

V4:
  pure Gomory fractional cuts

V5:
  bound-aware Gomory cuts

V6:
  MIR cuts / CutSat-style calculus

V7:
  proof-producing LIA
```

也就是说，最小可用 SMT(LIA) solver 是：

```text
simplex + integer check + branch split
```

但真正好用的 SMT(LIA) solver 应该是：

```text
simplex
+ cheap integer propagation
+ cube test
+ Gomory/MIR cuts
+ branch-and-bound
+ proof/model validation
```

一句话总结：

```text
simplex 负责证明有理可行性；
cube test 负责快速把有理可行性转成整数模型；
cuts 负责不分支地排除 fractional model；
branch-and-bound 负责在 cuts 不够强时做整数域搜索。
```
