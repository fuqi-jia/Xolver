# 1. `GomoryCut` 结果类型仍然不能表达“矛盾 cut”

Plan 里文字已经说：如果 cut 规范化为矛盾，比如 `0 >= positive`，不要注册 dynamic atom，而是返回 guarded conflict lemma；如果是 tautology 就跳过。这个思路是对的。

但接口还是：

```cpp
struct GomoryCut {
    LinearFormKey lhs;
    Relation rel;      // GEQ
    mpq_class rhs;
    std::vector<SatLit> premises;
    std::string debugInfo;
};
```

然后 `LiaSolver::check()` 仍然无条件执行：

```cpp
SatLit cutLit = registry_.getOrCreateLinearBoundAtom(
    cut->lhs, cut->rel, cut->rhs
);

TheoryLemma lemma =
    buildGuardedCutLemma(cut->premises, cutLit);
```

这和“矛盾 cut 不注册 dynamic atom”冲突。必须把 `GomoryCut` 改成带 kind 的结果类型，例如：

```cpp
enum class GomoryCutKind {
    LinearCut,          // lhs >= rhs
    GuardedConflict,    // ¬premise_1 ∨ ... ∨ ¬premise_k
    Skip
};

struct GomoryCut {
    GomoryCutKind kind;
    LinearFormKey lhs;
    Relation rel;
    mpq_class rhs;
    std::vector<SatLit> premises;
    std::string debugInfo;
};
```

然后 check 里必须分支：

```cpp
if (cut->kind == GomoryCutKind::GuardedConflict) {
    return TheoryCheckResult::mkLemma(
        buildGuardedConflictLemma(cut->premises)
    );
}

if (cut->kind == GomoryCutKind::LinearCut) {
    SatLit cutLit = registry_.getOrCreateLinearBoundAtom(
        cut->lhs, cut->rel, cut->rhs
    );
    return TheoryCheckResult::mkLemma(
        buildGuardedCutLemma(cut->premises, cutLit)
    );
}

if (cut->kind == GomoryCutKind::Skip) {
    // continue to branch split
}
```

否则实现者会继续按普通 `lhs >= rhs` 注册矛盾 cut，这正是你前面已经想避免的情况。

# 2. Unit Cube Test 仍缺少 active constraint 规范化

Phase 3 还是只写了：

```text
A x <= b
a_i · z <= b_i - 1/2 * ||a_i||_1
```

但没有规定 active atoms 怎么变成 `A x <= b`。这会直接影响正确性，尤其是 `>=`、`=`、strict inequality、false literal。

必须补：

```text
UnitCubeTester 必须只接受 normalized effective active constraints。

归一化规则：

1. 当前 SAT literal 为 false 时，先取反 relation。
   不要对原始 relation 直接建 cube system。

2. a·x <= c:
      保留。

3. a·x >= c:
      转成 -a·x <= -c。

4. a·x = c:
      加两条：
        a·x <= c
        -a·x <= -c

5. LIA strict:
      a·x < c   转成   a·x <= ceil(c) - 1
      a·x > c   转成   a·x >= floor(c) + 1
                再转成 -a·x <= -(floor(c)+1)

6. Disequality 不进入 cube system。
   disequality obligations 已经在 cube test 前处理。

7. 对 rational coefficients 先 clear denominators，再进入 integer-aware normalization。
```

没有这段，Cube Test 容易在 strict/equality/false literal 下构造错误系统。

---

# 3. GCD equality conflict 也要写 clear denominators

Plan 只在 GCD inequality tightening 里强调了 clear denominators。 但 equality conflict 同样需要。

比如：

```text
(1/2)x + (3/2)y = 5
```

必须先乘以 2：

```text
x + 3y = 10
```

再做 gcd。建议在 Phase 2 开头统一写：

```text
All IntegerReasoner rules operate on normalized effective active constraints:
  - false literal 已经取反；
  - strict LIA 已经转成非严格；
  - 所有 rational coefficients/rhs 已经 clear denominators；
  - gcd 在整数系数上计算。
```

这样 `gcd equality conflict`、`gcd inequality tightening`、`single-variable tightening` 都共用同一个规范化入口。

---

# 4. Cut validation 要明确验证 expanded cut，而不是 raw aux cut

Plan 已经要求 cut 注册前展开回原始变量空间，避免 dynamic atom 引用 solver-local aux，这点是对的。

但 validation 里还应该明确：

```text
当前 model 必须违反 cut，指的是 expanded cut under current original-variable model 被违反。
```

或者：

```text
validation evaluator 必须 definition-aware，保证 raw aux cut 和 expanded original cut 等价。
```

现在计划只写 “当前 model 必须违反 cut”，容易让实现者在 raw GS var cut 上验证，最后却注册 expanded atom。两者一旦 expansion 实现有 bug，validator 抓不到。

补一句即可：

```text
Cut validation must evaluate the exact expanded cut that will be registered in TheoryAtomRegistry.
Do not validate the raw aux-variable cut and then register a different expanded cut.
```

---

# 5. 4.5 里一处解释文字不严谨，但不算硬伤

Plan 说当前 `y_j=0` 违反 cut 的解释里写了：

```text
因为 f0 + 0 = f0 < 1，不满足 >= 1 - f0
```

严格来说，cut 的左边是 `Σ f_j y_j = 0`，右边是 `1-f0>0`，所以违反。`f0+0<1` 是推导整数性的解释，不是直接评价 cut 的方式。建议改成：

```text
当前 relaxation model 下 y_j=0，因此 cut 左边为 0；
由于 0 < f0 < 1，所以 1-f0 > 0；
因此 0 >= 1-f0 为 false，当前 model 被排除。
```

这不是 soundness bug，但改了更清楚。
