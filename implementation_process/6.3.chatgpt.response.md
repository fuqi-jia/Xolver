我建议选 **[4] Revise**，不要直接 Approve。

总体方向已经对了：统一 CDCL(T) 路径、删除 legacy scan path、所有 theory atom 进入 `TheoryAtomRegistry`、LRA/LIA/NRA 都走统一 `TheorySolver` 接口，这些都符合最终版求解器架构。当前计划也明确写了“不保留 legacy scan-based path”和“CaDiCaL ExternalPropagator 作为生产后端”。

但这版还有几个**必须修的正确性/API 细节**。

---

# 1. `connectPropagator` 和 `addObservedVar` 顺序反了

计划里 `Solver::checkSatInternal()` 的顺序是：

```cpp
for (const auto& rec : registry.records()) {
    sat->addObservedVar(rec.satVar);
}

CadicalTheoryPropagator propagator(...);
sat->connectPropagator(&propagator);
```

这个顺序不对。CaDiCaL 的 API 说明是：必须先连接 external propagator，然后相关变量才能被标记为 observed；`add_observed_var` 要求已经有 external propagator 连接，并且 observed variable 会被用于 notifications / propagation / clause addition。([GitHub][1])

应该改成：

```cpp
CadicalTheoryPropagator propagator(registry, theoryManager, lemmaDb);
sat->connectPropagator(&propagator);

Atomizer atomizer(*sat, registry);
registry.setContext(sat.get(), &atomizer);

for (ExprId a : ir->assertions()) {
    SatLit lit = atomizer.atomize(a, *ir);
    sat->addClause({lit});
}
```

并且：

```cpp
registry.registerParsedTheoryAtom(...)
```

内部或紧随其后必须调用：

```cpp
sat->addObservedVar(satVar);
```

动态 atom 也是一样：`getOrCreateLinearBoundAtom(...)` 创建 fresh SAT var 后，要立即 `addObservedVar(var)`。CaDiCaL 允许 observed variable 是 fresh variable，也允许在 solving 期间加入 observed variable。([GitHub][1])

---

# 2. `Atomizer::atomizeRec()` 只写了 linear path，缺 polynomial path

计划说 `TheoryAtomRecord` 支持：

```cpp
std::variant<LinearAtomPayload, PolynomialAtomPayload>
```

但 `Atomizer::atomizeRec()` 伪代码只有：

```cpp
if (extractLinearConstraint(...)) {
    registry.registerParsedTheoryAtom(... LinearAtomPayload ...);
}
```

这会导致 QF_NRA 的 nonlinear atom 没有进入 registry，`CdcacSolver` 永远收不到 active polynomial constraints。

必须补完整分支：

```cpp
if (extractLinearConstraint(eid, ir, lhs, rhs, rel)) {
    registry.registerParsedTheoryAtom(
        v, eid, detectLinearTheory(lhs), LinearAtomPayload{lhs, rel, rhs});
}
else if (extractPolynomialConstraint(eid, ir, poly, rhs, rel)) {
    registry.registerParsedTheoryAtom(
        v, eid, TheoryId::NRA, PolynomialAtomPayload{poly, rel, rhs});
}
else {
    // unsupported theory atom
    mark unsupported / return Unknown later
}
```

否则 “CdcacSolver implements new interface” 只是形式上完成，实际不会收到 atom。

---

# 3. Callback 名称必须以 vendored CaDiCaL 头文件为准

计划里写：

```cpp
cb_check_found_model
cb_has_external_clause
cb_add_external_clause_lit
notify_assignment(const std::vector<int>&)
```

但 CaDiCaL master 文档片段里出现的是 `connect_external_propagator`、`add_observed_var`、`force_backtrack`，并且文档注释提到 `cb_check_final_model`。([GitHub][1]) 你当前 vendored 版本可能确实有 `cb_check_found_model`，但计划必须加一条硬要求：

```text
Before implementation, inspect the vendored cadical.hpp and use the exact ExternalPropagator callback names and signatures from that file.
Do not invent callback signatures from memory.
```

否则实现时会在 Phase 1 就卡住。

建议改成：

```text
CadicalTheoryPropagator implements the exact API exposed by third_party/cadical/src/cadical.hpp.
The names below are conceptual; final code must match the vendored header.
```

# 4. `Unknown` 在 `cb_check_found_model` 里不能简单 `return false`

计划里：

```cpp
return false; // Unknown
```

这很危险。ExternalPropagator 里 `false` 的含义通常是：当前 SAT model 被外部约束拒绝，随后应通过 external clause 加入 lemma/conflict。IPASIR-UP 论文也说明，`cb_check_found_model` 返回 false 后，用户可以通过 external clause callbacks 添加新 clause，让 SAT solver 不重启地继续搜索。([repositum.tuwien.at][2])

如果 theory 返回 `Unknown`，但你 `return false` 又没有 pending clause，可能造成 CaDiCaL 继续询问、卡住，或者违反接口预期。

应该设计明确的 Unknown 退出通道：

```cpp
if (tr.kind == Unknown) {
    abortWithUnknown_ = true;
    requestTermination();  // through terminator / backend mechanism
    return true_or_false_according_to_api_contract;
}
```

更具体地说：

```text
Conflict/Lemma:
  return false and provide a pending external clause.

Consistent:
  return true.

Unknown:
  terminate solve and make CadicalBackend::solve() return Unknown.
  Do not reject the model without a clause.
```

这一点必须补，否则 NRA/CdcacSolver `check() returns Unknown` 时会不正确。

---

# 5. duplicate lemma 在 propagator 内也不能 `return false` 无 clause

计划里：

```cpp
if (tr.kind == Lemma) {
    if (lemmaDb_.insertIfNew(*tr.lemma)) {
        pendingClause_ = tr.lemma->lits;
        hasPendingClause_ = true;
    }
    return false;
}
```

如果 `insertIfNew` 返回 false，那么 `return false` 但没有 pending clause。这个和 Unknown 问题一样，会破坏 callback 语义。

虽然计划前面说 theory solver 应该先 `contains()` 避免返回 duplicate，但这里仍然必须防御：

```cpp
if (tr.kind == Lemma) {
    if (!lemmaDb_.insertIfNew(*tr.lemma)) {
        abortWithUnknown_ = true; // or internal error
        requestTermination();
        return true_or_false_according_to_api_contract;
    }

    pendingClause_ = tr.lemma->lits;
    hasPendingClause_ = true;
    return false;
}
```

不要出现 `return false` 但 `hasPendingClause_ == false` 的路径。

---

# 6. `cb_add_external_clause_lit()` 的 clause buffer 设计要更严谨

当前：

```cpp
if (!hasPendingClause_ || pendingClause_.empty()) {
    hasPendingClause_ = false;
    return 0;
}
SatLit lit = pendingClause_.back();
pendingClause_.pop_back();
return ...
```

可以工作，但建议改成 index，而不是 pop：

```cpp
std::vector<SatLit> pendingClause_;
size_t pendingClausePos_ = 0;
```

然后：

```cpp
if (!hasPendingClause_) return 0;

if (pendingClausePos_ == pendingClause_.size()) {
    hasPendingClause_ = false;
    pendingClause_.clear();
    pendingClausePos_ = 0;
    return 0;
}

return toCadicalLit(pendingClause_[pendingClausePos_++]);
```

这样更容易 debug，也不会反向输出 clause。虽然 clause 顺序逻辑上无所谓，但 reason/debug/proof trace 时保持原顺序更好。

---

# 7. observed vars 必须覆盖 dynamic atoms

计划目前只在 `checkSatInternal()` 里 atomize 后：

```cpp
for (const auto& rec : registry.records()) {
    sat->addObservedVar(rec.satVar);
}
```

这只能覆盖 parsed atoms。后续 branch split、GCD tightening、Gomory cut 生成 dynamic atom 时，必须也 observed。

因此 `TheoryAtomRegistry::getOrCreateLinearBoundAtom` 必须：

```cpp
SatLit TheoryAtomRegistry::getOrCreateLinearBoundAtom(...) {
    if (existing) return lit;

    SatVar v = sat_->newVar();
    records_.push_back(...);
    satVarToIdx_[v] = ...;

    sat_->addObservedVar(v);  // must happen here

    return SatLit::positive(v);
}
```

并且要保证 propagator 已经 connected，这和第 1 点的顺序绑定。

---

# 8. `CdcacSolver::assertLit` 需要处理 false literal

当前 NRA 伪代码：

```cpp
activeConstraints_.push_back({payload.poly, payload.rel, payload.rhs, value, reason});
```

这可以先存 `value`，但 `check()` 之前必须转成 effective relation：

```text
value = true:
  poly rel rhs

value = false:
  poly (negate rel) rhs
```

尤其：

```text
not(poly <= 0)  ==> poly > 0
not(poly = 0)   ==> poly != 0
not(poly != 0)  ==> poly = 0
```

如果 CDCAC 暂时不支持 disequality，需要明确返回 Unknown 或 split lemma。不能把 false literal 直接丢给 CDCAC 原始 relation。

---

# 9. `TheoryAtomRegistry` 仍然保留 synthetic ExprId，可以但不要依赖它路由

计划里：

```cpp
ExprId nextSyntheticExprId_ = 0x80000000;
```

可以保留用于 debug / trace，但最终路由必须只依赖：

```cpp
SatVar -> TheoryAtomRecord
```

dynamic atom 不应再通过 synthetic ExprId 查 payload。计划总体是这个方向，但最好补一句：

```text
Synthetic ExprId is diagnostic only. It is not used for theory routing or payload recovery.
```

# 10. `Phase 10` 中 “如果当前版本需要显式 while loop” 这段应该删除或改弱

计划后面仍写：

```cpp
while (true) {
    auto result = sat->solve();
    ...
    auto tr = theoryManager.check(lemmaDb);
    ...
}
```

这又回到了 lazy loop。你前面已经要求最终不保留这种路径。既然本版目标是 ExternalPropagator，就不要保留这个 fallback 伪代码。

可以改成：

```text
If the vendored CaDiCaL does not support the required ExternalPropagator API, fail this refactor early and upgrade CaDiCaL.
Do not reintroduce the old explicit SAT-theory loop as a fallback.
```

# 结论

这版**不需要推翻**，但还不能 Approve。
我的选择：**[4] Revise**。

它已经解决了之前最大的结构问题：统一路径、删除 legacy、NRA 也走新接口、CaDiCaL ExternalPropagator 作为生产后端。

但必须修掉这几个细节，尤其是：

```text

Unknown/duplicate lemma 不能 return false without clause；
NRA polynomial payload 注册；
dynamic atom 必须即时 observed；
```

这些补完后，我才建议 Approve。
