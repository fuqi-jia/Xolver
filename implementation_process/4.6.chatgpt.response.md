这版**还不能 Approve**，但已经只剩最后两处硬问题。常数项修正已经做对了：现在 `p=a·x+c` 的 split 正确写成 `a·x <= -g-c` 和 `a·x >= g-c`，这个之前的 soundness bug 已经修掉了。

但是还有两个地方必须改。

---

# 1. Step 7 仍然违反“只注册一个 arithmetic solver”

它前面明确写了：

```text
Exactly one arithmetic solver per logic.
```

但 Step 7 的 default 分支仍然是：

```cpp
else {
    // Default: QF_LRA
    theoryManager_.registerSolver(std::make_unique<LraSolver>());
    theoryManager_.registerSolver(std::make_unique<CdcacSolver>(createPolynomialKernel()));
}
```

这还是同时注册了 LRA 和 NRA solver。这个必须删掉。Phase 1 中，未知 logic 不要猜，更不要注册多个算术 solver。

应该改成：

```cpp
else {
    return Result::Unknown;  // unsupported / unspecified logic
}
```

或者：

```cpp
else {
    throw UnsupportedLogic(logic);
}
```

如果以后要自动 infer logic，另写 `inferLogicFromInput()`，但不要在 Phase 1 默认注册两个 solver。

---

# 2. disequality 检查必须先看当前 model 是否违反

现在 check flow 写的是：

```cpp
for (const auto& d : disequalities_) {
    auto dr = checkIntegerDisequality(d);
    ...
}
```

这会导致：只要 `p=0` 在整数中可能，它就返回 split lemma，即使当前 `GeneralSimplex` model 已经满足 `p != 0`。

这不是直接 unsound，因为 split lemma 本身有效；但它会造成严重问题：明明已有合法模型，也会不断返回不必要的 lemma，甚至可能重复添加同一条 split lemma，导致不能及时返回 SAT。

应该改成：

```cpp
for (const auto& d : disequalities_) {
    if (!gs_.value(d.auxVar).isZero()) {
        continue;  // current relaxation model already satisfies p != 0
    }

    auto dr = checkIntegerDisequality(d);
    ...
}
```

解释要写清楚：

```text
Only when the current relaxation model violates the disequality do we need a split.
For LIA, even if the relaxation value is zero, gcd/residue may prove p=0 impossible over integers; then the disequality is satisfied and no lemma is needed.
```

---

# 3. 风险表还有一句旧话要改

Risks 里仍写：

```text
Dynamic atom registration is complex ... all state is in LinearAtomManager
```

这和当前设计冲突。现在动态 atom 注册属于 `TheoryAtomRegistry`，`LinearAtomManager` 只负责 solver-local `form -> aux` 和 bound assertion。

改成：

```text
Dynamic atom registration is handled only by TheoryAtomRegistry.
LinearAtomManager remains solver-local and never creates SAT literals.
```
