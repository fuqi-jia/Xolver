我建议选 **[4] Revise**。整体路线是对的，尤其是 Phase 1.5 → Phase 2 → Phase 3 → Phase 4 的顺序合理，Plan 也明确把 Gomory 放在 validation、lemma checking、dynamic atom 测试之后，这是正确的。

但现在 **Phase 4 Gomory Cut 里有一个明确的 soundness bug**，所以不能 Approve。

# 1. 最大问题：Gomory cut 公式的 RHS 错了

Plan 里写：

```text
x_B = b' + Σ a'_j y_j

Gomory cut:
Σ frac(a'_j) * y_j >= frac(b')
```

这个公式在你当前 tableau 语义下是错的。你的 row 语义是：

```text
x_B = b' + Σ a'_j y_j
```

其中 `x_B` 和所有 `y_j` 都是整数变量，`y_j >= 0`。

设：

```text
b' = floor(b') + f0
a'_j = floor(a'_j) + f_j
```

则：

```text
x_B - floor(b') - Σ floor(a'_j) y_j
= f0 + Σ f_j y_j
```

左边是整数。由于 `0 < f0 < 1`，右边必须至少为 `1`，所以正确 cut 是：

```text
Σ frac(a'_j) * y_j >= 1 - frac(b')
```

不是：

```text
Σ frac(a'_j) * y_j >= frac(b')
```

反例：

```text
x_B = 0.8 + 0.1 y
```

令 `y = 2`，则：

```text
x_B = 1
```

这是合法整数解。

Plan 的 cut：

```text
0.1 y >= 0.8
```

会排除 `y=2`，因此 **unsound**。

正确 cut：

```text
0.1 y >= 0.2
```

保留 `y=2`。

所以 Phase 4.5、4.6、4.13 里的 Gomory 公式和测试都要改。Plan 当前写的是 `Σ frac(a'_j) y_j >= frac(b')`，这是不能接受的。

应该改成：

```text
f0 = frac(b')
if f0 == 0:
    skip row

Gomory cut:
    Σ frac(a'_j) * y_j >= 1 - f0
```

然后代回原变量：

```text
Σ_L f_j*(x_j - l_j) + Σ_U f_j*(u_j - x_j) >= 1 - f0
```

展开为：

```text
Σ_L f_j*x_j - Σ_U f_j*x_j
>=
(1 - f0) + Σ_L f_j*l_j - Σ_U f_j*u_j
```

# 2. Test 1 的 Gomory 例子也错了

Plan 里写：

```text
x_B = 5/2 - x

Gomory 应生成 cut:
x >= 1
```

这个测试不对。因为：

```text
x_B = 5/2 - x
```

如果 `x_B` 和 `x` 都是整数，那么这个 row 根本没有整数解。标准 Gomory cut 会生成一个矛盾 cut，而不是 `x >= 1`。

按正确公式：

```text
b' = 5/2
a' = -1
frac(-1) = 0
1 - frac(5/2) = 1/2
```

所以 cut 是：

```text
0 >= 1/2
```

这表示该 tableau row 在整数语义下不可满足。

建议把 Test 1 换成：

```text
x_B = 1/2 + 1/2 y
y >= 0
```

当前 `y=0` 时：

```text
x_B = 1/2
```

fractional。

正确 Gomory cut：

```text
1/2 y >= 1/2
```

即：

```text
y >= 1
```

这个测试才适合验证 cut 生成。

# 3. Candidate row 条件要改成“当前在 bound 上”

Plan 里有两种说法：

一处说：

```text
如果当前在 lower bound / upper bound
```

另一处说：

```text
row 中每个 non-basic var 必须在 finite integer lower/upper bound
```

后者不够严格。Phase 4 的 bound-aware Gomory 目前是按：

```text
if x_j at lower:
    x_j = l_j + y_j

if x_j at upper:
    x_j = u_j - y_j
```

来做的。为了保证当前 relaxation model 被 cut 排除，Phase 4 先要求：

```text
每个参与的 non-basic variable 当前必须正好 at finite lower 或 at finite upper。
```

也就是：

```text
atLower(x_j) && lower finite integer && lower reason exists
```

或者：

```text
atUpper(x_j) && upper finite integer && upper reason exists
```

不能只要求“有 finite bound”。

建议改成：

```text
Candidate row requirement:
  For every non-basic x_j in the row, choose a side only if:
    - beta[x_j] == lower[x_j] and lower is finite integer with reason; or
    - beta[x_j] == upper[x_j] and upper is finite integer with reason.
  Otherwise skip this row in Phase 4.
```

# 4. Gomory cut 里的变量可能是 internal aux，不能直接注册成外部 atom

这是另一个很重要的接口正确性问题。

Gomory row 来自 tableau：

```text
x_B = b + Σ a_j x_j
```

这里的 `x_j` 不一定是原始 SMT 变量，也可能是 GeneralSimplex 的 auxiliary variable，例如某个约束对应的 slack/linear-form aux。

但 `TheoryAtomRegistry::getOrCreateLinearBoundAtom` 目前的设计是基于 `LinearFormKey` 注册 dynamic atom。这个 `LinearFormKey` 必须有明确语义。你不能直接把 solver-local aux var id 放进去，除非整个 Atomizer/Registry 明确支持 solver-local internal vars。

更稳的做法是：**Gomory cut 在注册 dynamic atom 前，必须展开回原始线性变量空间。**

也就是说，`LinearAtomManager` 需要保存每个 GS var 的 original linear definition：

```text
original x:
    def(x) = x

aux s = lhs - rhs:
    def(s) = lhs - rhs
```

Gomory 生成 cut：

```text
Σ c_j * v_j >= C
```

其中 `v_j` 可能是 original var，也可能是 aux var。

注册前必须展开：

```text
Σ c_j * def(v_j) >= C
```

得到只含原始变量的 `LinearFormKey`，然后再调用：

```cpp
registry_.getOrCreateLinearBoundAtom(expandedLhs, GEQ, expandedRhs)
```

否则 dynamic atom 可能引用 `TheoryAtomRegistry` 和 `Atomizer` 无法稳定解释的 solver-local aux var。这会影响 soundness、model validation 和后续 proof。

建议加一节：

```text
4.x Cut expansion before atom registration

GomoryCutGenerator may produce a cut over GeneralSimplex variables.
Before registering cutLit, LiaSolver/LinearAtomManager must expand every GS variable to its original LinearFormKey.

Original Int variable:
    def(x) = x

Aux variable created by getOrCreateAuxVar(lhs,rhs):
    def(aux) = lhs - rhs

Then normalize:
    Σ c_j * def(v_j) >= C
into:
    expandedLhs >= expandedRhs

Only expanded cuts over original variables may be registered in TheoryAtomRegistry.
```

如果你想支持 internal-aux dynamic atoms，也可以，但那要明确设计 `TheoryAtomRegistry` 支持 solver-local variable namespace。当前 plan 没有这个设计，所以必须选择“展开回原始变量”。

# 5. Phase 2 的 gcd reasoning 需要说明清分母

Plan 里直接写：

```text
g = gcd(a_i)
```

这只有在所有系数和 rhs 都是整数时才成立。但你的系统内部用 `mpq_class`，所以 Phase 2 应该先规定：

```text
For LIA integer reasoning, first clear denominators.
```

例如：

```text
(1/2)x + (3/2)y <= 5
```

要先乘 `lcm(2,2)=2`：

```text
x + 3y <= 10
```

再做 gcd。

建议补：

```text
Before gcd equality conflict / gcd tightening / single-var tightening:
  clear denominators of all coefficients and rhs;
  normalize sign;
  then compute gcd over integer coefficients.
```

否则后面很容易对 `mpq_class` 直接 gcd，写出错误实现。

---

# 6. Unit cube test 要说明 active constraints 的规范化

Plan 里 unit cube test 只写了：

```text
A x <= b
a_i · z <= b_i - 1/2 ||a_i||_1
```

这方向对，但实现时必须说明：

```text
1. >= 要转成 <=；
2. = 要拆成 <= 和 >=；
3. strict integer inequality 要先转成非严格：
   a·x < b  =>  a·x <= b - 1
   a·x > b  =>  a·x >= b + 1
4. disequality 不进入 cube system，因为前面已处理 obligation；
5. 只使用当前 active literals。
```

这个不改不一定导致马上 unsound，但容易实现错。

---

# 结论

这版不能 Approve，建议选 **[4] Revise**。

必须修的 correctness 点是：

```text
1. Gomory fractional cut RHS 必须是 1 - frac(b')，不是 frac(b')。
2. Gomory Test 1 的 x_B = 5/2 - x 例子是错的，应换成 x_B = 1/2 + 1/2 y。
3. Candidate row 必须要求 non-basic vars 当前 at finite integer lower/upper bound，而不是仅有 finite bound。
4. Gomory cut 注册前必须展开 internal aux variables 到原始 linear form，或者明确支持 solver-local aux dynamic atom；当前 plan 没有后者，所以应采用展开方案。
5. GCD reasoning 要先 clear denominators。
```

可以直接给 Kimi 贴：

```text
Revise required before approve:

1. Fix Gomory cut formula for row
       x_B = b' + Σ a'_j y_j
   with y_j >= 0.
   Correct cut is:
       Σ frac(a'_j) y_j >= 1 - frac(b')
   not >= frac(b').

2. Update substitution:
       Σ_L f_j*x_j - Σ_U f_j*x_j
       >= (1 - f0) + Σ_L f_j*l_j - Σ_U f_j*u_j

3. Replace the test row x_B = 5/2 - x.
   It has no integer solution and should produce contradiction, not x >= 1.
   Use:
       x_B = 1/2 + 1/2 y
   expected cut:
       y >= 1.

4. Candidate row must require each non-basic variable to be currently at a finite integer lower or upper bound with a reason literal.
   Having a finite bound is not enough.

5. Before registering a Gomory cut atom, expand all GeneralSimplex variables in the cut back to original LinearFormKey definitions.
   Do not register dynamic atoms over solver-local aux variables unless TheoryAtomRegistry explicitly supports solver-local namespaces.

6. For gcd reasoning, clear denominators before computing gcd.

7. For unit cube test, normalize all active constraints to <= form; handle >=, =, and strict integer inequalities explicitly.
```

改完这些之后，这个计划就可以 Approve。
