这版**总体可以作为下一阶段主路线**，但我建议选 **[4] Revise**，因为还有几处阶段边界和正确性细节要收紧。不是大方向问题，主要是防止 Kimi 执行时把 Phase 1 写得过宽，或者写出会反复加同一 branch lemma 的实现。

这版的主线是对的：当前状态已经明确为 NIA-Core MVP 管道跑通、88 个单元测试和 10 个 CLI 回归通过；下一步目标聚焦“可验证、可回归、稳定解决有界整数非线性问题”；Phase 1 先做 DomainStore、SquareBoundReasoner、BoundedNiaSolver 和接入管道，Phase 2 再做最小 branch lemma，Phase 3/4 才做 ICP 与 Interval B&B。

# 主要需要 Revise 的点

## 1. Phase 1 的端到端测试里 `x²+y²<=5` 不应该放在 SquareBoundReasoner 验收里

Phase 1.2 的 SquareBoundReasoner 明确只处理：

```text
单变量、degree=2、首项系数为 1
```

也就是 `x² <= c`、`x² = c`、`x² != c`。计划后面却把：

```text
x²+y²<=5 → sat
```

放进 Phase 1.5 端到端验证。

这个例子本身 SAT，`x=0,y=0` 就满足，但它不能验证 SquareBoundReasoner 的能力；如果靠 local search 找到 0，也会通过，掩盖“sum-of-squares 传播还没实现”的事实。

建议改成：

```text
Phase 1.5 只验单变量平方：
  x² <= 4
  x² <= -1
  x² = 49
  x² = 50
  0<=x<=10 ∧ x²=49

x²+y²<=5 移到 Phase 3 sum-of-squares ICP 验收。
```

---

## 2. SquareBoundReasoner 的约束形式要写清楚：必须处理 normalized `p rel 0`

现在表格写的是：

```text
x² <= c
x² = c
x² != c
```

但 NiaSolver 管道里前面已经做了 normalize，内部通常是：

```text
p rel 0
```

所以实现时要识别的是：

```text
x² - c <= 0
x² - c = 0
x² - c != 0
```

建议明确：

```text
SquareBoundReasoner 输入是 NormalizedNiaConstraint。
它应识别形如:
  x² - c <= 0
  x² - c = 0
  x² - c != 0
其中 c 是整数常数。
不要假设输入直接保存成 “x² rel c”。
```

否则很容易写成只看 rhs 的接口，但当前 NIA normalizer 的内部表达不是那样。

---

## 3. `x² != c` 对 `c=0` 要去重

计划写：

```text
x² != c 若 c = r² → x ≠ r, x ≠ -r
```

当 `c=0` 时，`r=0`，不要插入两次 `x≠0`，也不要生成重复 reasons。

建议补：

```text
If c=0, add only one exclusion x != 0.
```

这是小问题，但容易影响 DomainStore reason 测试。

---

## 4. Phase 2 branch lemma 必须加 duplicate suppression

Phase 2 里：

```text
x <= k ∨ x >= k+1
```

是 sound 的。但如果每轮 check 都重复生成同一个 split，会导致 SAT 里重复加 clause，甚至卡住。

建议加入：

```cpp
std::unordered_set<BranchSplitKey> generatedSplits_;
```

其中：

```text
BranchSplitKey = (var, k)
```

如果已生成过：

```text
不要重复返回同一条 lemma；
换变量 / 换 k / 返回 Unknown。
```

尤其无界变量默认 `k=0` 时非常需要这个保护。

---

## 5. Branch lemma 的 dynamic atom 仍要补一句 rhs 归一化

计划已经写了必须通过 `TheoryAtomRegistry::getOrCreatePolynomialAtom` 注册动态 atom，但还要再强调一遍：

```text
当 dynamic atom (poly rel rhs) 回到 NiaSolver::assertLit 时，
必须转换成 (poly - rhs) rel 0，再进入 NiaNormalizer。
```

否则 `x <= k` 会被错误当成 `x <= 0`。

---

## 6. Phase 3 的 product pruning 暂时建议删掉或标成“禁止实现”

计划里写：

```text
x * y = c → 若已知 x ∈ [a,b]，可推 y ∈ [ceil(c/b), floor(c/a)]（需处理零和符号）
```

这个规则非常容易写错，特别是区间跨 0、负数、乘法不等式方向和整数除法边界。虽然它标了“可选”，但建议 Phase 3 不要做它。

改成：

```text
Product pruning 暂不实现。
Phase 3 只做 over-approx interval evaluation 和 pruning，不做反向乘法传播。
```

反向传播等以后有统一 ICP 框架再做。

---

## 7. Interval B&B 的 conflict 建议先统一 coarse active reasons

Phase 4 里 `any violated -> UnsatComplete(boxConflict)`，这里的 `boxConflict` 不好构造，因为 box 的边界可能来自内部 split，不一定有 SAT-level reason。为了第一版 sound，建议统一：

```text
如果整个 Interval B&B 完整关闭所有 boxes，返回 all active normalized reasons 的 coarse conflict。
子 box 内部不要尝试生成精细 boxConflict。
```

也就是：

```text
solveBox 内部只返回 Closed / Sat / Unknown；
顶层若所有 boxes Closed，才返回 UnsatComplete(allActiveReasonsConflict)。
```

这样实现简单且 sound。
