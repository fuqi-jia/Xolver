## P1: API 层补全

**目标**: Solver 从"只支持 parseFile + checkSat"升级为支持程序化 term 构造。

### 当前状态

```cpp
Sort Solver::boolSort() { return Sort{}; /* TODO */ }
Term Solver::mkInt(int64_t) { return Term{}; /* TODO */ }
void Solver::assertFormula(Term) { /* TODO */ }
```

所有 sort/term 构造都是空实现。

### 设计决策

**核心问题**: `Term` 当前是 `uint32_t` 包裹的 `ExprId`。但 `ExprId` 属于内部 `CoreIr`，不是 SOMTParser 的 DAG。

两种方案：

- **方案 A**: `Term` 直接封装 `ExprId`（内部 IR ID）。`mkInt` 调用 `CoreIr::mkIntConst()` 返回 `ExprId`。简单，但与 SOMTParser DAG 不互通。
- **方案 B**: `Term` 封装 SOMTParser 的 `Expr` handle（需要先暴露 parser 的 term API）。更标准，但 parser 当前不提供独立的 term 构造 API。

**推荐方案 A**：当前 `FrontendAdapter` 已经做 IR → CoreExpr 转换。反向只需要在 `CoreIr` 上添加 term 构造函数。

## P3: DL Fast Path for LIA / LRA

**目标**: QF_LIA 和 QF_LRA 中纯差分约束自动路由到 IDL/RDL solver，获得 O(VE) 性能。

### 设计决策

**分类器位置**: `Solver::checkSatInternal()` 中，在注册 solver 之前，先检查所有 atom 的线性形式是否都是差分形式。

**差分形式定义**:

```cpp
bool isDifferenceLogic(const LinearFormKey& form) {
    // form.coeffs 有且仅有 2 个非零项，系数为 +1 和 -1
    // 或者 1 个非零项（单变量 bound，即 x - 0 <= c）
    int nonZero = 0;
    bool hasOne = false, hasNegOne = false;
    for (const auto& [var, coeff] : form.coeffs) {
        if (coeff == 0) continue;
        ++nonZero;
        if (coeff == 1) hasOne = true;
        else if (coeff == -1) hasNegOne = true;
        else return false;
    }
    return nonZero <= 2 && (nonZero == 1 || (hasOne && hasNegOne));
}
```

## P4: NIA 第二轮增强

**目标**: 提升 NIA 的 UNSAT 证明能力和 completeness。

### 4.1 Factor Lemma

**规则**: `p * q = 0 → p = 0 ∨ q = 0`

**实现位置**: `AlgebraicIntegerReasoner::checkFactorRules()`

```cpp
NiaReasoningResult AlgebraicIntegerReasoner::checkFactorRules(
    const NormalizedNiaConstraint& c, TheoryLemmaDatabase& lemmaDb) {

    if (c.rel != Relation::Eq) return {NiaReasoningKind::NoChange, ...};
    // Check if poly is syntactically a product
    auto factors = factorize(c.poly);
    if (factors.size() < 2) return {NiaReasoningKind::NoChange, ...};

    // Build lemma: f1=0 ∨ f2=0 ∨ ... ∨ fk=0
    std::vector<SatLit> lits;
    for (auto& f : factors) {
        SatLit lit = registry_->getOrCreatePolynomialAtom(
            f, Relation::Eq, mpq_class(0), TheoryId::NIA);
        lits.push_back(lit);
    }
    return {NiaReasoningKind::Lemma, std::nullopt, TheoryLemma{lits}};
}
```

**依赖**: `PolynomialKernel::terms()`（已存在）用于检测乘积结构。

### 4.2 更强的 Modular Reasoning

**当前**: 仅对 `x^2 + y^2 = 3` 等简单形式做 mod-4。

**增强**:

```cpp
// General modular evaluation for arbitrary polynomial
checkModularAdvanced(const std::vector<NormalizedNiaConstraint>& constraints) {
```

**实现**: 在 `AlgebraicIntegerReasoner::checkModular()` 中，利用 `PolynomialKernel::terms()` 获取单变量/多变量的整数系数。

```cpp
    // Trail: sequence of (decision level, variable assignment)
    struct TrailEntry {
        int level;
        std::variant<SatLit, TheoryValue> decision;
    };

    // Theory plugin interface
    struct TheoryPlugin {
        virtual bool canAssign(const std::string& var,
                               const Trail& trail) = 0;
        virtual std::optional<TheoryValue> proposeValue(
            const std::string& var, const Trail& trail) = 0;
        virtual std::optional<TheoryConflict> checkConsistent(
            const Trail& trail) = 0;
    };

    void registerPlugin(TheoryId theory, std::unique_ptr<TheoryPlugin> plugin);
    McsatResult solve(const std::vector<TheoryAtomRecord>& atoms);

private:
    Trail trail_;
    std::unordered_map<TheoryId, std::unique_ptr<TheoryPlugin>> plugins_;
    // SAT solver integration (via IPASIR-UP or direct)
};
```

## P10: ProofManager

### ProofTrace 记录的事件类型

```cpp
enum class ProofEventKind {
    BoolLearnedClause,      // SAT learned clause
    TheoryConflict,         // theory conflict (with reasons)
    TheoryLemma,            // theory lemma

    ModularConflict,        // NIA modular contradiction

    IntervalConflict,       // Interval empty domain
```

## P11: OMT

**目标**: 支持单目标优化 `(minimize obj)` / `(maximize obj)`。

### 前提条件

- [ ] P1 API 稳定（需要 `assertFormula` + `checkSat`）
- [ ] P2 Model 稳定（需要 getValue 验证 bound）
- [ ] P9 MCSAT-2（LRA optimization 依赖 MCSAT trail）

### 设计

```cpp
// src/omt/Optimize.h
class Optimizer {
public:
    enum class ObjectiveKind { Minimize, Maximize };

    void setObjective(ObjectiveKind kind, Term obj);

    // Incremental optimization loop
    // 1. checkSat()
    // 2. getValue(obj) -> bound
    // 3. assertFormula(obj < bound) or obj > bound
    // 4. repeat until UNSAT
    OptimizeResult optimize(Solver& s);

private:
    ObjectiveKind kind_;
    Term obj_;
};
```

| OMT-1 | LIA 线性目标 |

| OMT-3 | NIA 多项式目标 |

## 风险与缓解

| 风险 | 缓解 |
|------|------|
| P1 API 改变化影响大量测试 | 保持 `parseFile` 路径不变，仅新增 API |

| P10 Proof 格式变化 | 先做内部 trace，等稳定后再 export |

## 立即执行建议

如果现在要开始下一个 Phase，**强烈建议从 P1 开始**：

```
P1 → P2 → (P3 || P5) → P4 → P6 → P7 → P8 → P9 → P10 → P11
```

1. P1 是所有上层工作的基础（没有 API 就无法写有意义的 integration test）。

3. P3 和 P5 可以并行（P3 是路由层，P5 是算法层，互不阻塞）。
