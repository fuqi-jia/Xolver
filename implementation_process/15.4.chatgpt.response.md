建议选 **[4] Revise**，但已经是**最后一轮小修**了。整体方向可以保留：本轮只做 `terms()`、多变量 GCD、公共 interval skeleton、迁移单变量 `IntervalEvaluator`，factor/modular/multivar interval 都暂缓，这个范围是合理的。

主要还差 3 个实现级细节。

---

# 1. `terms()` 不能直接加 pure virtual，除非所有 backend 都实现

计划里写：

```cpp
virtual std::optional<std::vector<MonomialTerm>> terms(PolyId a) const = 0;
```

如果现在还有 stub/mock polynomial backend，这会直接编译失败。更稳的做法是 base class 给默认实现：

```cpp
virtual std::optional<std::vector<MonomialTerm>>
terms(PolyId) const {
    return std::nullopt;
}
```

然后只让 `LibPolyKernel` override。这样符合你的“optional interface”设计。

---

# 2. GCD conflict 要显式处理常数多项式

`terms()` 现在正确规定常数项用 `powers.empty()` 表示；但 `checkGcdConflict` 还要写清楚：

```text
p = q + c
如果 q 没有任何非零项：
  c = 0  -> NoChange / tautology
  c != 0 -> Conflict
```

虽然前面 trivial constants 可能已经处理过，但这里补上更稳。否则 `{MonomialTerm{3,{}}}` 会得到 `g=0`，如果实现者直接 “g>0 才检查”，就会漏掉 `3=0`。

---

# 3. `IntervalEvaluator::run` 的接口要统一

计划里一处写：

```cpp
auto ir = intervalEvaluator_.run(constraint, box);
```

另一处写：

```cpp
auto ir = intervalEvaluator_.run(normalized, box);
```

这会混乱。建议本轮先用**单条 constraint 接口**，由 NIA adapter 遍历：

```cpp
for (const auto& c : normalized) {
    auto ir = intervalEvaluator_.run(c, box);
    if (ir.status == IntervalEvalStatus::DefinitelyViolated) {
        std::vector<SatLit> lits;
        lits.push_back(c.reason.negated());
        lits.insert(lits.end(), ir.usedReasons.begin(), ir.usedReasons.end());
        return NiaReasoningResult::conflict(TheoryConflict{lits});
    }
}
```

公共 interval 层只返回：

```cpp
IntervalEvalStatus status;
std::vector<SatLit> usedReasons;
```

不要让 interval evaluator 自己知道 NIA constraint list 或构造 conflict。
