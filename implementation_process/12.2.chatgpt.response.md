建议选 **[4] Revise**。这版比之前“大”了很多，已经覆盖 Branch Lemma、Interval Evaluation、Interval B&B、代数增强和 Local Search，整体方向对；但还有两个**正确性问题**和一个**执行范围问题**，不建议直接 Approve。

# 1. Interval status 里 `≠` 的 satisfied/violated 写反了

现在表格写的是：

```text
≠     0 ∉ [lo,hi]           [lo,hi] == [0,0]
```

表头是：

```text
rel   definitely violated   definitely satisfied
```

对 `p != 0`：

```text
definitely satisfied: 0 ∉ [lo,hi]
definitely violated:  [lo,hi] == [0,0]
```

所以这行应该改成：

```text
≠     [lo,hi] == [0,0]      0 ∉ [lo,hi]
```

否则会把明显满足 `p!=0` 的区间当成 violated，可能导致错误剪枝和 false UNSAT。这个必须修。

---

# 2. Branch lemma 的触发位置要加限制，避免无意义无限分裂

计划说当前阶段从 Phase 2 开始执行，并实现 `x ≤ k ∨ x ≥ k+1`。这个 lemma 本身全局有效，但如果在已经能由 univariate/modular/finite enumeration 处理的问题上过早触发，会造成大量无意义分支。

必须明确触发条件：

```text
buildBranchLemma 只在以下情况触发：
1. normalizer / linearDomain / squareBound / univariate / algebraic / intervalICP / bounded / localSearch 都没有返回 Conflict/Lemma/SAT；
2. 至少存在一个变量 domain 不是 singleton；
3. 如果所有变量 finite 且 totalSize <= threshold，必须先 bounded enumeration，不允许 branch 抢先；
4. 如果所有变量 finite 且 totalSize > threshold，Phase 2 可以 branch，但应优先选最大 finite interval；
5. 对完全无界问题，最多先生成少量中心 split，例如 k=0；重复 split 必须被 suppression 捕获，不能每轮重复。
```

尤其不要把 branch 当成证明 `x²=2` UNSAT 的工具。这个 plan 已经写了“不声称 branch 能证明 x²=2 UNSAT”，这是对的；但触发条件还要写死。

---

# 3. 计划仍然太像“路线图”，不是一次执行任务

它现在有 Phase 2–6，但 ExitPlanMode 的选项是直接 Approve 开始执行。按这个计划一次性执行，范围太大，Kimi 很容易从 Branch Lemma 写到 Interval B&B 再写 Local Search，风险很高。

我建议改成：

```text
总路线保留 Phase 2–6；
本次执行只执行 Phase 2；
Phase 3–6 作为后续路线，不进入本次实现。
```

也就是这次任务应该明确：

```text
Current execution scope:
  Phase 2 only:
    - NiaSolver::setRegistry
    - Solver.cpp wiring
    - buildBranchLemma
    - duplicate suppression
    - dynamic atom route tests
    - no Phase 3/4/5/6 implementation in this run
```

否则“更大更完善的 plan”可以作为路线图，但不能作为一次开发提交的 scope。
