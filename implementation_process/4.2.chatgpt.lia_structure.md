这个判断是对的：**LIA 不要塞进 SimplexSolver。应该创建独立的 `LiaSolver`，但不能复制一份 SimplexSolver 逻辑。**

你可以要求它按下面这个架构改 plan。语气可以坚决一点：

```text
这里请不要把 LIA 实现在 SimplexSolver 里面。SimplexSolver 必须保持纯 LRA solver；LIA 应该是独立的 LiaSolver。

最终架构应为：

  GeneralSimplex
      只负责 LRA relaxation：
      - linear form -> auxiliary variable
      - bound assertion
      - simplex check
      - LRA conflict explanation
      - model extraction

  LinearAtomManager / LinearTheoryCommon
      负责 LRA/LIA 共享的前端工作：
      - CoreExpr -> LinearForm 提取
      - LinearFormKey canonicalization
      - formToAux 映射
      - relation -> lower/upper bound 映射
      - equality / disequality obligation 记录
      - dynamic bound atom 注册接口

  LraSolver
      使用 GeneralSimplex + LinearAtomManager
      只处理 LRA：
      - assert bounds
      - check GeneralSimplex
      - disequality split: p < 0 ∨ p > 0
      - return LRA model / conflict / lemma

  LiaSolver
      使用 GeneralSimplex + LinearAtomManager + integer layer
      只处理 LIA：
      - 先跑 LRA relaxation
      - 检查 disequality obligation，使用 integer-strength split
      - 检查所有 Int 变量是否 integral
      - 若存在 fractional integer var，返回 branch split lemma
      - 后续加 gcd reasoning / cube test / cuts

不要让 LiaSolver 包含 SimplexSolver 实例。
不要让 SimplexSolver 进入 liaMode。
不要让一个 SimplexSolver 通过 setLiaMode(true) 变成 LIA solver。
```

关键点是：**共享代码可以抽出来，但 solver 语义要分开。**

# 不要用这个方案

```cpp
class LiaSolver {
    SimplexSolver lra_;
};
```

这个也不推荐。

原因：

```text
1. SimplexSolver 的 theory id 是 LRA，不应该被 LIA 包装复用。
2. SimplexSolver 里面会有 LRA disequality 逻辑，而 LIA disequality 应该更强。
3. LIA branch/cut/gcd reasoning 需要访问 GeneralSimplex 状态和 linear form 信息，包一层 SimplexSolver 会变得别扭。
4. 以后 proof/certificate 会混乱：LRA conflict、LIA branch、Gomory cut 都会挤在 SimplexSolver 里。
```

正确关系应该是：

```text
LraSolver  ----uses----> GeneralSimplex
LiaSolver  ----uses----> GeneralSimplex

LraSolver  ----uses----> LinearAtomManager
LiaSolver  ----uses----> LinearAtomManager
```

而不是：

```text
LiaSolver ----uses----> SimplexSolver
```

# TheoryManager 注册规则也要写死

对于不同 logic：

```cpp
if (logic == "QF_LRA" || logic == "LRA") {
    registerSolver(std::make_unique<LraSolver>(...));
}

else if (logic == "QF_LIA" || logic == "LIA") {
    registerSolver(std::make_unique<LiaSolver>(...));
}

else if (logic == "QF_NRA" || logic == "NRA") {
    registerSolver(std::make_unique<CdcacSolver>(...));
}
```

不要在 QF_LIA 下同时注册：

```text
LraSolver + LiaSolver
```

也不要注册：

```text
LiaSolver + CdcacSolver
```

Phase 1 必须保证：

```text
QF_LIA 只由 LiaSolver 处理。
QF_LRA 只由 LraSolver 处理。
QF_NRA 只由 NRA solver 处理。
```

否则多个 arithmetic solver 同时解释同一批 atoms，很容易出错。

# 结论

这次它的思路已经转向正确方向了：**独立 LiaSolver 是必须的。**

但你要卡住两个底线：

```text
第一，LiaSolver 不能复用 SimplexSolver 实例，只能复用 GeneralSimplex 和抽出的 LinearAtomManager。

第二，LIA branch lemma 不能用 fake literal / fake ExprId，必须走统一 atom registry。
```

这样改完之后，LRA 和 LIA 的架构就干净了，后面加 gcd、cube test、Gomory cut、MIR cut 都不会污染 SimplexSolver。
