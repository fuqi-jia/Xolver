建议选 **[4] Revise**。这个 IncrementalLinearizer 计划的大方向是对的：它被设计成独立 arithmetic component，不塞进 NIA/NRA；V1 只做 `x*y` 和 `x^2`；用 McCormick、square secant/tangent 生成 guarded dynamic linear lemmas；NIA cut 路由到 LIA，NRA cut 路由到 LRA。这个总体定位是正确的。

但现在还不能直接执行，有几个关键问题需要修。

# 1. 最大问题：NIA/NRA 中是否真的注册了 LIA/LRA solver？

计划说：

```cpp
owner == TheoryId::NIA ? TheoryId::LIA : TheoryId::LRA
```

这意味着 linearizer 生成的 cut 会路由给 LIA/LRA。方向对，但必须确认：

```text
QF_NIA 求解时，TheoryManager 中是否同时存在 LiaSolver？
QF_NRA 求解时，TheoryManager 中是否同时存在 LraSolver？
```

如果当前逻辑是：

```text
QF_NIA -> 只注册 NiaSolver
QF_NRA -> 只注册 NraSolver
```

那么 dynamic linear atom 虽然创建了，但没有 solver 接收它，cut 就不会起作用，甚至可能造成未处理 atom。

必须补成二选一：

```text
方案 A：
  QF_NIA 注册 NiaSolver + LiaSolver
  QF_NRA 注册 NraSolver + LraSolver

方案 B：
  NiaSolver / NraSolver 内部持有一个 linear relaxation solver，
  linearizer 生成的 cut 直接喂给内部 relaxation solver，
  不通过独立 TheoryId::LIA/LRA route。
```

我建议 V1 用 **方案 A**，但要明确 TheoryManager 支持同一逻辑下多个 arithmetic solver，并且 atom routing 不冲突。

# 2. `emitAllMcCormick=false` 但 `run()` 没有 model 参数

计划里写：

```cpp
bool emitAllMcCormick = false;  // false = emit violated cuts only (needs model)
```

但 `IncrementalLinearizer::run()` 的签名是：

```cpp
run(const std::vector<NormalizedArithConstraint>& constraints,
    const BoundStore& bounds,
    TheoryId owner,
    const LinearizationConfig& config)
```

没有当前 model。

所以 “emit violated cuts only” 无法实现。现在这会导致实现者要么乱用空 model，要么什么 cut 都不发。

V1 应该改成：

```cpp
struct LinearizationConfig {
    bool emitAllMcCormick = true;
    bool emitSquareSecant = true;
    bool emitSquareTangent = true;
    size_t maxLemmas = 10;
    size_t maxCutsPerTerm = 4;
};
```

或者改 `run()`：

```cpp
std::optional<TheoryModel> model
```

但你当前 `Solver::getModel()` 还是 stub，计划也承认 model filtering deferred，所以 V1 更适合 **默认 emit all McCormick cuts**。

# 3. Cache 不能只按 `(term, bounds)`，还要考虑 guard reason

现在 cache key 包括：

```text
kind, aux name, all bound values, cut index
```

但是 cut lemma 是 guarded 的：

```text
¬nonlinear_reason ∨ ¬bound_reason_1 ∨ ... ∨ cut_lit
```

如果同一个 `x*y` 出现在两个不同 nonlinear constraints 中：

```text
C1: x*y + z <= 5
C2: x*y - w >= 3
```

第一次为 `C1` 生成了 McCormick lemma，cache 记录这个 `(x*y,bounds,cutIndex)`。第二次处理 `C2` 时，如果 cache 阻止发 lemma，那么 `C2` 激活时 cut 可能没有被相应 guard 激活。

所以 cache key 至少要包含：

```text
nonlinearReason / constraint id
+ bound values
+ cut index
```

更准确：

```cpp
struct CutCacheKey {
    NonlinearTermKey term;
    SatLit nonlinearReason;
    std::vector<SatLit> boundReasons;
    std::vector<mpq_class> boundValues;
    int cutIndex;
};
```

如果不想把 reason 全放进 key，至少要区分：

```text
same cut_lit 但不同 guarded lemma
```

否则会丢 guard。

# 4. Adapter 里临时创建 `NiaLinearizationAdapter` 会丢缓存

计划中集成方式是：

```cpp
if (registry_) {
    NiaLinearizationAdapter adapter(*kernel_, registry_);
    auto lr = adapter.run(...);
    ...
}
```

这会导致每次 `check()` 都重新创建 adapter / linearizer / cache。

但计划又说：

```text
Return first lemma; rest cached for next round
```

这不成立。因为 adapter 是局部变量，函数返回后缓存就没了。

应该改成：

```cpp
class NiaSolver {
    std::unique_ptr<NiaLinearizationAdapter> linAdapter_;
};
```

或者：

```cpp
class NiaSolver {
    IncrementalLinearizer linearizer_;
    std::deque<TheoryLemma> pendingLinearizationLemmas_;
};
```

然后：

```text
1. 先返回 pending lemma；
2. pending 空了再 run linearizer；
3. run 生成多个 lemma 时，当前返回第一个，其余放 pending queue。
```

否则容易重复发相同 lemma，或者“剩余 lemma”直接丢失。

# 5. `NonlinearTermAbstraction` 的 detection 规则太窄

计划写：

```text
terms() has single term with two vars, exponents (1,1) → Product
terms() has single term with one var, exponent 2 → Square
```

这只能识别整个 polynomial 是单个 `x*y` 或 `x^2` 的情况。

但真正需要处理的是：

```text
x*y + z <= 5
x^2 + y^2 <= 1
3*x*y + 2*z <= 7
```

这里 nonlinear subterm 是 polynomial 中的某个 monomial，不是整个 polynomial。

所以应改成：

```text
For each monomial term in terms(poly):
  if powers = {x:1, y:1}:
      replace coeff * x*y by coeff * aux_xy
  if powers = {x:2}:
      replace coeff * x^2 by coeff * aux_x2
  if powers is linear or constant:
      keep original term
  otherwise:
      V1 unsupported for that term
```

同时 `abstract()` 不应该只返回 `std::vector<AuxTerm>`，还要返回 **linearized polynomial**：

```cpp
struct AbstractionResult {
    PolyId linearizedPoly;
    std::vector<AuxTerm> auxTerms;
    bool unsupported = false;
};
```

否则 Step 7 的 “Build linearized polynomial p_L” 没有具体实现依据。

# 6. Square test `x²+y²≤1, x=2` 不能靠当前方案保证 UNSAT

计划期望：

```text
x²+y²≤1, x=2 -> UNSAT (square secant)
```

但当前 square cuts 只在变量有 finite bounds 时生成。如果 `y` 没有 bounds，那么 `s_y` 可能是自由变量；即使 `x=2` 推出 `s_x=4`，`s_y` 可以被 LIA/LRA 赋成负数，使：

```text
s_x + s_y <= 1
```

仍然可满足。

要让这个测试成立，必须为 square aux 添加全局非负 cut：

```text
s = x²  =>  s >= 0
```

线性形式：

```text
-s <= 0
```

这个 cut 不需要 finite bounds，且对 Int/Real 都 sound。

建议 SquareCutGenerator 加：

```text
Always emit square nonnegativity:
  s >= 0
```

然后 finite bounds 下再额外发：

```text
secant upper
tangent lower
```

否则这个集成测试应该删掉或改成给 `y` 加 bounds，并确保有 `s_y >= 0`。

# 7. Bound reasons 为空不一定代表 unconditional

计划说：

```text
If bounds are unconditional (no reasons), omit bound guards.
```

这句话本身没错，但实现上不能把：

```text
reasons.empty()
```

等同于 unconditional。

因为 reasons 为空可能是：

```text
1. 真正 global/static bound；
2. DomainStore 没有记录 reason；
3. adapter 漏了 reason；
4. 从 model / heuristic 得来的 bound。
```

如果是 2/3/4，却省略 guard，就可能生成不受当前 bounds 保护的 McCormick cut，unsound。

建议 `BoundInfo` 加字段：

```cpp
bool lowerIsGlobal = false;
bool upperIsGlobal = false;
bool reasonsComplete = false;
```

V1 最简单规则：

```text
如果 bound 是 finite 但 reasons 为空，且不是显式 global bound：
  skip cut
或者：
  使用 all active reasons 作为 coarse guard
```

不要无条件 omit。

# 8. `p_L rel 0` lemma 不是普通 theory-valid lemma，要标注为 conservative extension

这个计划把：

```text
C -> p_L rel 0
C -> McCormick cuts
```

当作 theory lemma 加入。

这在“引入 fresh aux variable 的 equisatisfiable extension”意义下是可以的：任何原问题模型都可以扩展 `aux = 原非线性项值`，满足这些线性化约束。

但它不是一个在任意 aux 赋值下都 theory-valid 的普通 lemma。后续 ProofManager 如果要证明，就不能把它当成 LIA/LRA theory lemma。

建议在计划里明确：

```text
Linearization lemmas are conservative extension lemmas over fresh aux variables.
They preserve satisfiability but are not ordinary theory-valid consequences over arbitrary aux assignments.
Therefore:
  1. final SAT must validate original nonlinear constraints;
  2. ProofManager must treat them as definitional/extension lemmas;
  3. aux vars must never appear in public model.
```

这不是阻止执行，但要写清楚，避免以后 proof / soundness 混淆。

# 9. NIA cut 的 strict relation 要明确交给 LIA normalize

例如：

```smt2
(assert (> (* x y) 200))
```

线性化后可能是：

```text
t - 200 > 0
```

如果 cut route 到 LIA，那么 LIA 必须把 strict integer inequality 规约成：

```text
t - 200 >= 1
```

或者：

```text
t >= 201
```

计划没有写 strict handling。建议补一句：

```text
For owner=NIA, generated linear atoms must use integer-normalized relations:
  p > 0  -> p >= 1
  p < 0  -> p <= -1
or rely on LiaSolver's existing strict normalization, with regression tests.
```

# 结论

这份计划**方向正确，但要 Revise**。

```text
独立 linearizer/
V1 只做 x*y 和 x²
McCormick + square cuts

```

SAT validation / extension lemma 语义。

修完这些后，这个计划就可以执行。
