# Phase 3: Interval Evaluation + Pruning

## 目标

构建 IntervalZ + IntervalEvaluator 基础设施，在 NiaSolver check 管道中加入一轮 over-approx interval evaluation，用于：
1. 提前检测明显 violated 的约束（快速 UNSAT）
2. 为 Phase 4 的 Interval B&B 提供核心组件

## 背景

当前管道已有：
- LinearDomain（线性约束推 bound）
- SquareBound（单变量平方约束）
- SumOfSquaresBound（多变量平方和约束）
- Univariate（RRT 根求解）
- Algebraic（GCD / modular / factor skeleton）

对于更一般的多项式约束（如 `x³ + y² - z <= 10`），上述引擎无法直接处理。IntervalEvaluator 通过 over-approx evaluation 能安全地判断约束是否不可能满足。

## 设计

### 3.1 IntervalZ 数据结构

```cpp
struct IntervalZ {
    mpz_class lo;  // inclusive lower bound
    mpz_class hi;  // inclusive upper bound

    bool isEmpty() const { return lo > hi; }
    bool contains(const mpz_class& v) const { return v >= lo && v <= hi; }
    bool containsZero() const { return lo <= 0 && hi >= 0; }
    mpz_class width() const { return hi - lo; }
};
```

### 3.2 IntervalEvaluator

对多项式在每个变量的区间上求 over-approx 范围。

**核心操作（全部 over-approx）：**

| 操作 | 规则 |
|------|------|
| `add` | `[a,b] + [c,d] = [a+c, b+d]` |
| `sub` | `[a,b] - [c,d] = [a-d, b-c]` |
| `neg` | `-[a,b] = [-b,-a]` |
| `mul` | `[a,b] * [c,d] = [min(ac,ad,bc,bd), max(ac,ad,bc,bd)]` |
| `pow(k even)` | `0∈[l,u] → [0, max(l²,u²)]`; else `[min(l²,u²), max(l²,u²)]` |
| `pow(k odd)` | `[l^k, u^k]` |

**实现方式：**
- 不直接操作 libpoly 内部表示（太复杂且不跨后端）
- 利用 PolynomialKernel 的 `variables()` + `getIntegerCoefficients()` 分解多项式
- 对多变量多项式，使用逐项 interval evaluation

**简化策略：**
- 单变量多项式：用系数数组做 Horner / 逐项 evaluation
- 多变量多项式：如果 `getIntegerCoefficients` 返回 nullopt（多变量或系数不可用），尝试用 `kernel_.evalInteger()` 在 interval 的四个角点上 evaluation，取 min/max 作为 over-approx
  - 这对多项式不精确（可能 under-approx），所以只用于检测 definitely violated/satisfied
  - 或者更保守：直接返回 NoChange，不推 bounds

实际上，对于 interval evaluation 的 over-approx，最简单且安全的方式是：
- 从 DomainStore 获取每个变量的 interval
- 对多项式在 interval 上 evaluation
- 如果多项式结构简单（能分解为 interval 上的基本操作），精确计算
- 否则保守处理

由于 PolynomialKernel 接口不暴露项结构，我建议采用一个务实的方式：
- 对 `evalInteger(poly, assignment)` 做推广：给定每个变量的 interval，计算多项式的 over-approx range
- 这需要实现一个独立的 interval arithmetic evaluator，不依赖 PolynomialKernel 内部结构

**具体实现：**
- `IntervalEvaluator` 不直接操作 `PolyId`
- 而是操作 `NormalizedNiaConstraint`，尝试识别已知模式：
  - 线性形式：`a*x + c` → interval = `a * [x] + c`
  - 单变量多项式 → 用系数逐项 evaluation
  - 多变量多项式 → 用角点采样（保守 over-approx）
- 但这样每个模式都要写一遍，和现有引擎重复

**更好的方式：**
- 扩展 `PolynomialKernel` 接口，增加 `evalInterval(PolyId, const std::unordered_map<std::string, IntervalZ>&)` → `IntervalZ`
- 在 LibPolyKernel 中利用 libpoly 的 evaluation 能力实现
- 但这需要修改 PolynomialKernel 接口，影响面大

**最务实的方式（推荐）：**
- `IntervalEvaluator` 操作 `NormalizedNiaConstraint`
- 对每条约束，尝试获取变量列表
- 从 DomainStore 获取每个变量的 interval
- 用 PolynomialKernel::evalInteger 在 interval 的角点上采样，取 min/max
  - 对线性/凸函数，角点采样是精确的
  - 对非凸多项式，角点采样可能 under-approx → 必须加安全 margin
- **更简单**：只处理单变量和双变量，在 grid 上采样（如 3×3 grid），取 min/max

实际上，让我重新考虑。Interval evaluation 的真正价值不是精确计算多项式范围，而是快速判断约束是否 violated。一个简单且安全的策略：

```cpp
// 对 poly rel 0，用 domain 的 interval 做 evaluation
// 在 interval 的角点上采样，取 min/max
// 如果 min > 0 且 rel = Leq → violated
// 如果 max < 0 且 rel = Geq → violated
// 如果 0 ∉ [min, max] 且 rel = Eq → violated
```

角点采样的风险：对非凸多项式，最小值/最大值可能不在角点上。这会导致 under-approx（漏掉真正的 violated 情况），但不会导致 false UNSAT（因为 under-approx 只会漏报，不会误报）。

但等等，如果角点采样得到的 min = 5，而真正的 min = -10，那么我们会认为 `poly >= 0` 是 satisfied（因为 min > 0），但实际上它有负值。这不会导致 UNSAT 错误，但会漏掉 UNSAT 机会。

对于 `poly <= 0`，如果角点采样得到的 max = -5，而真正的 max = 10，我们会认为 satisfied，但实际上有正值。这也不会导致 UNSAT 错误。

所以角点采样是 **sound**（不会 false UNSAT），但 **incomplete**（可能漏掉 UNSAT）。这符合我们的正确性要求！

但问题是：如果角点采样得到的 interval 包含 0，我们无法判断是 violated 还是 satisfied，只能返回 unknown。

对于 Phase 3，这足够了。我们可以：
1. 对每个变量，从 DomainStore 获取 interval（如果没有 bound，用 [-M, M] 或 [-∞, ∞]，但 ∞ 无法处理）
2. 如果变量没有 bound，跳过这条约束的 interval evaluation
3. 如果有 bound，在角点上采样

但角点数量随变量指数增长（2^n）。对于 n>3 的情况，采样量太大。

**折中方案：**
- 只对 <= 2 个变量的约束做 interval evaluation
- 或者只对单变量约束做（Univariate 和 SquareBound 已经处理了大部分单变量情况）
- 多变量约束用 2×2 角点采样（4 个点）

让我采用这个方案。

### 3.3 IntervalEvaluator 实现

```cpp
class IntervalEvaluator {
public:
    explicit IntervalEvaluator(PolynomialKernel& kernel);

    NiaReasoningResult run(
        const std::vector<NormalizedNiaConstraint>& constraints,
        DomainStore& domains);

private:
    PolynomialKernel& kernel_;

    // Evaluate poly at a specific integer assignment
    std::optional<mpz_class> evalAt(const IntegerModel& model, PolyId poly) const;

    // Check if constraint is definitely violated by sampling corner points
    bool isDefinitelyViolated(
        const NormalizedNiaConstraint& c,
        const DomainStore& domains);

    // Build interval for a variable from domain
    std::optional<IntervalZ> getInterval(
        const std::string& var, const DomainStore& domains) const;
};
```

### 3.4 接入位置

放在 `AlgebraicIntegerReasoner` 之后、`EmptyDomain` 之前：

```
... → algebraic → intervalEvaluator → emptyDomain → bounded → ...
```

### 3.5 约束状态判断

| rel | definitely violated (角点采样) |
|-----|-------------------------------|
| `<=` | min > 0 |
| `>=` | max < 0 |
| `=` | 0 ∉ [min, max] |
| `!=` | [min, max] == [0,0]（unlikely in practice） |

注意："definitely violated" 意味着**所有**角点值都不满足约束。由于角点采样可能 under-approx，这不会导致 false UNSAT。

### 3.6 禁止反向传播

Phase 3 明确不做：
- 从 `p <= 0` 和 `p` 的 interval 反向推导变量的 bounds
- Product pruning（`xy <= c` 反推 x, y bounds）
- 任何可能 under-approx 的反向推理

### 3.7 测试

**单元测试：**
- `x² + y² <= -1` 在 `x,y ∈ [-5,5]` 上 → violated
- `x³ + 1 <= 0` 在 `x ∈ [0,5]` 上 → 角点 x=0: 1>0 → violated at some points? No, x=0: 0³+1=1>0 → violated for all x≥0 → actually x³+1 > 0 for all x ≥ 0 → violated for Leq → UNSAT
- `x³ - 8 <= 0` 在 `x ∈ [0,3]` 上 → x=0: -8<=0 ✓, x=3: 27-8=19>0 ✗ → not definitely violated

**端到端 SMT2：**
- `x >= 0, x³ + 1 <= 0` → UNSAT（角点采样：x=0→1>0, x=1→2>0, ... 所有 >0 → violated）
- `0 <= x <= 3, x³ - 8 <= 0` → SAT（x=0,1,2 满足，角点采样不 violated）

## 正确性红线

1. Interval evaluation 必须是 **over-approx**（宁可保守不要激进）。
2. 角点采样只会漏报 UNSAT，不会误报 UNSAT（sound）。
3. 不做任何反向传播（product pruning 禁止）。
4. 对 >2 变量的约束跳过 interval evaluation（避免指数级角点采样）。
