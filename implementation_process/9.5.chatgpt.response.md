建议选 **[4] Revise**，不要 Approve。这个 NIA-Core 方案整体方向是对的：它把 NIA 从“常量/平凡情况”推进到 univariate roots、domain store、modular reasoning、bounded solving、local search，并且强调 SAT 必须 exact validation、unsupported backend 返回 Unknown、local search failure 不能当 UNSAT，这些底线是正确的。方案目标和基础设施改动也明确写了 `evalInteger`、变量收集、动态 polynomial atom，以及 SAT 结果必须精确验证。

但现在还有几处**正确性硬伤/接口缺口**，必须修。

---

# 1. `DomainStore` 设计缺少 reason 字段

方案说：

```text
If any domain becomes empty, DomainStore builds TheoryConflict from all reasons that contributed to the emptiness.
Never emit conflict from an incomplete reason set.
```

这个原则是对的。但当前 `IntDomain` 只有：

```cpp
hasLower, hasUpper, lower, upper, finiteValues, excludedValues
```

没有保存任何 reason。也就是说，`restrictToFiniteSet`、`addLowerBound`、`addUpperBound`、`excludeValue` 之后，后面 domain empty 时根本不知道该用哪些 literal 组成 conflict。

必须改成：

```cpp
struct DomainReason {
    std::vector<SatLit> reasons;
};

struct BoundWithReason {
    mpz_class value;
    std::vector<SatLit> reasons;
};

struct FiniteSetWithReason {
    std::set<mpz_class> values;
    std::vector<SatLit> reasons;
};

struct ExclusionWithReason {
    mpz_class value;
    std::vector<SatLit> reasons;
};
```

或者至少：

```cpp
lowerReason;
upperReason;
finiteSetReason;
excludedValueReasons[value];
```

否则 `DomainStore::buildEmptyDomainConflict()` 很容易生成不完整 conflict，导致 unsound。

---

# 2. Bounded solver 的 UNSAT 必须区分“完整穷举完毕”和“预算耗尽”

方案中 direct enumeration 写：

```cpp
for each assignment ...
    if validator.validate(...):
        return Sat
return Unsat(conflict over all active reasons)
```

这个在**完整枚举了整个有限域**时是 sound 的。但后面 interval B&B 有 budget 和 pruning。必须明确：

```cpp
enum class BoundedSolveStatus {
    Sat,
    UnsatComplete,
    UnknownBudget,
    UnknownUnsupported
};
```

只有在：

```text
1. finite domains 已完整覆盖；
2. 所有 branches 都被验证/pruned；
3. pruning 是 over-approx sound；
4. 没有 budget cutoff；
```

才能返回 `UnsatComplete`。

否则只能 `Unknown`。尤其 interval B&B 如果中途因为预算停止，不能返回 UNSAT。方案虽然写了 budget exhaustion returns Unknown，但 `BoundedNiaSolver` 的 status 和 conflict 构造还需要更明确。

---

# 3. NiaNormalizer 必须明确处理 effective literal，而不是原始 atom

方案说输入是 `ActiveNiaConstraint`，输出 `p rel 0`，并做 strict normalization：

```text
p < 0 → p <= -1
p > 0 → p >= 1
```

这对 QF_NIA 是对的，但必须补一句：

```text
NiaNormalizer 只处理当前 SAT assignment 下的 effective active constraint。
如果原始 atom 被赋 false，必须先取反 relation。
```

例如：

```smt2
(not (<= p 0))
```

不能仍按 `p <= 0` 进入 NIA；必须变成：

```text
p > 0
```

再归一化成：

```text
p >= 1
```

否则所有后续 univariate、modular、bounded reasoning 都会在错误约束上工作。

---

# 4. Univariate root reasoning 的 domain update 必须带 reason

`x^2 = 4` 推出：

```text
x ∈ {-2, 2}
```

`x^2 != 4` 推出：

```text
x != -2, x != 2
```

这些 domain update 都依赖当前 equality/disequality literal。方案里 `DomainUpdated` 会继续 pipeline，这是对的；但必须规定：

```cpp
DomainStore::restrictToFiniteSet(var, roots, {reasonLit});
DomainStore::excludeValue(var, root, {reasonLit});
```

否则后面如果：

```text
x^2 = 4
x != -2
x != 2
```

导致 domain empty，conflict 必须包含：

```text
¬lit(x^2=4) ∨ ¬lit(x!=-2) ∨ ¬lit(x!=-2)
```

如果 reason 没有存好，conflict 就不可靠。

---

# 5. Factor lemma 的动态 polynomial atom 路由还不够详细

方案写：

```text
p*q = 0 → ¬(p*q=0) ∨ (p=0) ∨ (q=0)
p*q != 0 → ¬(p*q!=0) ∨ (p!=0), ¬(p*q!=0) ∨ (q!=0)
Uses TheoryAtomRegistry::getOrCreatePolynomialAtom
```

公式本身是 sound 的。但要补接口闭环：

```text
getOrCreatePolynomialAtom(p, rel, rhs, TheoryId::NIA)
  必须创建真实 SAT var；
  必须注册到 Atomizer / TheoryAtomRegistry；
  下一轮 SAT assignment 中该 literal 必须能 route 回 NiaSolver::assertLit；
  NiaSolver 必须能把该 dynamic atom 转成 ActiveNiaConstraint。
```

否则会出现 lemma 里有新的 `p=0` literal，但后续 SAT 选中它时 NiaSolver 不知道如何 assert。

也要明确：

```text
dynamic polynomial atom 不允许 fake ExprId；
PolyId 必须是 kernel 中已注册、可 eval、可 variables()、可 normalize 的真实 polynomial。
```

---

# 6. Modular reasoning 的 conflict reason 要包含所有参与 equality

方案写：

```text
For each active equality p(x)=0:
  Test mod m ...
If no residue satisfies all equalities simultaneously → UNSAT
```

这里“所有 equalities simultaneously”是对的，但 conflict 不能只返回某一个 equality 的 reason。必须收集本次 modular check 使用的全部 equality reasons：

```cpp
conflict = { ¬reason(e1), ¬reason(e2), ..., ¬reason(ek) };
```

否则会生成过强 conflict。

同时要明确：只对 normalized effective equalities 做，且 coefficient/rhs 已经 clear denominators。方案已经强调 modular reasoning 只用于 equalities，不用于证明 `p != 0` false，这点是正确底线。

---

# 7. Polynomial coefficient extraction fallback 要更保守

方案说如果 libpoly 没有直接 term iteration，可以用 evaluation + interpolation。这个可以做，但必须写成：

```text
只有在 degree(poly, var) 精确可得，并且确认 polynomial 是 univariate in var 时，才能 interpolation。
如果 degree 或 univariate check 不可靠，返回 nullopt。
```

不能为了得到系数而猜 degree。否则 RRT 可能漏根，导致错误 UNSAT。

建议补：

```text
interpolation fallback:
  1. variables(poly) 必须精确返回单变量 {x};
  2. degree(poly,x) 必须精确；
  3. 用 d+1 个点做 exact rational/integer interpolation；
  4. 重构 polynomial 后必须与原 polynomial 做等价验证，或至少在足够测试点做 sanity check；
  5. 失败则 nullopt。
```

而且 `getIntegerCoefficients()` 如果发现系数不是整数，应返回 `nullopt`，不能强转。

---

# 8. Branch lemma 仍然需要具体化

最后 pipeline 写：

```cpp
if (auto lemma = buildBranchLemma(*normalized, domains_, lemmaDb)) {
    return mkLemma(*lemma);
}
return unknown();
```

但 plan 没写 branch lemma 的形式。

对于整数变量，安全的 branch 是全局有效：

```text
x <= k ∨ x >= k+1
```

如果根据当前 domain `[l,u]` 做 split：

```text
x <= mid ∨ x >= mid+1
```

这也是全局有效，不需要 guard。

但如果根据 finiteValues / excludedValues 生成：

```text
x = v1 ∨ x = v2 ∨ ...
```

这不是全局有效，必须 guarded by domain reasons。建议 plan 明确：

```text
NIA-Core Phase 只允许全局 integer split:
  x <= mid ∨ x >= mid+1

任何 domain-specific split 必须带上 domain reasons：
  ¬r1 ∨ ... ∨ branch_literals
```

可以直接贴：

```text
Revise required before approve:

1. DomainStore must store reasons for every domain update:
   lower/upper bounds, finite set restrictions, and excluded values.
   buildEmptyDomainConflict() must use complete reasons. The current IntDomain has no reason fields.

   Budget exhaustion or unsupported interval evaluation returns Unknown.

```

# 结论

这版方案**不应 Reject**，因为总体结构和正确性红线已经比较成熟；但也**不能直接 Approve**。最关键的 blocker 是 `DomainStore` 没有 reason 字段，但后面又要从 domain empty 生成 conflict；这是直接的 correctness gap。其次是 bounded solver 的 UNSAT 完整性、dynamic polynomial atom route、以及 effective literal normalization。

我的选择：**[4] Revise**。
