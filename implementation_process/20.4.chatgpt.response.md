建议 **再 Revise 一小轮**。这版已经把上一轮最关键的 “LIA/LRA 看不到原始 bounds” 问题补上了：现在明确要求 QF_NIA/QF_NRA co-register LIA/LRA，并通过 mirror lemma 把原始 linear bounds 送到 LIA/LRA；这点是对的，也是 McCormick 能真正证明冲突的前提。

但现在还有几个执行级问题，主要不是大方向，而是接口一致性和测试预期。

---

# 1. `markEmitted(lemma)` 仍然和 cache API 不匹配

计划里 cache API 是：

```cpp
markEmitted(
  NonlinearTermKey key,
  SatLit nonlinearReason,
  vector<mpq_class> boundValues,
  int cutIndex
)
```

但 `NiaSolver::check()` 伪代码里仍然写：

```cpp
linAdapter_->markEmitted(lemma);
```

这不够，因为单个 `TheoryLemma` 里面没有：

```text
NonlinearTermKey
nonlinearReason
boundValues
cutIndex
```

所以 adapter 没法从 `lemma` 反推出 cache key。

需要把返回类型改成带 metadata：

```cpp
struct PendingLinearizationLemma {
    TheoryLemma lemma;
    std::optional<CutCacheKey> cacheKey;
};

struct LinearizationResult {
    LinearizationStatus status;
    std::vector<PendingLinearizationLemma> lemmas;
};
```

然后：

```cpp
if (!lemmaDb.contains(item.lemma)) {
    lemmaDb.insertIfNew(item.lemma);
    pendingLinLemmas_.push_back(item.lemma);

    if (item.cacheKey) {
        linAdapter_->markEmitted(*item.cacheKey);
    }
}
```

或者更简单：`IncrementalLinearizer::run()` 内部只对真正放入 `result.lemmas` 的前 `maxLemmas` 标记 emitted。不要让 adapter 用 `TheoryLemma` 反查 key。

这是目前最大剩余问题。

---

# 2. “Missing bounds -> NoChange” 测试不准确

计划现在说：

```text
Missing bounds -> NoChange | x∈[0,10], y unbounded → NoChange
```

但同一版又规定：即使缺少 bounds，也仍然允许生成：

```text
abstraction lemma: ¬C ∨ p_L rel 0
```

而且 square nonnegativity 也不需要 bounds。

所以如果测试的是整个 `IncrementalLinearizer::run()`，缺少 `y` bound 时不一定是 `NoChange`，因为它可能仍然返回 abstraction lemma。

建议把测试改成更精确：

```text
Missing product bounds:
  McCormickGenerator returns empty;
  no bounds-dependent envelope cuts are emitted;
  abstraction lemma may still be emitted.
```

如果你想保留 `NoChange` 测试，那就应该直接测试 `McCormickGenerator`，而不是测试完整 linearizer。

---

# 3. mirror lemma 必须基于 effective active relation

计划说 mirror active linear bounds：

```text
B -> B_LIA
```

这是对的。但要强调这个 `B` 必须是当前 SAT assignment 下的 **effective atom**。

例如原始 atom 是：

```text
x <= 3
```

如果 SAT 赋值为 false，当前 active constraint 是：

```text
x > 3
```

NIA 下要变成：

```text
x >= 4
```

mirror 给 LIA 的也必须是这个 effective relation，而不是原始 `x<=3`。

建议补一句：

```text
Mirror active normalized/effective linear constraints, not raw syntactic atoms.
If a linear atom is assigned false, first negate and normalize it, then mirror the resulting bound to LIA/LRA.
```

否则会产生错误 mirror。

---

# 4. `BoundInfo` 的 V1 policy 要写死为 “incomplete reason -> skip cut”

现在计划写：

```text
If finite bound has no complete reason:
  skip cut or use coarse all-active guard
```

二选一会让 agent 实现发散。V1 建议明确选择：

```text
V1: if finite bound is not global and reasonComplete=false, skip bounds-dependent cut.
```

coarse all-active guard 后面再做。这样简单、安全。

---

# 5. `Duplicate cache` 测试要区分 same reason / different reason

计划里 cache key 已包含 `nonlinearReason`，这很好，因为不同非线性约束需要不同 guard。

但测试写：

```text
Same term+bounds twice → only one lemma
```

应该改成：

```text
Same term + same nonlinearReason + same bounds -> one lemma
Same term + different nonlinearReason + same bounds -> separate guarded lemmas
```

否则实现者可能又把 `nonlinearReason` 从 cache key 里删掉。

---

# 6. co-register 需要验证 TheoryManager 真的是按 `TheoryId` dispatch

计划依赖：

```text
TheoryManager dispatches by TheoryAtomRecord.theory
```

如果当前代码实际上仍然是“每个 solver 遍历所有 atoms，然后自己 ignore”，也许能工作；但如果某些 solver 对 unsupported atom 返回 Unknown，就会出问题。

建议加一句执行前检查：

```text
Before co-registering LIA/LRA with NIA/NRA, verify TheoryManager only sends atoms to matching TheoryId, or every solver must safely ignore non-owned atoms.
```

并加 regression：

```text
QF_NIA original nonlinear atom is not sent to LiaSolver.
linearizer-generated LIA cut is not sent to NiaSolver.
```

# 可以直接给 agent 的 Revise 文本

```text
Final revise before approve:

   markEmitted(lemma) is insufficient.
   Either:
     a) return PendingLinearizationLemma { TheoryLemma lemma, optional<CutCacheKey> cacheKey },
```

结论：**这版已经很接近，但还需要 Revise 这些细节后再 Approve**。最大 blocker 是 `markEmitted(lemma)` 和 cache key 不匹配；这个不修，后面很容易丢 cut 或重复发 cut。
