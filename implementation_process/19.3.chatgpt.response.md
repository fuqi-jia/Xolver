建议 **再 Revise 一小轮**。这版整体已经接近可执行：Z/Q 结果拆开了，`IcpEngineZ/IcpEngineQ` 也明确了；V1 只做 forward eval + square / sum-of-squares pattern，不做 Add/Sub/Mul backward；`UnknownBudget` 也改成 adapter 里 `NoChange`，这几个关键问题都已经修正。

但现在还有几个 correctness 细节必须修。

---

# 1. `x² - c >= 0` 在 `c < 0` 时不是 Empty

计划里写：

```text
x^2 - c rel 0:
  c < 0 with rel = <=, =, >= -> Empty
```

这里 `>=` 是错的。

如果 `c < 0`，那么：

```text
x² - c = x² + |c|
```

所以：

```text
x² - c >= 0
```

是恒真，不是冲突。

正确规则：

```text
x² <= c:
  c < 0 -> Empty

x² = c:
  c < 0 -> Empty

x² >= c:
  c < 0 -> Tautology / NoChange

x² != c:
  c < 0 -> Tautology / NoChange
```

同理，`Σxi² >= c` 在 `c < 0` 时也是恒真，不是 Empty。

---

# 2. `x² <= c` 的整数边界应为 `[-floor_sqrt(c), floor_sqrt(c)]`

计划里写：

```text
x ∈ [-ceil_sqrt(c), floor_sqrt(c)]
```

对整数 NIA 来说，`x² <= c` 应该是：

```text
x ∈ [-floor_sqrt(c), floor_sqrt(c)]
```

例如 `c=2`，合法整数是：

```text
{-1,0,1}
```

不是 `[-2,1]`。`[-2,1]` 虽然 over-approx，不会 unsound，但非常弱且不对称，容易让后续测试困惑。

所以 NIA 规则写成：

```text
r = floor_sqrt(c)
x ∈ [-r, r]
```

NRA 的 `IntervalQ` 则需要 rational outer bound，另行处理，不要混成同一个公式。

---

# 3. `x² = c` 对整数不是普通区间 update，而是 finite set

计划写：

```text
x^2 = c -> bound update + if c not perfect square -> Empty
```

如果 `c = 4`，仅更新成：

```text
x ∈ [-2,2]
```

是不精确的。正确整数域是：

```text
x ∈ {-2, 2}
```

如果你的 `DomainStore` 支持 finite set，NIA adapter 应该返回 finite-set update；如果 ICP V1 的 `BoundUpdateZ` 只支持 interval，那至少要写清楚：

```text
V1 interval-only ICP may update x to [-r,r];
exact finite-set refinement remains in old SquareBoundReasoner / DomainStore-specific adapter.
```

否则你把旧 `SquareBoundReasoner` 替换掉后会丢精度。既然当前计划保留 old squareBound 并行，这个暂时不影响 soundness，但要写明。

---

# 4. `SumSquaresContractor` 的 `c < 0 -> Empty` 也要按 relation 区分

当前写：

```text
Σ xi^2 - c rel 0:
  c < 0 -> Empty
```

这只对：

```text
Σxi² <= c
Σxi² = c
```

成立。

但对：

```text
Σxi² >= c
```

如果 `c < 0`，它是恒真。

对：

```text
Σxi² != c
```

如果 `c < 0`，也恒真。

建议改为：

```text
Σxi² <= c:
  c < 0 -> Empty
  c >= 0 -> each xi ∈ [-floor_sqrt(c), floor_sqrt(c)] for Z

Σxi² = c:
  c < 0 -> Empty
  c >= 0 -> same bounding update

Σxi² >= c:
  c < 0 -> NoChange / Tautology
  c >= 0 -> V1 NoChange

Σxi² != c:
  c < 0 -> NoChange / Tautology
  otherwise V1 NoChange
```

---

# 5. `ContractorResult` 还没有定义清楚

计划定义了：

```cpp
ContractorZ::contract(...) -> ContractorResult
ContractorQ::contract(...) -> ContractorResult
```

但 `ContractorResult` 没有拆 Z/Q，也没有说明里面的 update 类型。

既然 `IcpResultZ/Q` 已经拆开，建议 contractor 也拆：

```cpp
struct ContractorResultZ {
    IcpStatus status;
    std::optional<TheoryConflict> conflict;
    std::vector<BoundUpdateZ> updates;
};

struct ContractorResultQ {
    IcpStatus status;
    std::optional<TheoryConflict> conflict;
    std::vector<BoundUpdateQ> updates;
};
```

然后：

```cpp
class ContractorZ {
    virtual ContractorResultZ contract(ReasonedBoxZ& box) = 0;
};

class ContractorQ {
    virtual ContractorResultQ contract(ReasonedBoxQ& box) = 0;
};
```

否则实现时又会回到一个模糊的 `Interval` 类型。

---

# 6. `SquareContractor` / `SumSquaresContractor` 的 Z/Q 规则不能完全共用

NIA 和 NRA 的 sqrt 处理不同：

```text
NIA:
  x² <= c -> integer rounding, floor_sqrt
  x² = c -> if c not perfect square -> Empty

NRA:
  x² <= c -> rational/algebraic bound, over-approx only
  x² = c -> c >= 0 is satisfiable over reals, not require perfect square
```

所以不要只写一个泛化 `SquareContractor`。建议明确：

```text
SquareContractorZ
SquareContractorQ
SumSquaresContractorZ
SumSquaresContractorQ
```

V1 可以优先实现 `Z`，`Q` 只做 forward conflict 或保守 NoChange。
