# NLColver 求解器审计报告

**日期**: 2026-05-26
**范围**: `src/` 全量(7 个并行只读审计,按子系统切分)
**目的**: 找出所有影响**正确性**(健全性/完备性)与**效率**的问题,供后续提升使用。
**方法**: 7 个独立 agent 分别审计 NRA-CDCAC 核心、NRA 后端+Lazard、NIA、LRA/LIA/线性/IDL/RDL、组合+核心+EUF+混合、多项式内核+search+presolve、前端+IR+SAT+API;外加一次跨目录的硬编码上界全局扫描。

> **关键判定框架**:对 SMT 求解器,「有没有上界」不是问题,**超过上界时返回什么**才是。超限回退到 `unknown` = 健全(只是不完备);超限返回确定的 `sat`/`unsat` = 潜在**不健全**。本报告据此区分严重度。

## 一、健全性风险

> 这些是可能直接返回**错误判定**或**错误模型**的问题。优先处理。

### 1.1 理论组合:模型提取与等式传播(确认已知 bug 类)

- **[高] `src/theory/core/TheoryManager.cpp:360-384`** — `getModel()` 用「first-wins」合并各子理论模型,**从不调和共享变量的取值**。EUF 可强制 `a=b`,而 LRA/LIA/NIRA 给 `a`、`b` 各自独立的单纯形值,合并出的模型违反原公式。这正是记忆中记录的模型提取 bug(uflra_007 `a=b`)。
  *修复*: 合并后用 `sharedEqMgr_` 的等价类并查集,每个类只赋一个代表值;或让 EUF 输出类代表的数值。

- **[高] `src/theory/arith/lra/LraSolver.cpp:440-443`、`src/theory/arith/nira/NiraSolver.cpp:686-689`、`src/theory/arith/nia/NiaSolver.cpp:686-689`** — `getDeducedSharedEqualities()` 一律返回 `{}`。QF_UFLRA / QF_UFNRA / QF_UFNIA / QF_UFNIRA 下,算术理论从不向 EUF 传播蕴含的接口等式(只有 LIA 实现了,`LiaSolver.cpp:699-756`)。Nelson-Oppen 完备性要求每个理论共享所有蕴含的接口变量等式。
  *后果*: **漏 UNSAT**(组合不完备 → 错误 SAT 或 Unknown)。
  *修复*: 给 LRA/NIRA/NIA 实现等式推导(类似 LIA 的 `proveFixedValue` 分组);否则在共享变量未解析时回退 Unknown。

- **[高] `src/api/Solver.cpp:864-879`** — 共享项对的 arrangement / case-split **整段被注释禁用**,组合完全依赖理论推导出的等式(而上一条说明这些为空)。非凸理论(LIA/NIA)需要 arrangement 分裂才能完备。
  *后果*: 隐式假设凸性;非凸共享变量交互会**漏 UNSAT**。
  *修复*: 启用有界 arrangement 枚举(SAT 仍由 validator 兜底);或在仍有未分裂共享项且无等式推出时显式返回 Unknown。

- **[高] `src/theory/euf/EufSolver.cpp:570-656` + `src/theory/euf/IncrementalEGraph.cpp:386-413`** — `tryEvaluateBuiltin` 内建常量折叠用 `MergeReason{BuiltinEval, lit=SatLit()}`(空 lit)合并节点;而 `explainEdge` 只处理 `AssertedEquality` 和 `Congruence`,`BuiltinEval` 边落到空 `argPairs` 循环返回 `{true, {}}`(空 reason)。
  *后果*: 任何证明路径跨越 builtin-eval 边的冲突/等式,产生的冲突子句**缺失真实前提** → 不健全的学习子句。
  *修复*: 在 `explainEdge` 中处理 `BuiltinEval`,补上固定操作数的参数类等式 reason。

- **[中] `src/theory/core/TheoryManager.cpp:337-342`** — `deducedEqCache_` 只在 `clearSolvers()` 清空,**回溯时(`backtrackToLevel`)从不清空**。高层推导出的等式 lemma 被永久抑制;回溯越过该层后(lemma 可能已被 SAT DB 遗忘)该等式再也不会被重新提出。
  *后果*: 必需的接口等式在回溯后被静默丢弃 → 组合不完备/不健全。
  *修复*: 给 cache 打 level 标签并在 `backtrackToLevel` 回滚;或直接去掉(lemmaDb 已去重)。

- **[中] `src/theory/core/TheoryManager.cpp:86-92`** — `snapshotForLevel` 在找不到快照时返回一个**函数内 `static LevelSnapshot empty` 的引用**(非线程安全、可变共享),静默给出空(level 0)快照,导致 `backtrackToLevel`(128-129)可能把 `sharedEqMgr_` 回滚到空而非正确层。
  *修复*: 保证 level-0 快照恒存在;或返回 `std::optional` 并显式处理 miss。

- **[中] `src/theory/combination/Purifier.cpp:76-80`** — `theoryOf()` 对既非 UFApply 也非已知算术 kind 的表达式(纯 Bool/array/BV/select)一律返回 `EUF`,且 `containsArithmetic` 不含 array/BV/select。混合项被误分类、未纯化。
  *修复*: 按 kind 显式分类;未处理的 kind 应 assert/Unknown 而非默认 EUF。

- **[中] `src/theory/combination/Purifier.cpp:30-49`** — `registerEufVars` 只对 UFApply 的参数注册共享候选,不注册只出现在算术侧的变量。仅在算术侧出现的共享变量可能未被登记为共享 → 等式永不交换。
  *修复*: 把每条断言的所有变量都登记为共享候选,而非只登记 UFApply 参数。

- **[中] `src/theory/arith/nira/NiraSolver.cpp:462-473`** — 奇次(≥1)单变量约束对 `Eq`/`Geq`/`Leq` 无条件返回 `Sat`,**忽略该变量上的活跃线性 bound**。例如 `x^3=8 ∧ x<0` 会被判为 Sat。
  *修复*: 只有当该变量无其他活跃约束时才信任单约束 Sat,否则回退 CDCAC/Unknown。

### 1.2 NRA 后端:代数数比较与精确过滤(可能假 UNSAT)

- **[高] `src/theory/arith/nra/backend/LibpolyBackend.cpp:1018-1027`** — `compareRealAlg` 在两代数数**隔离区间相同**(`lower==lower && upper==upper`)时即返回 `Equal`,即使定义多项式不同。不同多项式的两个真实根可能共享同一隔离盒。
  *后果*: 把两条真实 cell 边界塌缩成一条 → covering 假装「盖满整条线」→ **假 UNSAT**。
  *修复*: 仅从精确判定(已有的 gcd-root-membership,或细化到区间分离)得出 Equal;绝不从「区间巧合相同」推断相等。

- **[高] `src/theory/arith/nra/engine/CoveringManager.cpp:108-112`** — `coversAllLine` 的相邻间隙检测仅在 `Equal` 且两端皆开时才认为有间隙;它正是消费上一条 false-`Equal` 的**最后一道闸门**(UNSAT 之前)。
  *修复*: 要求 boundCompare 由精确相等支撑;把 `Unknown`/不确定相等当作 `DoesNotCover`/`Unknown`。

- **[高] `src/theory/arith/nra/backend/LibpolyBackend.cpp:1112-1118`** — 代数-代数重叠分支用 `if (a.definingPoly == b.definingPoly) return rootIndex< ? Less : Greater`,**假设 `rootIndex` 对同一定义多项式按实值单调**。但 `rootIndex` 是按各调用点 `poly::isolate_real_roots` 返回顺序赋的,不保证是两操作数的升序实根序。
  *后果*: 错误的根序破坏 cell 构造/covering → 潜在假 UNSAT。
  *修复*: 用实际隔离区间/符号细化比较,而非存储的 rootIndex(除非能证明它确是该多项式的升序实根序号)。

- **[高] `src/theory/arith/nra/backend/LibpolyBackend.cpp:1422`** — Norm 精确根过滤(`isolateRealRootsViaNorm`)把区间细化封在 `for (int d=0; d<80; ++d)`,若 80 次内未推翻 `0∈V` 就接受 `isRoot=true`。这违反 LAZARD.md 的 H2「精确过滤必须是判定」契约——耗尽预算的**未决候选被静默当作根保留**。
  *后果*: 制造伪 section 边界 → 不健全 covering。
  *修复*: 循环退出仍未证明 `0∉V` 且区间未收敛到点时,返回 `supported=false`(Unknown),不保留候选。

- **[中] `src/theory/arith/nra/backend/LibpolyBackend.cpp:1063-1074, 1094-1099`** — `compareRealAlg` 的有理-代数「符号回退」假设定义多项式在 `[lower,upper]` 内恰有一根,用 `sign(f(q))==sign(f(lower))` 推序。若区间内多于一根则推序错误。
  *修复*: 用 `countRealRootsInInterval==1` 验证单根后再用符号启发。

- **[中] `src/theory/arith/nra/valuation/TowerAlgebraicKernel.cpp:142`** — 塔域 kernel 从不验证 `minimalPolys[i]` 在所选实嵌入下确实在低层塔上不可约;精确零测试(`reduce().isZero()`)只判定**商环**中的相等。若传入可约/非极小定义多项式,环零 ≠ 域零,`isZero()` 可能把非零实值报告为零。
  *后果*: 不健全的证书基础(**目前潜伏**——见 §1.4)。
  *修复*: 信任零测试用于判定前,强制/验证不可约性(LAZARD.md H1)。

### 1.3 NIA:界传播、空冲突子句与归一化

- **[高] `src/theory/arith/nia/NiaSolver.cpp:250-288`** — 乘积界传播匹配二次项 `a*b` 与常数项,但**从不验证多项式只有这两项**。对 `a*b + a - 6 = 0` 会推出 `a*b = 6` 的上界(`ub1 = 6/lb(b)`),而真实关系是 `a*b = 6 - a`。错误的紧上界排除真解。
  *后果*: 经后续空域/枚举 → **假 UNSAT**。
  *修复*: 加 `terms.size()==2` 守卫(§4.6 已有同款守卫)。

- **[高] `src/theory/arith/nia/reasoners/BoundedNiaSolver.cpp:42-46`** — 某变量域为空时,`enumerate` 返回 `UnsatComplete` 带**空冲突子句** `TheoryConflict{{}}`。TheoryManager 取反空子句交给 SAT = 无理由的无条件全局 UNSAT。
  *后果*: 若此路径可达(如 exclusion 使域在前置 `isEmpty()` 检查后变空)→ **不健全 UNSAT**。
  *修复*: 从变量的 bound/exclusion reason 构造冲突(参考 `CompleteFiniteDomainEnumerator.cpp:138-142`),或走 `domains.buildEmptyDomainConflict()`。

- **[中] `src/theory/arith/nia/core/DomainStore.cpp:231-244`** — `buildEmptyDomainConflict` 只报告**第一个**空域(238 行 `break`),且若 `collectEmptyReasons` 返回空(某种未覆盖的空形)则产出空冲突子句。
  *修复*: 发出前 assert/守卫 `collectEmptyReasons` 非空;否则回退到全部活跃 reason。

- **[中] `src/theory/arith/nia/preprocess/NiaNormalizer.cpp:41-72`** — 多元多项式 `clearDenominators` **不验证整系数直接透传**,而 `normalizeStrict` 随后做 `p>0 ⇒ p-1≥0`、`p<0 ⇒ p+1≤0` 强化。该 +1 强化仅当 `p` 整值时成立;有理系数多元(如 `(1/2)xy`)被不健全强化。
  *后果*: 潜在**假 UNSAT**。
  *修复*: 多元情形先验证/清分母(非整值则回退 Unknown)再做 strict→non-strict。

- **[中] `src/theory/arith/nia/reasoners/BoundedNiaSolver.cpp:90-97`** — 完全-UNSAT 冲突只收集约束 reason(`c.reason`),**遗漏使域有限的 bound reason**。当前约束 reason 恰为超集时无害,但脆弱;若某 bound 来自非约束 reason 的 literal 则冲突错误。
  *修复*: 同时附上每个枚举变量的 lower/upper/finite-set/exclusion reason。

### 1.4 NRA-CDCAC 核心:缓存与采样回退(部分潜伏)

- **[高,潜伏] `src/theory/arith/nra/core/CdcacCache.cpp:65`** — `hashKey(PolyId, prefix, var)` **完全忽略 prefix 采样点**,`getRoots`/`setRoots` 仅按 hash 存取**无相等性复查**。不同 prefix 哈希相同 → 返回另一采样点的根集。多元多项式的实根依赖 prefix。
  *后果*: 错误的根 → 潜在不健全。**当前 get/set 读路径未接线**(只有 push/pop/clear 在用),是定时炸弹。
  *修复*: 哈希实际 prefix 值并存完整 key 做 `operator==` 碰撞检查;或在修正前删除缓存。

- **[高,潜伏] `src/theory/arith/nra/core/CdcacCache.cpp:55`** — `hashKey(SignCacheKey)` 只哈希 `prefixSample.varOrder.size()`,不哈希采样值,map 哈希→值无相等检查。任何碰撞静默返回错误符号 → 翻转关系求值 → 不健全。同样目前未接线。
  *修复*: 启用读路径前先存真实 key + 相等检查(链式桶)。

- **[中] `src/theory/arith/nra/core/CdcacSolver.cpp:194-204`** — `sampleValueToRealValue` 在 `definingPoly==NullUniPolyId` 时回退到代数根隔离区间的有理中点。返回的不是真实根。
  *后果*: SAT 模型可能违反公式(模型提取 bug 类);仅因 ModelValidator 复查而安全——但此处退化根会让 validator 拒绝真 SAT 实例,强制 Unknown。
  *修复*: 无定义多项式时传播 Unknown,而非假有理数。

- **[中] `src/theory/arith/nra/core/CdcacCore.cpp:646-660, 687-701`** — first/final 无穷扇区循环 `for (attempt<3)` 在 UNSAT 扇区最多递归 3 次,但只用第一次 UNSAT 构造冲突 cell;若后续尝试返回 SAT 则整体返回 SAT。三个采样都落在同一 cell,采样驱动可能分歧。
  *修复*: 无穷扇区只采样一次;若要重试,从驱动判定的同一采样构造/验证 cell。

### 1.5 LRA/LIA:静默丢弃、陈旧 bound、debug-only 健全性断言

- **[高] `src/theory/arith/lra/LraSolver.cpp:58`** — `assertLit` 对 payload 非 `LinearAtomPayload` 的原子**静默 `return;`**(不记入 `theoryTrail_`)。SAT 层以为 LRA 在约束它,实则不可见。这就是记忆中记录的「fast-unknown = silent atom drop」类。
  *后果*: 可在该原子未满足时返回 `sat`/`consistent`。
  *修复*: 返回 `Unknown`/记为未处理,而非静默忽略;或断言 LRA 只收线性 payload。

- **[高] `src/theory/arith/lia/LiaSolver.cpp:60`** — 同款静默 `return;` 丢弃非线性 payload。同样的错误判定风险。

- **[中] `src/theory/arith/lra/LraSolver.cpp:68-82`(及 `LiaSolver.cpp:70-89`)** — 对已见 satVar 的 trail 条目就地去重替换,**不重置 `appliedCursor_`**。若 satVar 以不同 `value`/关系重断言且其槽位 < `appliedCursor_`(已重放进单纯形),新 bound 永不应用 → 单纯形保留陈旧 bound。
  *后果*: 潜在不健全 `sat`。
  *修复*: 就地替换改变了 `value`/`isDiseq` 且索引 < `appliedCursor_` 时,重置 `appliedCursor_=0`(强制全量重放)。

- **[中] `src/theory/arith/lra/LraSolver.cpp:248-249`(及 `LiaSolver.cpp:212-213, 267-269, 319-321`)** — 单纯形冲突子句的健全性仅由 debug `assert(ok ...)` 守卫;Release(NDEBUG)下 `normalizeTheoryClause` 失败(含互补 literal)被忽略,返回畸形冲突子句。含 literal 及其反的子句是重言式,作为 UNSAT 理由发出即不健全,且 Release 静默通过。
  *修复*: 运行期处理 `!ok`(回退 `allActiveReasons()` 或 Unknown),不依赖 assert。

- **[中] `src/theory/arith/lia/InternalMilpEngine.cpp:70-71`** — `if (reason.var==0) reason = SatLit{0,true};` 用 var 0 作「无 reason」哨兵,`getConflictReasons`(265-271)过滤掉 `var==0`。若真实 SAT literal 合法地 `var==0`,其 reason 被强制哨兵并从冲突中静默过滤 → 欠约束(潜在不健全)冲突子句。
  *修复*: 用不会与有效 SatVar 碰撞的专用哨兵/optional。

### 1.6 前端:常量截断与自由原子

- **[高] `src/frontend/preprocess/IntDivModLowerer.cpp:251, 254`** — `emitConstDivisorConstraints` 用 `mkIntConst(k.get_si())` 与 `mkIntConst(absKMinusOneVal.get_si())` 构造除数与 `|k|-1` 界,**静默截断不放进 `int64_t` 的常量**。`k` 来自 `evalIntConstTerm`(无界 `mpz_class`,如 `99999999999999999999`)。
  *后果*: 大常量除数产生垃圾 `k*q+r` 方程与错误 `r<=|k|-1` 界 → **静默错误判定**。
  *修复*: `mkIntConst` 接受 `mpz_class`/字符串不做 `get_si()`;或 `!k.fits_slong_p()` 时拒绝折叠,落到变量除数路径。

- **[高] `src/frontend/atomization/ArithAtomExtractor.cpp:113-116`** — `Tautology`/`Conflict` 状态下 extractor `return true`(已处理)但**不注册理论原子、不发单元子句固定 SatVar 极性**。注释说「caller 处理」,但 caller(`Atomizer.cpp:342`)在 `handled==true` 时什么也不做。
  *后果*: 重言原子的 SAT 变量可被赋 false(冲突原子可被赋 true)而理论无感知——正是已修过的 nra_012 自由原子不健全类。
  *修复*: Tautology 加单元子句 `{positive(v)}`,Conflict 加 `{negative(v)}`,再 return。

- **[中] `src/expr/ir.h:69-73`** — `CoreIr::add` **完全无 hash-consing**,每次追加新节点返回新 ExprId。但 CLAUDE.md 声称「Expr 是 hash-consed」,且 `UnconditionalConstantPropagation.cpp:266-273` 的 `(= a a)→true` 折叠明确依赖「CoreIr DAG 去重相同子表达式」。该折叠只在两操作数恰好共享 ExprId(经 per-pass cache)时触发;结构相等但重建的子项得到不同 ID 就跳过折叠。
  *后果*: 关系恒等折叠与跨 pass DAG 共享的健全性靠运气;可能漏折叠泄漏不健全自由原子。
  *修复*: 在 `add()` 加真正的 hash-cons map(`unordered_map<CoreExpr, ExprId>`);或停止依赖/声明去重。

- **[中] `src/frontend/preprocess/UnconditionalConstantPropagation.cpp:561-575`** — `materializeConstant` 只在 `sort==intSortId_ && den==1 && fits_slong_p()` 时发 `ConstInt`;Int 排序的大整数(分母 1 但超 int64)落到通用分支,造出带 `intSortId_` 排序的 `ConstReal` 节点——kind/sort 不匹配。下游按 `kind==ConstInt` 路由的 extractor 不识别它。
  *修复*: `sort==intSortId_ && den==1` 时始终发 `ConstInt`(以字符串/mpz 存值,不用 int64)。

- **[中] `src/parser/adapter.cpp:276-281, 309-310`** — 所有数值常量(NUMBER)以 `num.toString()` 字符串 payload 存,而 `mapKind` 返回 `Kind::ConstReal`,仅靠排序细化(95-98/110-131)翻成 `ConstInt`。于是 `ConstInt` 节点带 `std::string` payload,而所有消费者(`evalIntConstTerm`、`tryAsConstNumeric`、`ArithCastNormalizer`)用 `std::get_if<int64_t>` 读 → 整数字面量上静默 no-op。
  *后果*: 整数常量折叠 / div-mod 求值对解析器产出的整数字面量静默失效(代码里多处 "Defensive: SOMTParser maps integer literals to ConstReal" 注释正是在绕这个)。
  *修复*: `extractPayload` 在 Int 排序时把数字解析为 `int64_t`(或专用 mpz payload)。

### 1.7 search/poly:未验证模型与常量解析

## 三、效率问题

### 3.1 NRA:O(n!) 行列式(头号瓶颈)

- **[中] `src/theory/arith/nra/engine/ReasonManager.cpp:44-50`** — 一个**死循环**构造 `test` 副本却什么都不做(只有注释),随后 53-60 用 vector 副本 + `std::remove` + 每 reason 全重验做 O(n²) minimize。
